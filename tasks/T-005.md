## 1. Objetivo
Criar o `SimNetwork` v1 no pacote `@plataforma/testkit`. É uma implementação `in-memory` do contrato `NetworkAdapterPort` que conecta múltiplos peers (instâncias) dentro do mesmo processo Node.js para testes de integração ultra-rápidos.
*(derivado de `docs/plano-de-implementacao.md` §2.1 Anel 1 — Simulação Determinística; contrato `NetworkAdapterPort` em T-004 §1)*.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/testkit/src/SimNetwork.ts 
---
import { NetworkAdapterPort, PeerId, WireData, MessageHandler } from '@plataforma/protocol';
import { VirtualClock } from './clock';

/** Hub de rede simulada in-memory para N peers. */
export class SimNetwork {
  /** @param clock Relógio virtual para latência. Se omitido, entrega é síncrona/assíncrona imediata. */
  constructor(clock?: VirtualClock);

  /** Cria um adapter que implementa NetworkAdapterPort para o peer informado. */
  createAdapter(peerId: PeerId): NetworkAdapterPort;

  /** Particiona a rede: mensagens que cruzam fronteiras de grupos são descartadas. */
  partition(...groups: PeerId[][]): void;

  /** Restaura conectividade total entre todos os peers. */
  heal(): void;
}
```

Cada adapter retornado por `createAdapter` implementa `NetworkAdapterPort`:
- `connect(to)`: registra intenção de conexão (no-op imediato em v1; mantido por contrato).
- `listen()`: no-op (adapters são automaticamente registrados no hub ao serem criados).
- `send(to, data)`: entrega `data` no `onMessage` do adapter de `to`. Se `clock` foi provido no `SimNetwork`, agenda via `clock.setTimeout(callback, 0)` — entrega assíncrona com latência zero fixa.
- `onMessage(handler)`: registra callback para mensagens recebidas.
- `close()`: remove o adapter do hub; `send` para peer fechado é no-op silencioso.

> **Semântica de timing da entrega (CONTRATO — fonte dos achados M1/M2 da 1ª revisão):** a entrega é
> assíncrona (microtask ou `clock.setTimeout(_,0)`). O que é lido **no momento da ENTREGA** (não no `send`):
> - **`onMessage` do destino** — se o handler for trocado entre o `send` e a entrega, o handler **vigente
>   na entrega** é chamado (não o que existia no `send`).
> - **estado `closed` do destino** — se o peer for fechado entre o `send` e a entrega, a entrega é
>   **cancelada** (não chega ao handler).
>
> O que é avaliado **no momento do `send`** (snapshot): a **regra de partição**. Uma mensagem já agendada
> NÃO é afetada por um `partition()`/`heal()` posterior — o "pacote já está no fio". Mudar isso é decisão
> de produto (não desta v1).

Casos de teste (numerados):
1. `createAdapter` retorna objeto que implementa `NetworkAdapterPort` (todos os 5 métodos presentes).
2. 2 peers: A envia `send('B', data)` → B recebe via `onMessage` com o mesmo `data` e `peerId`='A'.
3. 3 peers em cadeia: A→B, B→C, C→A — todos recebem exatamente 1 mensagem cada.
4. `partition(['A'], ['B'])`: A→B é descartado (B não recebe). A→A (self-send) funciona.
5. `partition(['A','B'], ['C'])`: A→B funciona, A→C é descartado, C→B é descartado.
6. `heal()` após partição: conectividade total restaurada, A→B funciona novamente.
7. `close()` em um adapter: `send` para peer fechado é no-op (não dispara erro nem callback).
8. `VirtualClock` integrado: `send` agenda via `clock.setTimeout`; `clock.advance(0)` dispara a entrega.

> **Casos 9–12 — semântica de timing da entrega (anti-regressão; cobrem a classe de bug das sondas
> S1/S2 da 1ª revisão).** Em todos, NÃO dar `await` no `send` antes da mutação — o `await` esvaziaria a
> microtask de entrega antes da mutação e a janela testada não seria exercitada (lição da própria
> correção). Padrão: `const p = a.send(...); /* muta */; await p;` e então aguardar a entrega.

9. **Handler tardio (M1):** `b.onMessage(antigo)`; `send` A→B (sem await); `b.onMessage(novo)` síncrono; após a entrega, o **novo** handler é chamado 1× e o antigo 0×.
10. **Close in-flight (M2):** `b.onMessage(spy)`; `send` A→B (sem await); `b.close()` síncrono; após aguardar, o spy é chamado **0×** (entrega cancelada).
11. **Partição é snapshot do `send`:** sem partição ativa, `send` A→B (sem await); `partition(['A'],['B'])` síncrono depois; após a entrega, B **recebe** (a regra de partição posterior NÃO afeta mensagem já agendada).
12. **Não-membro passa pela partição:** `partition(['A'],['B'])` com C fora de qualquer grupo; `send` A→C → C **recebe** (a partição só descarta quando origem E destino estão em grupos diferentes; não-membro não é bloqueado).

### Pegadinhas conhecidas
- **Leitura de estado: agendamento vs. disparo.** A entrega é diferida (microtask/`setTimeout(_,0)`). NÃO capture `onMessage` nem `closed` numa variável no `send` — leia-os **dentro** do callback de entrega (ver "Semântica de timing" na Seção 1). Capturar no `send` foi exatamente o que gerou os achados M1/M2. A partição, ao contrário, é snapshot do `send` (intencional).
- **Testes da janela de timing.** Para exercitar a janela entre `send` e entrega, NÃO dê `await` no `send` antes de mutar o estado (trocar handler / `close` / `partition`) — o `await` drena a microtask de entrega antes da mutação e o teste vira falso-verde. Use `const p = a.send(...); muta(); await p;`.
