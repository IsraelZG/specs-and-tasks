## 1. Objetivo
Implementar o **adapter SQLite concreto do `StoragePort`** (`@plataforma/protocol`, T-004) — hoje só
existe a **interface** e um **mock de teste** (`packages/protocol/tests/ports.test.ts`); nenhum
backend real persiste a lineage. O adapter deve: abrir o banco em **WAL mode**, implementar
`exec`/`transaction`/`migrate`, e expor um **wal_checkpoint** (PRAGMA) para o gerenciamento de WAL
que T-304 (guard concorrente) e T-701 (`device_state.db`) assumem mas não podem fornecer por serem
storage-agnósticos.

> **Por que esta task existe (achado de 2026-06-29):** ao endurecer T-304 descobriu-se que todo o
> `@plataforma/core` roda hoje sobre um `StoragePort` **mockado** — não há SQLite real, WAL não é
> ativado em lugar nenhum (`grep` por `journal_mode`/`PRAGMA` no monorepo = vazio). É uma lacuna de
> fundação: sem o backend concreto, sync/persistência reais não rodam fora de teste.

## 6. Feedback de Especificação (Spec Feedback Loop)

**⛔ Status: `blocked-decision`** — 2 decisões de roadmap travam o endurecimento. A implementação é
mecânica (Sonnet) depois de fechadas.

### Decisão 1 — Driver SQLite
- **Contexto:** máquina-alvo é **Windows 11 ARM64** (ver `PITFALLS.md` / memória do projeto).
- **Opções:** (a) **`node:sqlite`** — built-in no Node 22+, sem compilação nativa (ótimo no ARM64),
  porém ainda experimental; (b) **`better-sqlite3`** — maduro e síncrono, mas precisa de build nativo
  (já houve atrito de toolchain ARM64 no projeto); (c) **ambos por ambiente** (node:sqlite no servidor,
  WASM no browser — ver Decisão 2).
- **Recomendação técnica:** (a) `node:sqlite` para o backend de servidor/teste — zero toolchain no
  ARM64, alinha com Node 22+. Revisitar se o experimental incomodar.

### Decisão 2 — Backend de browser (local-first)
- **Contexto:** o produto é local-first no browser; lá não há `node:sqlite` nem better-sqlite3.
- **Opções:** (a) WASM (**wa-sqlite**/sql.js + **OPFS** para persistência) **nesta** task; (b) task
  **separada** (`T-004b`), entregando aqui só o backend Node e deixando o browser para depois.
- **Recomendação técnica:** (b) — separar. Esta task entrega o backend Node (destrava T-304/T-701 e
  testes); o backend WASM/OPFS é uma fronteira própria (perf, OPFS, bundling) e merece task dedicada.

**Próximo passo:** `/arquiteto-decisoes T-004a` → fechar as 2 → re-endurecer → `hardened`.
