## 1. Objetivo
Implementar o **adapter SQLite concreto do `StoragePort`** (`@plataforma/protocol`, T-004) — hoje só
existe a **interface** e um **mock de teste** (`packages/protocol/tests/ports.test.ts`); nenhum
backend real persiste a lineage. O adapter deve: abrir o banco em **WAL mode**, implementar
`exec`/`transaction`/`migrate`, e expor um **wal_checkpoint** (PRAGMA) para o gerenciamento de WAL
que T-304 (guard concorrente) e T-701 (`device_state.db`) assumem mas não podem fornecer por serem
storage-agnósticos.

> **Por que esta task existe (achado de 2026-06-29):** ao endurecer T-304 descobriu-se que todo o
> `@plataforma/core` roda hoje sobre um `StoragePort` **mockado** — não há SQLite real, WAL não é
> ativado em lugar nenhum (`grep` por `journal_mode`/`PRAGMA` no monorepo = vazio). É uma lacuna de
> fundação: sem o backend concreto, sync/persistência reais não rodam fora de teste.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - NÃO adicione `better-sqlite3` nem nenhuma dependência nova ao `package.json` — `node:sqlite` é
>   built-in, zero dependência.
> - NÃO implemente backend de browser/WASM aqui — é T-004b.
> - NÃO altere a interface `StoragePort` em `packages/protocol/src/ports.ts`.

### Pegadinhas conhecidas
- `node:sqlite` exige Node 22+; confirme o runtime antes de iniciar (`node -v`). Se o ambiente
  estiver em Node 20, isso é BLOCKER de ambiente — `pause`/`block`, não troque de driver sozinho.
- `DatabaseSync` de `node:sqlite` é **síncrona** — `exec`/`transaction`/`migrate` da interface são
  `Promise`-based; envolva as chamadas síncronas em `Promise.resolve(...)` ou `async` wrapper, não
  mude a assinatura da interface.
- `:memory:` não persiste WAL em disco — para o teste de `walCheckpoint`/`journal_mode = wal`, use
  um arquivo real em `os.tmpdir()` e limpe no `afterEach`.
- `transaction` deve fazer rollback real em exceção — `node:sqlite` não tem savepoint automático;
  use `BEGIN`/`COMMIT`/`ROLLBACK` explícitos dentro do método.

1. **[TDD]** Escreva `packages/core/tests/sqliteStorage.test.ts` reaproveitando os casos de
   `packages/protocol/tests/ports.test.ts` contra uma instância real de `SqliteStorage`.
2. Implemente `packages/core/src/sqliteStorage.ts`: `SqliteStorage implements StoragePort`,
   abre com `PRAGMA journal_mode = WAL`, implementa `exec`/`transaction`/`migrate` sobre
   `node:sqlite`, expõe `walCheckpoint(mode)`.
3. Re-exporte em `packages/core/src/index.ts`. Rode o Gate.

## 6. Feedback de Especificação (Spec Feedback Loop)

### Decisão 1 — Driver SQLite
**DECIDIDO (arquiteto, 2026-06-30):** `node:sqlite` (built-in Node 22+) para o backend de
servidor/teste. Zero compilação nativa, zero toolchain ARM64 — elimina o atrito de build que
`better-sqlite3` já causou neste projeto (ver `PITFALLS.md`). Revisitar só se o status
"experimental" do módulo virar bloqueio prático.

### Decisão 2 — Backend de browser (local-first)
**DECIDIDO (arquiteto, 2026-06-30):** separar em **T-004b**. Esta task (T-004a) entrega
exclusivamente o backend Node — destrava T-304 e T-701 o mais rápido possível. O backend
WASM/OPFS (wa-sqlite/sql.js + OPFS) é uma fronteira própria (performance, bundling, ciclo de vida
do OPFS) e fica para T-004b.
