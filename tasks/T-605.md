## 1. Objetivo
Implementar fluxo de intent não-comutativo: `CONTENT:INTENT` + arestas `SPENDS→head`, `CREDITS→entity_id`, `APPROVES` (K=1 comum), aplicador determinístico criando `MUTATES` + `TRANSFERS`; merge aditivo de créditos concorrentes (nunca LWW); `pending` local-não-replicado vs `finalized` durável.
*(Normativo: `docs/plano-de-implementacao.md` §9 T-605 (linha 298) + §2.5 vetor 8. **Modelo de grafo canônico:** `docs/caderno-2-protocol/01-graph-ontology.md §64-71` — SPENDS→head (serializa débito), CREDITS→entity_id (crédito comutativo), reusa APPROVES/RESOLVES/TRANSFERS/RESULTED_FROM. Regra de serialização: `caderno-4/03 §3.5`.)*

> **⚠️ CONFRONTAÇÃO (endurecimento 2026-07-08, claude-opus).** `complexity: 5` (o Dimensionamento
> do CLAUDE.md manda quebrar) + fundação não-materializada. Ver Seção 6, D1–D5:
> 1. **Zen Engine (T-604) não existe em código:** `ready` mas `packages/core/src/rules/` ausente;
>    `applyIntent` valida "via Zen Engine" — contrato não resolve.
> 2. **`amount: number` (plaintext) CONTRADIZ o aceite** "conflito estrutural detectável **sem
>    decifrar valores**" (vetor 8). Valores devem ser opacos/cifrados — e o "merge **aditivo**"
>    sobre valores cifrados exige esquema (commitment/homomórfico) não especificado.
> 3. **Arestas SPENDS/CREDITS/APPROVES/TRANSFERS não existem no schema/código** (só `MUTATES`).

### Contratos exatos (multi-concern — provável decomposição; ver D4)
```ts
// --- packages/core/src/intent/intentFlow.ts ---
import type { EntityId, ULID, StoragePort } from '@plataforma/protocol'; // T-004/T-102 (derivado)
// ⚠️ ZenEngine (T-604) — contrato não materializado (D1); representação de valor cifrado (D2)

export interface IntentPayload {
  type: 'transfer' | 'credit' | 'debit';
  amount: number;
  targetEntityId: EntityId;       // destinatário (CREDITS) ou origem (SPENDS)
  referenceHeadId: ULID;          // head da linhagem como âncora de serialização
}

export interface IntentNode {
  intentId: ULID;
  intent: IntentPayload;
  state: 'pending' | 'approved' | 'finalized' | 'rejected';
  createdBy: EntityId;
  spendsAnchors: ULID[];          // arestas SPENDS→head
  creditsAnchors: EntityId[];     // arestas CREDITS→entity_id
  approvals: Map<EntityId, Uint8Array>; // APPROVES: entityId → assinatura
}

export interface IntentResult {
  success: boolean;
  mutatesEdgeId?: string;         // aresta MUTATES criada
  transfersEdgeId?: string;       // aresta TRANSFERS criada
  pendingIntents: EntityId[];     // intents ainda pendentes
  reason?: string;
}

export interface IntentValidator {
  /** Cria intent CONTENT:INTENT no grafo. */
  createIntent(intent: IntentPayload, authorEntityId: EntityId): Promise<IntentNode>;

  /** Aplica intent: valida invariantes (Zen Engine T-604) + serialização. */
  applyIntent(intentId: ULID, storage: StoragePort): Promise<IntentResult>;

  /** Aprova intent (K=1 comum; K>1 para quorum). */
  approveIntent(intentId: ULID, approverEntityId: EntityId, signature: Uint8Array): Promise<void>;

  /** Finaliza intent: persiste estado `finalized` durável. */
  finalizeIntent(intentId: ULID): Promise<void>;

  /** Merge aditivo: soma créditos concorrentes (nunca LWW). */
  mergeConcurrentIntents(intents: IntentNode[]): IntentPayload;

  /** Detecta conflito estrutural SPENDS (double-spend). */
  detectSpendsConflict(intents: IntentNode[]): boolean;
}
```
- `SPENDS→head`: âncora de serialização, previne double-spend (vetor adversarial 8).
- `APPROVES`: K=1 no caso comum (posse determinística da linhagem por `hash(entity_id)`).
- `applyIntent`: valida via Zen Engine (T-604) + cria `MUTATES` + `TRANSFERS`.
- `mergeConcurrentIntents`: créditos concorrentes somados aditivamente, nunca LWW.
- `pending` vs `finalized`: pending é local-não-replicado; finalized é durável no grafo.
