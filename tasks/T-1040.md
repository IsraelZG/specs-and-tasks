# T-1040 · SPIKE: distribuição unificada de plugins nativos (LiveKit SFU, llama.cpp) — marketplace-only sem exceção de payload

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (pacotes core/protocol) e `playwright` (E2E/Frontend)
- **Capacidade-alvo:** haiku | sonnet | opus-spike *(ver regra "Dimensionamento de Tarefas" no CLAUDE.md: spec sem decisões em aberto, contratos explícitos, sem API externa não-fixada, verificação por comando)*

## 1. Objetivo
Explorar como a plataforma distribui plugins que **não são bundles JS/WASM** — binários nativos e serviços de terceiros já existentes — sob a **mesma** garantia de "única via de distribuição, sem sideload" que já vale para plugins `browser`/`node` em JS.

> **ORIGEM (auditoria + arquiteto, 2026-07-02, absorção de `rfc-plugin_architecture_blueprint`, subtask AB-09).** O contrato canônico (`docs/caderno-3-sdk/12-plugins-e-computacao.md §2`) já promete: *"Única via de obtenção é o módulo Marketplace... não existe sideload — o runtime só carrega bundles cuja procedência é um SPEC:PLUGIN válido, assinado e listado."* Mas o próprio caderno-3/12 §3 **já lista como exemplo de categoria `infra`** software que **não é TypeScript**: *"signaling/rendezvous, **SFU LiveKit**, seeder/tracker WebTorrent, relay"* — LiveKit SFU é Go. O usuário quer, além disso, distribuir **llama.cpp** (C++) e outros binários nativos pelo mesmo mecanismo. A RFC absorvida (`rfc- plugin_architecture_blueprint`, §3.1, já superseded por outro motivo) também tinha precedente disso: o `lookout-plugin` envolvia `child_process.spawn` de um binário Go. **O contrato de distribuição (assinatura + listagem + procedência via SPEC:PLUGIN) nunca foi detalhado para payloads não-JS/WASM** — hoje é uma lacuna, não uma decisão tomada.

**Meta:** um **único modelo de distribuição** (SPEC:PLUGIN assinado, marketplace-only, sem sideload) que cubra JS/WASM **e** binários/serviços nativos multi-plataforma, sem criar uma segunda via "para os nativos". Explorar casos concretos: LiveKit SFU (Go, processo de longa duração com canais de rede próprios — já é `infra`), llama.cpp (C++, binário por plataforma/arquitetura, possivelmente com pesos de modelo como blob separado), e generalizar.

## 6. Feedback de Especificação (perguntas a explorar — não pré-decididas)
- **Assinatura de binário multi-plataforma:** um `SPEC:PLUGIN` hoje assina "o bundle". Para um binário nativo, isso vira "N bundles, um por par OS/arch" sob o mesmo `SPEC:PLUGIN`? Como o runtime escolhe o correto no load?
- **Verificação de procedência sem rebuild determinístico:** bundles JS/WASM podem (em tese) ser auditados/rebuildados; binários de terceiros (LiveKit SFU release oficial, llama.cpp release) não são buildados pela plataforma. A garantia "assinado e listado" muda de "nós verificamos o código" para "nós atestamos a procedência do binário publicado por X" — isso é uma mudança de modelo de confiança que precisa ficar explícita no ADR, não escondida.
- **Peso/tamanho:** modelos de LLM (llama.cpp) são GBs — o media plane (blob por chunk) já foi desenhado pensando nisso, ou assume payloads pequenos (mídia de usuário)? Verificar limites/premissas atuais.
- **Runtime `infra` já cobre o caso operacional** (processo de longa duração, canais próprios) — o que falta é só a distribuição/assinatura, ou o sandbox de `infra` (§6) também precisa de revisão para "processo nativo com mais autoridade que um Worker/isolate JS"?
- **AB-09 (localização no monorepo) fica resolvida por este ADR** — se o modelo unificado emergir, a resposta de "onde vive o código first-party" decorre dele, não precisa de decisão paralela.
