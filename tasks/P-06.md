# P-06 · Pool de worktrees quentes (slots fixos) + desacoplar wt new do Docs

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+ · **Package Manager:** `pnpm` (11, `allowBuilds` no `pnpm-workspace.yaml`)
- **Repo:** o `worktree.mjs` **vive no controle** (`C:\Dev2026\Docs\tools\scripts\`) mas opera sobre o **superapp** (`C:\Dev2026\superapp` + `C:\Dev2026\.superapp-worktrees\`).
- **⚠️ Windows ARM64 + pnpm:** junction de `node_modules` NÃO funciona com pnpm (memória de ambiente). Slots de **caminho fixo** — renomear diretório quebra os symlinks do `.pnpm`.
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Tirar da rota crítica os ~30–58 s de setup por task (worktree 28–58 s, dos quais install 19–33 s + pré-builds), e **de quebra** eliminar o incidente recorrente "wt new bloqueado por Docs sujo" (ocorreu nas 3 execuções, causou perda de trabalho 1×). Duas partes: (A) **pool de slots quentes** de caminho fixo, renovados assincronamente pós-merge; (B) `wt new`/`wt claim` **nunca** exige Docs limpo nem sugere `stash`/`restore`/`clean` no controle.

## 4. Estratégia de Testes Estrita
- [x] **Framework:** funcional (Node puro), testado manualmente contra o superapp real (não há mock viável de worktree pnpm).
- [x] **Casos:** (1) `wt claim <ID>` num slot livre: `git reset --hard origin/master` + cria branch `task/<ID>` **no slot de caminho fixo**, em ~2 s, com `node_modules`/`dist` já quentes (sem install); (2) `wt release <ID>`: volta o slot ao pool; (3) `wt refresh`: async, se lockfile mudou roda install, roda `turbo build` full → slot quente de novo; (4) **Docs sujo** (arquivo não commitado em `tasks/*.md`) **não** impede `wt claim` e **nenhuma** ação destrutiva no Docs é oferecida; (5) exaustão do pool (todos ocupados) → mensagem clara ou cria worktree efêmera fora do pool (decidir — ver §6).
- [x] **Fora de Escopo:** tamanho do pool dinâmico; GC de branches órfãs.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER (INVIOLÁVEL — o incidente de perda de dados nasceu daqui):**
> - **NUNCA** rodar `git stash`/`restore`/`clean`/`reset` no repo **Docs** (controle). O patch B existe justamente para o `wt` tratar o Docs como read-only: `fetch`, informar divergência, e parar aí.
> - **NÃO** renomear diretórios de slot (quebra `.pnpm` no Windows) — slots têm caminho fixo; troca-se o conteúdo git, não o path.
> - **NÃO** compartilhar `node_modules` mutável entre slots (junction/symlink não funciona com pnpm aqui).

Passos: (A) criar N slots (`_slot-1..3`) via `git worktree add` de caminho fixo, cada um com install feito uma vez. `wt claim` = achar slot livre (marcador em `.superapp-worktrees/.pool.json`), `git fetch origin && git reset --hard origin/master && git switch -c task/<ID>`. `wt release` = `git switch --detach` + marcar livre. `wt refresh` = pós-merge async: `git reset --hard origin/master`; se `pnpm-lock.yaml` mudou, `pnpm install`; `turbo run build`. (B) no path de criação, remover qualquer checagem/mutação do Docs.

## 6. Feedback de Especificação
- **[DECISÃO menor — resolver no início, não bloqueia]** Exaustão do pool: (i) erro "pool cheio, aguarde `release`" **ou** (ii) fallback para worktree efêmera fora do pool (custo de install). Recomendação: (ii) com aviso, para não travar paralelismo. Se preferir (i), registrar. Não é decisão de arquiteto — é default operacional; o worker escolhe (ii) salvo objeção.
- Se o `git reset --hard` num slot com `dist/` quente invalidar o cache do turbo (mtime), medir: pode anular o ganho. Se anular, PARE e registre (pode exigir `turbo` com cache por hash de conteúdo, não mtime).
