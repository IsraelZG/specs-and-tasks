# C-17 · Fix identityGuard: `assertValidModelIdentity` rejeita formato `<papel>:<modelo>`

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v22+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest`
- **Package:** `@plataforma/plugin-tasks` (em `packages/plugin-tasks/`)
- **Capacidade-alvo:** haiku

## 1. Objetivo
Corrigir o falso-positivo em `assertValidModelIdentity` (`identityGuard.ts:28`) que rejeita atores no formato canônico `<papel>:<modelo>` (ex: `agile_reviewer:gemini`). A guarda usa `actor.startsWith(prefix)` e o prefixo `"agile_reviewer"` casa com `"agile_reviewer:gemini"`, bloqueando erroneamente.

## 2. Contexto RAG (Spec-Driven Development)
- **CLAUDE.md §Identidade (linhas 48-51):** autoriza `agile_reviewer:<modelo>` como formato canônico — o serviço autoriza pelo prefixo antes de `:`, o ledger exibe só o modelo.
- **`identityGuard.ts:27-33`** — fonte do bug: `for (const prefix of BLOCKED_ROLE_PREFIXES) { if (actor.startsWith(prefix)) throw ... }` bloqueia `agile_reviewer:gemini` porque `startsWith("agile_reviewer")` é verdadeiro.
- **`guards.test.ts:69-82`** — testes existentes cobrem `agile_reviewer` (papel puro) e `deepseek` (modelo válido), mas não cobrem o formato canônico `<papel>:<modelo>`.

## 5. Lógica da Correção (derivada do contrato CLAUDE.md)
No loop de `BLOCKED_ROLE_PREFIXES`, após `actor.startsWith(prefix)`:

- Se `actor === prefix` → é papel puro → bloquear (throw GuardError)
- Se `actor[prefix.length] === ':'` → é formato canônico `<papel>:<modelo>` → permitir (continue)
- Senão → é papel com sufixo inválido → bloquear (throw GuardError)
