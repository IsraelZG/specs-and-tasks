### Achados a corrigir (19 itens)
- [ ] [m1][T-407] Spec inconsistente: §1 declara `generateOobLink(multiaddr, ephemeralPubKey, relayHint?)` com 3 params, mas `OutOfBandLink.nonce` é obrigatório e o formato `plataforma://connect#…&nonce=<b64>…` exige-no. Impl adicionou `nonce: Uint8Array` como 3º param obrigatório (outOfBand.ts:38-43) — interpretação correta. Reendurecer a spec para alinhar a assinatura com a impl antes que callers dependam dela (tasks/T-407.md:44-48 vs outOfBand.ts:38-43)
- [ ] [m1][T-407] Tamanho do `nonce` não validado em runtime — spec §1 comenta `// 16 bytes` mas `parseOobLink`/`fromQrPayload` aceitam qualquer `Uint8Array`. Não é bug (interface sem invariante) mas vale validar no boundary se `nonce` for reaproveitado por consumidor de baixo nível (outOfBand.ts:65, 69, 101)
- [ ] [m1][T-407] Mudança fora do escopo declarado em §3: `packages/protocol/src/index.ts:33-34` re-exporta `generateOobLink`/`parseOobLink`/`toQrPayload`/`fromQrPayload`/`OobLinkError`/`OutOfBandLink` — necessário para consumo externo mas não constava em [CREATE]/[EDIT] do escopo (src/index.ts:33-34)
- [ ] [i1][T-407] `outOfBand.test.ts:47,73` usam `import('../src/outOfBand.js').OutOfBandLink` inline em vez de `import type { OutOfBandLink }` no topo — cosmético (outOfBand.test.ts:47, 73)
- [ ] [m1][T-208] Mudança de escopo não declarada: ClockPort estendido com setTimeout/clearTimeout — spec §3 só declarava GenesisWriter (ports.ts:58-78)
- [ ] [m1][T-212] Handover do executor contém "snapshot.ts não existe na worktree" — factualmente incorreto (arquivo existe e workaround já removido por T-308-rework-2); item de DoD §7 fica satisfeito pela rework-2 de T-308, não por T-212 — housekeeping, não bloqueia (tasks/T-212.md:119)
- [ ] [m1][T-305a] Defensive handling: extractEntityIds aceita capability com `resource: 'entity:'` (prefixo + ID vazio) e devolve `['']`, o que crasha scopeRBSRTree via BTree.validateId. Fix de 1 linha: `cap.resource.length > ENTITY_PREFIX.length` antes do add (ucanScope.ts:24-32)
- [ ] [m1][T-302a] Mudança fora do escopo declarado na §3: `packages/protocol/package.json` (devDep `@plataforma/testkit`), `packages/protocol/src/index.ts` (re-export do novo módulo) e `packages/testkit/src/index.ts` (export de `SimNetwork`) — suporte necessário mas deveria constar em [CREATE]/[EDIT] (package.json:16, src/index.ts:24-25, testkit/src/index.ts:3)
- [ ] [m2][T-302a] `respondNodes` exportado em `index.ts` mas não declarado nos Contratos exatos da §1 (aparece só na impl) — adicionar à spec ou remover do `index.ts` (src/rbsr/exchange.ts:161-175, src/index.ts:24)
- [ ] [m3][T-302a] Spec §1 declara `ids: ULID[]` em `requestNodes`; impl usa `string[]`. Equivalente em runtime (ULID é string opaca) mas desvia da assinatura nominal — alinhar (src/rbsr/exchange.ts:152-158)
- [ ] [m1][T-303] `RangeFooter` usa `rangeStartId`/`rangeEndId` em vez de `range: IdRange` conforme spec §1 — consumidores que esperam `footer.range.lo`/`.range.hi` quebram (rangeFooter.ts)
- [ ] [i1][T-303] `sessionNonce` aceito com qualquer tamanho — spec §1 declara "16 bytes" como contrato; adicionar validação `sessionNonce.length === 16` (rangeFooter.ts)
- [ ] [m1][T-306] `BTree` importado mas não usado em `waves.ts` — usar `type` na import ou remover (waves.ts:1)
- [ ] [i1][T-306] `executeWave2` itera `heads` mesmo com `getEntityHeads = null` (cai em `heads = []` e retorna `completed: true`) — comportamento aceitável mas não documentado; vale uma linha de comentário explicitando o fallback (waves.ts:160)
- [ ] [i4][T-306] `BTree.reconcile` usa XOR aggregation vulnerável a collisions com Fingerprints de baixa entropia — probe P1 do Reviewer 3 demonstrou que `isIdempotent` retorna `true` para trees divergentes quando os XORs colidem (FPs de 1 byte são insuficientes). Em produção FPs são SHA-256 (32 bytes) e colisão é ~0; mas se algum caller em application code gerar FPs com entropia reduzida, a anti-entropy fica silenciosamente quebrada. Documentar "Fingerprints DEVEM ter ≥128 bits de entropia" na doc do `BTree.reconcile` ou trocar XOR por SHA-256 (ADR) (BTree.ts:75-86, waves.ts:69,253)
- [ ] [m1][T-306] Rework escolheu injetar remote state via parâmetro (`rootRange: IdRange` + `remoteRootXor: Fingerprint` em executeWave0/isIdempotent; `remoteTree?: BTree` em executeWave1/executeWave2). Decisão NÃO foi adicionada ao §6 da spec nem §1 foi atualizada. Quando T-306 for aprovada, formalizar como Decisão 6 do arquiteto (waves.ts:55-90, 94-167, 171-240, 244-252)
- [ ] [m1][T-302b] Lint warning pré-existente em `codec.ts:29` (unused eslint-disable directive, T-203) — NÃO introduzido por T-302b mas re-rodado pelo gate. Housekeeping (codec.ts:29)
- [ ] [m2][T-302b] `applyNodes` chama `insertNode(storage, node)` SEM `parentId` — caller que precisar de lineage chain deve chamar `insertNode` diretamente. Spec §1 já implica (camada RBSR = sig validation, lineage = T-108a) (exchange.ts:171)
- [ ] [m3][T-302b] Wire format gap: `respondNodes` (T-302a) envia apenas `{id, fingerprint}`, não o `SignedNode` completo. `applyNodes` fica testada e pronta, mas integração end-to-end com `reconcile` de T-302a depende de decisão arquitetural — T-302c futuro (3 opções: evoluir respondNodes, nova MSG_FULL_NODES, reusar T-802 blob) (T-302b §6)
