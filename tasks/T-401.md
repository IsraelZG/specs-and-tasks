## 1. Objetivo
Implementar signaling de WebRTC no peer do sistema: troca de SDP/ICE via WS; salas derivadas de capability (`RendezvousId = SHA-256(rendezvous_secret ‖ ASSET:PERMISSION_ID)`); secret distribuído junto ao UCAN
*(derivado de `docs/plano-de-implementacao.md` §7 T-401)*.

### Contratos exatos
```ts
// --- apps/system-peer/src/signaling.ts 
---
import { PeerId } from '@plataforma/protocol';

export interface SignalingRoom {
  roomId: string;                // SHA-256(rendezvous_secret ‖ permission_id)
  peers: Set<PeerId>;
  createdAt: number;
  /** SHA-256(secret ‖ roomId) — permite autenticar ingressantes sem o backend
   *  conhecer a permissão de antemão (vetor adversarial 10).
   *  *(derivado de §1 prosa; adicionado no rework M3)* */
  storedHash: string;
}

export interface SignalingMessage {
  type: 'offer' | 'answer' | 'ice_candidate' | 'room_join' | 'room_leave';
  roomId: string;
  from: PeerId;
  to?: PeerId;             // opcional, broadcast se omitido
  payload: unknown;              // SDP ou ICE candidate
}

/**
 * Abstração de transporte para o signaling.
 *
 * `send(peerId, msg)` entrega a mensagem ao peer correspondente. Erros de
 * entrega são responsabilidade do transporte (o serviço não os propaga —
 * relay é fire-and-forget).
 */
export interface SignalingTransport {
  send(peerId: PeerId, msg: SignalingMessage): void;
}

export interface SignalingService {
  /** Cria ou entra em sala de signaling. Requer secret válido.
   *  Operação puramente in-memory (sem IO assíncrono) — retorna boolean. */
  joinRoom(peerId: PeerId, roomId: string, secret: Uint8Array): boolean;

  /** Sai de sala. Idempotente. */
  leaveRoom(peerId: PeerId, roomId: string): void;

  /** Encaminha mensagem SDP/ICE para peers na sala. */
  relayMessage(msg: SignalingMessage): void;

  /** Lista peers em uma sala (requer autenticação). */
  listPeers(roomId: string, secret: Uint8Array): PeerId[];

  /** Deriva RendezvousId. */
  deriveRoomId(secret: Uint8Array, permissionId: string): string;

  /** Registra o transporte usado para entregar mensagens a um peer.
   *  Desacopla a lógica de negócio do transporte (WebSocket, in-memory stub, etc.). */
  setTransport(peerId: PeerId, transport: SignalingTransport): void;
}
```
- `deriveRoomId`: `SHA-256(rendezvous_secret ‖ ASSET:PERMISSION_ID)` — vetor adversarial 10: sem `rendezvous_secret`, sala inacessível.
- `joinRoom`: Se a sala for nova, cria a sala no backend e inicializa seu `storedHash` localmente como `SHA-256(secret ‖ roomId)`. Se a sala já existir, valida se `SHA-256(secret ‖ roomId) === storedHash` antes de adicionar o peer. Isso permite autenticar os ingressantes sem exigir que o backend conheça a permissão de antemão.
- Mensagens relay: SDP offer/answer e ICE candidates entre peers na mesma sala.
- Sala derivada de capability: o secret é distribuído junto ao UCAN que concede a permissão.
