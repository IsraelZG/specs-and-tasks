# T-401 · Signaling no peer do sistema

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro) + `playwright` (E2E)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Implementar signaling de WebRTC no peer do sistema: troca de SDP/ICE via WS; salas derivadas de capability (`RendezvousId = SHA-256(rendezvous_secret ‖ ASSET:PERMISSION_ID)`); secret distribuído junto ao UCAN
*(derivado de `docs/plano-de-implementacao.md` §7 T-401)*.

### Contratos exatos
```ts
// --- apps/system-peer/src/signaling.ts ---
import { DevicePeerId } from '@plataforma/protocol'; // T-105a
import { NetworkAdapterPort } from '@plataforma/protocol'; // T-004

export interface SignalingRoom {
  roomId: string;                // SHA-256(rendezvous_secret ‖ permission_id)
  peers: Set<DevicePeerId>;
  createdAt: number;
}

export interface SignalingMessage {
  type: 'offer' | 'answer' | 'ice_candidate' | 'room_join' | 'room_leave';
  roomId: string;
  from: DevicePeerId;
  to?: DevicePeerId;             // opcional, broadcast se omitido
  payload: unknown;              // SDP ou ICE candidate
}

export interface SignalingService {
  /** Cria ou entra em sala de signaling. Requer secret válido. */
  joinRoom(peerId: DevicePeerId, roomId: string, secret: Uint8Array): Promise<boolean>;

  /** Sai de sala. */
  leaveRoom(peerId: DevicePeerId, roomId: string): void;

  /** Encaminha mensagem SDP/ICE para peers na sala. */
  relayMessage(msg: SignalingMessage): void;

  /** Lista peers em uma sala (requer autenticação). */
  listPeers(roomId: string, secret: Uint8Array): DevicePeerId[];

  /** Deriva RendezvousId. */
  deriveRoomId(secret: Uint8Array, permissionId: string): string;
}
```
- `deriveRoomId`: `SHA-256(rendezvous_secret ‖ ASSET:PERMISSION_ID)` — vetor adversarial 10: sem `rendezvous_secret`, sala inacessível.
- `joinRoom`: valida `SHA-256(secret ‖ roomId) === storedHash` antes de adicionar peer.
- Mensagens relay: SDP offer/answer e ICE candidates entre peers na mesma sala.
- Sala derivada de capability: o secret é distribuído junto ao UCAN que concede a permissão.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest (Node puro) + Playwright (E2E).

Casos de teste (Vitest):
1. `deriveRoomId` determinístico: mesmo secret + permissionId → mesmo roomId.
2. `deriveRoomId` com secret diferente → roomId diferente.
3. `joinRoom` com secret correto → sucesso.
4. `joinRoom` com secret errado → rejeitado (vetor adversarial 10).
5. `relayMessage` envia para todos os peers na sala.
6. `listPeers` retorna lista correta.

Casos de teste (Playwright):
7. Dois browsers estabelecem DataChannel real via signaling (Anel 3).
8. Peer sem secret não consegue entrar na sala.

## 5. Instruções de Execução
