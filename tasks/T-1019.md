# T-1019 · Ciclo completo de branch: merge squash + push + rename main

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Test Runner:** `vitest`
- **Capacidade-alvo:** sonnet *(envolve lógica git não-trivial: merge squash, push, detecção de conflito, rename de branch)*

## 1. Objetivo
Completar o ciclo de vida git das branches-por-task iniciado pela T-1012.
Hoje o `worktree.service.ts` cria branches e worktrees (`ensureTaskWorktree`), mas:

1. **Não faz merge** da branch na trunk após QA aprovar
2. **Não faz `git push`** das branches para o remote
3. **Não usa `--squash`** (o merge deveria consolidar em um único commit limpo)
4. **A branch principal é `master`**, precisa ser renomeada para `main`

Esta task adiciona ao `worktree.service.ts` as funções de merge/push e integra
no `TaskService.transition` para que o ciclo seja automático (com push **opcional**):
`start` → cria branch (+ push se `NEXUS_PUSH_ENABLED`) → `approve` → squash merge **local**
+ cleanup (+ push da trunk e delete da branch remota se `NEXUS_PUSH_ENABLED`).

## 3. Escopo de Arquivos (Inputs e Outputs) — assinaturas FIXAS

> **POLÍTICA (decidida): merge LOCAL por padrão; push atrás de gate próprio.** O `approve` faz
> o squash merge **local** na trunk e limpa a branch local + worktree — operação reversível.
> O `git push` (da trunk e o delete da branch remota) **só ocorre se `NEXUS_PUSH_ENABLED === '1'`**,
> um gate **separado** do `NEXUS_WORKTREE_ENABLED`. Sem esse gate, nada toca o `origin` do GitHub.

- **[UPDATE]** `apps/nexus-backend/src/services/worktree.service.ts` — adicionar:
  ```ts
  /** Opções de merge para o squash. */
  export interface MergeOptions {
    /** Branch base. Default: NEXUS_TRUNK_BRANCH ou a branch ATUAL do repo (detectada). */
    trunk?: string;
    /** Mensagem de commit do squash. */
    commitMessage?: string;
  }

  /** Resolve a trunk: opts.trunk ?? env NEXUS_TRUNK_BRANCH ?? `git rev-parse --abbrev-ref HEAD`. */
  // (interna; NÃO hardcode 'main' — o repo está em 'master' até o rename one-shot da seção 5.5)

  /**
   * Squash-merge LOCAL da branch task/<id> na trunk e limpeza. Push é OPCIONAL e gated.
   * Fluxo:
   *   1. resolver trunk (ver acima)
   *   2. git checkout <trunk>
   *   3. SE NEXUS_PUSH_ENABLED==='1': git pull --ff-only   (best-effort; se não houver
   *      upstream, captura o erro e segue — não aborta o merge local)
   *   4. git merge --squash task/<id>
   *   5. git commit -m "<commitMessage>"
   *   6. SE NEXUS_PUSH_ENABLED==='1': git push                (trunk)
   *   7. git branch -D task/<id>                              (sempre — limpeza local)
   *   8. SE NEXUS_PUSH_ENABLED==='1': git push origin --delete task/<id>  (best-effort)
   *   9. git worktree remove <wtPath> (se existir)
   *
   * Conflito no passo 4: `git merge --abort` e LANÇA Error descritivo (NÃO silencia —
   * o chamador trata). A trunk deve voltar limpa.
   */
  export async function mergeTaskBranch(
    id: string,
    opts?: WorktreeOptions & MergeOptions,
  ): Promise<void>

  /**
   * Faz push da branch task/<id> para o `origin`. Gated:
   *   - SE NEXUS_PUSH_ENABLED !== '1' → NO-OP (retorna imediatamente; opcional console.debug).
   *   - Senão: `git push -u origin task/<id>`. Idempotente (re-push de branch existente é ok).
   * Best-effort: em erro, console.warn e NÃO relança.
   */
  export async function pushTaskBranch(
    id: string,
    opts?: WorktreeOptions,
  ): Promise<void>
  ```

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [x] **Framework:** Vitest

      do `git init` (que aqui é `master`): use `git init -b main <dir>` (ou `git -c init.defaultBranch=main init`),
      um commit vazio, e configure `user.email`/`user.name`. Para os testes de push, simule o remote
      com `git init --bare` num segundo dir tmp + `git remote add origin <bare-path>` e um push inicial

      no `afterEach`); os de merge-local rodam com o gate **desligado**.
- [x] **Cobertura:**
  - **merge local (push OFF):** após `mergeTaskBranch`, a trunk tem exatamente 1 commit a mais,
    a branch local `task/<id>` não existe, o worktree foi removido, e o `origin` **não** foi tocado.
  - **push ON:** `pushTaskBranch` envia a branch ao bare remote; `mergeTaskBranch` empurra a trunk
    e remove a branch remota.
  - **conflito:** `mergeTaskBranch` lança Error e faz `--abort` (trunk limpa, branch persiste).
  - **gate de push OFF:** `pushTaskBranch` é no-op (não falha, não cria branch remota).
  - **idempotência:** chamar `pushTaskBranch` 2x (com push ON) não dá erro.
- [x] **Ambiente:** Node puro (sem browser). Pular testes se `git` não existir (`it.skip`).
- [x] **Fora de Escopo:** Integração real com GitHub; a fiação no `transition` (ganchos `start`/`approve`)
      é coberta por inspeção; o fallback `done→blocked` pode ter um teste unitário simulando merge que lança.

### Pegadinhas conhecidas *(armadilhas que derrubam um modelo leve — leia antes de codar)*
- **Trunk ≠ 'main' agora.** O repo está em **`master`** e o `git init` desta máquina cria `master`.
  Se você hardcodar `'main'`, o `git checkout main` falha (no teste E no repo real até o rename).
  Resolva a trunk por `opts.trunk ?? NEXUS_TRUNK_BRANCH ?? git rev-parse --abbrev-ref HEAD`, e no
  teste use `git init -b <trunk>` para fixar.
- **Dois gates distintos.** `NEXUS_WORKTREE_ENABLED` liga os ganchos no `transition`;
  `NEXUS_PUSH_ENABLED` (separado!) é o ÚNICO que autoriza tocar o `origin`. Não confunda nem unifique.
- **`title` vem do registro.** No gancho do `approve`, pegue o título com
  `this.getTask(id).frontmatter.title` — não invente nem deixe `undefined` no commit message.
- **Ordem importa no `approve`.** O gancho de merge roda ANTES do `rebuildIndexes()` final do
  `transition`; só assim o fallback `done→blocked` aparece no INDEX. Não chame `rebuildIndexes()` à parte.
- **`async` já é a assinatura.** `transition` já é `async` (T-1012); apenas `await` suas chamadas novas.
