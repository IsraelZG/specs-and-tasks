## 1. Objetivo
Adicionar perfis de NAT (`open`, `cone`, `symmetric`) ao `SimNetwork`, controlando quais peers podem se comunicar diretamente
*(extraído de T-006 §1)*.

### Contratos exatos
```ts
// --- packages/testkit/src/SimNetwork.ts (extensão) ---
import type { PeerId } from '@plataforma/protocol';

export enum NatType {
  Open = 'open',
  Cone = 'cone',
  Symmetric = 'symmetric',
}

export interface LinkConfig {           // derivado de T-006a §1
  latencyMs?: number;     // default 0
  jitterMs?: number;      // ±jitter uniforme
  dropRate?: number;      // 0.0–1.0 (probabilidade de drop)
  bandwidthBps?: number;  // Bytes por segundo (0 = ilimitado)
}

export interface PeerConfig {           // derivado de T-006a §1 (estendido aqui)
  link?: LinkConfig;      // T-006a
  natType?: NatType;      // ← T-006b
}

export class SimNetwork {               // derivado de T-005 (v1) + T-006a (extensão)
  setPeerConfig(peerId: PeerId, config: PeerConfig): void;   // já existia em T-006a
  createAdapter(peerId: PeerId): NetworkAdapterPort;          // send() rastreia outbound (knownPeers); deliver() enforça NAT
}
```
- `Open` (default implícito quando `natType` ausente ou `Open`): qualquer peer pode conectar.
- `Cone`: peer só recebe de destinos que ele próprio já enviou antes (outbound tracking).
- `Symmetric`: idem Cone — peer só recebe de pares que ele próprio já enviou antes. A spec original diferenciava "endereço (ip:port)" mas no SimNetwork in-memory a granularidade é por `PeerId`.
- **Outbound tracking:** mantido em `Map<PeerId, Set<PeerId>>` no `SimNetwork`; cada `send(from, to)` adiciona `to` ao `Set` de `from`.
- **NAT enforcement:** em `deliver(to, from, …)`, antes da entrega, verifica `toCfg.natType`:
  - `Open` (ou ausente): passa.
  - `Cone`/`Symmetric`: se `from ∉ knownPeers.get(to)`, drop silencioso (não chama handlers do destino).
- **Mudança de NAT em runtime:** `setPeerConfig` substitui a config; a próxima `deliver` usa o novo perfil. O `knownPeers` acumulado NÃO é resetado (intencional: o histórico de outbound persiste).
- T-006b **NÃO** introduz `knownPeers` reset nem reverte a lógica de degradação do T-006a — composições são ortogonais.

Casos de teste (4):
1. **Symmetric:** peer `B` com `natType: Symmetric`. Setup: `B.send(A)` (B conhece A). A→B funciona, C→B bloqueado (B não conhece C).
2. **Cone:** peer `B` com `natType: Cone`. Setup: `B.send(A)`, `B.send(C)`. A→B funciona, C→B funciona.
3. **Open:** peer `B` com `natType: Open`. A→B funciona sem B ter contatado A; C→B funciona sem B ter contatado C.
4. **Runtime switch:** `B` começa `Symmetric` (sem conhecer C) → C→B bloqueado. `setPeerConfig(B, { natType: Open })` → C→B agora funciona.

> **Cobertura de bordas (não-bloqueante para a task, mas o worker pode adicionar):**
> - `setPeerConfig` antes de qualquer `send`: NAT em `Open` por default, sem `knownPeers` registrado.
> - `close()` do adapter: limpar `knownPeers` daquele peer (evita leak).
> - `partition`/`heal` ortogonais a NAT (já é o comportamento atual).
