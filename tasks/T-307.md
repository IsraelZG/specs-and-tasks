# T-307 · Coordenação de sync + failover

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo

## 1. Objetivo
Implementar coordenação de sync e failover no pacote `@plataforma/transport`: eleição determinística do líder local (menor `entity_id` ativo), sessão RBSR única com o líder, timer de 5s com PING 2s, shadowban temporário, rollback WAL, re-eleição e retomada do último sub-range
*(derivado de `docs/plano-de-implementacao.md` §6 T-307)*.

### Contratos exatos (corrigidos de T-205, T-302a, T-004, T-105)

> **Discrepâncias com a spec original corrigidas:**
> 1. `SwarmRegistry` está em `packages/transport/src/SwarmRegistry.ts` (T-205) — **NÃO em core**.
> 2. Não há `DevicePeerId` separado — T-105 define `PeerId` em `@plataforma/protocol` (o nome
>    "Device" é semântico, não há tipo distinto).
> 3. Não há `PeerRecord` exportado por T-205 — a interface é `PeerEntry` (`peerId`, `state`,
>    `latencyMs`, `tier`, `relayScore`, `identityEpoch`).
> 4. `T-105a` não existe como task — T-105 (parent) é o contrato de identidade.
> 5. T-307 vive no pacote `@plataforma/transport` (mesmo pacote de SwarmRegistry) — não importa
>    de `@plataforma/core`.

```ts
// --- packages/transport/src/syncCoordinator.ts (NOVO) ---
import type { StoragePort, PeerId, ClockPort } from '@plataforma/protocol'; // T-004, T-105
import { RBSRSession } from '@plataforma/protocol'; // T-302a
import { SwarmRegistry, PeerEntry } from './SwarmRegistry.js'; // T-205 (mesmo pacote)

/** Opções de tuning do SyncCoordinator. Todos têm defaults razoáveis. */
export interface SyncCoordinatorOptions {
  /** Registry de peers ativos (T-205). */
  readonly swarm: SwarmRegistry;
  /** Timer total para conclusão de uma eleição. Default 5000ms. */
  readonly electionTimerMs?: number;
  /** Intervalo entre PINGs ao líder. Default 2000ms. */
  readonly pingIntervalMs?: number;
  /** PINGs falhos consecutivos para shadowban. Default 3. */
  readonly shadowbanThreshold?: number;
  /** Duração do shadowban. Default 30000ms. */
  readonly shadowbanDurationMs?: number;
  /** Clock para timers determinísticos (T-004 ClockPort). Default = system clock. */
  readonly clock?: ClockPort;
}

/** Resultado de uma eleição. */
export interface LeaderInfo {
  readonly peerId: PeerId;
  readonly electedAt: number;     // timestamp (clock.now())
  readonly isSelf: boolean;
}

/** Evento de failover disparado em onFailover. */
export interface FailoverEvent {
  readonly reason: 'leader_unreachable' | 'leader_shadowbanned' | 'leader_disconnected';
  readonly previousLeader: PeerId;
  readonly newLeader: PeerId;
  /** True se rollback WAL foi executado antes da re-eleição. */
  readonly walRollback: boolean;
  /** ULID-range do último sub-range em progresso antes do failover (null se nada em progresso). */
  readonly lastSubRange: string | null;
}

export class SyncCoordinator {
  constructor(storage: StoragePort, opts: SyncCoordinatorOptions);

  /**
   * Executa eleição determinística: peers ativos no swarm ordenados por
   * `entry.deviceId` (lex order). Menor vence. Se vazio, retorna `null`.
   */
  electLeader(): Promise<LeaderInfo | null>;

  /**
   * Inicia sessão RBSR (T-302a) com o líder. Se `leaderPeerId` não está no swarm ou
   * está shadowbanned, lança.
   */
  startSession(leaderPeerId: PeerId): Promise<RBSRSession>;

  /**
   * Inicia loop de monitoramento: PING a cada `pingIntervalMs`; se `shadowbanThreshold`
   * PINGs falharem, marca como shadowbanned e dispara failover (callback registrado em
   * onFailover). Para de monitorar quando `stop()` é chamado.
   */
  monitorLeader(): Promise<void>;

  /** Para o loop de monitoramento. Idempotente. */
  stop(): void;

  /** Registra callback para failover. Pode ser chamado múltiplas vezes (broadcast). */
  onFailover(cb: (event: FailoverEvent) => void): () => void;

  /**
   * Executa rollback do state interno (em memória) até o último checkpoint. NÃO
   * executa `wal_checkpoint(TRUNCATE)` no SQLite (ver T-304 §6 para a discussão de WAL).
   * O rollback aqui é da posição de progresso RBSR (último sub-range), não do disco.
   */
  rollbackProgress(): Promise<void>;

  /**
   * Retoma reconciliação do último sub-range persistido em checkpoint. Retorna qtd
   * de nós importados na retomada.
   */
  resumeFromCheckpoint(): Promise<number>;
}
```

**Casos de teste (8 enumerados, baseados em §4 original):**

1. **`electLeader` com 3 peers** — `PeerEntry.deviceId` ordenados: peer com id "01H...A" vence. `LeaderInfo.isSelf === true` se o próprio peer é o vencedor.
2. **`electLeader` com swarm vazio** — retorna `null` (sem líder).
3. **Líder cai durante RBSR** — após `startSession`, simular `peerEntry.state = 'disconnected'` no swarm; `onFailover` callback dispara com `reason: 'leader_disconnected'`, `walRollback: true`.
4. **Failover retoma sem perda** — após failover, `resumeFromCheckpoint` re-aplica o último sub-range; resultado final do RBSR (nós importados) é igual ao que seria se não houvesse failover (no caso teste = sequência sem interrupção).
5. **Failover retoma sem duplicação** — verifica via `getDeferredRanges` (T-306) ou tracking próprio: nenhum nó importado duas vezes.
6. **PING 2s: 1 falha mantém líder; 3 falhas → shadowban** — com VirtualClock, avançar 2s e simular PING falho 3x; 1ª e 2ª falha: líder mantido, `consecutiveFailures++`; 3ª falha: `reason: 'leader_shadowbanned'`, `electLeader` re-elege novo.
7. **Shadowban expira** — `VirtualClock.advance(shadowbanDurationMs)`, peer volta a ser elegível. Verificar via `swarm.get(deviceId).state === 'connected'` (ou flag explícita de shadowban expiry).
8. **`rollbackProgress` + `resumeFromCheckpoint` roundtrip** — sem failover, `rollbackProgress()` limpa o checkpoint interno; `resumeFromCheckpoint()` retorna 0 (nada a retomar).

## 6. Feedback de Especificação

**Decisão 1 (correção de imports):** a spec original importava `SwarmRegistry` de `@plataforma/core` —
**errado**: T-205 o define em `@plataforma/transport` (mesmo pacote de T-307). Aplicado:
imports de `./SwarmRegistry.js` (relativo ao pacote). Também corrigido: não há `DevicePeerId`
separado, é `PeerId` (T-105); não há `PeerRecord`, é `PeerEntry` (T-205); `T-105a` não existe.

**Decisão 2 (rollback WAL escopo):** a spec original menciona "rollback WAL" e
`wal_checkpoint(TRUNCATE)`. O `SyncCoordinator` **NÃO** deve tocar o WAL do SQLite diretamente
— isso é responsabilidade de T-304 (ConcurrentReconciliationGuard, hoje `blocked-decision`).
O rollback desta task é apenas da **posição de progresso RBSR** (qual sub-range estava sendo
processado), persistido em memória. Aplicado: `rollbackProgress()` (não `rollbackWAL()`) com
contrato explícito de que NÃO toca o storage.

**Decisão 3 (PING reuso):** "PING ao líder" não é uma mensagem RBSR nova. Recomenda-se reusar
`MSG_ROOT_FINGERPRINT` de T-302a: enviar root fingerprint; se não chegar resposta em
`pingIntervalMs`, considera falha. Justificativa: PING separado duplica o canal, e o root
fingerprint já é barato (32 bytes) e prova que o peer está processando mensagens RBSR.

**Decisão 4 (persistência de checkpoint):** `lastCheckpoint` fica em memória, não no storage.
Em caso de crash do peer, é perdido. Recomenda-se (em ADR futuro) mover para tabela
`sync_progress` (vinculada à decisão 2 de T-304), mas para esta task YAGNI.

**Decisão 5 (shadowban expiry):** O spec original diz "shadowban temporário expira → peer pode
ser re-eleito". Isso requer: (a) timer de expiry, (b) re-eleição automática ou apenas remoção
da flag? Recomendação: timer de expiry gerenciado por `monitorLeader` (que já tem o clock);
ao expirar, marca peer como não-shadowbanned. Re-eleição só ocorre no próximo `electLeader()`.

**Ações:**
- T-307 implementável com as 5 decisões aplicadas; nenhuma decisão em aberto.
