# T-307 · Coordenação de sync + failover

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Implementar coordenação de sync e failover no pacote `@plataforma/transport`: eleição determinística do líder local (menor `entity_id` ativo), sessão RBSR única com o líder, timer de 5s com PING 2s, shadowban temporário, rollback WAL, re-eleição e retomada do último sub-range
*(derivado de `docs/plano-de-implementacao.md` §6 T-307)*.

### Contratos exatos
```ts
// --- packages/transport/src/syncCoordinator.ts ---
import { SwarmRegistry, PeerRecord } from '@plataforma/core'; // T-205
import { DevicePeerId } from '@plataforma/protocol'; // T-105a
import { RBSRSession } from '@plataforma/protocol'; // T-302a
import { StoragePort } from '@plataforma/protocol'; // T-004

export interface SyncCoordinatorOptions {
  swarm: SwarmRegistry;
  electionTimerMs?: number;    // default 5000 (5s para eleição)
  pingIntervalMs?: number;     // default 2000 (2s entre PINGs)
  shadowbanThreshold?: number;  // PINGs falhos consecutivos para ban temporário (default 3)
  shadowbanDurationMs?: number; // default 30000 (30s)
}

export interface LeaderInfo {
  peerId: DevicePeerId;
  electedAt: number;
  isSelf: boolean;
}

export interface FailoverEvent {
  reason: 'leader_unreachable' | 'leader_shadowbanned' | 'leader_disconnected';
  previousLeader: DevicePeerId;
  newLeader: DevicePeerId;
  walRollback: boolean;
  lastSubRange: string | null;
}

export class SyncCoordinator {
  constructor(storage: StoragePort, opts: SyncCoordinatorOptions);

  /** Executa eleição determinística: menor entity_id ativo no swarm vence. */
  electLeader(): Promise<LeaderInfo>;

  /** Inicia sessão RBSR com o líder eleito. */
  startSession(leaderPeerId: DevicePeerId): Promise<RBSRSession>;

  /** Monitora líder: PING a cada 2s; 3 falhas → shadowban → re-eleição. */
  monitorLeader(): Promise<void>;

  /** Callback quando failover ocorre. */
  onFailover(cb: (event: FailoverEvent) => void): void;

  /** Executa rollback WAL até último checkpoint de range. */
  rollbackWAL(): Promise<void>;

  /** Retoma reconciliação do último sub-range pendente após failover. */
  resumeFromCheckpoint(): Promise<number>;
}
```
- Eleição determinística: peers ativos no swarm ordenados por `entity_id`; o menor é líder.
- Sessão RBSR: única com o líder; se líder cai, nova eleição.
- Failover: ao matar líder no meio do RBSR, novo líder assume e retoma sem perda nem duplicação.
- Rollback WAL: checkpoint do range atual antes de re-eleição.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest + SimNetwork (T-005) + VirtualClock (T-003).
- [x] **Ambiente:** `pnpm --filter @plataforma/transport test`.

Casos de teste:
1. `electLeader` com 3 peers: menor `entity_id` vence.
2. `electLeader` com swarm vazio → sem líder.
3. Líder cai durante RBSR → `onFailover` dispara com `walRollback=true`.
4. Após failover, retoma do último sub-range sem perda de dados.
5. Após failover, retoma sem duplicação (nós já recebidos não são reaplicados).
6. PING 2s: 1 falha → mantém líder; 3 falhas consecutivas → shadowban → re-eleição.
7. Shadowban temporário expira → peer pode ser re-eleito.
8. `rollbackWAL` + `resumeFromCheckpoint` roundtrip.
