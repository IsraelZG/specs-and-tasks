# T-004b · Adapter SQLite WASM/OPFS do StoragePort (browser/local-first)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+ (build) + Browser headless (Playwright para testes)

- **Monorepo:** Turborepo

- **Capacidade-alvo:** sonnet

## 1. Objetivo
Implementar o **adapter SQLite de browser** do `StoragePort` — a contraparte local-first do adapter
Node entregue por T-004a. No browser não há `node:sqlite`; o backend roda **SQLite compilado em
WASM** com persistência via **OPFS** (Origin Private File System). Deve satisfazer **o mesmo
contrato** `StoragePort` (`exec`/`transaction`/`migrate`) e passar **a mesma suíte de conformidade**
que T-004a criou — a diferença é só o backend e o ambiente de teste (browser headless, não Node).

> **Decisão deferida de T-004a (2026-06-30):** o backend Node (`node:sqlite`) foi entregue em T-004a;
> o backend WASM/OPFS foi separado aqui porque tem fronteira própria (performance do WASM, ciclo de
> vida do OPFS, bundling, SharedArrayBuffer/COOP-COEP headers) que não devia atrasar T-304/T-701.

### Contratos exatos
```ts
// --- packages/core/src/sqliteWasmStorage.ts 
---
import { StoragePort, SqlRow, SqlParams } from '@plataforma/protocol'; // T-004
import type { Migration } from './schema.js'; // T-106

/**
 * Adapter StoragePort para browser via SQLite WASM + OPFS.
 * Mesmo contrato de T-004a (SqliteStorage), backend diferente.
 * *(derivado de T-004 §1 — StoragePort; T-004a §1 — padrão SqliteStorage)*
 */
export class SqliteWasmStorage implements StoragePort {
  /**
   * Abre/cria banco via OPFS usando `wa-sqlite` (decisão do arquiteto,
   * 2026-07-01 — ver §6 Decisão A), num Web Worker dedicado (Decisão B).
   * @param dbPath — caminho relativo no OPFS (ex.: "device_state.db")
   */
  constructor(dbPath: string);

  /** Executa SQL e retorna linhas. *(derivado de T-004 §1 — StoragePort.exec)* */
  exec(sql: string, params?: SqlParams): Promise<SqlRow[]>;

  /** Transação com rollback em exceção. *(derivado de T-004a §5 — BEGIN/COMMIT/ROLLBACK)* */
  transaction<T>(fn: () => Promise<T>): Promise<T>;

  /** Aplica migrations pendentes. Idempotente. *(derivado de T-106 — MIGRATIONS)* */
  migrate(migrations: Migration[]): Promise<void>;

  /**
   * No-op documentado no browser (Decisão C, §6) — OPFS é single-process,
   * WAL resolve concorrência multi-processo que não existe aqui.
   * *(derivado de T-004a §1 — walCheckpoint)*
   */
  walCheckpoint(mode?: 'PASSIVE' | 'FULL' | 'RESTART' | 'TRUNCATE'): Promise<void>;

  /** Fecha o banco e libera recursos. */
  close(): Promise<void>;
}
```
- `exec`: executa SQL com params; retorna linhas como `SqlRow[]`.
- `transaction`: `BEGIN`/`COMMIT`/`ROLLBACK` explícitos (padrão T-004a).
- `migrate`: aplica `MIGRATIONS` de T-106; idempotente (roda 2x sem duplicar).
- `walCheckpoint`: no-op documentado (Decisão C, §6) — divergência intencional do contrato de T-004a.
- `close`: libera handle OPFS + WASM.

## 6. Feedback de Especificação

### Decisões de Arquiteto (2026-07-01, resolvidas via `/arquiteto-decisoes`)

**Decisão A — Biblioteca WASM: DECIDIDO `wa-sqlite`.** Wrapper moderno, suporta OPFS VFS
nativamente, assíncrono; não serializa o DB inteiro para persistir (necessário para
`device_state.db` com lineage/archives crescendo). Fonte canônica para `package.json` (dep) e
`sqliteWasmStorage.ts` (API de inicialização). `sql.js` e `@sqlite.org/sqlite-wasm` descartados.

**Decisão B — Threading: DECIDIDO Worker dedicado.** `SqliteWasmStorage` roda em Web Worker com
bridge via `postMessage` pro main thread — acesso síncrono ao OPFS (`createSyncAccessHandle`).
Checar headers COOP/COEP (cross-origin isolation) contra a config de T-008 (App Bancada PWA) se
usar `SharedArrayBuffer`.

**Decisão C — WAL no browser: DECIDIDO no-op documentado.** `walCheckpoint` vira no-op no
browser — coerente com single-process do OPFS (WAL resolve concorrência multi-processo, que não
existe aqui). Documentar a divergência do contrato original de T-004a.
