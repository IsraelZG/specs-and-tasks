## 1. Objetivo
Emissão, validação e delegação de UCANs (User Controlled Authorization Networks) baseados em assinaturas Ed25519, com enforcement do `edge_filter` (obrigatório para `depth > 1`) e do predicado de bloqueio (v4).
*(derivações de `docs/conceitos/ucan.md` e `docs/conceitos/predicado-de-bloqueio.md` — a RAG original `docs/caderno-4-governance/01-ucan.md` **NÃO EXISTE no disco**; confirmado em 2026-06-22.)*

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/core/src/auth/ucan.ts 
---

/** Algoritmo de assinatura do UCAN. v0: só Ed25519. */
export type UcanAlgo = 'ed25519';

/** Capability atômica do UCAN: `capability` (recurso) + `caveat` (restrição opcional). */
export interface UcanCapability {
  readonly resource: string;   // ex: "ASSET:PERMISSION:01ARZ..."
  readonly ability: string;    // ex: "read", "write", "delegate"
  readonly caveat?: Record<string, unknown>;
}

/** Payload canônico do UCAN (a serializar, sem assinaturas). */
export interface UcanPayload {
  readonly iss: string;        // issuer — chave pública Ed25519 do emitente (hex)
  readonly aud: string;         // audience — chave pública do destinatário (hex)
  readonly sub?: string;        // subject (quem de fato usa; pode == aud)
  readonly exp: bigint;         // expiration (HLC ou unix ms — ver §6 decisão em aberto)
  readonly nbf?: bigint;        // not before
  readonly capabilities: readonly UcanCapability[];
  readonly depth: number;       // 0..6 (caderno ucan.md: "depth ≤ 6")
  readonly root: string;        // ULID do nó raiz do traversal
  readonly direction: 'in' | 'out' | 'both';
  readonly edgeFilter?: ReadonlyArray<readonly [edgeType: string, targetNodeType: string]>;
  /** Se `false`, este UCAN NÃO pode ser sub-delegado (um filho que o referencie como `parent` é inválido). Default na emissão: `true`. (caderno `ucan.md` §Delegação) */
  readonly delegatable: boolean;
  /** Cadeia de delegação: payload do pai, se este UCAN foi delegado. */
  readonly parent?: UcanPayload;
}

/** UCAN emitido: payload + assinatura. */
export interface SignedUcan {
  readonly payload: UcanPayload;
  readonly signature: Uint8Array; // 64 bytes Ed25519 sobre `canonicalize(payload)`
  readonly algo: UcanAlgo;
}

/** Resultado de `validateUcan`. */
export type UcanValidation =
  | { ok: true; ucan: SignedUcan }
  | { ok: false; reason:
      | 'signature_invalid'        // assinatura Ed25519 não confere
      | 'expired'                  // exp passou
      | 'not_yet_valid'            // nbf ainda no futuro
      | 'chain_broken'             // assinatura do parent não confere com iss do payload atual
      | 'edge_filter_missing'      // depth > 1 e edgeFilter ausente/vazio (ucan.md §Invariante)
      | 'edge_filter_violation'    // algum nó acessível viola edge_filter
      | 'audience_mismatch'        // aud ≠ chave pública de quem está usando
      | 'depth_exceeded'           // depth > 6
      | 'delegatable_false'        // parent tinha `delegatable: false` mas há sub-delegação
  };

/** Erro na construção do UCAN. */
export class UcanError extends Error {}

/**
 * Emite um UCAN a partir de um par (issuerKey, payload). Assina Ed25519.
 * Valida estrutura do payload antes de assinar:
 *  - `iss` deve ser 32 bytes hex (chave pública Ed25519).
 *  - `exp` deve ser > 0.
 *  - `capabilities` não pode ser vazio.
 *  - `depth` entre 0 e 6.
 *  - Se `depth > 1`, `edgeFilter` é obrigatório e não pode ser vazio.
 *  - Se `parent` é fornecido, `parent.iss` deve == `aud` deste payload (chain).
 *  - Se `parent` é fornecido, `parent.delegatable` deve ser `true` (senão é sub-delegação proibida → `validateUcan` retorna `delegatable_false`).
 */
export async function issueUcan(
  issuerPrivateKey: Uint8Array,
  payload: UcanPayload,
): Promise<SignedUcan>;

/** Versão síncrona de issueUcan (usa `ed25519Sign` síncrono do `@plataforma/crypto`). */
export function issueUcanSync(
  issuerPrivateKey: Uint8Array,
  payload: UcanPayload,
): SignedUcan;

/**
 * Valida um UCAN: assinatura, expiração, chain de delegação, edge_filter,
 * profundidade, audience. **NÃO** consulta SQLite nem key-vault — é pure-math
 * (criptográfica). O predicado de bloqueio (`blocks-aresta` do autor) é
 * responsabilidade do `KeyVault` (T-110) — esta task só valida o UCAN em si.
 */
export function validateUcan(ucan: SignedUcan): UcanValidation;

/** Canonicalização determinística para assinatura (ordem léxica de chaves, bigint como string). */
export function canonicalizeUcanPayload(payload: UcanPayload): Uint8Array;
```

### Casos de teste (numerados)
1. `issueUcan` retorna `SignedUcan` com assinatura de 64 bytes.
2. `validateUcan` aceita UCAN recém-emitido (roundtrip).
3. Assinatura adulterada (1 byte trocado) → `{ok:false, reason:'signature_invalid'}`.
4. `exp` no passado → `{ok:false, reason:'expired'}`.
5. `nbf` no futuro → `{ok:false, reason:'not_yet_valid'}`.
6. `aud` diferente da chave pública de quem está submetendo → `{ok:false, reason:'audience_mismatch'}`.
7. `depth > 1` sem `edgeFilter` → `{ok:false, reason:'edge_filter_missing'}`.
8. `depth > 1` com `edgeFilter` válido (não-vazio) → ok.
9. `depth > 1` com `edgeFilter` vazio `[]` → `{ok:false, reason:'edge_filter_missing'}`.
10. **Chain de delegação válida:** parent.assinado por iss-pai + iss-pai = aud deste → ok.
11. **Chain quebrada:** parent.assinado por chave diferente de parent.iss → `{ok:false, reason:'chain_broken'}`.
12. `depth > 6` → `{ok:false, reason:'depth_exceeded'}`.
13. **Predicado de bloqueio — `validateUcan` NÃO rejeita por BLOCKS** (verificar que BLOCKS é responsabilidade do KeyVault, não desta task) — isso confirma a separação de concerns. Se o teste mostrar que validateUcan rejeita por BLOCKS, está errado.
14. **`delegatable: false` na chain:** parent emitido com `delegatable: false` + um UCAN filho que o referencia como `parent` → `{ok:false, reason:'delegatable_false'}`. (espelha o campo `delegatable` do `UcanPayload` e a regra de `issueUcan`).

### Pegadinhas conhecidas
- `canonicalizeUcanPayload` precisa de **ordem determinística** de chaves (ordem léxica) para que a assinatura seja reproduzível — similar a `canonicalizeNode` da T-107.
- `bigint` no payload (exp/nbf) deve serializar como string decimal (ou int64) — não como number (perda de precisão).
- `iss`/`aud` são chaves públicas Ed25519 (32 bytes hex) — validar tamanho antes de assinar.
- `parent.iss` deve ser **igual a** `payload.aud` (não apenas compatível) — chain é literal, não semântica.

## 6. Feedback de Especificação (Spec Feedback Loop)

> **✅ DECISÕES RESOLVIDAS (2026-06-23 — arquiteto):** (1) **`exp`/`nbf` = `bigint` de unix epoch ms** (tempo de parede; KeyVault T-110 compara com TTL real; HLC é para ordenação causal, não validade de token). (2) **`validateUcan` NÃO valida BLOCKS** — predicado de bloqueio é do KeyVault (T-110); confirmado pelo caso de teste 13.

- **ABERTO — `exp` e `nbf` em HLC vs unix ms:** `docs/conceitos/ucan.md` não fixa o tipo. O caderno-2/02 menciona HLC, mas o predicado-de-bloqueio.md fala em "TTL da chave em RAM". **Sugestão:** usar `bigint` representando `Date.now()` em ms (unix epoch ms), que é o que o KeyVault T-110 vai comparar com TTL. Se quiser HLC, é uma task separada (T-501b). ABERTO: arquiteto decide.
- **RESOLVIDO — `delegatable`:** o campo `readonly delegatable: boolean` foi adicionado ao `UcanPayload` (Seção 1), default `true` na emissão, e `validateUcan` rejeita sub-delegação de um parent com `delegatable: false` (reason `delegatable_false`, caso de teste 14). Isso elimina a inconsistência anterior (o enum `UcanValidation` já tinha `delegatable_false` mas o payload não tinha o campo). Se o arquiteto **não** quiser controle de delegação na v0, remova o campo, o reason e o caso 14 juntos — mas não deixe um sem o outro.
- **ABERTO — `blocks-aresta` (predicado de bloqueio) na validação:** o teste caso 13 confirma que `validateUcan` **não** rejeita por BLOCKS — isso é escopo do KeyVault (T-110). Garantir via teste explícito que esta separação é respeitada.
