# T-804 · Cloud WebSeed + Edge translation

## 0. Ambiente de Execução Obrigatório

## 1. Objetivo
Implementar Cloud WebSeed + Edge translation no peer do sistema: upload consolidado pelo agente do sistema; objeto único com tag_region; Edge stateless content-blind traduzindo Range↔peça e injetando token de bucket; fallback swarm→WebSeed
*(derivado de `docs/plano-de-implementacao.md` §11 T-804)*.

### Contratos exatos
```ts
// --- apps/system-peer/src/media/webSeed.ts 
---
import { Chunker, EncryptedChunk, InfoHash } from '@plataforma/media'; // T-801; InfoHash = Sha256Digest (§6 #1)
import { ed25519Sign, ed25519Verify, Ed25519PrivateKey, Ed25519PublicKey } from '@plataforma/crypto'; // T-101 — sem lib JWT nova; token JWT-shaped assinado com os wrappers Ed25519 já existentes (§6 #3)

/** Payload do bucketToken (JWT-shaped: base64url(header).base64url(payload).base64url(signature), assinado via ed25519Sign — não usa lib jsonwebtoken, que não tem suporte Ed25519 limpo no monorepo). */
export interface BucketTokenPayload {
  infoHash: InfoHash;
  scope: 'read';
  exp: number; // unix seconds
}

export interface WebSeedUpload {
  infoHash: InfoHash;
  totalSize: number;
  chunkCount: number;
  tagRegion: Uint8Array;        // concatenação ordenada das authTag de 16B por chunk, offset = index*16 (§6 #2)
}

export interface ChunkRange {
  start: number;                // byte offset
  end: number;                  // byte offset (exclusive)
}

export interface ChunkPiece {
  pieceIndex: number;
  range: ChunkRange;
}

export interface EdgeTranslation {
  /** Traduz HTTP Range request → índice de peça WebTorrent. */
  rangeToPieces(range: ChunkRange, chunkSize: number): ChunkPiece[];

  /** Traduz peça WebTorrent → HTTP Range. */
  pieceToRange(pieceIndex: number, chunkSize: number): ChunkRange;
}

export interface WebSeedService {
  /** Upload consolidado via agente do sistema. */
  upload(data: Uint8Array, tagRegion: Uint8Array): Promise<WebSeedUpload>;

  /** Serve chunk por range request HTTP. bucketToken verificado via ed25519Verify (§6 #3), sem estado no Edge. */
  serveChunk(infoHash: InfoHash, range: ChunkRange, bucketToken: string): Promise<Uint8Array>;

  /** Fallback: quando zero seeders P2P, serve do WebSeed. */
  serveFallback(infoHash: InfoHash, range: ChunkRange): Promise<Uint8Array>;

  /** Verifica se objeto existe. */
  hasObject(infoHash: InfoHash): Promise<boolean>;

  /**
   * Derruba o WebSeed do objeto (força fallback P2P) — endpoint HTTP consumido pelo
   * botão "Derrubar WebSeed" da Bancada (T-807 §6 Decisão #2). `DELETE /webseed/:infoHash`.
   */
  killWebSeed(infoHash: InfoHash): Promise<void>;
}
```
- `upload`: agente do sistema recebe arquivo consolidado → armazena objeto único com `tag_region` = concatenação ordenada das `authTag` de 16B por chunk, offset = `index*16` (§6 #2).
- `serveChunk`: Edge stateless traduz Range request → peça e serve chunk. Requer `bucketToken` JWT-shaped assinado via `ed25519Sign` (§6 #3), verificado via `ed25519Verify` — sem estado no Edge, expiração (`exp`) e escopo (`scope`) no payload.
- `serveFallback`: quando swarm tem zero seeders P2P, download funciona do WebSeed.
- Edge **nunca** recebe a chave AES — content-blind (asserção de interface).
- `InfoHash` (§6 #1) é `Sha256Digest` (`Uint8Array`, T-101/T-801), exportado de `@plataforma/media` — mesma fonte usada por T-807.

## 6. Feedback de Especificação

### Decisões de Arquiteto (2026-07-01, resolvidas via `/arquiteto-decisoes`)

**Decisão #1 — `InfoHash`: DECIDIDO opção (B).** `InfoHash` é exportado de `@plataforma/media`
(tipo já implementado em T-801 como `Sha256Digest`/`Uint8Array`) — mesma decisão aplicada em
T-807 (Bancada), que também referencia `InfoHash`. Uma fonte de verdade, sem duplicar definição.

**Decisão #2 — `tagRegion`: DECIDIDO opção (A).** Concatenação direta ordenada das `authTag` de
16 bytes por chunk: `offset = index * 16`. Zero overhead de serialização, acesso O(1) por offset
calculado — caso de uso é sempre denso e ordenado.

**Decisão #3 — `bucketToken`: DECIDIDO opção (A).** Token JWT assinado pela chave privada da
cloud/peer do sistema, contendo expiração e permissões de leitura do `infoHash`. Verificável sem
estado no Edge, suporta expiração e escopo granular (HMAC simples descartado).

**Nota de implementação (re-endurecimento, 2026-07-02):** o monorepo não usa nenhuma lib JWT
(`jsonwebtoken`/`jose` não aparecem em nenhum `package.json`) e o suporte EdDSA dessas libs é
inconsistente. Formato JWT-shaped (`base64url(header).base64url(payload).base64url(signature)`)
implementado com os wrappers Ed25519 já existentes (`ed25519Sign`/`ed25519Verify`, T-101) — sem
dependência nova.
