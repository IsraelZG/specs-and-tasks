## 1. Objetivo
Implementar `isAttenuation` (valida que UCAN filho é subset das capacidades do pai) e validação de cadeia de delegação (A→B→C) no pacote `@plataforma/core`
*(extraído de T-501 §1)*.

### Contratos exatos
```ts
// --- packages/core/src/auth/ucan-attenuation.ts ---
import { SignedUcan } from './ucan-codec.js'; // T-501a

/** Verifica se `child` é atenuação válida de `parent` (subset de capacidades). */
export function isAttenuation(child: SignedUcan, parent: SignedUcan): boolean;

/** Valida cadeia de delegação: cada UCAN é atenuação do anterior, todos verificam. */
export function validateChain(chain: SignedUcan[], rootPubKey: Ed25519PublicKey): Promise<boolean>;
```
Regras de atenuação:
- `child.att` deve ser subset de `parent.att` (mesmo `with`, `can` igual ou mais restrito).
- Child não pode ter `can` que o pai não tem.
- Child não pode ter `with` que o pai não cobre (prefix matching).

## 6. Feedback de Especificação
- **ABERTO — algoritmo `isAttenuation` sem fonte canônica:** nenhuma fonte (T-501 done, `docs/conceitos/ucan.md`, cadernos, ADRs) define o algoritmo de comparação de atenuação entre UCANs. As regras atuais ("subset de capabilities", "prefix matching no campo `with`") foram inventadas pelo autor da subtask. Impossível implementar sem escalar para arquiteto.
- **ABERTO — regra de `prefix matching` para `with`:** o termo "prefix matching" não aparece em NENHUM documento do repositório. A semântica exata (prefixo de path? prefixo de string? wildcard?) é indefinida. Sem isso, `isAttenuation` não pode ser implementada corretamente.
- **ABERTO — `validateChain` vs `validateUcan`:** T-501 (canônico) já define `validateUcan` com validação de chain de delegação (reason `chain_broken`). A função `validateChain(chain[], rootPubKey)` de T-501c sobrepõe ou duplica essa lógica? Precisa de decisão arquitetural.
