#### Achados da §5 — auditoria de código (revisão fria, anti-ancoragem)

- **[m1] → fixed (correto)**
  - `runner.ts:38-40`: `export const _internals = { startCancelWatcher }` está presente; chamada interna em `runner.ts:127` usa `_internals.startCancelWatcher(...)`. OK.

- **[m2] → MAS o teste é VACUOSO — BLOQUEANTE (M1)**
  - `runner.test.ts:124-138` — fluxo do teste 3:
    ```ts
    const result = await run({...cancelWatcher: false...});  // run completa primeiro
    const spy = vi.spyOn(_internals, 'startCancelWatcher');    // spy DEPOIS
    expect(spy).not.toHaveBeenCalled();                        // sempre 0
    ```
  - **Defeito:** o spy é criado **depois** que `run()` já executou. Como `_internals.startCancelWatcher` é chamado sincronicamente dentro de `run()` (antes do `await` resolver), qualquer chamada real teria acontecido antes do spy existir. A asserção `not.toHaveBeenCalled()` é **sempre verdadeira**, independente do comportamento real — o teste não falha mesmo se a regressão (chamar `startCancelWatcher` com `cancelWatcher:false`) for reintroduzida.
  - **Probe executada localmente** (criada em `tests/probe-vacuous.test.ts`, removida após validação): três variantes confirmam o defeito.
    - (A) `vi.spyOn` ANTES de `run()` + `cancelWatcher:true` → spy **detecta** a chamada. ✅
    - (B) `vi.spyOn` DEPOIS de `run()` + `cancelWatcher:true` → spy **não detecta** (sempre 0). Vacuoso. ❌
    - (C) `vi.spyOn` ANTES de `run()` + `cancelWatcher:false` → spy corretamente **não detecta**. ✅
  - O teste 3 atual segue o padrão (B). Spec §3 exige o padrão (C) — spy antes da chamada.
  - **Fix exigido (1 linha):** mover `const spy = vi.spyOn(_internals, 'startCancelWatcher')` para ANTES de `run(...)` no teste 3.
  - **Severidade:** MAJOR (não-bloqueante para merge, mas bloqueante para o objetivo do achado m2). Como o spec §4 caso 2 é o entregável de m2, e o teste não cumpre esse caso, devolvo para rework.

- **[m3] → fixed (correto, com observação sobre a trade-off)**
  - `runner.ts:3`: `import type { PluginTools } from '@plataforma/plugin-fs-tools';` ✅
  - `runner.ts:17`: `tools: Pick<PluginTools, 'readFile'>;` ✅
  - `runner.ts:127`: cast `as unknown as` **removido**, substituído por `as CancelWatcherOpts['tools']` (narrowing legítimo de `PluginTools` para `Pick<…>`). ✅
  - `runner.ts:26`: `?.` **removido** de `tools.readFile.execute`; adicionado cast de tipo de função `as (args: { path: string }) => Promise<unknown>`. Trade-off necessário — sem o `?.`, o `Tool.execute` (do `ai` SDK) tem assinatura `(input, options) => Promise<OUTPUT>` que requer narrowing para chamar com apenas `{ path }`. Não viola spec (spec só pedia para remover `as unknown as` e `?.`, não para evitar narrowing de função). Aceito.
  - Teste 5: `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync(1000)` + `expect(readFileSpy).toHaveBeenCalled()` + `vi.useRealTimers()`. ✅ Cumpre spec §4 caso 3 (a chamada agendada acontece de fato sob fake timers).
