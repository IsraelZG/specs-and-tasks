## 1. Objetivo
Implementar o protocolo de ondas de sincronização no pacote `@plataforma/protocol`: Onda 0 (anti-entropy O(1) + catch-up de identidades), Onda 1 (cabeçalhos críticos/tela ativa com priorização por escopo), Onda 2 (histórico podado em background), com máscara de ranges conhecidos-divergentes-mas-adiados
*(derivado de `docs/plano-de-implementacao.md` §6 T-306)*.

### Contratos exatos (derivados de T-301, T-302a, T-203, T-202)

> **Discrepância corrigida:** spec original referia "catch-up de identidades via T-203". **T-203 é o codec MessagePack, NÃO o pipeline de identidade.** O pipeline vive em **T-202 (Noise_XX handshake)**, que troca `DevicePeerId` + `identity_epoch_index`. Adicionado T-202 à lista de `dependencies:`. `executeWave0` pode usar `remoteEpochIndex` (cacheado no `RBSRSession` após handshake T-202) como sinal de catch-up necessário. Detalhes em §6.
```ts
// --- packages/protocol/src/rbsr/waves.ts ---
import { RBSRBTree, Fingerprint } from './BTree.js'; // T-301a
import { RBSRSession, RBSRState } from './exchange.js'; // T-302a
import { NetworkAdapterPort } from '../ports.js'; // T-004

export enum WaveKind {
  WAVE_0 = 0,   // anti-entropy O(1): root exchange + catch-up de identidades (T-203)
  WAVE_1 = 1,   // cabeçalhos críticos / tela ativa (priorização por escopo)
  WAVE_2 = 2,   // histórico podado em background
}

export interface WaveConfig {
  kind: WaveKind;
  priorityScope?: string[];     // entity_ids prioritários (WAVE_1)
  maxEntities?: number;         // limite de entidades por onda (WAVE_2 default=100)
  deferOnDivergence?: boolean;  // adiar ranges divergentes para onda futura
}

export interface DeferredRange {
  rangeStartId: string;
  rangeEndId: string;
  deferReason: string;
  deferredAtWave: WaveKind;
}

export interface WaveResult {
  wave: WaveKind;
  entitiesSynced: number;
  deferredRanges: DeferredRange[];
  completed: boolean;
}

export interface WaveCoordinator {
  /** Executa Onda 0: troca de root fingerprint. */
  executeWave0(session: RBSRSession): Promise<WaveResult>;

  /** Executa Onda 1: reconciliação de cabeçalhos prioritários. */
  executeWave1(session: RBSRSession, config: WaveConfig): Promise<WaveResult>;

  /** Executa Onda 2: histórico podado em background. */
  executeWave2(session: RBSRSession, config: WaveConfig): Promise<WaveResult>;

  /** Verifica se estado é idêntico (encerra em O(1)). */
  isIdempotent(session: RBSRSession): Promise<boolean>;

  /** Retorna ranges adiados pendentes. */
  getDeferredRanges(): DeferredRange[];

  /** Limpa ranges adiados quando escopo é aberto. */
  clearDeferredForScope(scopeEntityIds: string[]): void;
}
```
- Onda 0: envia root fingerprint; se igual, DONE (anti-entropy O(1)). Inclui catch-up de identidades (`identity_epoch_index` divergente → pipeline de catch-up do T-203).
- Onda 1: reconciliação de cabeçalhos críticos e da tela ativa, priorizando escopo declarado.
- Onda 2: reconciliação de histórico podado, em background, com limite de entidades.
- Máscara de defer: ranges divergentes são marcados como "adiados" e não re-disparados em resumes subsequentes até abertura do escopo.

## 4. Estratégia de Testes

**Casos de teste (8 enumerados, baseados em §4 original):**

1. **Onda 0 idempotente** — 2 B-Trees idênticas → `isIdempotent` retorna `true`, `executeWave0` retorna `{entitiesSynced: 0, completed: true}`.
2. **Onda 0 diverge → Wave 1** — root fingerprints diferentes → `executeWave0` retorna `completed: false`, `deferredRanges` contém o range. `executeWave1(session, {priorityScope: []})` é chamado e não avança (priorityScope vazio).
3. **Onda 1 reconcilia escopo prioritário** — 5 entity_ids em `priorityScope` → `executeWave1` reconcile cada um; `entitiesSynced === 5`. Outros entity_ids fora do escopo ficam para Onda 2.
4. **Onda 2 background com limite** — `executeWave2(session, {maxEntities: 3})` → reconcile apenas 3 entities, `completed: false` se há mais pendentes.
5. **Convergência completa em 3 ondas** — 2 B-Trees com 1 entity divergente; rodar `Wave 0 → Wave 1 (priorityScope=[entity]) → Wave 2 (maxEntities=10)`. Após 3 ondas, `isIdempotent` retorna `true`.
6. **Defer: range divergente não re-processado** — 2 B-Trees com 2 entities divergentes; `Wave 1` com `priorityScope=[entity1]` reconcile entity1, ADIA entity2 (deferOnDivergence=true). Re-executar `Wave 1` com `priorityScope=[entity1]` novamente → NÃO re-processa entity2 (deferred).
7. **`clearDeferredForScope` reativa range** — após caso 6, chamar `clearDeferredForScope([entity2])`. Re-executar `Wave 1` com `priorityScope=[entity2]` → agora reconcile entity2.
8. **3 ondas sequenciais com divergência plantada** — start com A e B idênticos; C tem 1 entity a mais. Após 3 ondas entre A e C, A.size() == C.size() e `isIdempotent(A↔C) === true`.

## 6. Feedback de Especificação

**Decisão 1 (correção de spec):** A spec original declarava `dependencies: [T-302a, T-203]` e
mencava "catch-up de identidades (T-203)". **T-203 é o codec MessagePack, NÃO o pipeline de
identidade.** O pipeline de identidade (com `identity_epoch_index`) é de **T-202 (Noise_XX
handshake)**. **Aplicado:** adicionado T-202 à lista de dependências. `executeWave0` pode
checar `remoteEpochIndex` (fornecido pelo handshake T-202, cacheado no `RBSRSession`) para
detectar catch-up necessário; se `localEpochIndex !== remoteEpochIndex`, marcar como
divergência de identidade e adiar para Onda 2 (que faz o catch-up via T-202 re-handshake).
**Sem fonte canônica para o contrato exato de "catch-up de identidades"** (não há T-XXX dedicado
a "pipeline de epoch index"); recomendação é: o catch-up é simplesmente "re-handshake via T-202",
não uma nova mensagem RBSR.

**Decisão 2 (acesso a entity_heads):** A Onda 2 precisa iterar `entity_heads` (tabela SQL em
core/lineage). Mas o pacote `@plataforma/protocol` é layer-1 e **não importa `@plataforma/core`**
(separação de camadas). Opções: (a) `WaveCoordinator` recebe `getEntityHeads: () => Promise<EntityHead[]>`
como injeção de função (vinda do core em application code); (b) adiar iteração de entity_heads para
um sub-task que move a chamada para `@plataforma/core` (T-306a?). **Recomendação:** (a) — a task
implementa a interface e o caller (application code) injeta a fonte de entity_heads. Mantém
`@plataforma/protocol` agnóstico ao storage.

**Decisão 3 (persistência de `deferredRanges`):** Spec original não diz se ranges adiados
sobrevivem entre execuções. Recomendação pragmática: **in-memory** (perde ao reiniciar o peer,
que re-deriva via Onda 0). Persistência seria ADR à parte.

**Decisão 4 (background):** "Onda 2: background" é ambíguo. Em Node.js single-threaded,
"background" = "não bloqueia o event loop enquanto processa" = simplesmente `async function`.
NÃO usar Worker threads (YAGNI). O caller (application code) decide se quer esperar ou
fire-and-forget via `void executeWave2(...)`.

**Ações:**
- T-306: implementar conforme §1-§5 com as 4 decisões acima aplicadas.
- Nenhum sub-task novo necessário se a decisão 2 for aceita (injeção).
