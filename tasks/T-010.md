# T-010 · Peer do sistema v0 + admin

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (pacotes core) / `playwright` (E2E/Frontend)

## 1. Objetivo
Implementar a base do servidor de cloud ("Peer do sistema") em Node.js. Neste estágio inicial (M0), ele é um servidor Express simples que responde a rotas de healthcheck e expõe a rota administrativa \`POST /admin/reset\`, essencial para o orquestrador de testes E2E.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** importe SQLite reais ainda (\`better-sqlite3\`). O reset de banco de dados real entra quando as portas de storage estiverem maduras. A rota de reset da T-010 agora deve apenas imprimir logs que "Ocorreu um reset" e devolver sucesso, para habilitar testes da pipeline que dependem do endpoint existir.
> - **NÃO** instale frameworks monolíticos (como NestJS). O escopo do System Peer é ser um hub de eventos em Express + WebSockets.
> - **NÃO** exponha o token de admin em código-fonte. Use variáveis de ambiente (ex: `process.env.ADMIN_TOKEN`).

1. **[TDD]** Crie \`admin.test.ts\` com \`Supertest\`. Afirme os códigos HTTP (200, 403, 404) para as rotas \`/health\` e \`/admin/reset\`.
2. Configure o pacote \`system-peer\` como um pacote Node.js no workspace. 
3. Instale `express`, `cors` e as tipagens pertinentes.
4. Crie o `server.ts` inicializando a porta (ex: 3000) com graceful shutdown básico.
5. Implemente a lógica de gatekeeper de proteção da rota `/admin` (checar env e headers).
6. Faça o Vitest (no modo Node) rodar e aprovar.
