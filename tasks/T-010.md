# T-010 · Peer do sistema v0 + admin

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro — unidade) + `supertest` (integração HTTP)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Implementar a base do servidor de cloud ("Peer do sistema") em Node.js. Neste estágio inicial (M0), é um servidor Express que atua como **Hub de Controle** para testes E2E:
1. Responde a healthcheck (`GET /health` → 200).
2. Expõe `POST /admin/reset` protegida por `ADMIN_TOKEN` e bloqueada em `NODE_ENV=production`.
3. Expõe namespace WebSocket `/control` (relay de comandos com `correlation_id` e timeout de 5s).
*(Derivado de `docs/plano-de-implementacao.md` §3 E0.3 T-010 e §2.4 tabela "Peer do sistema")*

```ts
// --- apps/system-peer/src/controlRelay.ts ---
import type { WebSocket } from 'ws';
import type { IncomingMessage } from 'http';

/** Par (clientId efêmero → WebSocket ativo) para roteamento de relay. */
export type ClientRegistry = Map<string, WebSocket>;

/**
 * Inicializa o WebSocket server acoplado ao servidor HTTP.
 * - Clientes conectam em '/control' com header Authorization: Bearer <sessionToken>.
 * - Gera clientId efêmero (crypto.randomUUID()) na conexão.
 * - Mantém ClientRegistry: clientId → WebSocket.
 * - Remove do registry ao desconectar.
 * Derivado de T-010 §5 item 5.
 */
export function initControlRelay(
  httpServer: import('http').Server,
  adminToken: string
): { registry: ClientRegistry };

/**
 * Relay de comando: chamado pela rota POST /control/relay/:clientId/execute.
 * - Requer header Authorization: Bearer <ADMIN_TOKEN>.
 * - Retorna 404 se clientId não encontrado no registry.
 * - Gera correlationId (crypto.randomUUID()).
 * - Faz push via WebSocket: { corr_id: string, cmd: string, payload: unknown }.
 * - Aguarda resposta do client com mesmo corr_id.
 * - Timeout 5000 ms → retorna HTTP 504.
 * - Sucesso → retorna HTTP 200 { corr_id, result: unknown }.
 * Derivado de T-010 §5 item 6 e plano §3 E0.3 T-010.
 */
export async function relayCommand(opts: {
  clientId: string;
  cmd: string;
  payload: unknown;
  registry: ClientRegistry;
  timeoutMs?: number; // default: 5000
}): Promise<{ corr_id: string; result: unknown }>;

// --- apps/system-peer/src/server.ts ---
/**
 * Entry point. Inicializa Express + WebSocket server na porta PORT (env, default 3000).
 * Monta /health, mountAdminRoutes(), mountSeedRoute(), initControlRelay().
 * Imprime ADMIN_TOKEN no console no boot.
 */
```

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [x] **Framework:** Vitest + Supertest (HTTP) + `ws` client in-memory (WebSocket)
- [x] **Ambiente do Teste:** Node puro, `pnpm --filter system-peer test`
- [x] **Fora de Escopo:** Interações de UI, SQLite real (M0: stub de reset)

Casos de teste (numerados):

**Rotas HTTP (Supertest):**
1. `GET /health` → 200 `{ ok: true }`.
2. `POST /admin/reset` sem token → 403.
3. `POST /admin/reset` com token errado → 403.
4. `POST /admin/reset` com token correto + `NODE_ENV=test` → 200 `{ ok: true }`.
5. `POST /admin/reset` com token correto + `NODE_ENV=production` → 403.
6. `POST /control/relay/:clientId/execute` sem ADMIN_TOKEN → 403.
7. `POST /control/relay/nao-existe/execute` com ADMIN_TOKEN → 404.

**WebSocket relay (ws client in-process):**
8. Cliente conecta ao endpoint `/control` → recebe `clientId` no registry.
9. `POST /control/relay/:clientId/execute` com cmd válido → cliente WS recebe `{ corr_id, cmd, payload }`, responde com `{ corr_id, result: 'ok' }` → rota retorna 200 `{ corr_id, result: 'ok' }`.
10. Timeout: cliente conectado mas não responde em 5s → rota retorna 504 (usar VirtualClock ou fake timer do Vitest).
11. `POST /admin/seed/empty` com token correto + `NODE_ENV=test` → 200 `{ ok: true, seed: 'empty' }`.
12. `POST /admin/seed/empty` sem token → 403.
13. `POST /admin/seed/empty` + `NODE_ENV=production` → 403.
