# T-010 · Peer do sistema v0 + admin

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (pacotes core) / `playwright` (E2E/Frontend)

## 1. Objetivo
Implementar a base do servidor de cloud ("Peer do sistema") em Node.js. Neste estágio inicial (M0), ele é um servidor Express simples que atua como **Hub de Controle** para testes E2E:
1. Responde a rotas de healthcheck e expõe a rota administrativa `POST /admin/reset`.
2. Expõe o namespace `/control` protegido por `ADMIN_TOKEN` gerado no boot, permitindo ao QA remoto despachar comandos para clientes web específicos via WebSocket relay, com suporte a timeout e `correlation_id`.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** importe SQLite reais ainda (`better-sqlite3`). A rota de reset deve apenas imprimir logs.
> - **NÃO** instale frameworks monolíticos (como NestJS). O escopo do System Peer é ser um hub de eventos em Express + WebSockets.
> - **NÃO** exponha o `ADMIN_TOKEN` em código-fonte. Gere-o no boot (se não for passado no env) e imprima no console. O token nunca deve ser enviado ao browser.

1. **[TDD]** Crie `admin.test.ts` com `Supertest`. Afirme os códigos HTTP (200, 403, 404) para `/health`, `/admin/reset` e `/control/*`.
2. Configure o pacote `system-peer` como um pacote Node.js no workspace. 
3. Instale `express`, `ws`, `cors` e as tipagens pertinentes.
4. Crie o `server.ts` inicializando na porta (ex: 3000) e monte um servidor WebSocket (`ws`) acoplado ao servidor HTTP.
5. Implemente a lógica de registro de clientes WS: o browser conecta passando um token de sessão escopado e gera um `client_id` efêmero. O servidor guarda o mapa `clientId -> WebSocket`.
6. Implemente a rota `POST /control/relay/:client_id/execute` (protegida pelo `ADMIN_TOKEN`). Ela deve:
   - Gerar um `correlation_id`.
   - Fazer push do comando via WebSocket pro `client_id` alvo.
   - Aguardar a resposta assíncrona do client contendo o mesmo `correlation_id`. Se demorar > 5s, retornar HTTP 504. Em sucesso, devolver HTTP 200 com a resposta.
7. Faça o Vitest (no modo Node) rodar e aprovar.
