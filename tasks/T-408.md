## 1. Objetivo
Implementar tracker WSS privado no peer do sistema: endpoint tracker para descoberta de seeders por InfoHash, com autenticação por UCAN; consumido no M8 (WebTorrent)
*(derivado de `docs/plano-de-implementacao.md` §7 T-408)*.

### Contratos exatos
```ts
// --- apps/system-peer/src/tracker.ts ---
import { PeerId } from '@plataforma/protocol';
import { SignedUcan } from '@plataforma/core';

export interface TrackerAnnounce {
  infoHash: string;              // InfoHash do conteúdo (hex)
  peerId: PeerId;
  uploaded: number;              // bytes uploaded
  downloaded: number;            // bytes downloaded
  left: number;                  // bytes restantes
  event: 'started' | 'stopped' | 'completed' | 'update';
  ucan: SignedUcan;              // UCAN autorizando acesso ao swarm
}

export interface TrackerPeer {
  peerId: PeerId;
  ip: string;
  port: number;
  left: number;
  lastSeenAt: number;
}

export interface TrackerScrape {
  infoHash: string;
  ucan: SignedUcan;
}

export interface TrackerStats {
  complete: number;              // seeders
  incomplete: number;            // leechers
  downloaded: number;            // vezes que foi completado
}

export interface TrackerService {
  /** Anuncia presença no swarm. */
  announce(req: TrackerAnnounce): Promise<{ interval: number; peers: TrackerPeer[] }>;

  /** Consulta estatísticas de um swarm. */
  scrape(req: TrackerScrape): Promise<{ files: Record<string, TrackerStats> }>;

  /** Valida UCAN antes de permitir announce/scrape. */
  validateUcan(ucan: SignedUcan, infoHash: string): Promise<boolean>;

  /** Remove peers inativos (TTL 30min sem announce). */
  purgeStale(): number;
}
```
- `announce`: registra peer como seeder/leecher; valida UCAN; retorna lista de peers no swarm.
- `scrape`: retorna estatísticas (completed, incomplete, downloaded).
- `validateUcan`: verifica assinatura + expiração + escopo inclui o infoHash.
- `purgeStale`: remove peers sem announce há > 30min.
- Endpoints: `GET /tracker/announce` e `GET /tracker/scrape` via WSS.

  chama `abortHandshake(socket, 400)` — **destrutivo**, não um "ignora e deixa passar". Quando
  `signalingRelay` repassava para `originalUpgradeListeners` (que, após o reorder, passou a
  incluir `[control, tracker]`), o `control` matava o socket com 400 **antes** do tracker ter
  chance de responder — **independente da ordem de registro**. O reorder sozinho reduzia o
  sintoma (tracker agora está na lista de "originais") mas não removia a causa (control ainda
  aborta primeiro).
- **Fix:** migrei `controlRelay.ts` para o mesmo padrão cooperativo já usado por
  `signalingRelay`/`trackerRelay` — `noServer: true` + `httpServer.on('upgrade', ...)` com
  checagem manual de path e `return` simples no mismatch (em vez do `abortHandshake` destrutivo
  do modo `{server, path}` do `ws`).
- **Nova cobertura:** `apps/system-peer/tests/trackerWiring.test.ts` — abre uma conexão WS real
  via `createApp()` (não mocka `TrackerServiceImpl`) e confirma que `/tracker` de fato responde.
  Essa é a lacuna que permitiu B1 e depois B2 passarem despercebidos: `tracker.test.ts` sempre
  testou `TrackerServiceImpl` isoladamente (Gate sempre verde), nunca o wiring real do endpoint.

**Achado novo, fora do escopo deste rework (disclosure para o próximo parecer):** ao tornar
`controlRelay` cooperativo, removi o único comportamento que rejeitava paths de upgrade
desconhecidos (o `abortHandshake` do modo antigo, que batia em qualquer path não-`/control`,
inclusive lixo, e funcionava como catch-all acidental). Agora um cliente que faz upgrade para um
path inexistente (ex.: `/nao-existe`) não recebe resposta alguma — o socket TCP fica aberto
indefinidamente (nenhum dos 3 handlers cooperativos o reivindica nem o destrói). Cheguei a
escrever um teste para isso e o socket realmente trava (`server.close()` no `afterAll` não
resolve em 10s). Removi esse teste do deliverable (não fazia parte do achado B2) em vez de
tentar consertar aqui — a correção correta exige um dispatcher central único de `upgrade` que
saiba se algum handler já reivindicou o socket antes de decidir abortar, não um patch pontual.
Recomendo task de follow-up para roteamento central de upgrade em `server.ts` (`/control`,
`/signaling`, `/tracker` e catch-all 400 para o resto).
