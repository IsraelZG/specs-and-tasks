# T-601 · Detecção Estrutural de Fork + Merge

## 1. Objetivo
Detectar forks estruturais no DAG (duas ou mais arestas `MUTATES` ativas saindo do mesmo `source_id` cujos targets não têm relação de ancestralidade entre si) e criar um nó de merge determinístico que resolve o fork apontando para ambas as pontas.
*(derivado de `docs/plano-de-implementacao.md` §9 M6 — T-601: "Duas MUTATES ativas no mesmo source_id sem ancestralidade → fork; mergeador eleito deterministicamente (preferência PROFILE:SYSTEM); nó de merge com duas MUTATES e HLC superior; convergência pela projeção de heads; head provisório = ramo de maior HLC")*

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/core/src/merge.ts (CREATE) ---
import type { StoragePort } from '@plataforma/protocol';
import { SignedNode, NodeHash, hashNode, ZERO_HASH } from './lineage.js';
import { ULID } from './ulid.js';
import { HLCTimestamp } from './hlc.js';

/** Resultado da detecção de fork estrutural. */
export interface ForkInfo {
  /** Nó de onde o fork se origina (source_id comum das arestas MUTATES). */
  readonly forkPoint: SignedNode;
  /** IDs dos nós que bifurcaram (targets das arestas MUTATES). */
  readonly branchTips: ULID[];
  /** Nós completos das pontas do fork. */
  readonly branchNodes: SignedNode[];
}

/**
 * Detecta forks estruturais a partir de um nó.
 * Percorre arestas MUTATES: duas ou mais arestas com mesmo source_id
 * cujos targets não têm relação de ancestralidade → fork.
 * Retorna null se não houver fork.
 *
 * @param sourceId — nó a partir do qual verificar forks (tipicamente um head ou nó recente).
 *   Pode ser qualquer nó no DAG; a função segue transitivamente para encontrar forks.
 * @param maxDepth — profundidade máxima de busca (default 100). Evita scans infinitos.
 *
 * Derivado de: plano §M6 T-601 "Duas MUTATES ativas no mesmo source_id sem ancestralidade → fork"
 */
export async function detectStructuralFork(
  storage: StoragePort,
  sourceId: ULID,
  maxDepth?: number,
): Promise<ForkInfo | null>;

/**
 * Resolve deterministicamente um fork criando um nó de merge.
 *
 * Regras de resolução (derivadas de plano §M6 T-601):
 * 1. O nó de merge tem nodeType = 'MERGE' e payload vazio.
 * 2. parentHash do merge = hashNode(pai comum = forkPoint).
 * 3. HLC do merge > max(HLC de ambas as branchTips).
 * 4. Cria DUAS arestas MUTATES (forkPoint → merge) — ambas referenciam
 *    o mesmo nó de merge como target. O merge "reunifica" os ramos;
 *    a partir daqui, a linhagem segue linear.
 * 5. entity_id do merge = entity_id do forkPoint (herança).
 * 6. Mergeador determinístico: se houver múltiplos PROFILE:SYSTEM elegíveis,
 *    o de menor PeerId (ordem lexicográfica) é o eleito. Se nenhum, o próprio
 *    peer local assina o merge.
 *
 * @returns o SignedNode de merge criado e já persistido.
 *
 * Derivado de: plano §M6 T-601 "mergeador eleito deterministicamente (preferência
 *   PROFILE:SYSTEM); nó de merge com duas MUTATES e HLC superior"
 */
export async function resolveFork(
  storage: StoragePort,
  fork: ForkInfo,
  signerPublicKey: Uint8Array,
  sign: (message: Uint8Array) => Promise<Uint8Array>,
): Promise<SignedNode>;

/**
 * Projeta o head provisório durante um fork ativo.
 * Enquanto o fork não é resolvido, o head "visível" é o ramo de maior HLC.
 * Usado por getHead durante a coexistência de branches.
 *
 * Derivado de: plano §M6 T-601 "head provisório = ramo de maior HLC"
 */
export async function projectProvisionalHead(
  storage: StoragePort,
  fork: ForkInfo,
): Promise<SignedNode>;
```

> **Nota sobre `resolveFork` e `insertNode`:** `resolveFork` deve chamar `insertNode` para
> criar o nó de merge (reusando a validação de HLC, entity_id, e manutenção de entity_heads
> já existentes). As duas arestas MUTATES adicionais são criadas **dentro de
> `resolveFork`**, em transação separada (para não acoplar com `insertNode`).

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [x] **Framework:** Vitest (Node puro).
- [x] **Ambiente do Teste:** `pnpm --filter @plataforma/core test`. SQLite `:memory:` via `better-sqlite3`.
- [x] **Fora de Escopo:** SimNetwork, RBSR real, rede, UCAN.

Casos de teste (numerados):
1. **Sem fork — `detectStructuralFork` retorna null:** 1 raiz + 1 filho linear → sem fork.
2. **Fork detectado — 2 filhos do mesmo pai sem ancestralidade:** raiz → filho A, raiz → filho B (A e B não têm arestas MUTATES entre si) → `detectStructuralFork` retorna fork com 2 branchTips.
3. **Fork com 3 ramos:** raiz → A, raiz → B, raiz → C (todos independentes) → `detectStructuralFork` retorna 3 branchTips.
4. **Sem fork quando há ancestralidade:** raiz → A → B, raiz → C (B descende de A, e C é irmão de A — isso NÃO é fork porque A e C não têm ancestralidade, mas B descende de A... wait, isso é um fork sim). Reformulando: raiz → A, A → B (linear). Sem outras arestas saindo de raiz → sem fork.
5. **Fork com 2 níveis de profundidade:** raiz → A, A → B. raiz → C (fork na raiz). A → D (fork em A). `detectStructuralFork(raiz, 2)` deve encontrar o fork profundo e retornar as branchTips apropriadas (testando expressamente `maxDepth > 1` como decidido na Seção 6).
6. **resolveFork — cria nó de merge:** cenário 2 (fork com 2 ramos) → `resolveFork` cria nó MERGE com 2 arestas MUTATES saindo do forkPoint para o merge. `getLineage` a partir do merge retorna [merge, forkPoint, raiz].
7. **resolveFork — HLC do merge > max(HLC dos ramos):** fork com branch A (HLC=1000) e B (HLC=2000) → merge tem HLC > 2000.
8. **resolveFork — parentHash do merge = hashNode(forkPoint):** validação de parentHash no merge.
9. **resolveFork — entity_heads atualizado:** após merge, `entity_heads` aponta para o nó de merge.
10. **projectProvisionalHead — retorna ramo de maior HLC:** fork com A (HLC=1000) e B (HLC=2000) → head provisório = B.

## 6. Feedback de Especificação (Spec Feedback Loop)

> **✅ DECISÃO #1 DERIVADA (2026-06-25 — endurecedor): Pacote e arquivo.**
> `packages/core/src/merge.ts` (CREATE). O `dag/merge.ts` da spec original e `sqlite-driver.ts`
> não existem no disco. Derivado de: todas as funções de linhagem existentes estão em
> `packages/core/src/lineage.ts` (T-108). `merge.ts` fica no mesmo nível, sem subdiretório `dag/`.
>
> **✅ DECISÃO #2 DERIVADA: Resolução de escopo.**
> A dependência **T-302 (RBSR) foi removida** porque `detectStructuralFork` e `resolveFork`
> operam puramente sobre o storage local (SQLite), sem rede. A integração com RBSR é escopo
> da própria T-302 (que consumirá `detectStructuralFork` como API). A dependência real é
> **apenas T-108** (que provê `insertNode`, `hashNode`, `ZERO_HASH`, `bytesEqual`, e a tabela
> `entity_heads` mantida em código — ADR 0002).
>
> **✅ DECISÃO #3 DERIVADA: Fork detection = 2+ MUTATES sem ancestralidade.**
> Derivado literal de plano §M6: "Duas MUTATES ativas no mesmo source_id sem ancestralidade".
> `detectStructuralFork` percorre `MUTATES` com `source_id = sourceId`, coleta os targets, e
> verifica se há ancestralidade entre eles (via `getLineage`). Se 2+ targets não têm relação
> de ancestralidade → fork.
>
> **✅ DECISÃO #4 DERIVADA: Nó de merge tem nodeType = 'MERGE'.**
> Derivado de plano §M6: "nó de merge com duas MUTATES e HLC superior". O tipo exato do nó
> (`nodeType`) não está especificado em fonte normativa. Decisão do endurecedor: `'MERGE'`
> (consistente com a convenção de tipos em maiúsculas do sistema — `MUTATES`, `ASSET`, etc.).

> **✅ DECISÕES RESOLVIDAS (2026-06-25 — arquiteto):**
> **(1) Profundidade de detecção de fork:** A função `detectStructuralFork` deve ser capaz de trabalhar em qualquer nível de `depth` (busca recursiva suportada até `maxDepth`), e isto DEVE ser garantido com casos de teste expressos validando `depth > 1` (ex: atualizar teste 5). Na prática do sistema atual o caller usará `depth = 1`, mas o contrato da função será robusto para *n* níveis.
> **(2) Eleição do mergeador:** Aprovada a sugestão do endurecedor. A API `resolveFork` recebe `signerPublicKey` e a função `sign` como parâmetros. Ela não decide a política; o caller decide quem assina e a T-601 apenas fornece o mecanismo de persistência.
