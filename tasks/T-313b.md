# T-313b · Archive Cargo: assignCustodian

> Subtask de T-313 (decomposta 2026-06-29, arquiteto opção B). Atribui custodiante a um
> archive via consistent hashing sobre `blindScopeId`. O custodiante recebe apenas o
> `archiveId` + `ciphertext` — jamais vê `pub_key`/autoria. Reusa T-305a (UCAN-scoped
> sync) para garantir que arestas `ARCHIVES` só saem para peers com capability.

## 1. Objetivo

Implementar a atribuição cega de archive a custodiante via consistent hashing sobre
`blindScopeId`: dado o conjunto de custodiantes elegíveis (peers com capability para
o scope cego), o archive vai deterministicamente para o mesmo custodiante sempre
(módulo rebalanço). O custodiante recupera o archive por `archiveId` — não precisa
de `pub_key` nem de identidade do originador.

*(derivado de `tasks/T-313.md` §1 + §4 casos 5-6; `docs/plano-de-implementacao.md`
§2.5 vetor 6 estendido — peer sem UCAN não enxerga arestas ARCHIVES; `tasks/T-305a.md`
— UCAN-scoped sync; `tasks/T-301.md` — Fingerprint/Hash para o hash ring)*

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/core/src/archive/assignCustodian.ts ---
import type { EntityId } from '@plataforma/protocol';                  // T-102
import type { BlindScopeId, ArchiveId } from './archiveCargo.js';      // T-313a

/**
 * Hash ring mínimo para consistent hashing.
 * Cada peer é mapeado para `replicas` pontos no ring (default 16) usando
 * `blake2s256(EntityId ‖ counter)` como hash de 32 bits.
 * Lookup: `blake2s256(blindScopeId)[0..4]` encontra o próximo ponto no ring
 * (caminho clockwise); o peer dono desse ponto é o custodiante.
 */
export interface HashRing {
  /** Pontos ordenados (counter ascendente, hash ascendente). */
  readonly points: ReadonlyArray<{ counter: number; peerId: EntityId; hash: number }>;
}

/** Constrói hash ring a partir de peers elegíveis. `replicas` default 16. */
export function buildHashRing(custodians: EntityId[], replicas?: number): HashRing;

/**
 * Atribui custodiante: deterministicamente o mesmo `blindScopeId` → mesmo `EntityId`.
 * Lança `Error('NO_CUSTODIANS')` se a lista de custodiantes for vazia.
 */
export function assignCustodian(
  blindScopeId: BlindScopeId,
  custodians: EntityId[],
): EntityId;

/**
 * Versão que aceita um ring pré-computado (evita rebuild em hot path).
 * Comportamento idêntico a `assignCustodian` mas O(log N) em vez de O(N × replicas).
 */
export function assignCustodianWithRing(
  blindScopeId: BlindScopeId,
  ring: HashRing,
): EntityId;

/**
 * Validação UCAN-scoped: custodiante tem capability para servir o `blindScopeId`?
 * Reusa `scopeRBSRTree` de T-305a: se o `edge_filter` do UCAN permite o scope,
 * retorna `true`. Sem capability → `false` (peer não enxerga arestas ARCHIVES).
 */
export function canServeArchive(
  custodianId: EntityId,
  blindScopeId: BlindScopeId,
  ucanCapabilities: ReadonlyArray<{ resource: string; caveat?: string }>,
): boolean;
```

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** expor `pub_key`/autoria em nenhuma interface. `assignCustodian` recebe só `BlindScopeId` (hash) + `EntityId` (peer id público), nunca o entity_id real do originador.
> - **NÃO** usar `Math.random` — `blake2s256` é determinístico; para seeds de teste, derive de `ULID` (T-102).
> - **NÃO** implementar rebalanceamento de ring (fora de escopo desta task; deferir para T-313+ se necessário).

### Pegadinhas conhecidas
- **Hash 32 bits para o ring:** truncar `blake2s256` aos primeiros 4 bytes (big-endian) e converter para `uint32`. Documentar o trade-off (colisões raras em rings pequenos, mas é o padrão da literatura — Karger et al. 1997).
- **Wrap-around:** o ring é circular. Implementar busca binária no array ordenado; se o hash do scope for maior que o último ponto, wrap para o primeiro.
- **`replicas`:** mais replicas = distribuição mais uniforme, mas O(replicas × N) para build. Default 16 (paper) ou 64 (produção). Começa com 16; documentar.
- **Tie-break:** dois peers com mesmo hash no mesmo counter — `counter` é a tie-break (estável).
