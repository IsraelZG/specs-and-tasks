## 1. Objetivo
Resolver uma decisão arquitetural sobre o limite core↔storage, entregando um **ADR** (não código nesta task). O princípio da plataforma é *"ninguém escreve SQL diretamente; o resto do sistema é abstraído pelo TinyBase"*. Hoje isso vale para o **resto do sistema** (a UI só fala com TinyBase), mas **não para o core**: `StoragePort` é **SQL-shaped**.

> **ACHADO (auditoria 2026-07-02) — acoplamento de storage.** `packages/protocol/src/ports.ts:51` define `StoragePort.exec(sql: string, params)` — a porta **é** SQL. Consequência: os serviços do core escrevem SQL cru por toda parte — **86 ocorrências em 8 arquivos** (`merge.ts`, `lineage.ts`, `invite.ts`, `schema.ts`, `blindArchives.ts`, `concurrentGuard.ts`, `deviceState/schema.ts`, `sqliteStorage.ts`). A `visao-arquitetural.md §3` documenta isso como intencional ("o core acessa o SQLite diretamente via StoragePort"). Portanto: **trocar de engine (Postgres, ou NoSQL/KV/documento) hoje é mudança de paradigma**, porque o contrato de storage é a linguagem SQL, não um modelo de query.

## 6. Feedback de Especificação

> **✅ DECIDIDO (arquiteto — Israel, 2026-07-02): Opção C (híbrido), refinada.**
> Query-model **tipado e agnóstico de engine** para o **grafo append-only** (nodes/edges); as
> **projeções locais não fazem parte do contrato replicado** — são materializações da **camada de
> índice (TinyBase-side)**, definidas *sobre* o query-model do grafo, **não** SQL embutido no core.
> Consequência canônica (fonte para o ADR e para o endurecimento das subtasks de implementação):
> - `StoragePort` **deixa de expor `exec(sql)`**; expõe operações do domínio do grafo
>   (`putNode`/`getNode`/`rangeScan(lo,hi)`/`getEdges` + aplicação de nó assinado). Adapters
>   concretos: SQLite (traduz p/ SQL), e um KV/documento como segundo backend de prova.
> - **Projeções** (views, `entity_heads`, `chat_conversations`, índices) são reconstruídas **por
>   engine** na camada de leitura — num deployment SQLite vivem no mesmo SQLite (custo zero); num
>   deployment NoSQL são re-materializadas via TinyBase/índice nativo. **Nenhum deployment é forçado
>   a arrastar um SQLite só para projeções.** Não existe "dois bancos".
> - RBSR opera sobre `rangeScan` ordenado — engine-neutro por construção (KV ordenado atende).
>
> **Entregável desta task (opus-spike):** ADR `docs/adr/adr-00X-storage-engine-agnosticism.md`
> fixando o contrato do query-model do grafo + a fronteira projeção↔índice. A implementação
> (reescrita dos sítios de grafo + adapter KV de prova) vira épico/subtasks derivadas do ADR.

---

**Contexto da decisão (registro do que foi ponderado):**
- **Pergunta original:** manter `StoragePort` SQL-shaped, ou introduzir query-model agnóstico?
- **É paradigma?** *Parcialmente.* A fronteira UI→TinyBase **já é** agnóstica (TinyBase é um store reativo tabular/KV, não SQL). O acoplamento está **só** no core→StoragePort. RBSR precisa de *range scans ordenados* — um KV ordenado (LMDB/RocksDB/IndexedDB) atende **naturalmente**, às vezes melhor que SQL. O que é genuinamente SQL-específico são as **projeções relacionais** (views, CTE de traversal para filtro UCAN — ver `sync-dirigido-por-ucan`).
- **Opção A — manter SQL-as-port (status quo).** Menor custo; assume SQLite/Postgres para sempre. NoSQL fica fora. Viola parcialmente o princípio "não acoplar a SQLite" (acopla ao *dialeto*, não ao driver).
- **Opção B — query-model agnóstico (repository/porta tipada).** `StoragePort` deixa de expor `exec(sql)`; expõe operações do domínio do grafo (`putNode`, `getNode`, `rangeScan(lo,hi)`, `getEdges`, `applyProjection`). Adapters: um SQLite (traduz p/ SQL), um KV/documento. Custo alto: reescrever os 8 sítios + mover as projeções relacionais para o adapter. Ganho: NoSQL-ready de verdade; o core nunca mais vê SQL.
- **Opção C — híbrido.** Query-model tipado para o **grafo append-only** (nodes/edges — o que precisa portabilidade e é KV-friendly) + manter SQL só para as **projeções locais** (que são um detalhe de cada adapter, nunca replicadas). Provavelmente o melhor custo/benefício.

**Recomendação do auditor:** Opção C. Preserva o princípio no dado que importa (o grafo replicado fica engine-agnóstico), sem pagar a reescrita das projeções, que são locais e descartáveis por definição.

**Entregável desta task:** ADR registrando a escolha + o contrato da porta resultante (se B/C) ou a justificativa explícita de acoplamento (se A). A implementação vira épico/subtasks derivadas.
