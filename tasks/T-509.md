# T-509 · Central Custody (corporativo)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo

## 1. Objetivo
Implementar Central Custody no peer do sistema: chave mestra do funcionário cifrada sob chave da empresa; provisionamento; reset pelo admin; login usuário/senha (PBKDF2 desbloqueio) + 2FA TOTP + recuperação por e-mail (T-508)
*(derivado de `docs/plano-de-implementacao.md` §8 T-509)*.

### Contratos exatos
```ts
// --- apps/system-peer/src/custody/centralCustody.ts ---
import { Ed25519PrivateKey, Ed25519PublicKey } from '@plataforma/crypto'; // T-101a

export interface CustodyAccount {
  accountId: string;             // entity_id do funcionário
  encryptedMasterKey: Uint8Array; // chave mestra cifrada com chave da empresa
  passwordHash: Uint8Array;       // PBKDF2 do password do usuário
  totpSecret: string;            // segredo TOTP (base32)
  recoveryEmail: string;
  provisionedAt: number;
  lastLoginAt: number;
}

export interface ProvisioningRequest {
  employeeId: string;
  email: string;
  adminToken: string;            // token de admin do sistema
}

export interface LoginRequest {
  email: string;
  password: string;
  totpCode: string;              // código TOTP de 6 dígitos
}

export interface RecoveryRequest {
  email: string;
  recoveryToken: string;         // token enviado por email (single-use, TTL curto)
  newPassword: string;
}

export interface CentralCustodyService {
  /** Provisiona nova conta de funcionário (admin). */
  provision(req: ProvisioningRequest): Promise<CustodyAccount>;

  /** Login: PBKDF2 + TOTP. */
  login(req: LoginRequest): Promise<{ success: boolean; account?: CustodyAccount; reason?: string }>;

  /** Recuperação: token por email → reset de senha. */
  recover(req: RecoveryRequest): Promise<{ success: boolean; reason?: string }>;

  /** Admin: reset de conta (força recovery). */
  adminReset(accountId: string, adminToken: string): Promise<void>;

  /** Admin: ver estado de uma conta. */
  getAccount(accountId: string, adminToken: string): Promise<CustodyAccount>;
}
```
- Chave mestra do funcionário: gerada localmente, cifrada com chave AES da empresa no servidor.
- Provisionamento: admin fornece employeeId + email → conta criada, email de boas-vindas enviado via SMTP (T-508).
- Login: PBKDF2(password) + TOTP code → desbloqueio da chave mestra.
- Recuperação: token enviado por email → reset de senha + reconfiguração TOTP.
- Limite honesto documentado: admin pode resetar conta (custódia corporativa).

## 4. Estratégia de Testes
- [x] **Framework:** Vitest + SQLite `:memory:` + SMTP stub.
- [x] **Ambiente:** `pnpm --filter @plataforma/system-peer test`.

Casos de teste:
1. `provision` cria conta com `encryptedMasterKey` e `passwordHash`.
2. `login` com senha correta + TOTP válido → `success: true`.
3. `login` com senha errada → `success: false`, `reason: 'invalid_password'`.
4. `login` com TOTP errado → `success: false`, `reason: 'invalid_totp'`.
5. E2E: provisionar → logar → perder senha → recuperar por e-mail → 2FA → acesso restaurado.
6. `adminReset` força recovery com novo token.
7. `adminReset` com token inválido → rejeitado.
8. Limite honesto: admin pode ler `CustodyAccount` (documentado no painel admin).
