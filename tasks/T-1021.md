# T-1021 · Runner: gate de verificação determinístico (build+test no worktree)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Test Runner:** `vitest`
- **Capacidade-alvo:** sonnet *(spawn de processo cross-platform tem nuances; assinaturas fixas)*

## 1. Objetivo
Módulo que roda os comandos de "Verificação automática" de uma task **no diretório dela**
(o worktree) e devolve `{ ok, output, steps }`. É o **ground truth** do runner: quem garante
"verde" é este gate, não o auto-relato do agente. Inclui um parser que extrai os comandos do
corpo da spec (bloco ` ```bash ` da seção "Verificação automática"), com fallback.

## 3. Escopo de Arquivos (Inputs e Outputs) — assinaturas FIXAS
- **[CREATE]** `apps/nexus-backend/src/runner/verify-gate.ts`
  ```ts
  export interface VerifyStep { command: string; ok: boolean; }
  export interface VerifyResult {
    ok: boolean;            // true só se TODOS os steps passaram
    steps: VerifyStep[];
    output: string;         // stdout+stderr concatenado, últimas `tailLines` por comando
  }
  export interface VerifyOptions {
    cwd: string;                   // worktree da task
    commands: string[];            // ex.: ['pnpm --filter nexus-backend build', '... test']
    timeoutMsPerCommand?: number;  // default 600000
    tailLines?: number;            // default 40
  }

  /**
   * Roda cada comando EM SEQUÊNCIA; PARA no primeiro que falhar (curto-circuito).
   * Cada comando é uma linha de shell completa e CONFIÁVEL (vem da spec, sem interpolação
   * de input externo) → use `child_process.exec(command, { cwd, timeout })` (shell:true),
   * que resolve `pnpm`/`pnpm.cmd` cross-platform. Exit !== 0 → step.ok=false, ok geral=false.
   */
  export function runVerificationGate(opts: VerifyOptions): Promise<VerifyResult>

  /**
   * Extrai os comandos do corpo da task: pega o PRIMEIRO fence ```bash que aparece DEPOIS
   * do título "Verificação automática" e devolve as linhas não-vazias que não começam com '#'.
   * Se não encontrar e `fallbackPackage` for dado → ['pnpm --filter <pkg> build',
   * 'pnpm --filter <pkg> test']. Se nada → [].
   */
  export function parseVerifyCommands(taskBody: string, fallbackPackage?: string): string[]
  ```

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Vitest (Node puro). NÃO depender de `pnpm` real nos testes.
- [ ] **Cobertura (`runVerificationGate`):** use comandos triviais e portáveis:
  - sucesso: `['node -e "process.exit(0)"']` → `ok:true`, 1 step ok.
  - falha + curto-circuito: `['node -e "process.exit(1)"', 'node -e "process.exit(0)"']`
    → `ok:false`, o 2º comando NÃO roda (steps.length===1).
  - `cwd` é respeitado (rode `node -e "process.stdout.write(process.cwd())"` num tmpdir e
    confira que a saída contém o tmpdir).
  - timeout: comando que dorme além de `timeoutMsPerCommand` curto → step falha (`ok:false`).
- [ ] **Cobertura (`parseVerifyCommands`):** extrai as 2 linhas de um corpo com o bloco;
  ignora linhas `#`; sem bloco + fallback → as 2 linhas pnpm; sem bloco e sem fallback → `[]`.
- [ ] **Ambiente:** Node puro. Pular se `node` não resolver (não deve ocorrer).
- [ ] **Fora de Escopo:** rodar `pnpm` de verdade; comparar veredito com o reviewer (é do T-1023).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - NÃO transicionar tasks aqui — este módulo só executa comandos e reporta.
> - NÃO truncar a saída antes de detectar o exit code — capture exit primeiro, depois corte `tail`.

### Pegadinhas conhecidas
- **Aqui `exec` (shell) é intencional**, ao contrário da regra da T-1012: o comando é uma
  linha inteira e confiável vinda da spec (não há interpolação de input do usuário), e
  precisamos do shell para resolver `pnpm`/`pnpm.cmd` cross-platform. Documente isso no código.
- **Curto-circuito:** pare no primeiro fail; não rode os comandos seguintes.
- **`exec` não rejeita "limpo" em exit!=0** — ele chama o callback com `error` cujo `.code` é o
  exit. Trate `error` como step-falho, não como crash; capture `stdout`+`stderr` mesmo no erro.
- **Timeout do `exec`** mata o processo e chama o callback com erro — conte como step-falho.
