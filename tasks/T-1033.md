# T-1033 · Quebrar ciclo core↔protocol: mover applyNodes para core

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (pacotes core/protocol) e `playwright` (E2E/Frontend)
- **Capacidade-alvo:** haiku | sonnet | opus-spike *(ver regra "Dimensionamento de Tarefas" no CLAUDE.md: spec sem decisões em aberto, contratos explícitos, sem API externa não-fixada, verificação por comando)*

## 1. Objetivo
Desfazer a **dependência circular `@plataforma/core` ↔ `@plataforma/protocol`**. A direção correta é uma seta só: `protocol ← core ← transport`. Hoje `core` depende de `protocol` (ports — correto) e `protocol` reimporta `core`, fechando o ciclo.

> **ACHADO (auditoria 2026-07-01) — GRAVE / arquitetura.** `packages/protocol/src/rbsr/exchange.ts:4` importa `verifyNode`/`insertNode`/`SignedNode` de `@plataforma/core`. As próprias specs se contradizem: **T-302b** manda importar de `@plataforma/core` dentro do protocol; **T-305b** explicitamente evita ("importar de @plataforma/core causaria dep circular") e manda **duplicar** os tipos — por isso `packages/protocol/src/auth/ucanScope.ts:5-19` carrega um subset copiado de `SignedUcan`/`UcanCapability` que já pode divergir de `packages/core/src/auth/ucan.ts`. Sintoma já no ledger: `[m2][T-022][monorepo]` — ciclo `protocol ↔ core ↔ testkit#build` quebra `pnpm build` da raiz via turbo.

**Meta:** `applyNodes` (que valida assinatura e insere no storage) **pertence ao `core`**, não ao `protocol` — mover para lá elimina o import invertido em `exchange.ts` e a duplicação de tipos UCAN. Após a mudança, `pnpm build` da raiz (turbo) deve rodar sem ciclo no grafo.

### Contratos exatos (assinaturas TS fixadas, derivadas de `packages/protocol/src/rbsr/exchange.ts:1-179` + `packages/protocol/src/auth/ucanScope.ts:1-87` + `packages/core/src/lineage.ts`)

**Origem `applyNodes` atual** (`packages/protocol/src/rbsr/exchange.ts:157-179`):
```ts
import { verifyNode, insertNode } from '@plataforma/core';
import type { SignedNode } from '@plataforma/core';
import type { StoragePort } from '../ports.js';

export async function applyNodes(
  nodes: readonly SignedNode[],
  storage: StoragePort,
): Promise<{ imported: number; rejected: number }>;
```

**Origem tipos UCAN duplicados** (`packages/protocol/src/auth/ucanScope.ts:5-19`):
```ts
// subset local — fonte única deveria ser @plataforma/core
interface UcanCapability { readonly resource: string; readonly ability: string; readonly caveat?: ... }
interface UcanPayload { readonly capabilities: ...; readonly exp?: bigint }
interface SignedUcan { readonly payload: UcanPayload }
```

**Meta do hardening — fonte única em core, consumers substituem:**
```ts
// --- packages/core/src/rbsr/applyNodes.ts (NOVO) ---
import { verifyNode, insertNode } from './signature.js';
import { getLineage, getHead } from './lineage.js';
import type { SignedNode } from './signature.js';
import type { StoragePort } from '@plataforma/protocol';

/**
 * Aplica nós binários recebidos no storage local.
 *   1. Verifica assinatura Layer 1 (verifyNode). Se inválida → rejected e segue.
 *   2. Tenta insertNode (Layer 2). Se lança → rejected e segue.
 *   3. Em sucesso, conta como imported.
 * @param nodes  Nós a importar.
 * @param storage  StoragePort do grafo local.
 */
export async function applyNodes(
  nodes: readonly SignedNode[],
  storage: StoragePort,
): Promise<{ imported: number; rejected: number }>;

export { applyNodes }; // re-export via packages/core/src/index.ts (T-501, T-107, T-108a já reexportam)

// --- packages/protocol/src/auth/ucanScope.ts (REESCRITA) ---
// ❌ REMOVER: tipos UCAN locais (linhas 5-19)
// ✅ IMPORTAR: tipo real de @plataforma/core
import type { SignedUcan, UcanCapability } from '@plataforma/core';
// scopeRBSRTree, canAccess, extractEntityIds permanecem em protocol (consumidos por sync UCAN)

// --- packages/protocol/src/rbsr/exchange.ts (ATUALIZAÇÃO) ---
// ❌ REMOVER: import de verifyNode/insertNode/SignedNode de @plataforma/core (linhas 4-5)
// ✅ IMPORTAR: applyNodes de @plataforma/core (invertido mas permitido — core depende de protocol,
//             e protocol agora consome de core SÓ via applyNodes — sem ciclo porque applyNodes
//             não importa nada de protocol além do tipo StoragePort que já vive em protocol)
// ❌ REMOVER: a função applyNodes local (linhas 157-179)
```

> **Análise de ciclo:** o tipo `StoragePort` vive em `protocol` (T-004), é PURO (sem dep). `core` importa `StoragePort` de `protocol` (correto: core depende de protocol para Ports). `applyNodes` (em core) usa apenas `StoragePort` de protocol + `verifyNode`/`insertNode` de core — **não fecha ciclo**. `exchange.ts` (em protocol) chama `applyNodes` de core — também **não fecha ciclo** (core exporta applyNodes; protocol importa). O ciclo original vinha de `exchange.ts` importar `verifyNode`/`insertNode`/`SignedNode` de core **E** `core` importar `StoragePort` de protocol. Movendo `applyNodes` para core e fazendo protocol só importar **esse** símbolo de core, o ciclo fecha.
