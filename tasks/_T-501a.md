## 1. Objetivo
Implementar `encodeUcan`/`decodeUcan` no pacote `@plataforma/core`: serialização/desserialização de UCAN como JWT compact (header.payload.signature base64url). Puramente codec — sem validação criptográfica
*(extraído de T-501 §1)*.

### Contratos exatos
```ts
// --- packages/core/src/auth/ucan-codec.ts ---
import { Ed25519Signature } from '@plataforma/crypto'; // T-101a
import { ULID } from '../ulid.js'; // T-102

export interface UcanHeader { alg: 'EdDSA'; typ: 'JWT'; }

export interface UcanPayload {
  iss: ULID; aud: ULID;
  att: Array<{ with: string; can: string; nb?: Record<string, unknown> }>;
  exp: number; nbf?: number; nnc?: string; fct?: unknown[];
}

export interface SignedUcan {
  header: UcanHeader;
  payload: UcanPayload;
  signature: Ed25519Signature;
}

/** Serializa UCAN para JWT compact (base64url). */
export function encodeUcan(ucan: SignedUcan): string;

/** Desserializa JWT → SignedUcan. Valida formato, não assinatura. */
export function decodeUcan(token: string): SignedUcan;
```

Casos de teste:
1. `encodeUcan` + `decodeUcan` roundtrip.
2. JWT com header ausente → `decodeUcan` lança erro.
3. JWT com payload malformado → erro.
4. JWT com 2 pontos (sem signature) → erro.
5. `encodeUcan` produz 3 segmentos base64url separados por `.`.

## 6. Feedback de Especificação
- **ABERTO — alinhamento de tipos com T-501 (canônico):** esta subtask usa `Ed25519Signature` (tipo de `@plataforma/crypto`) e `att: Array<{with, can, nb?}>`, enquanto T-501 (done) define `signature: Uint8Array` e `capabilities: UcanCapability[]`. Se T-501a for um codec JWT intermediário que antecede a consolidação em T-501, isso é aceitável como simplificação interna — mas o contrato de saída (o `SignedUcan` que T-501b/T-501c consomem) precisa ser compatível com ambos. Decidir: (a) T-501a exporta seu próprio `SignedUcan` interno e T-501b faz adaptação; ou (b) T-501a deve usar exatamente os tipos de T-501. Escalar para arquiteto.
