# T-1048 · Composition root + inversão de dependência no superapp

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (pacotes core) / `playwright` (E2E/Frontend)

## 1. Objetivo
Adotar/formalizar **ports-and-adapters + composition root** no superapp para que a direção de camadas `@plataforma/protocol ← crypto ← core ← transport ← workers` ([visao-arquitetural.md](../docs/visao-arquitetural.md) §1) seja **invariante verificável por ferramenta**: nenhum pacote importa um adapter concreto "pra cima", nenhum ciclo entre pacotes. O core conhece **só as portas** (`StoragePort`, `GraphStorePort`, `NetworkAdapters` — [ADR-005](../docs/adr/adr-005-storage-engine-agnosticism.md)); os adapters concretos são instanciados e injetados na **raiz de composição** (entry do app), nunca importados pelo core.
Esta task é a **spike**: o entregável é o ADR canônico + o inventário de violações reais + a **decomposição** em sub-tasks de correção (uma por par de pacotes). As correções complexas em si são as filhas.

- **[CREATE]** sub-tasks filhas usando `node tools/scripts/generate-task.mjs` para quaisquer violações complexas de ciclo de dependências/upward imports que demandem refatorações profundas de código (uma task filha por par de pacotes violadores).

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Madge (`npx madge`) para verificação de acoplamento estático, além do `pnpm -r build` e `pnpm -r test` como rede de segurança para regressões.
- [ ] **Métricas/Cobertura:** Nenhuma métrica de cobertura de linhas lógica é aplicável a esta spike (que é predominantemente documental/ADR e de mapeamento de dependências). O critério de sucesso é a detecção e mapeamento de ciclos e a verificação do build e dos testes existentes.
- [ ] **Ambiente do Teste:** CLI no ambiente local do monorepo.
- [ ] **Fora de Escopo:** Refatorações de acoplamentos complexos não triviais (estes devem ser apenas mapeados e decompostos em tasks filhas).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** altere a lógica de negócio ou de criptografia dos pacotes.
> - **NÃO** realize refatorações de arquitetura complexas ou de larga escala que não caibam na spike; use a decomposição.
> - **NÃO** instale/introduza novos pacotes de runtime ou dependências de terceiros sem aprovação expressa do arquiteto via ADR.

1. **[Reconhecimento]** Execute `npx madge --circular --extensions ts packages apps` na raiz do repositório de código do superapp para gerar o inventário de dependências cíclicas.
2. **[Análise]** Identifique todos os imports de adapters ou classes concretas dentro das camadas de `@plataforma/core` ou `@plataforma/protocol` que violam a direção de camadas canônica.
3. **[Documentação]** Redija o ADR `docs/adr/0019-composition-root-superapp.md` detalhando o padrão de Composition Root para o superapp, especificando o local exato onde as dependências devem ser instanciadas e injetadas, e documentando exceções aceitáveis.
4. **[Correção Trivial]** Aplique correções imediatas e triviais para injeção de portas se identificadas durante o processo.
5. **[Decomposição]** Para cada par de pacotes envolvido em ciclos complexos ou violações de upward import não triviais, execute o script de geração de tarefas (`node tools/scripts/generate-task.mjs`) para criar sub-tasks filhas dedicadas a remover a violação.

## 6. Feedback de Especificação (Spec Feedback Loop)
- **Não redesenhar** as camadas — a direção já é canônica (§1). Só remover violações.
- Se uma "violação" for design intencional (ex.: barrel de tipos compartilhados), documentar no ADR como exceção — não forçar refactor.
- Este é o pré-requisito conceitual de T-1049 (guardrails): sem saber os ciclos reais, habilitar `import/no-cycle` deixa o baseline vermelho.
- **Classificação (pass-2):** `draft:hardened`.
- **Motivo:** Spike de arquitetura e ADR do Composition Root com escopo e produtos definidos de forma estrita.
