## 1. Objetivo
Adicionar a notação ZEN `$doc.<campo>` (ex.: `$doc.title`) como terceiro namespace de fonte para reagir a mutations do CRDT em sessões-doc (Automerge).
O runtime assina as mutations do Automerge e repassa o estado consolidado da sessão-doc para o avaliador de páginas.
Diferenciar `$doc` (estado da sessão-doc/Automerge) de `$state` (estado local transiente de render) e `$zen` (parâmetros da página/rota).

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/pages/src/evaluator.ts ---

export interface EvaluationContext {
  /** Estado local transiente da página (ex: seleção, aba ativa, rascunhos de campo). */
  $state: Record<string, unknown>;
  /** Parâmetros da página e da rota (ex: rota ativa, query params). */
  $zen: Record<string, unknown>;
  /** Estado consolidado da sessão-doc colaborativa / Automerge (ex: título, corpo do doc). */
  $doc: Record<string, unknown>;
}

/**
 * Avalia uma expressão ZEN no contexto de renderização de página.
 * Suporta três namespaces principais no escopo global:
 * - $state (estado local)
 * - $zen (parâmetros de página)
 * - $doc (estado consolidado da sessão-doc / Automerge)
 * 
 * Exemplo de expressão: "$doc.title == 'Home' && $state.tab == 'edit'"
 * 
 * Se a expressão acessar campos não definidos em $doc, retorna undefined sem disparar erro.
 *
 * @param expression A expressão contendo referências a namespaces (ex: "$doc.title").
 * @param context O contexto com os valores dos três namespaces.
 * @returns O resultado da avaliação da expressão.
 */
export function evaluateZenExpression(
  expression: { $zen: string } | string,
  context: EvaluationContext
): unknown;
```

- **Comentários de Revisão:**

**[B1] BLOCKER — Bug em `parsePrimary`: double-consume em string literals quebra expressões compostas com `&&`/`||`.**

Arquivo: `packages/pages/src/evaluator.ts:208-212`

```ts
if (token.type === 'string') {
    this.consume();  // ← OK: consome o token string
    this.consume();  // ← BUG: consome também o token seguinte (&&, ||, etc.)
    return token.value;
}
```

O segundo `this.consume()` avança o parser além do string literal, engolindo o próximo token (que pode ser `&&`, `||`, `)`, etc.). Isso faz com que expressões como `"$doc.title == 'Home' && $state.tab == 'view'"` retornem `true` em vez de `false`, porque o `&&` é consumido silenciosamente e o lado direito do AND nunca é avaliado.

**Correção:** remover a segunda chamada `this.consume()` (linha 210). O `parsePrimary` já consumiu o token no `peek()` via `this.consume()` da linha 209 — não precisa consumir novamente.

**Impacte:** todos os 7 testes da Seção 4 passam exceto o caso 7 (linha 65). O teste 7 com OR (linha 68) passa por coincidência — o bug consome o `||` mas o lado esquerdo já é `true`, então o resultado bate. Expressões reais de visibilidade de página com `&&`/`||` envolvendo string literals falhariam em produção.

**Veredito: REFATORAÇÃO NECESSÁRIA** — corrigir o double-consume e re-rodar o gate (build+test+lint).
