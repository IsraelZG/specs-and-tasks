# T-007 · Asserções de convergência (Testkit)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (pacotes core) / `playwright` (E2E/Frontend)

## 1. Objetivo
Criar uma suíte de *helpers* em \`testkit\` para facilitar testes determinísticos de convergência do grafo. O principal é o método \`expectConverged(peers, scope)\` que validará se múltiplos nós chegaram a um estado matematicamente idêntico.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** tente escrever o código de RBSR (reconciliação de árvore B) aqui. O testkit não implementa o protocolo, ele só fornece utilitários para comparar estados de memória em testes.
> - **NÃO** utilize libs de deep-compare lentas como o `lodash.isEqual` que quebrem se os objetos tiverem chaves cíclicas (embora o SQLite não tenha isso, previna importações pesadas no testkit). Use o \`assert.deepStrictEqual\` do Node.js ou \`expect().toEqual()\` do Vitest.

1. **[TDD]** Crie \`tests/assertions.test.ts\`. Escreva testes passando mock arrays (representando dumps de DB SQLite) que divergem minimamente (um registro faltando, ou `hlc` divergente). A asserção criada deve lançar erro.
2. Implemente \`expectConverged(peers, scope)\`. Ele deve aceitar um array de interfaces de persistência (ou stubs simples se a persistência real não existir) e invocar a leitura dos root fingerprints.
3. Se houver divergência nos root fingerprints, o utilitário deve automaticamente invocar a exportação (dump) da tabela de \`nodes\` e \`edges\` dos peers divergentes e fazer um `diff` no terminal para ajudar o desenvolvedor a saber *onde* divergiu.
4. Exporte essa função do \`index.ts\` do testkit para ser consumida pela suíte P2P.
