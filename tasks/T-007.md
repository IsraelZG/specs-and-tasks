## 1. Objetivo e Contratos
Criar uma suíte de *helpers* em \`testkit\` para facilitar testes determinísticos de convergência do grafo. O principal é o método \`expectConverged(peers, scope)\` que validará se múltiplos peers chegaram a um estado matematicamente idêntico.
*(derivado de `docs/plano-de-implementacao.md` §2.1 Anel 1 — Simulação Determinística)*.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/testkit/src/assertions.ts 
---
import { PeerId, SqlRow } from '@plataforma/protocol';

/** Tabelas do grafo que podem ser comparadas. */
export type TableName = 'nodes' | 'edges';

/** Escopo de convergência: conjunto de tabelas a comparar. */
export type ConvergenceScope = ReadonlySet<TableName>;

/**
 * Interface que um peer deve implementar para participar de testes de convergência.
 * Em v1, aceita tanto peers reais (com StoragePort) quanto stubs manuais.
 */
export interface ConvergencePeer {
  /** Identificador do peer para mensagens de diagnóstico. */
  readonly peerId: PeerId;

  /**
   * Retorna o root fingerprint para o escopo informado.
   * Fingerprint = SHA-256 de 256 bits representado como Uint8Array (32 bytes).
   */
  getRootFingerprint(scope: ConvergenceScope): Uint8Array | Promise<Uint8Array>;

  /**
   * Exporta (dump) completo de uma tabela para o escopo informado.
   * Chamado apenas quando há divergência de fingerprint, para gerar diff.
   */
  dumpTable(table: TableName, scope: ConvergenceScope): SqlRow[] | Promise<SqlRow[]>;
}

/**
 * Asserção de convergência: compara root fingerprints de múltiplos peers.
 *
 * - Se todos os fingerprints forem idênticos → resolve silenciosamente.
 * - Se houver divergência → faz dump das tabelas divergentes e lança erro
 *   com diff descritivo usando `assert.deepStrictEqual` do Node.js.
 * - Se `peers` tiver menos de 2 elementos → lança erro descritivo.
 *
 * @throws AssertionError com diff detalhado das tabelas divergentes.
 */
export async function expectConverged(
  peers: ConvergencePeer[],
  scope: ConvergenceScope,
): Promise<void>;
```

Casos de teste (numerados):
1. **Convergência feliz (2 peers):** dois peers com mesma tabela `nodes` (mesmo conteúdo) e `edges` vazia → `expectConverged` resolve sem erro.
2. **Convergência feliz (3 peers):** três peers idênticos → resolve.
3. **Nó extra (divergência):** peer A tem 2 nós, peer B tem 3 nós (1 extra) → `expectConverged` lança erro com diff mostrando o nó extra.
4. **Nó com campo divergente:** peer A e B têm mesmo número de nós, mas `hlc` de um nó difere → lança erro com diff mostrando o campo divergente.
5. **Aresta extra (divergência):** `edges` difere (peer B tem aresta extra) → lança erro com diff.
6. **Apenas `edges` diverge, escopo só inclui `nodes`:** scope = `new Set(['nodes'])`, edges divergem → resolve (edges fora do escopo).
7. **Array vazio:** `peers = []` → lança erro descritivo (`expectConverged requires at least 2 peers`).
8. **Array com 1 peer:** `peers = [p]` → lança erro descritivo.
9. **DumpTable async:** peer cujo `dumpTable` retorna `Promise` → funciona corretamente (aguarda resolução antes de comparar).
10. **getRootFingerprint async:** peer cujo `getRootFingerprint` retorna `Promise` → funciona corretamente.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** tente escrever o código de RBSR (reconciliação de árvore B) aqui. O testkit não implementa o protocolo, ele só fornece utilitários para comparar estados de memória em testes.
> - **NÃO** utilize libs de deep-compare lentas como o `lodash.isEqual` que quebrem se os objetos tiverem chaves cíclicas (embora o SQLite não tenha isso, previna importações pesadas no testkit). Use o \`assert.deepStrictEqual\` do Node.js ou \`expect().toEqual()\` do Vitest.

1. **[TDD]** Crie \`tests/assertions.test.ts\`. Escreva testes passando mock arrays (representando dumps de DB SQLite) que divergem minimamente (um registro faltando, ou `hlc` divergente). A asserção criada deve lançar erro.
2. Implemente \`expectConverged(peers, scope)\`. Ele deve aceitar um array de interfaces de persistência (ou stubs simples se a persistência real não existir) e invocar a leitura dos root fingerprints.
3. Se houver divergência nos root fingerprints, o utilitário deve automaticamente invocar a exportação (dump) da tabela de \`nodes\` e \`edges\` dos peers divergentes e fazer um `diff` no terminal para ajudar o desenvolvedor a saber *onde* divergiu.
4. Exporte essa função do \`index.ts\` do testkit para ser consumida pela suíte P2P.
