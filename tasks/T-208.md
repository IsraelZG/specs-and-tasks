## 0. Ambiente de Execução Obrigatório

## 1. Objetivo
Implementar a máquina de estados do First Peer Protocol *(derivado de
`docs/conceitos/first-peer-protocol.md`; `docs/conceitos/genesis-state.md`;
`docs/plano-de-implementacao.md` §5 T-208)* que rege o cold start da topologia
P2P: `JOINING → WAITING_FOR_SWARM(8s) → CONNECTED | GENESIS | OFFLINE_RETRY`
com backoff 10→60s.

O FPP é a lógica de decisão — **não** contém descoberta de rede (mDNS/DHT/WS)
nem heartbeat (T-206). Ele consome eventos do [[swarm-registry|SwarmRegistry]]
(T-205) para detectar peers e delega a escrita de gênese ao seam `GenesisWriter`
(decisão 2) — não escreve SQL nem monta nós assinados.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/transport/src/fpp.ts ---

import { PeerId } from '@plataforma/protocol';
import type { SwarmRegistry } from './SwarmRegistry';     // T-205
import type { GenesisWriter } from '@plataforma/protocol'; // novo port (ver decisão 2)
import type { ClockPort } from '@plataforma/protocol';     // T-004 §1

/** Estados da máquina FPP. */
export enum FppState {
  /** Estado inicial — ainda não iniciado. */
  JOINING          = 'JOINING',
  /** Busca ativa por peers (timer de 8 s rodando). */
  WAITING_FOR_SWARM = 'WAITING_FOR_SWARM',
  /** Peer encontrado — operação normal. */
  CONNECTED        = 'CONNECTED',
  /** Fundação da rede (bootstrap token ou ação explícita). */
  GENESIS          = 'GENESIS',
  /** Busca passiva com recuo exponencial (sem token). */
  OFFLINE_RETRY    = 'OFFLINE_RETRY',
}
```

> **Fonte dos estados:** `docs/conceitos/first-peer-protocol.md` §Estados do Protocolo.
> Nomes e semântica exatamente como no state diagram do verbete.

```ts
/** Configuração de bootstrap do FPP. */
export interface FppConfig {
  /** true se o peer detém o bootstrap token (chave de fundação). */
  readonly hasBootstrapToken: boolean;
  /** true se o usuário optou explicitamente por "Criar Nova Rede". */
  readonly explicitCreateNetwork: boolean;
}

/**
 * Seam de escrita da gênese (decisão 2). O FPP **não** monta nós do grafo nem emite SQL:
 * nós (`nodes`) exigem payload MessagePack + Ed25519 `public_key`/`signature` (schema T-106),
 * cuja construção pertence à camada de linhagem (ULID T-102, HLC T-103, sign T-101), não a uma
 * máquina de estados de transporte. O FPP apenas dispara a intenção de gênese; a implementação
 * real (PROFILE/SPECIFICATION/NETWORK_BIRTH assinados, atômica) é de T-209/linhagem.
 * Definido em `@plataforma/protocol` (port compartilhado). `writeGenesis` é **atômico**.
 */
export interface GenesisWriter {
  /** Persiste atomicamente os registros de gênese. Idempotente quanto ao NETWORK_BIRTH (imutável). */
  writeGenesis(input: { adminProfile: PeerId; explicit: boolean }): Promise<void>;
}

/** Eventos emitidos pelo FPP para consumo externo (ex: SyncWorker). */
export type FppEvent =
  | { type: 'state_changed'; from: FppState; to: FppState }
  | { type: 'genesis_write_completed' }
  | { type: 'provisional_peer_cleared' };

export type FppEventHandler = (event: FppEvent) => void;

/**
 * Máquina de estados do First Peer Protocol.
 *
 * Consome SwarmRegistry para detectar peers, GenesisWriter para disparar a
 * gênese (sem montar nós/SQL), e ClockPort para temporização testável.
 *
 * @remarks
 *   - Parâmetros temporais derivados de first-peer-protocol.md:
 *     WAITING_FOR_SWARM = 8 s, OFFLINE_RETRY backoff 10→20→40→máx 60 s.
 *   - Vetor adversarial 12 (plano-de-implementacao §2.5): peer sem token
 *     em rede vazia NUNCA entra em GENESIS — vai para OFFLINE_RETRY.
 */
export class FirstPeerProtocol {
  constructor(
    config: FppConfig,
    registry: SwarmRegistry,
    genesis: GenesisWriter,
    clock: ClockPort,
    /** Identidade do peer local: usada para filtrar self-events do SwarmRegistry
     *  e como adminProfile no writeGenesis. */
    localDeviceId: PeerId,
  );

  /** Estado atual da máquina. */
  get state(): FppState;

  /**
   * Inicia o protocolo: transita JOINING → WAITING_FOR_SWARM,
   * subscreve eventos do SwarmRegistry, dispara timer de 8 s.
   * @returns Promise que resolve no **primeiro estado assentado** — `CONNECTED`, `GENESIS` ou a
   *   **primeira entrada em** `OFFLINE_RETRY` (decisão 6). NUNCA fica pendente indefinidamente.
   *   Transições subsequentes (ex.: OFFLINE_RETRY → CONNECTED) são observadas via `onEvent`.
   */
  start(): Promise<FppState>;

  /** Força transição para CONNECTED (ex.: peer encontrado via SwarmRegistry). */
  onPeerFound(peerId: PeerId): void;

  /** Registra callback para eventos da máquina. Retorna unsubscribe. */
  onEvent(handler: FppEventHandler): () => void;

  /** Encerra o protocolo: limpa timers, unsubscribe do registry. */
  destroy(): void;
}
```

> **✅ Contrato confirmado (Seção 6).** `start()` resolve no primeiro estado assentado (decisão 6);
> injeção é `GenesisWriter` + `ClockPort` (decisão 2 — não `StoragePort` cru). Zero decisões abertas.

### Parâmetros temporais fixos *(derivados de first-peer-protocol.md)*

| Parâmetro | Valor | Fonte |
|:---|:---|:---|
| Timeout `WAITING_FOR_SWARM` | **8 000 ms** | first-peer-protocol.md §WAITING_FOR_SWARM ("timer com duração de 8 segundos") |
| Backoff base `OFFLINE_RETRY` | **10 000 ms** | first-peer-protocol.md §OFFLINE_RETRY ("10 s → 20 s → 40 s → máximo de 60 s") |
| Backoff máximo | **60 000 ms** | idem |
| Fator de recuo | **2×** (10→20→40→60) | idem (sequência implícita: dobra a cada tentativa, teto em 60 s) |

### Regras de transição *(derivadas de first-peer-protocol.md §Estados)*

1. `JOINING` + `start()` → `WAITING_FOR_SWARM` (dispara timer 8 s, subscreve SwarmRegistry).
2. `WAITING_FOR_SWARM` + `onPeerFound()` → `CONNECTED` (timer cancelado).
3. `WAITING_FOR_SWARM` + timer expira + (`hasBootstrapToken` OU `explicitCreateNetwork`) → `GENESIS`.
4. `WAITING_FOR_SWARM` + timer expira + sem token E sem ação explícita → `OFFLINE_RETRY`.
5. `GENESIS` + `onPeerFound()` → `CONNECTED` (limpa `PROVISIONAL_SYSTEM_PEER`).
6. `OFFLINE_RETRY` + `onPeerFound()` → `CONNECTED`.
7. `OFFLINE_RETRY` + timer expira → recuo exponencial (10→20→40→60 s), reavalia.

### Ações do estado GENESIS *(derivadas de genesis-state.md §Contrato)*

Ao entrar em `GENESIS`, o FPP executa **duas** ações (decisão 2):
1. `await genesis.writeGenesis({ adminProfile, explicit })` — a impl (T-209/linhagem) persiste
   atomicamente, **com assinatura Ed25519**, os nós `PROFILE` (admin), `SPECIFICATION` (workspace)
   e `SPECIFICATION:NETWORK_BIRTH` (imutável — `specification-network-birth.md`). Ao concluir, emite
   `genesis_write_completed`.
2. `registry.upsert(localDeviceId, { isProvisionalSystemPeer: true })` — marca a flag
   `PROVISIONAL_SYSTEM_PEER` (decisão 4; campo aditivo no `PeerEntry` de T-205).

> **Por que não SQL no FPP (decisão 2):** o schema `nodes` (T-106, `done`) exige `payload`
> (MessagePack), `public_key` e `signature` Ed25519 — construir um nó assinado é trabalho da
> linhagem (T-101/T-102/T-103), não de uma máquina de estados. O FPP delega via `GenesisWriter`;
> os testes usam um **stub** que registra a chamada. A imutabilidade de `NETWORK_BIRTH` é garantida
> pela impl/grafo, não pelo FPP.

Casos de teste (numerados, implementar 1:1):
1. **Estado inicial:** `new FirstPeerProtocol({ hasBootstrapToken: false, explicitCreateNetwork: false }, ...)` → `state === FppState.JOINING`.
2. **JOINING → WAITING_FOR_SWARM:** `start()` → `state === FppState.WAITING_FOR_SWARM`.
3. **Peer encontrado durante espera:** `start()`; `onPeerFound('peer-B')` antes do timer expirar → `state === FppState.CONNECTED`, timer cancelado.
4. **Timer expira + bootstrap token → GENESIS:** config `hasBootstrapToken: true`; `start()`; `clock.advance(8000)` → `state === FppState.GENESIS`.
5. **Timer expira + sem token → OFFLINE_RETRY:** config `hasBootstrapToken: false`; `start()`; `clock.advance(8000)` → `state === FppState.OFFLINE_RETRY`.
6. **Timer expira + explicitCreateNetwork → GENESIS:** config `{ hasBootstrapToken: false, explicitCreateNetwork: true }`; `start()`; `clock.advance(8000)` → `state === FppState.GENESIS`.
7. **GENESIS dispara a escrita:** entrar em GENESIS → `genesis.writeGenesis(...)` chamado **exatamente uma vez** (stub registra a chamada e os args); evento `genesis_write_completed` emitido ao resolver.
8. **GENESIS é idempotente quanto à escrita:** re-entrada em GENESIS (ou GENESIS já assentado) **não** chama `writeGenesis` de novo — o FPP garante chamada única. *(A imutabilidade do nó `NETWORK_BIRTH` no grafo é responsabilidade da impl do `GenesisWriter`/T-106, não do FPP — specification-network-birth.md §Contrato item 2.)*
9. **GENESIS → CONNECTED (segundo peer):** em GENESIS; `onPeerFound('peer-C')` → `state === FppState.CONNECTED`; `registry.upsert(local, { isProvisionalSystemPeer: false })` chamado (flag limpa) + evento `provisional_peer_cleared`.
10. **OFFLINE_RETRY backoff:** `start()` sem token → timer expira → OFFLINE_RETRY com primeiro backoff de 10 s; `clock.advance(10000)` → segundo backoff de 20 s; `clock.advance(20000)` → terceiro de 40 s; `clock.advance(40000)` → quarto de 60 s (teto).
11. **OFFLINE_RETRY → CONNECTED:** em OFFLINE_RETRY, no meio do backoff; `onPeerFound('peer-D')` → `state === FppState.CONNECTED`, timer cancelado.
12. **Vetor adversarial 12:** config `{ hasBootstrapToken: false, explicitCreateNetwork: false }` + SwarmRegistry vazio + timer expira → OFFLINE_RETRY. Verificar que GENESIS **nunca** é atingido *(derivado de plano-de-implementacao.md §2.5 vetor 12)*.
13. **Evento state_changed:** subscrever `onEvent` → `start()` emite `{ type: 'state_changed', from: JOINING, to: WAITING_FOR_SWARM }`; demais transições também emitem.
14. **destroy() limpa recursos:** após `start()` + `destroy()`, timer cancelado, unsubscribe do SwarmRegistry, handlers não recebem mais eventos.
15. **start() chamado 2×:** segundo `start()` é no-op ou lança — estado não corrompido.

2. **Persistência de GENESIS → seam `GenesisWriter` injetado; FPP NÃO escreve SQL (revisa as opções).**
   Descoberta na verificação: **T-106 está `done`** e seu schema `nodes` exige `payload` (MessagePack),
   `public_key` e `signature` Ed25519 — i.e. **nós assinados**. Construir um nó assinado é trabalho da
   linhagem (ULID T-102, HLC T-103, sign T-101), não de uma máquina de estados de transporte. Logo as
   opções (a) "SQL parametrizado no FPP" e (b) "stubar exec()" estão **erradas de camada**: o FPP
   produziria nós sem assinatura válida, violando a invariante do grafo. **Resolução = (c) refinada:**
   o FPP injeta um port estreito `GenesisWriter` (definido em `@plataforma/protocol`) e chama
   `writeGenesis(...)` **uma vez**; a impl real (assinada, atômica) é T-209/linhagem. Testes usam stub.
   *(Esta é a decisão de maior julgamento — registrada para auditoria.)*

3. **Bootstrap token → booleano `hasBootstrapToken` no config (opção a).** Caderno-2 trata o token
   como construto UCAN derivado de convite/root; **validar** o token é da camada de Linhagem
   Criptográfica, não da máquina de estados. O FPP recebe o booleano **já computado** por um seam de
   validação upstream (T-209/linhagem). Mantém o teste do vetor adversarial 12 limpo. Futureproof: o
   booleano é a fronteira; a prova criptográfica é injetada por quem o calcula.

4. **PROVISIONAL_SYSTEM_PEER → campo aditivo `isProvisionalSystemPeer?: boolean` em `PeerEntry` (opção a).**
   A flag precisa ser **observável** pelo registry (consumida por T-206/207/404 — genesis-state.md),
   então estado interno do FPP (b) está errado. O `SwarmRegistry` (T-205) já tem
   `upsert(deviceId, partial)` — nenhum método novo (c) é necessário. Como T-205 está `ready` (ainda
   não implementado), o campo é **adicionado agora ao `PeerEntry` de T-205** (zero rework). O FPP
   set/clear via `registry.upsert(local, { isProvisionalSystemPeer })`.

5. **FPP ↔ descoberta → o FPP só consome eventos do SwarmRegistry; descoberta é injetada (opção a).**
   A Seção 5 já proíbe descoberta. O "dispara buscas em paralelo" do verbete é o papel **conceitual**;
   a fronteira de **implementação** mantém o FPP como pura lógica de decisão reagindo a eventos do
   registry. Sem callback `onStartDiscovery` (b — acoplamento especulativo, YAGNI). As camadas de
   descoberta (T-204/mDNS/T-209) alimentam o registry; o registry é o seam.

6. **`start()` → resolve no primeiro estado assentado (opção c).** O risco de memory-leak (b: bloqueia
   para sempre em OFFLINE_RETRY) é real; (a: resolve logo após o timer) torna o retorno inútil. **(c)** é
   o meio futureproof: `start()` resolve com o primeiro de {`CONNECTED` | `GENESIS` | 1ª entrada em
   `OFFLINE_RETRY`} e emite eventos para transições posteriores. Compatível com a assinatura
   `Promise<FppState>` ("estado final" = primeiro assentado).
