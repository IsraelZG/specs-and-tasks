## 1. Objetivo
Implementar política de serialização em SPEC no pacote `@plataforma/protocol`: bloco `serialization: {mode, set, k, fault_model, lease}` lido da SPEC do ativo; defaults por modalidade (corporativo `leader`=super peer; P2P `quorum` no anel de custódia); invariante de não-conflito permanece no core independente da política
*(derivado de `docs/plano-de-implementacao.md` §9 T-607)*.

### Contratos exatos
```ts
// --- packages/protocol/src/serializationPolicy.ts 
---
import type { StoragePort } from './ports.js';
// NÃO importar SignedNode/UnsignedNode de @plataforma/core aqui — protocol não pode depender de
// core (direção fixa: protocol ← crypto ← core ← transport, ver T-1033 §1 e visao-arquitetural §1).
// applyPolicy devolve um payload protocol-native pronto pra assinar; quem assina (KeyVault) e
// envolve em SignedNode é o CALLER em core (Decisão §6 #1).

export enum SerializationMode {
  LEADER = 'leader',         // líder único (corporativo: super peer)
  QUORUM = 'quorum',         // quorum no anel de custódia (P2P)
  NONE = 'none',             // sem serialização (apenas comutativos)
}

export interface SerializationPolicy {
  mode: SerializationMode;
  set: string[];             // entity_ids do conjunto de validadores/líderes
  k: number;                 // tamanho do quorum (default 1 para LEADER)
  faultModel: 'crash' | 'byzantine';
  lease: number;             // lease do líder em ms (evita split-brain)
}

export const DEFAULT_CORPORATE_POLICY: SerializationPolicy = {
  mode: SerializationMode.LEADER,
  set: [],
  k: 1,
  faultModel: 'crash',
  lease: 5000,
};

export const DEFAULT_P2P_POLICY: SerializationPolicy = {
  mode: SerializationMode.QUORUM,
  set: [],
  k: 3,
  faultModel: 'byzantine',
  lease: 0,
};

/** Lê política do nó SPECIFICATION do ativo. */
export function readSerializationPolicy(
  storage: StoragePort,
  assetEntityId: string,
): Promise<SerializationPolicy>;

/** Valida política (não permite k=0 sem NONE). */
export function validatePolicy(policy: SerializationPolicy): { valid: boolean; errors: string[] };

/** Verifica se peer é líder/quorum member dado a política. */
export function isAuthorized(
  policy: SerializationPolicy,
  peerEntityId: string,
): boolean;

/** Payload pronto pra assinar (protocol-native — não é o SignedNode de core). */
export interface UnsignedSpecUpdate {
  assetEntityId: string;
  nodeType: 'SPECIFICATION';
  payload: Uint8Array;          // bloco `serialization` codificado
}

/**
 * Aplica política (troca sem recompilar). Constrói o payload não-assinado —
 * a chave privada nunca circula fora do KeyVault; o caller (em `@plataforma/core`)
 * assina via KeyVault e persiste como `SignedNode` (decisão do arquiteto, 2026-07-01 —
 * ver §6, Decisão #1). `protocol` não importa tipos de `core` (T-1033).
 */
export function applyPolicy(
  storage: StoragePort,
  assetEntityId: string,
  policy: Partial<SerializationPolicy>,
): Promise<UnsignedSpecUpdate>;
```
- `readSerializationPolicy`: consulta `SPECIFICATION` do entity_id do ativo → extrai `serialization` block; se ausente, retorna `DEFAULT_P2P_POLICY` (decisão do arquiteto, 2026-07-01 — ver §6, Decisão #2). `DEFAULT_CORPORATE_POLICY` só via bloco explícito na SPEC.
- `validatePolicy`: `k=0` só permitido quando `mode=NONE`; `k=0` malicioso não desativa invariante de core.
- `isAuthorized`: no modo `LEADER`, retorna true se peer é o líder; no modo `QUORUM`, se peer está no set.
- Invariante de não-conflito: permanece no core independente da política.

## 6. Feedback de Especificação

### Decisões de Arquiteto (2026-07-01, resolvidas via `/arquiteto-decisoes`)

**Decisão #1 — assinatura em `applyPolicy`: DECIDIDO opção (B).** `applyPolicy` retorna um
payload não-assinado (`UnsignedSpecUpdate`) para o KeyVault assinar — a chave privada nunca
circula fora do KeyVault, mesmo princípio de T-511/T-512 (chaves ficam no vault).

**Correção (re-endurecimento, 2026-07-02):** a formulação original citava `SignedNode` de
`@plataforma/core` — mas `packages/protocol` não pode importar tipos de `@plataforma/core`
(direção fixa `protocol ← crypto ← core ← transport`, ver `docs/visao-arquitetural.md §1` e
T-1033 §1, que está corrigindo exatamente essa direção de import). `applyPolicy` devolve o tipo
protocol-native `UnsignedSpecUpdate` (§1); o wrap em `SignedNode` e a assinatura via KeyVault
acontecem no caller, em `@plataforma/core`.

**Decisão #2 — política default: DECIDIDO opção (A).** `readSerializationPolicy` retorna
`DEFAULT_P2P_POLICY` quando o bloco `serialization` está ausente — consistente com o projeto ser
peer-to-peer "soberano" por padrão (mesma filosofia de T-511). `DEFAULT_CORPORATE_POLICY` é opção
explícita, não fallback implícito.
