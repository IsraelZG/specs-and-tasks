# T-607 · Política de serialização em SPEC

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro)

## 1. Objetivo
Implementar política de serialização em SPEC no pacote `@plataforma/protocol`: bloco `serialization: {mode, set, k, fault_model, lease}` lido da SPEC do ativo; defaults por modalidade (corporativo `leader`=super peer; P2P `quorum` no anel de custódia); invariante de não-conflito permanece no core independente da política
*(derivado de `docs/plano-de-implementacao.md` §9 T-607)*.

### Contratos exatos
```ts
// --- packages/protocol/src/serializationPolicy.ts ---
import type { StoragePort } from './ports.js';

export enum SerializationMode {
  LEADER = 'leader',         // líder único (corporativo: super peer)
  QUORUM = 'quorum',         // quorum no anel de custódia (P2P)
  NONE = 'none',             // sem serialização (apenas comutativos)
}

export interface SerializationPolicy {
  mode: SerializationMode;
  set: string[];             // entity_ids do conjunto de validadores/líderes
  k: number;                 // tamanho do quorum (default 1 para LEADER)
  faultModel: 'crash' | 'byzantine';
  lease: number;             // lease do líder em ms (evita split-brain)
}

export const DEFAULT_CORPORATE_POLICY: SerializationPolicy = {
  mode: SerializationMode.LEADER,
  set: [],
  k: 1,
  faultModel: 'crash',
  lease: 5000,
};

export const DEFAULT_P2P_POLICY: SerializationPolicy = {
  mode: SerializationMode.QUORUM,
  set: [],
  k: 3,
  faultModel: 'byzantine',
  lease: 0,
};

/** Lê política do nó SPECIFICATION do ativo. */
export function readSerializationPolicy(
  storage: StoragePort,
  assetEntityId: string,
): Promise<SerializationPolicy>;

/** Valida política (não permite k=0 sem NONE). */
export function validatePolicy(policy: SerializationPolicy): { valid: boolean; errors: string[] };

/** Verifica se peer é líder/quorum member dado a política. */
export function isAuthorized(
  policy: SerializationPolicy,
  peerEntityId: string,
): boolean;

/** Aplica política (troca sem recompilar). */
export function applyPolicy(
  storage: StoragePort,
  assetEntityId: string,
  policy: Partial<SerializationPolicy>,
  // TODO: signingKey?: Ed25519PrivateKey (ver Decisão Aberta 1)
): Promise<void>;
```
- `readSerializationPolicy`: consulta `SPECIFICATION` do entity_id do ativo → extrai `serialization` block.
- `validatePolicy`: `k=0` só permitido quando `mode=NONE`; `k=0` malicioso não desativa invariante de core.
- `isAuthorized`: no modo `LEADER`, retorna true se peer é o líder; no modo `QUORUM`, se peer está no set.
- Invariante de não-conflito: permanece no core independente da política.

## 6. Feedback de Especificação

> [!WARNING]
> **Decisões abertas para o Arquiteto:**
>
> 1. **Chave de Assinatura em `applyPolicy`**:
>    - Para que `applyPolicy` atualize a especificação de um ativo, ela precisa criar e persistir um novo nó `SPECIFICATION` que muta o nó anterior (usando `insertNode`). Porém, isso requer assinar o nó, o que exige a chave privada correspondente. A assinatura do método não recebe nenhuma chave privada ou delegate.
>    - **Opções**:
>      - (A) Adicionar `privateKey: Ed25519PrivateKey` como argumento em `applyPolicy`.
>      - (B) Fazer com que o `applyPolicy` apenas retorne o `SignedNode` construído ou delegar a assinatura para um serviço do KeyVault.
>
> 2. **Seleção de Política Default na ausência do bloco `serialization`**:
>    - Se a SPECIFICATION lida não possui o bloco `serialization`, qual default deve ser retornado por `readSerializationPolicy`?
>    - **Opções**:
>      - (A) Retornar `DEFAULT_P2P_POLICY` por padrão de descentralização.
>      - (B) Retornar `DEFAULT_CORPORATE_POLICY` por padrão comercial.
>      - (C) Adicionar um parâmetro `defaultPolicy` na função `readSerializationPolicy`.
