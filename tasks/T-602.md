## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro)

## 1. Objetivo
Implementar ciclo de commit Automerge: `pending_changes` local; ephemeral Changes entre co-editores (canal T-403); gatilhos de commit por inatividade/limiar; consolidação `Automerge.save()` → nó-versão assinado + `MUTATES` + `AUTHORED` com sumário; limpeza de staging.
*(Normativo: `docs/plano-de-implementacao.md` §9 T-602, linha 295. **Pré-requisito documental: ADR-001** `docs/adr/adr-001-automerge-unico.md` — existe. Aceite: dois editores concorrentes convergem; tabela `nodes` recebe só commits, nunca micro-changes.)*

> **⚠️ CONFRONTAÇÃO (endurecimento 2026-07-08, claude-opus).** Três divergências reais contra o
> código/arquitetura atuais (ver Seção 6, D1–D4):
> 1. **Violação de camada:** o contrato coloca `commitCycle.ts` em `@plataforma/core` mas importa
>    `AutomergeDocId` de `@plataforma/transport`. `visao-arquitetural.md §1` fixa a ordem
>    `protocol ← crypto ← core ← transport` — **core NÃO depende de transport** (confirmado:
>    `core/package.json` não lista `@plataforma/transport`). Esse import **inverte a seta / fecha
>    ciclo** (mesma classe do bug de T-1033).
> 2. **Automerge não integrado:** `Automerge.save()` está na spec, mas **nenhum `package.json` tem
>    dependência `automerge`/`@automerge/automerge`** (grep vazio) e **T-403 está `ready` mas NÃO
>    implementado** — `packages/transport/src/automerge/` não existe. A "casca" que forneceria o
>    doc Automerge vivo (p/ chamar `.save()`) não resolve no estado atual.
> 3. **Sem helper de aresta AUTHORED nem fonte de chave de assinatura** — ver D3/D4.

### Contratos exatos *(D1–D4 materializadas — recon arquiteto 2026-07-12, claude-fable)*
```ts
// --- packages/workers/src/sync/commitCycle.ts  (D1: @plataforma/workers, camada 5 —
//     pode importar core E transport sem inverter seta; visao-arquitetural.md §1) ---
// Dependências NOVAS em packages/workers/package.json (hoje o pacote não tem nenhuma):
//   "@automerge/automerge": "^3.0.0"          // API estável verificada na fonte:
//     load<T>(data: Uint8Array): Doc<T> · save<T>(doc): Uint8Array ·
//     applyChanges<T>(doc, changes, opts?): [Doc<T>] · getChanges · getLastLocalChange · merge
//   "@plataforma/protocol", "@plataforma/crypto", "@plataforma/core",
//   "@plataforma/transport": "workspace:*"
import * as A from '@automerge/automerge';
import type { EntityId, ULID, StoragePort, GraphStorePort, ClockPort } from '@plataforma/protocol';
import type { Ed25519PrivateKey } from '@plataforma/crypto';
import { signNode, signEdge, insertNodeWithEdges } from '@plataforma/core'; // signature.ts:147/156 · lineage.ts (nova primitiva abaixo)
import type { AutomergeShell } from '@plataforma/transport';             // T-403

// ── ATOMICIDADE (F2) — nova primitiva transaction-aware no CORE ──────────────
// PROBLEMA PROVADO: insertNode abre a própria StoragePort.transaction (lineage.ts:105) e
// as impls fazem BEGIN/COMMIT crus SEM savepoint/contador (sqliteStorage.ts:110-121) —
// aninhar transação quebra ("cannot start a transaction within a transaction"). O
// GraphStoreTx do callback só tem putNode/putEdge (ports.ts:174-177), insuficiente para as
// leituras de linhagem do insertNode.
// SOLUÇÃO MÍNIMA (dona: core, junto dos invariantes de linhagem — NENHUMA mudança em
// GraphStorePort/T-1043):
//
//   // packages/core/src/lineage.ts — [UPDATE nesta task]
//   /** Corpo idêntico ao insertNode (validações B1/HLC/ciclo, nó, entity_members, MUTATES,
//    *  entity_heads) + grava `extraEdges` (AUTHORED, CO_SIGNED_BY…) DENTRO da mesma
//    *  transação. As leituras usam storage.exec (mesma conexão ⇒ juntam-se ao BEGIN aberto,
//    *  exatamente como o insertNode já faz hoje); as arestas extras usam o tx.putEdge do
//    *  callback (GraphStoreTx). Qualquer throw ⇒ ROLLBACK de TUDO (nó, arestas, projeções).
//    *  insertNode(s, n, p) passa a delegar: insertNodeWithEdges(s, n, [], p) — zero mudança
//    *  de comportamento; suíte existente de lineage/merge continua verde (gate cobre). */
//   export async function insertNodeWithEdges(
//     storage: StoragePort & GraphStorePort,
//     node: SignedNode,
//     extraEdges: SignedEdge[],
//     parentId?: ULID,
//   ): Promise<void>;
//
// O CommitCycle NÃO abre transação própria — delega a transação inteira à primitiva.

export interface PendingChange {
  changeId: string;
  docId: string;
  authorId: EntityId;
  payload: Uint8Array;          // Automerge change bytes (A.getLastLocalChange / evento local)
  timestamp: number;
}

export interface CommitTrigger {
  inactivityMs: number;          // default 3000 (3s de inatividade)
  changeThreshold: number;       // default 50 changes pendentes
  maxPendingMs: number;          // default 30000 (30s máximo pendente)
}

export interface CommitResult {
  nodeId: ULID;                  // nó-versão assinado
  authoredEdgeId: string;        // aresta AUTHORED com sumário
  mutatesEdgeId: string;         // criada por insertNode; recuperar pós-insert:
                                 //   SELECT id FROM edges WHERE edge_type='MUTATES' AND target_id=?
  changesConsolidated: number;
}

export interface CommitCycleConfig {  // D4: injeção explícita de dependência
  storage: StoragePort & GraphStorePort;  // insertNode exige StoragePort; AUTHORED usa putEdge (ports.ts:194)
  clock: ClockPort;                       // gatilhos via clock.setTimeout — VirtualClock nos testes (T-003)
  signingKey: Ed25519PrivateKey;
  authorId: EntityId;
  shell: AutomergeShell;                  // canal ephemeral T-403
  docId: string;                          // doc casca do documento em edição
}

/** Lote de commit CONGELADO (F2). O que os co-signers assinam e o que o finalize persiste
 *  são OS MESMOS BYTES — `snapshot` é capturado no prepare; changes posteriores não o alteram. */
export interface PreparedCommit {
  docId: string;
  snapshot: Uint8Array;          // A.save(doc) capturado AQUI; finalize persiste EXATAMENTE isto
  snapshotHash: Uint8Array;      // sha256(snapshot) — crypto/wrappers.ts:138; é o que se co-assina
  changeIds: string[];           // ids do lote (p/ clearStaging seletivo)
  changeHashes: Uint8Array[];    // sha256 de cada change (payload da AUTHORED)
  parentNodeId: ULID;            // head da entity no momento do prepare
}

export class CommitCycle {
  constructor(config: CommitCycleConfig);

  /** DONO DO DOC VIVO (D2): o CommitCycle possui o A.Doc<T> — reidrata com
   *  A.load(payload do último nó-versão) e o mantém em RAM. A casca (T-403) só
   *  transporta bytes; nunca possui nem acessa o documento de conteúdo. */

  /** Change local: A.applyChanges no doc vivo + staging + shell.broadcastEphemeral
   *  (envelope { kind: 'change', change: PendingChange }). */
  stageChange(change: PendingChange): void;

  /** Change de co-editor, vinda de shell.onEphemeralMessage: A.applyChanges (idempotente,
   *  dedup por changeId) + staging. */
  receiveRemoteChange(change: PendingChange): void;

  /** Congela o lote corrente: snapshot + hash + ids/hashes das changes staged ATÉ AGORA.
   *  Síncrono; NÃO limpa staging, NÃO persiste. Changes que chegarem depois ficam fora
   *  deste lote (vão para o próximo ciclo). */
  prepareCommit(): PreparedCommit;

  /** Persiste EXATAMENTE `prepared.snapshot`: monta SignedNode(payload=snapshot, signNode) +
   *  AUTHORED (signEdge) + `extraEdges` (ex.: CO_SIGNED_BY vindas do T-603) e grava TUDO com
   *  UMA chamada a insertNodeWithEdges (transação única do core; rollback total em falha).
   *  Só após sucesso: clearStaging(docId, prepared.changeIds). */
  finalizeCommit(prepared: PreparedCommit, extraEdges?: SignedEdge[]): Promise<CommitResult>;

  /** Açúcar sem co-assinatura: finalizeCommit(prepareCommit()). */
  forceCommit(): Promise<CommitResult>;

  /** Gatilhos disparados ⇒ mesmo caminho prepare→finalize. */
  checkTriggers(): Promise<CommitResult | null>;

  /** Limpeza SELETIVA por id de change do lote (caderno-2 §3.3 passos 5–6). Full-wipe
   *  descartaria changes de peers chegadas durante o commit — perda silenciosa. */
  clearStaging(docId: string, changeIds: string[]): void;

  configureTriggers(triggers: Partial<CommitTrigger>): void;
}
```
- **Formato da AUTHORED (D3):** aresta `edge_type='AUTHORED'`, `source_id=authorId (PROFILE)`,
  `target_id=nodeId do commit`, payload = JSON UTF-8
  `{ changeHashes: string[] /* sha256 hex de cada change */, summary: string, count: number }`,
  assinada com `signEdge` e persistida com `storage.putEdge` (GraphStorePort — ports.ts:194).
  O schema aceita qualquer `edge_type` (schema.ts:18, TEXT sem CHECK) — zero migração.
- **Reconexão (ataque coberto):** peer que perdeu ephemerals durante queda NÃO recupera pelo
  canal (ephemeral é fire-and-forget). Recupera por: (a) suas changes continuam no staging
  local até aparecerem em um commit (limpeza seletiva); (b) ao receber o nó-versão via RBSR,
  faz `A.load(snapshot)` + `A.merge` com o doc local — changes não consolidadas sobrevivem e
  entram no ciclo seguinte.
- `pending_changes`: buffer local de changes não-commitadas.
- Gatilhos: inatividade (3s), limiar (50 changes), timeout máximo (30s) — todos via ClockPort.
- Tabela `nodes` recebe só commits, nunca micro-changes.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest + VirtualClock (T-003).
- [x] **Ambiente:** `pnpm --filter @plataforma/workers test`.

Casos de teste:
1. `stageChange` acumula changes no buffer.
2. Gatilho por inatividade: 3s sem change → commit automático.
3. Gatilho por limiar: 50 changes → commit automático.
4. Gatilho por timeout máximo: 30s → commit automático.
5. `receiveRemoteChange` de co-editor → buffer inclui change remota.
6. `forceCommit` consolida changes e cria nó-versão + arestas.
7. `clearStaging` seletivo: só remove os `changeIds` do lote; change de peer chegada DURANTE o commit sobrevive no staging.
8. Dois editores digitando concorrentemente → convergem (tabela `nodes` recebe só commits, nunca micro-changes).
9. Reconexão: peer B cai, perde N ephemerals, volta, recebe o nó-versão (snapshot) e faz `A.merge` com o doc local → converge e as changes locais não consolidadas entram no ciclo seguinte.
10. **Change pós-prepare:** `prepareCommit()` congela o lote; change staged depois NÃO entra no snapshot persistido nem em `changeIds` — aparece no lote do ciclo seguinte.
11. **Rollback atômico (adversarial):** storage instrumentado para falhar no `putEdge` da AUTHORED (e, noutra rodada, numa aresta extra) → `finalizeCommit` rejeita E o banco fica com ZERO nó novo, ZERO aresta, `entity_heads`/`entity_members` intactos; staging não é limpo; re-tentativa após remover a falha commita normalmente.
12. **Snapshot exato:** o payload do nó persistido é byte-a-byte `prepared.snapshot` (não um re-`A.save` do doc que já recebeu changes novas) — `sha256(payload) === prepared.snapshotHash`.
