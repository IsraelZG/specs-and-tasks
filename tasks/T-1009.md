### Pegadinhas conhecidas
- **Decoder chain:** A função `psReplace` precisa emular a cadeia exata: o `pattern` vem
  de JSON source (com escapes), é decodificado por `JSON.parse` para string PS, depois
  PowerShell usa essa string como regex. O helper precisa aceitar o pattern COMO
  APARECE NO JSON SOURCE (com `\\\\` em JS source = 2 backslashes) ou COMO DECODIFICADO
  (1 backslash)? Decidir e documentar. **Recomendação:** aceitar o pattern como ele
  apareceria no `.claude/settings.json` (i.e., 4 backslashes em JS source para 1
  backslash em PS). A API é `psReplace(filePath, jsonEscapedPattern, replacement)`.
- **JS regex vs PowerShell regex:** `RegExp` em JS é case-sensitive; PowerShell
  `-match` é case-insensitive por padrão. O helper precisa emular essa semântica.
  Use `new RegExp(pattern, 'i')` para case-insensitive. Documente a diferença.
- **Backslash escape em JSON:** o pattern em JSON é `'\\\\\\\\\\\\'` (8 source chars) para
  match de 2 backslashes; ou `'\\\\'` (4 source chars) para match de 1 backslash. A
  tabela de decoding é a parte mais valiosa do helper — documente com exemplos.

**[m1] MINOR — Desvio do contrato TS: parâmetro `pattern` em vez de `jsonEscapedPattern`.**
- **Local:** `packages/testkit/src/psRegex.ts:13-21, 30-34`.
- **Evidência:** spec §1 fixa:
  ```ts
  export function psReplace(filePath: string, jsonEscapedPattern: string, replacement: string): string;
  export function psMatch(filePath: string, jsonEscapedPattern: string): boolean;
  ```
  A implementação tem `pattern: string` e o JSDoc (linhas 10-11, 27-28) diz explicitamente
  que "A API aceita o pattern JÁ decodificado (pós JSON.parse)". A spec §1 texto
  descreve `jsonEscapedPattern` como "como aparece no JSON source, ex.: 4 backslashes
  em JS string para match de 1 backslash" — o que implicaria `JSON.parse` interno
  (8 backslashes em JS source = 4 chars JSON = 2 backslashes decoded = regex `\\`).
  A spec §4 caso 1, no entanto, usa `'\\\\'` (2 backslashes em JS = 1 char JSON =
  1 backslash decoded = regex `\` que é inválido) — então a spec tem uma inconsistência
  interna: o texto §1 contradiz o teste §4. A impl escolheu a interpretação do §4 (pattern
  decoded) que é o que os testes exercem. Funcionalmente correto, mas o nome do parâmetro
  diverge do §1.
- **Ação corretiva:** ou (a) atualizar spec §1 para alinhar com a impl (renomear para
  `pattern` e documentar que é o pattern pós-JSON.parse), ou (b) implementar `JSON.parse`
  interno na função (mais complexo, diverge dos testes §4). Recomendação: (a) — alinhar
  a spec com a impl. Não-bloqueante → ledger.
