# T-904 · Caos programado (Chaos/Fuzzer sobre SimNetwork)

> **Origem:** contrato de framework herdado da antiga `T-905` (mal-rotulada como "Bateria de
> Testes Adversariais"). O escopo canônico de T-904 (`plano-de-implementacao.md` §12) é
> exatamente este: modo "chaos" no SimNetwork com falhas aleatórias rodando a suíte de
> convergência por N minutos com seed. A enumeração dos 12 vetores adversariais NÃO mora aqui —
> ela é responsabilidade da `T-902` (suíte consolidada). Esta task é o *motor* de injeção
> aleatória de falhas; T-902 é a *suíte nomeada* de testes-alvo determinísticos.

## 1. Objetivo
Implementar um motor de caos programado no pacote `@plataforma/testkit`: injeta falhas
aleatórias mas **reprodutíveis por seed** (partições, drops, corrupção, atraso, restart de peer,
rotação de época) e, ao final de cada execução, verifica que **nenhum invariante foi violado** —
convergência pós-cura (T-007), assinaturas válidas (T-107), heads consistentes, não-duplo-gasto
(invariante T1, T-604) e resolução determinística de fork (T-601)
*(derivado de `docs/plano-de-implementacao.md` §12 — "T-904 · Caos programado"; usa SimNetwork de
T-006 e `expectConverged` de T-007)*.

### Contratos exatos
```ts
// --- packages/testkit/src/chaos/Fuzzer.ts 
---
import { SimNetwork } from '../SimNetwork.js';            // T-006
import { SeededRandom } from '../random.js';              // T-003
import { expectConverged, PeerState } from '../assertions.js'; // T-007
import { PeerId } from '@plataforma/protocol';            // T-004

export interface ChaosScenario {
  name: string;
  iterations: number;   // nº de ações aleatórias aplicadas
  seed: string;         // seed fixa → execução 100% reprodutível
  peers: PeerState[];   // peers sob teste (para checagem de invariantes ao final)
}

export type ChaosAction =
  | { type: 'partition'; groups: PeerId[][] }
  | { type: 'heal' }
  | { type: 'dropMessages'; peerId: PeerId; rate: number }
  | { type: 'corruptMessage'; peerId: PeerId; rate: number }
  | { type: 'delayMessages'; peerId: PeerId; maxMs: number }
  | { type: 'restartPeer'; peerId: PeerId }
  | { type: 'rotateEpoch'; peerId: PeerId };  // plano §12: "rotações de época" (usa T-505)

export interface ChaosResult {
  scenario: string;
  passed: boolean;
  iterations: number;
  violations: string[];     // invariantes violados (vazio = passou)
  convergenceTimeMs: number;
}

/** Sorteia `iterations` ChaosActions (SeededRandom), aplica no SimNetwork, cura a rede,
 *  e ao final roda os checadores de invariante. Determinístico por seed. */
export function runChaosScenario(scenario: ChaosScenario, sim: SimNetwork): Promise<ChaosResult>;
```
- A geração de cada `ChaosAction` usa **exclusivamente** `SeededRandom` — duas execuções com a
  mesma seed produzem a mesma sequência de ações e o mesmo `ChaosResult`.
- Toda passagem de tempo é via `VirtualClock` (T-003) — sem `setTimeout` real; "N minutos" do
  critério de aceite são minutos *virtuais* avançados pelo clock.
- Checadores de invariante ao final: `expectConverged(peers)` (T-007) + verificação de
  assinaturas válidas + ausência de duplo-gasto (consulta de `SPENDS` por head, T-604) +
  fork resolvido deterministicamente (T-601).

## 4. Estratégia de Testes
- [x] **Framework:** Vitest (Node puro). SQLite `:memory:` + SimNetwork multi-peer.

Casos de teste:
1. Seed fixa + 1000 iterações → `ChaosResult` idêntico em duas execuções (reprodutibilidade).
2. Partição arbitrária + heal → peers convergem (`violations` vazio).
3. `corruptMessage rate=1.0` num peer → nós corrompidos rejeitados, peers íntegros convergem.
4. `rotateEpoch` no meio do caos → catch-up correto, sem violação de invariante.
5. Cenário que planta duplo-gasto sob partição → `violations` reporta a quebra de T1.
6. "Soak" curto: 5 min virtuais de caos com seed → 0 violações.
