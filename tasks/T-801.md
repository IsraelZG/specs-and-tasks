## 1. Objetivo
Fatiar arquivos binários em chunks de tamanho fixo (1 MiB), cifrar cada chunk independentemente com AES-256-GCM, e construir uma Merkle Tree sobre os hashes dos ciphertexts para verificação de integridade e sync descentralizado
*(derivado de `docs/caderno-3-sdk/05-media-transport-plane.md` §2 Modelo de Chunking, §3 Criptografia por Chunk; `docs/caderno-3-sdk/19-streaming-reference-spec.md` §2 VOD)*.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/media/src/blob/Chunker.ts ---
import type { AesKey, AesNonce } from '@plataforma/crypto'; // T-101

/** Tamanho padrão de chunk = 1 MiB. Derivado de caderno-3-sdk/05 §2. */
export const DEFAULT_CHUNK_SIZE = 1_048_576; // 1 MiB

/** Um chunk cifrado com seus metadados. */
export interface EncryptedChunk {
  index: number;          // posição do chunk no arquivo (0-based)
  offset: number;         // offset em bytes no plaintext original
  ciphertext: Uint8Array; // bytes cifrados com AES-256-GCM
  nonce: AesNonce;        // 12 bytes — nonce único por chunk
  authTag: Uint8Array;    // 16 bytes — tag de autenticação GCM
  ciphertextHash: Uint8Array; // SHA-256 do CIPHERTEXT — FOLHA da Merkle Tree → InfoHash.
                              // Derivado de caderno-3-sdk/05 §2.3 (linha 22): "o InfoHash é
                              // calculado sobre o ciphertext — nunca sobre o plaintext."
  plaintextHash: Uint8Array;  // SHA-256 do plaintext do chunk — usado só para verificação de
                              // integridade pós-decifração. NÃO entra na Merkle Tree.
}

/**
 * Fatiador de arquivos binários.
 * Derivado de: caderno-3-sdk/05 §2 — chunk_size = potência-de-2, default 1 MiB.
 */
export class Chunker {
  readonly chunkSize: number;

  constructor(chunkSize?: number);

  /**
   * Divide `data` em chunks de tamanho `chunkSize`.
   * Último chunk pode ser menor.
   */
  slice(data: Uint8Array): Uint8Array[];

  /**
   * Cifra um chunk com AES-256-GCM.
   * Gera nonce internamente (12 bytes aleatórios).
   */
  encryptChunk(
    chunk: Uint8Array,
    index: number,
    offset: number,
    key: AesKey,
  ): Promise<EncryptedChunk>;

  /**
   * Processa arquivo completo: slice → encrypt → retorna lista de EncryptedChunk.
   */
  process(
    data: Uint8Array,
    key: AesKey,
  ): Promise<EncryptedChunk[]>;

  /** Reconstitui o plaintext original a partir dos chunks cifrados. */
  reassemble(chunks: EncryptedChunk[], key: AesKey): Promise<Uint8Array>;
}
```

```ts
// --- packages/media/src/blob/MerkleTree.ts ---
import type { Sha256Digest } from '@plataforma/crypto';

/** Nó da Merkle Tree. */
export interface MerkleNode {
  hash: Sha256Digest;   // SHA-256 do nó
  left?: MerkleNode;    // filho esquerdo
  right?: MerkleNode;   // filho direito
}

/**
 * Merkle Tree construída sobre os hashes dos ciphertexts dos chunks.
 * A raiz (InfoHash) identifica unicamente o arquivo cifrado.
 * Derivado de: caderno-3-sdk/05 §2.3 — InfoHash calculado sobre ciphertext.
 */
export class MerkleTree {
  /**
   * Constrói a árvore a partir dos hashes dos CIPHERTEXTS dos chunks
   * (`EncryptedChunk.ciphertextHash`), na ordem de `index`.
   * A raiz é o InfoHash do arquivo cifrado — caderno-3-sdk/05 §2.3 (linha 22).
   * NÃO passar `plaintextHash` aqui — isso violaria a norma e quebraria interop torrent/IPFS.
   */
  static build(leafHashes: Sha256Digest[]): MerkleTree;

  /** Raiz da árvore — o InfoHash do arquivo. */
  get root(): Sha256Digest;

  /** Prova de inclusão para um chunk específico (caminho de hashes). */
  getProof(chunkIndex: number): Sha256Digest[];

  /** Verifica uma prova de inclusão contra a raiz. */
  static verifyProof(
    root: Sha256Digest,
    leaf: Sha256Digest,
    index: number,
    proof: Sha256Digest[],
  ): boolean;
}
```

Casos de teste — Chunker (numerados):
1. `slice` de 1024 bytes com chunkSize=256 → 4 chunks de 256 bytes cada.
2. `slice` de 1000 bytes com chunkSize=256 → 3 chunks de 256 + 1 chunk de 232 bytes.
3. `process` de 512 bytes: cada chunk retorna `EncryptedChunk` com `index`, `offset`, `ciphertext`, `nonce`, `authTag`, `ciphertextHash` **e** `plaintextHash` preenchidos. `ciphertextHash` deve ser igual a `sha256(ciphertext)`.
4. Roundtrip: `process` → `reassemble` → bytes idênticos ao original.
5. `reassemble` com chunks em ordem trocada → falha (authTag não confere ou hash diferente).
6. Arquivo de 5 MiB dividido em chunks de 1 MiB (5 chunks) e reconstituído com mesmo SHA-256.
7. Dois chunks idênticos cifrados com mesma chave → ciphertexts diferentes (nonce único por chunk).

Casos de teste — MerkleTree (numerados — as folhas são sempre `ciphertextHash`):
8. `MerkleTree.build` com 1 leaf → raiz = hash do leaf.
9. `MerkleTree.build` com 4 leaves → raiz consistente; `getProof(0)` retorna 2 hashes.
10. `verifyProof` com prova válida → true.
11. `verifyProof` com prova adulterada → false.
12. `MerkleTree.build` com 3 leaves (ímpar) → árvore balanceada com duplicação do último leaf.
13. **InfoHash sobre ciphertext (anti-regressão da norma §2.3):** dado o resultado de `process(data, key)`, a raiz de `MerkleTree.build(chunks.map(c => c.ciphertextHash))` **difere** da raiz de `MerkleTree.build(chunks.map(c => c.plaintextHash))` (prova que a árvore consome o ciphertext, não o plaintext) e é igual à raiz reconstruída a partir de `sha256(c.ciphertext)` de cada chunk.

## 6. Feedback de Especificação (Spec Feedback Loop)
> **✅ DECISÃO RESOLVIDA (2026-06-22 — arquiteto) — Merkle/InfoHash sobre ciphertext:**
> O contrato original era contraditório: `EncryptedChunk` só carregava `plaintextHash`, o comentário
> de classe da `MerkleTree` dizia "ciphertext", e o doc de `build()` dizia "plaintextHash". A fonte
> normativa é inequívoca — caderno-3-sdk/05 §2.3 linha 22: *"o InfoHash é calculado sobre o ciphertext
> — nunca sobre o plaintext"* (e linha 21: "peças do torrent construídas sobre o ciphertext").
> **Resolução:** a Merkle Tree é construída sobre `ciphertextHash` (= `sha256(ciphertext)`); a raiz é o
> InfoHash. `plaintextHash` é mantido apenas para verificação de integridade pós-decifração e **não**
> entra na árvore. Contrato e casos de teste (3 e 13) já refletem isso. Não há mais decisão em aberto.

- **Fora de escopo (não é decisão pendente) — dois modos de chave (convergent vs unique):** caderno-3-sdk/05 §3.2 define `K_content` (convergente, dedup) e `K_file` (único, privacidade). A escolha depende da `SPECIFICATION:MEDIA_DELIVERY` e **não é desta task** — o `Chunker` recebe a chave pronta externamente. Deliberadamente excluído, não pendente.
- **Fora de escopo (não é decisão pendente) — streaming vs buffer:** esta v1 aceita `Uint8Array` de entrada; streaming real (ReadableStream) fica para task futura (T-802 ou similar). Deliberadamente adiado, não pendente.
