# T-305a · Sync UCAN: scopeRBSRTree (filtro por UCAN)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo

## 1. Objetivo
Implementar `scopeRBSRTree` no pacote `@plataforma/protocol`: reconstrói RBSRBTree apenas com nós autorizados pelo UCAN, impedindo vazamento de subgrafos não permitidos
*(extraído de T-305 §1; T-301a — RBSRBTree; T-501a — UCAN codec)*.

### Contratos exatos
```ts
// --- packages/protocol/src/auth/ucanScope.ts 
---
import { SignedUcan } from '@plataforma/core'; // T-501a
import { RBSRBTree } from '../rbsr/BTree.js'; // T-301a

/** Reconstrói B-Tree apenas com entity_ids no escopo do UCAN. */
export function scopeRBSRTree(tree: RBSRBTree, ucan: SignedUcan): RBSRBTree;

/** Extrai entity_ids autorizados do UCAN (prefixo 'entity:' em att[].with). */
export function extractEntityIds(ucan: SignedUcan): string[];
```

**[m1] MINOR — Tratamento defensivo de capability malformada.**
- **Local:** `packages/protocol/src/auth/ucanScope.ts:24-32` (`extractEntityIds`).
- **Evidência:** Sonda adversarial descobriu que uma capability com `resource: 'entity:'`
  (prefixo presente, mas ID vazio) faz `scopeRBSRTree` jogar `Error: id must be non-empty`
  (origem em `BTree.validateId` linha 76). O fluxo: `extractEntityIds` retorna `['']`
  → `scopeRBSRTree` chama `tree.get('')` → throw. Em produção, um UCAN vindo da rede
  com payload adulterado ou bug de issuer crasha o sync worker.
- **Não bloqueia porque:** (a) spec §4 não cobre este caso; (b) validação estrutural de
  UCAN pertence à T-501 (signature/exp/etc), não ao filter; (c) o throw é consistente
  com a invariante do BTree (rejeita IDs vazios). Mas é hygiene barato.
- **Ação corretiva (1 linha):** em `extractEntityIds`, exigir `cap.resource.length > ENTITY_PREFIX.length`
  antes de fazer o `add`. Custo trivial, sem mudança de contrato. → ledger.

**Resolução das ambiguidades da spec §1 (não-bloqueantes, esclarecidos):**
- **Tipo de retorno `BTree` (não `RBSRBTree` como na spec):** a classe real em
  `BTree.ts:90` chama-se `BTree`. A spec usou placeholder; impl alinhada com a realidade.
  Nota para o arquiteto atualizar a spec nas próximas passadas.
- **Tipos `SignedUcan`/`UcanPayload`/`UcanCapability` definidos localmente (não
  importados de `@plataforma/core`):** confirmado que seria **dep circular** — `core`
  já depende de `protocol` (ver `core/package.json:11`). A spec cometeu este erro;
  a impl está correta em definir o subset mínimo. Para T-501-evoluir-`SignedUcan`,
  o subset em `ucanScope.ts` pode precisar de sincronização manual — risco
  arquitetural, não código, → INFO (housekeeping para hardening de T-501).
