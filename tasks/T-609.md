# T-609 · Runtime Zen canônico + invariante T1 no core — sucessor de T-604

> **Por que existe (2026-07-14, arquiteto claude-fable).** T-604 foi obsoletada em lote no commit
> `9750676` ("limpeza ORQ/Nexus/Zen") sob a justificativa de que "Zen Engine" foi superado pelo
> Estaleiro/`@plataforma/plugin-workflows`. Isso vale para o **plano de orquestração de agentes**
> (JDM do Estaleiro), mas **não** para o **plano de produto**: a invariante T1
> (`anterior + delta == novo`, saldo nunca negativo) é capability do protocolo (plano §2.5 vetor
> adversarial 11; §9 M6) e vive em `@plataforma/core`. Verificado na master:
> `packages/plugin-zen-engine` depende de `@plataforma/estaleiro-contracts` (plano-Estaleiro,
> importável pelo core = inversão de arquitetura) e `packages/core` não tem motor de invariante
> algum. Dependentes de produto (T-605, T-606, T-902, T-904, T-AD-02, T-LOG-02, T-OFF-03) foram
> remapeados para esta task. O contrato de T-604 pós-reendurecimento de 2026-07-02 foi reconciliado
> com o runtime de workflows já implementado, para haver um único encapsulamento de GoRules.

## 1. Objetivo
Extrair um runtime Zen canônico, sem dependência do Estaleiro, e consumi-lo tanto pelo core quanto
pelos workflows. O core implementa invariantes transacionais obrigatórias — incluindo
`anterior + delta == novo` e saldo não-negativo — enquanto políticas configuráveis usam a Zen
Expression Language. O pacote `@plataforma/plugin-zen-engine` é absorvido e removido depois da
migração de todos os consumidores.

### Contratos
```ts
// --- packages/zen-runtime/src/index.ts ---
// Único pacote autorizado a importar @gorules/zen-engine@1.0.0-beta.4.
export interface WorkflowDefinition {
  id: string;
  version: number;
  content: string;
  createdAt: string;
  notes?: string;
}

export interface WorkflowStore {
  get(id: string, version?: number): Promise<WorkflowDefinition | null>;
  list(): Promise<WorkflowDefinition[]>;
  put(definition: WorkflowDefinition): Promise<void>;
}

export interface WorkflowEvaluationResult {
  workflowId: string;
  version: number;
  result: unknown;
  elapsedMs: number;
}

export class WorkflowEngine {
  constructor(store: WorkflowStore);
  evaluate(workflowId: string, context: unknown, version?: number): Promise<WorkflowEvaluationResult>;
  precompile(workflowId: string, version?: number): Promise<void>;
}

export function evaluatePolicyExpression(expression: string, context: unknown): boolean;

// --- packages/core/src/rules/InvariantEngine.ts ---
export interface BalanceTransition {
  previous: bigint;
  delta: bigint;
  next: bigint;
  causalRef: string;
}

export interface ZenRule {
  id: string;
  expression: string;
  onFailure: 'reject' | 'warn';
}

export interface ZenContext {
  facts: Record<string, unknown>;   // estado anterior
  changes: Record<string, unknown>;  // delta proposto
  result: Record<string, unknown>;   // estado resultante
}

export class InvariantEngine {
  constructor();

  /** Carrega políticas adicionais; invariantes de protocolo não entram aqui. */
  loadRules(rules: ZenRule[]): void;

  /** Avalia políticas configuráveis via runtime Zen canônico. */
  evaluate(ctx: ZenContext): Promise<{ passed: boolean; failures: ZenRule[] }>;

  /** Invariante T1 fixa: matemática, não-negatividade e causalidade. */
  validateBalanceTransition(transition: BalanceTransition): { passed: boolean; failures: string[] };
}
```
- `evaluatePolicyExpression` encapsula `evaluateUnaryExpressionSync`; nenhum consumidor acessa a
  lib externa diretamente.
- T1 não é string configurável: usa `bigint` e exige simultaneamente
  `previous + delta === next`, `next >= 0n` e `causalRef` não vazio.
- Isomórfico: no browser, avaliações JDM rodam fora da thread principal. A avaliação de workflow é
  assíncrona; somente o wrapper da expressão unária preserva o contrato síncrono da biblioteca.
- `@plataforma/zen-runtime` não importa `estaleiro-contracts`; é infraestrutura compartilhada.

### Pegadinhas conhecidas
- `evaluateUnaryExpressionSync` existe em `@gorules/zen-engine` (verificado em `index.d.ts:487`):
  `evaluateUnaryExpressionSync(expression: string, context: any): boolean`. Retorna `true` se
  a expressão for verdadeira no contexto dado.
- O pacote `@gorules/zen-engine` usa NAPI-RS com bindings nativos; o campo `browser` aponta
  pro build WASM/WASI. Para Windows ARM64, o fallback WASM funciona (verificado: `node -e "const { evaluateUnaryExpressionSync } = require('@gorules/zen-engine'); console.log(typeof evaluateUnaryExpressionSync)"` → `function`).
- O pacote WASM é `@gorules/zen-engine-wasm32-wasi@1.0.0-beta.4` (NÃO `@gorules/zen-engine-wasm`).
  Usar `packages/plugin-zen-engine/package.json` apenas como fonte da migração das dependências;
  esse pacote deve ser deletado ao final da task.
- Fixar `@gorules/zen-engine` em `1.0.0-beta.4` exato no runtime canônico; não usar range `^` em
  código de consenso. Erro de carregamento/avaliação falha fechado.
- O runtime preserva a classe de avaliação JDM usada pelos workflows e a função standalone usada
  pelas políticas. O core não importa o plugin nem a lib externa diretamente.

## 6. Feedback de Especificação (Spec Feedback Loop)

**DECIDIDO (arquiteto, 2026-07-16) — runtime único, plugin removido e T1 fixa.**

- GoRules permanece o motor open source canônico, mas fica encapsulado em
  `@plataforma/zen-runtime`, com versão exata, seleção NAPI/WASM e testes de paridade. Core e
  workflows consomem a mesma API; um avaliador próprio paralelo é proibido.
- `@plataforma/plugin-zen-engine` perde sua função após a extração: `WorkflowEngine` e tipos vão para
  o runtime; `createFileStore` e `createDecisionHook` não têm consumidor de produção e são apagados.
  Todos os consumidores devem ser refatorados na mesma entrega antes de deletar o pacote.
- Invariantes de protocolo não são regras de texto substituíveis. T1 é implementada em código
  tipado com `bigint`, valida igualdade aritmética, não-negatividade e referência causal. Zen avalia
  políticas adicionais declaradas por SPEC, sempre depois das validações criptográficas e causais.
- A identidade/hash da SPEC e a versão semântica do runtime precisam integrar a evidência canônica
  usada pelos validadores; atualização incompatível não pode alterar silenciosamente a semântica de
  uma regra já assinada.
