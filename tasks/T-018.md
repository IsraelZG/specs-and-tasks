# T-018 · Expandir cobertura de testes do design-system para os componentes restantes

O conteúdo original abaixo é mantido como referência de design; a implementação real está nas subtasks. Esta task só deve ser movida para `done` quando todas as 5 subtasks estiverem `done`.

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` + `@testing-library/react` (JSDOM) + `@testing-library/user-event`

## 1. Objetivo
O pacote `packages/design-system` tem 46 componentes em `src/components/`. Apenas 6 têm teste hoje (`tests/components/{Button,Checkbox,Input,Modal,Switch,Tabs}.test.tsx`). Esta task expande a cobertura para os **39 componentes restantes**, seguindo exatamente o molde já estabelecido nesses 6 arquivos (Vitest + `@testing-library/react`, `describe`/`it`, `render`+`screen`, `userEvent` para interação, `vi.fn()` para callbacks).

`Sidebar` **não** está nesta lista — seu teste é criado pela T-017 (que primeiro decide o destino da prop `collapsible` e só então escreve o teste; criar `Sidebar.test.tsx` aqui duplicaria/colidiria com aquele trabalho).

**Tamanho:** 39 componentes em uma única task é grande para uma sessão de agente. Esta task é dimensionada para ser executada **em lotes** (sugestão: 6-8 componentes por sessão, agrupados por tier — ver Seção 4). Entre lotes, se o contexto/tokens do Worker se esgotar, use `pause` com o resumo de quais componentes já têm teste — o próximo agente retoma de onde parou (Regra 2 do CLAUDE.md). Não é necessário (nem esperado) terminar os 39 em uma única chamada de `finish`.

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Vitest + `@testing-library/react` (JSDOM) + `@testing-library/user-event`, replicando o setup de `tests/setup.ts`
- [ ] **Métricas/Cobertura:** todo componente testado precisa cobrir os critérios da Seção 4 correspondentes ao seu tier — não é necessário 100% de branch coverage, é necessário cobrir os critérios listados
- [ ] **Ambiente do Teste:** JSDOM — nenhum dos 39 componentes precisa de browser real (sem animação não-determinística relevante para o teste; se algum precisar de `requestAnimationFrame` como o `Modal`, replicar o padrão `flushEnterAnimation` de `Modal.test.tsx`)
- [ ] **Fora de Escopo:** testes visuais/screenshot, Storybook, qualquer regressão de pixel — isso é deliberadamente outro tipo de teste (explicado fora desta task, sem task associada por ora)

### Critérios de aceite por tier (aplicar a cada componente do tier)

**Tier A (interação):**
1. Smoke: renderiza sem erro com props mínimas obrigatórias.
2. Interação real: a ação primária do componente (clique, digitação, seleção, arraste de slider, etc.) dispara o callback público esperado (`onChange`/`onValueChange`/`onCheckedChange`/`onSelect`, conforme a metadata) com o valor correto.
3. Estado `disabled` (quando a metadata listar essa prop): a interação não dispara o callback.
4. A11y: o role ARIA esperado é resolvível via `getByRole` (ex.: `radio`, `combobox`, `slider`, `tab`) e o nome acessível (via `aria-label`/label associada) está presente.
5. Ref forwarding: se o componente usa `React.forwardRef`, uma ref chega no nó DOM real esperado (verificar no código-fonte se há `forwardRef` antes de escrever este caso — nem todo componente do tier necessariamente usa).

**Tier B (overlay/disclosure):**
1. Smoke: fechado/closed não renderiza o conteúdo do overlay (`queryByRole(...)` retorna `null`), igual ao padrão de `Modal.test.tsx` linha 16-23.
2. Abrir o overlay (via prop controlada `open`/`isOpen` ou via interação, conforme a API do componente) renderiza o conteúdo com o role esperado (`dialog`, `menu`, `tooltip`, etc.) e atributo `aria-*` correspondente (`aria-expanded`, `aria-modal`, conforme aplicável).
3. Fechar via a interação documentada (clique fora, Escape, clique no trigger de novo — verificar a metadata/código de cada componente para saber qual mecanismo ele de fato implementa antes de escrever o teste) dispara o callback de fechamento esperado.
4. Ref forwarding, se aplicável (mesmo critério do Tier A).

**Tier C (display/estrutural):**
1. Smoke: renderiza sem erro com props mínimas.
2. A11y: role/atributo esperado presente (ex.: `Avatar` com `alt` ou `aria-label`, `Progress` com `role="progressbar"` e `aria-valuenow`, `Separator` com `role="separator"` — verificar a metadata de cada um, não assumir).
3. Ref forwarding, somente se o componente usa `React.forwardRef` (a maioria deste tier é função simples sem ref — não inventar um caso se não houver `forwardRef` no código).
4. **Não é necessário** teste de interação para este tier — são componentes sem estado interativo próprio.

> Nenhum critério acima é um teste 1:1 fixo — é um checklist mínimo. Se a metadata do componente documentar um `antiPattern` ou `selectionCriteria` relevante para comportamento (não apenas para uso), considere um teste extra que prove que o componente não viola esse anti-pattern.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - Não escreva testes "genéricos" copiando literalmente o mesmo `it()` para todos os componentes do tier sem ler o componente real — a lista de critérios é um mínimo, não um template para `find-and-replace` do nome.
> - Não corrija bugs de produto encontrados durante o teste — documente na Seção 6 e siga.
> - Não crie `Sidebar.test.tsx` aqui (pertence à T-017).
> - Não crie testes visuais/screenshot — fora de escopo (ver Seção 4).

### Pegadinhas conhecidas
- Componentes com portal (Tier B) podem renderizar fora do container de `render()` do testing-library (via `createPortal`) — `screen.getByRole` ainda funciona porque busca no `document.body` inteiro, mas não assuma que o conteúdo está dentro do container retornado por `render()`.
- `Checkbox.test.tsx` mostra o padrão correto para testar `indeterminate` (propriedade DOM, não atributo HTML) e para testar dois formatos de ref (callback ref e `useRef`) — usar como referência sempre que um componente do Tier A tiver uma prop booleana que reflete em propriedade DOM não-atributo.
- `Modal.test.tsx` mostra o padrão para componentes cujo listener de teclado só é registrado após uma animação de entrada (`flushEnterAnimation`) — componentes do Tier B com transição de entrada podem precisar do mesmo cuidado.
