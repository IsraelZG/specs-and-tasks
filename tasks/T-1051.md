# T-1051 · Guard de commit no `finish`

## 1. Objetivo
Impedir que `finish` transicione uma task para `review` quando o worktree do superapp associado
tem trabalho **não commitado**. Aconteceu 2x em sequência (EST-34 rodada 1, EST-33 rodada 1):
o worker chamou `finish` alegando "gate passou", mas o código só existia como diff não commitado
na worktree — o `integrar-task` não tinha o que mergear. Cada ocorrência custou um round completo
de review→rework.

**Por que o guard não pode viver em `plugin-tasks/src/guards/`:** confirmado lendo
`packages/plugin-tasks/src/service.ts` (superapp) — `TaskService.transition()` opera só sobre o
`Task` lido do storage (SQLite), que só tem `section8_handover` como texto. O serviço **não tem
acesso ao repo de código** (pode nem estar na mesma máquina). O guard tem que viver no **CLI
wrapper** (`tools/scripts/manage-task.mjs`, neste repo Docs), que roda no mesmo host da worktree e
pode inspecionar git antes de delegar ao serviço.

## 3. Escopo (Inputs/Outputs)
- **[UPDATE]** [tools/scripts/manage-task.mjs](file:///C:/Dev2026/Docs/tools/scripts/manage-task.mjs):
  Importar `execFileSync` de `child_process`. No bloco `action === 'finish'`, antes de qualquer processamento e antes de instanciar `TaskService` ou tentar importar o backend, verificar se a worktree existe:
  1. Derivar o path absoluto da worktree: `path.resolve(repoRoot, '..', '.superapp-worktrees', taskId)`.
  2. Se o diretório não existir (retorno falso para `existsSync`) -> Prosseguir normalmente (isento do guard).
  3. Se existir -> Executar e verificar o status do Git:
     - Rodar `git status --porcelain` via `execFileSync('git', ['status', '--porcelain'], {cwd: worktreeDir, encoding: 'utf-8'})`.
     - Rodar `git status --porcelain --untracked-files=all` via `execFileSync('git', ['status', '--porcelain', '--untracked-files=all'], {cwd: worktreeDir, encoding: 'utf-8'})`.
     - Se qualquer um dos comandos retornar saída não-vazia (indicando arquivos modificados, deletados ou untracked) -> Recusar a transição imprimindo erro via `console.error` explicando quais arquivos estão sujos com a instrução clara para commitar antes de finalizar, e sair imediatamente via `process.exit(1)`.
     - Se limpo -> Prosseguir com a validação do Gate e a transição normal do serviço.
- **[CREATE]** [tools/scripts/manage-task-finish-guard.test.mjs](file:///C:/Dev2026/Docs/tools/scripts/manage-task-finish-guard.test.mjs):
  Arquivo de teste funcional em Node.js puro usando `node:assert`. Deve criar uma worktree fake temporária sob `../.superapp-worktrees/T-TEST-GUARD` e testar os 4 cenários da Seção 4.

## 4. Testes
Os testes devem ser executados usando Node.js puro (`node tools/scripts/manage-task-finish-guard.test.mjs`) cobrindo:
1. **Cenário 1 (Worktree suja com modificações ou arquivos untracked):**
   - Setup: Criar diretório `../.superapp-worktrees/T-TEST-GUARD-DIRTY`, iniciar um repositório git (`git init`), e criar um arquivo não commitado (untracked ou modificado).
   - Execução: Rodar `node tools/scripts/manage-task.mjs finish T-TEST-GUARD-DIRTY test-agent "mensagem"` via `execSync` ou `execFileSync`.
   - Asserção: O comando deve sair com exit code 1, escrever mensagens no `stderr` identificando os arquivos sujos, e NÃO chamar a importação do backend nem mudar o status no banco de dados.
2. **Cenário 2 (Worktree limpa):**
   - Setup: Criar diretório `../.superapp-worktrees/T-TEST-GUARD-CLEAN`, iniciar um repositório git, adicionar e commitar um arquivo para que o status fique 100% limpo.
   - Execução: Rodar `node tools/scripts/manage-task.mjs finish T-TEST-GUARD-CLEAN test-agent "mensagem"`.
   - Asserção: O comando deve ultrapassar o guard de dirty worktree com sucesso e falhar no passo subsequente de validação do Gate (por exemplo, "gate: worktree nao e um repo git" ou "gate ausente"), comprovando que o guard de dirty não o bloqueou.
3. **Cenário 3 (Worktree inexistente):**
   - Setup: Garantir que o diretório `../.superapp-worktrees/T-TEST-GUARD-NONE` não exista.
   - Execução: Rodar `node tools/scripts/manage-task.mjs finish T-TEST-GUARD-NONE test-agent "mensagem"`.
   - Asserção: O comando deve prosseguir diretamente (bypassando o guard de dirty e gate) e falhar apenas ao tentar carregar a tarefa no banco de dados com erro de tarefa não encontrada (`TaskNotFoundError`), comprovando que o guard de git não foi ativado para tarefas de tooling.
4. **Cenário 4 (Outras transições não-finish com worktree suja):**
   - Setup: Criar diretório `../.superapp-worktrees/T-TEST-GUARD-DIRTY` com arquivos sujos.
   - Execução: Rodar `node tools/scripts/manage-task.mjs start T-TEST-GUARD-DIRTY test-agent "mensagem"`.
   - Asserção: O comando não deve acionar o guard de dirty worktree, prosseguindo diretamente para o carregamento do banco de dados (onde falhará com tarefa não encontrada ou transição inválida do banco, não pelo guard de git).

## 5. DoD
- O comando `manage-task.mjs finish` impede com segurança transições para tasks que possuem worktrees com arquivos sujos ou untracked.
- Tarefas de tooling (sem worktree correspondente) continuam fluindo sem impedimentos.
- O arquivo de testes `manage-task-finish-guard.test.mjs` roda com sucesso na raiz do repositório Docs.

## 6. Feedback de Especificação
- Não há flag de bypass para o guard de dirty, garantindo consistência com as regras de ambiente do `agile-reviewer.md`.
- A verificação de dirty ocorre antes de carregar o módulo do backend, o que agiliza a rejeição sem carregar o banco de dados.
