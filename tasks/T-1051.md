# T-1051 · Guard de commit no `finish`

## 1. Objetivo
Impedir que `finish` transicione uma task para `review` quando o worktree do superapp associado
tem trabalho **não commitado**. Aconteceu 2x em sequência (EST-34 rodada 1, EST-33 rodada 1):
o worker chamou `finish` alegando "gate passou", mas o código só existia como diff não commitado
na worktree — o `integrar-task` não tinha o que mergear. Cada ocorrência custou um round completo
de review→rework.

**Por que o guard não pode viver em `plugin-tasks/src/guards/`:** confirmado lendo
`packages/plugin-tasks/src/service.ts` (superapp) — `TaskService.transition()` opera só sobre o
`Task` lido do storage (SQLite), que só tem `section8_handover` como texto. O serviço **não tem
acesso ao repo de código** (pode nem estar na mesma máquina). O guard tem que viver no **CLI
wrapper** (`tools/scripts/manage-task.mjs`, neste repo Docs), que roda no mesmo host da worktree e
pode inspecionar git antes de delegar ao serviço.

## 2. Contexto RAG
- `tools/scripts/manage-task.mjs` (CLI atual — delega direto pro `TaskService.transition`, sem
  branching por verbo).
- `packages/plugin-tasks/src/service.ts` (superapp) e `guards/evidenceGuard.ts` — confirma que o
  guard existente só valida texto da Seção 8, não git.
- `CLAUDE.md` Regra 2 (Estado e Handoff) — já atualizada com o invariante que esta task torna executável.
- `.claude/skills/integrar-task/SKILL.md` — "Caminho A-tooling" define quais tasks NÃO têm
  worktree/branch no superapp (tooling do controle); o guard precisa reconhecer esse caso e não
  bloquear.
- Precedentes: `tasks/EST-34.md` §8 (Parecer B1), `tasks/EST-33.md` §8 (Parecer B1) — ambos com o
  achado exato que este guard deveria ter pego automaticamente.

## 3. Escopo (Inputs/Outputs)
- **[UPDATE]** `tools/scripts/manage-task.mjs`: antes de chamar `svc.transition(taskId, action, ...)`
  quando `action === 'finish'`:
  1. Derivar o path convencional da worktree: `path.resolve(repoRoot, '..', '.superapp-worktrees', taskId)`.
  2. Se o diretório **não existir** → prosseguir normalmente (task de tooling-do-controle, sem
     worktree — não bloquear).
  3. Se existir → rodar `git status --porcelain` (via `execFileSync('git', ['status', '--porcelain'], {cwd: worktreePath})`)
     e checar `git status --porcelain --untracked-files=all` também (untracked conta — ver EST-34
     B1, que tinha um pacote inteiro untracked).
  4. Se a saída for não-vazia → **recusar**: `console.error` explicando quais arquivos estão sujos
     e a instrução ("commitar antes de finish"), `process.exit(1)` **sem chamar** `svc.transition`.
  5. Se vazia → prosseguir com a chamada normal ao serviço.
- **[CREATE]** teste (`tools/scripts/tests/manage-task-finish-guard.test.mjs` ou local equivalente
  aos testes já existentes do repo — confirmar convenção de test runner do Docs antes de criar)
  cobrindo os 3 casos da Seção 4.

## 4. Testes
1. Worktree existe + `git status --porcelain` retorna algo (modificado OU untracked) → `finish`
   recusa, exit code 1, **não** chama `svc.transition` (task não muda de status).
2. Worktree existe + limpa → `finish` prossegue e chama `svc.transition` normalmente.
3. Worktree **não existe** (path `.superapp-worktrees/<taskId>` ausente) → `finish` prossegue sem
   checar git (não regride tasks de tooling-do-controle).
4. Outras ações (`start`, `pause`, `approve`, etc.) **não** disparam o guard — só `finish`.

## 5. DoD
`finish` com worktree suja falha de forma clara e não muda o status da task; worktree limpa ou
ausente não regride o fluxo atual.

## 6. Feedback de Especificação
- Sem flag de bypass — mesma disciplina do `agile-reviewer.md` ("Bypass de ambiente é proibido,
  mesmo sem alternativa"). Se o worker achar um caso legítimo de exceção, é `pause`, não bypass.
- Não mover a checagem para dentro do `TaskService`/`plugin-tasks` — já confirmado que o serviço
  não tem acesso ao repo de código; mover pra lá seria reintroduzir o problema por trás de uma
  abstração. O guard fica no CLI do Docs.
- Se a convenção de path da worktree mudar no futuro (hoje `.superapp-worktrees/<taskId>` sibling
  de `Docs`/`superapp`), este guard quebra silenciosamente (path não existe → guard vira no-op).
  Não é regressão de comportamento, mas registre no handoff se notar o path divergindo.
