# T-011 · Incorporar @plataforma/design-system (lib) ao monorepo

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (lógica) e `playwright` (E2E/Frontend)
- **Capacidade-alvo:** haiku

## 1. Objetivo
Trazer a biblioteca de design system (hoje em repo externo `c:\Dev2026\Design System`, pasta `design-system/`) para dentro do monorepo como `packages/design-system`, sob o scope `@plataforma/design-system`, integrada ao workspace pnpm + Turborepo. **Cópia simples** dos arquivos (o histórico git do DS é descartado; upstream `IsraelZG/design-system.git` fica disponível só para consulta). Ao fim, o build da lib (tokens via Style Dictionary + bundle Vite) roda verde dentro do monorepo.

## 4. Estratégia de Testes Estrita (Test-Driven Development)
### Casos de teste (numerados):
1. `pnpm install` na raiz resolve workspace sem erro.
2. `pnpm --filter @plataforma/design-system build` executa `build:tokens` + `build:index` + `build:react` com exit 0.
3. `pnpm --filter @plataforma/design-system validate` (validate-metadata.mjs) retorna exit 0.
4. `pnpm --filter @plataforma/design-system typecheck` (tsc) retorna exit 0.
5. `build/web/theme-light.css` e `build/web/theme-dark.css` são gerados.
6. `src/metadata/components.index.json` é gerado/atualizado.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** use `git subtree`/`submodule` nem traga o `.git` do DS — é cópia simples.
> - **NÃO** crie um segundo lockfile; o monorepo tem um único `pnpm-lock.yaml`.
> - **NÃO** migre o build da lib de Vite para `tsup` — a lib precisa de Tailwind v4 + bundle de CSS (exceção justificada ao padrão, formalizada na T-013).
> - **NÃO** edite o conteúdo dos componentes/tokens — só renomeie pacote/imports e fie a infraestrutura.

1. Copiar `design-system/` externo → `packages/design-system/` (sem `.git`, sem `node_modules`, sem `build`).
2. Em `packages/design-system/package.json`: `name` → `@plataforma/design-system`, `private: true`, `type: module`, scripts trocam `npm` → `pnpm`, manter `exports`. Adicionar `style-dictionary` aos `devDependencies` (se ausente).
3. `tsconfig.json` do pacote: `extends: "../../tsconfig.base.json"`, `compilerOptions.jsx: "react-jsx"`, `compilerOptions.lib: ["ESNext", "DOM", "DOM.Iterable"]`, `include: ["src"]`. Remover `paths` (alias `@ds/*` será substituído). Manter `skipLibCheck: true`.
4. Substituir imports `@ds/*` por caminhos relativos:
   - `@ds/lib/utils` → `../../lib/utils` (a partir de `src/components/*/X.tsx`)
   - `@ds/components/X/Y` → `../X/Y` (a partir de outro componente em `src/components/`)
   - Remover alias `@ds` do `vite.config.ts` (`resolve.alias`) e do `tsconfig.json` (`paths`).
5. Deletar `package-lock.json` (qualquer nível).
6. `pnpm install` na raiz; depois `pnpm --filter @plataforma/design-system build` e `pnpm --filter @plataforma/design-system validate`.
7. Verificar geração de `build/web/theme-{light,dark}.css` e `src/metadata/components.index.json`.

## 6. Feedback de Especificação (Spec Feedback Loop)
> **ATENÇÃO:** Se a T-001 não estiver concluída (sem `tsconfig.base.json`/`pnpm-workspace.yaml`/`turbo.json` na raiz), **PARE** — mude o status para `blocked`.
- **Decisão em aberto — TypeScript version:** fonte usa `typescript: ^6.0.3`, monorepo base (T-001) usa `^5.8.0`. O worker deve usar `^5.8.0` (alinhado ao monorepo) a menos que `tsc` falhe com erro de sintaxe TS 6 — nesse caso, pause e reporte.
- **Decisão em aberto — `verbatimModuleSyntax`:** `tsconfig.base.json` tem `verbatimModuleSyntax: true`. O código-fonte original do DS usa `import React from 'react'` em alguns componentes (import de valor + tipo). Se `tsc` quebrar com `TS1484`, converter imports de tipo para `import type` ou ajustar para `import * as React` conforme necessário. NÃO mude o `tsconfig.base.json`.
- *Demais itens: derivados e sem ambiguidade.*
