## 1. Objetivo
Trazer a biblioteca de design system (hoje em repo externo `c:\Dev2026\Design System`, pasta `design-system/`) para dentro do monorepo como `packages/design-system`, sob o scope `@plataforma/design-system`, integrada ao workspace pnpm + Turborepo. **Cópia simples** dos arquivos (o histórico git do DS é descartado; upstream `IsraelZG/design-system.git` fica disponível só para consulta). Ao fim, o build da lib (tokens via Style Dictionary + bundle Vite) roda verde dentro do monorepo.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** use `git subtree`/`submodule` nem traga o `.git` do DS — é cópia simples.
> - **NÃO** crie um segundo lockfile; o monorepo tem um único `pnpm-lock.yaml`.
> - **NÃO** migre o build da lib de Vite para `tsup` — a lib precisa de Tailwind v4 + bundle de CSS (exceção justificada ao padrão, formalizada na T-013).
> - **NÃO** edite o conteúdo dos componentes/tokens — só renomeie pacote/imports e fie a infraestrutura.

1. Copiar `design-system/` externo → `packages/design-system/` (sem `.git`, sem `node_modules`, sem `build`).
2. Em `packages/design-system/package.json`: `name` → `@plataforma/design-system`, `private: true`, `type: module`, manter `exports`/`scripts` (build:tokens, build:react, build:index, validate, typecheck).
3. `tsconfig.json` do pacote passa a `extends` o `../../tsconfig.base.json`.
4. Normalizar o alias interno `@ds/*` usado no código-fonte original copiado. Substituir as importações (find and replace) por caminhos relativos padrão (ex: `../../lib/...`). Não reconfigurar plugins de alias no Vite ou TS. Garantir que `tsc` e `vite build` resolvam corretamente.
5. `pnpm install` na raiz; depois `pnpm --filter @plataforma/design-system build` e `... validate`.
6. Verificar geração de `build/web/theme-{light,dark}.css` e `src/metadata/components.index.json`.
