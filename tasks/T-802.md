# T-802 · Remontagem verificada por Merkle/InfoHash (fronteira de confiança do BlobStorage)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo — pacote alvo: `packages/media/`
- **Test Runner:** `vitest` (Node puro)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Adicionar a **camada de verificação de integridade** que falta na T-801. `Chunker.reassemble` (T-801)
apenas tolera entrega fora de ordem e valida sanidade de índices (`0..n-1`) — ele **não** detecta
reordenação maliciosa nem adulteração de ciphertext, porque AES-256-GCM autentica o conteúdo de cada
chunk isoladamente mas **não liga um chunk à sua posição**. Esta task implementa a **fronteira de
confiança**: verificar os chunks recebidos contra o **InfoHash** (raiz da Merkle Tree sobre os
`ciphertextHash`, em ordem de `index`) **antes** de confiar nos bytes — e só então remontar.

> **Contexto da decisão (T-801 §6, 2026-06-23):** decisão "C agora + B como T-802". `reassemble` ficou
> sendo o helper local tolerante; a garantia contra adulteração de ordem/conteúdo é desta task.

*(Derivado de `docs/caderno-3-sdk/05-media-transport-plane.md`: "o **InfoHash** é calculado **sobre o
ciphertext** — nunca sobre o plaintext" (linha 22) e "validação por chunk: a tag autentica cada peça
antes de entregar ao player" (linha 30). Verificado no disco 2026-06-23.)*

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/media/src/blob/verifyReassemble.ts ---
import type { AesKey, Sha256Digest } from '@plataforma/crypto'; // T-101
import type { EncryptedChunk } from './Chunker.js';             // T-801

/** Erro de integridade: índices inconsistentes, ciphertext adulterado, ou InfoHash divergente. */
export class BlobIntegrityError extends Error {
  /** Motivo legível por máquina, para o caller decidir (rejeitar peer, retentar, etc.). */
  readonly reason:
    | 'index_not_contiguous'     // o conjunto de `index` ≠ 0..n-1 (faltando/duplicado)
    | 'ciphertext_hash_mismatch' // sha256(ciphertext) ≠ ciphertextHash declarado em algum chunk
    | 'infohash_mismatch'        // raiz Merkle recomputada ≠ expectedInfoHash (inclui reordenação)
    | 'empty';                   // lista de chunks vazia
  constructor(reason: BlobIntegrityError['reason'], message: string);
}

/**
 * Verifica a integridade do conjunto de chunks contra o InfoHash esperado, SEM decifrar.
 * Útil para validar antes de aceitar/baixar (não precisa da chave).
 * Passos (todos obrigatórios, nesta ordem):
 *  1. `chunks` não vazio (senão `BlobIntegrityError('empty')`).
 *  2. Sanidade: ordenado por `index`, o conjunto deve ser exatamente `0..n-1`
 *     (contíguo, único) — senão `BlobIntegrityError('index_not_contiguous')`.
 *  3. Para cada chunk: `sha256(chunk.ciphertext)` deve === `chunk.ciphertextHash`
 *     — senão `BlobIntegrityError('ciphertext_hash_mismatch')` (detecta ciphertext adulterado
 *     e `ciphertextHash` mentiroso, ANTES de qualquer decifração).
 *  4. `MerkleTree.build(ordenado.map(c => c.ciphertextHash))` (T-801) → a raiz deve === `expectedInfoHash`
 *     — senão `BlobIntegrityError('infohash_mismatch')` (detecta reordenação maliciosa e chunk trocado).
 * Retorna `true` se íntegro. NUNCA retorna `false` — divergência sempre lança (com `reason`).
 */
export function verifyChunks(
  chunks: EncryptedChunk[],
  expectedInfoHash: Sha256Digest,
): Promise<true>;

/**
 * Verifica (via `verifyChunks`) e SÓ ENTÃO remonta o plaintext (via `Chunker.reassemble` de T-801).
 * É a única forma segura de reconstruir um blob vindo de fonte não confiável (P2P/torrent).
 * Lança `BlobIntegrityError` (sem decifrar) se a verificação falhar; caso contrário devolve os bytes.
 */
export function verifyAndReassemble(
  chunks: EncryptedChunk[],
  key: AesKey,
  expectedInfoHash: Sha256Digest,
): Promise<Uint8Array>;
```

### Casos de teste (numerados — 1:1 em `verifyReassemble.test.ts`)
1. **Roundtrip íntegro:** `process(data, key)` (T-801) → `infoHash = (await MerkleTree.build(chunks.map(c=>c.ciphertextHash))).root` → `verifyAndReassemble(chunks, key, infoHash)` devolve bytes === `data`.
2. **InfoHash errado:** conjunto válido, mas `expectedInfoHash` com 1 byte trocado → `BlobIntegrityError` com `reason:'infohash_mismatch'`.
3. **Reordenação maliciosa (o caso que `reassemble` sozinho NÃO pega):** trocar os campos `index` de dois chunks (mantendo o conjunto `0..n-1`) → `verifyAndReassemble` lança `infohash_mismatch` (a raiz Merkle depende da ordem dos `ciphertextHash`).
4. **Ciphertext adulterado (1 byte):** adulterar 1 byte do `ciphertext` de um chunk (sem corrigir `ciphertextHash`) → `ciphertext_hash_mismatch`, **antes** de qualquer decifração.
5. **Índice faltando/duplicado:** remover um chunk do meio (conjunto ≠ `0..n-1`) → `index_not_contiguous`.
6. **`ciphertextHash` mentiroso:** chunk com `ciphertextHash` que não bate `sha256(ciphertext)` → `ciphertext_hash_mismatch`.
7. **`verifyChunks` sem decifrar:** retorna `true` para conjunto íntegro (não precisa de chave); lança `BlobIntegrityError` para as adulterações dos casos 2–6; lista vazia → `reason:'empty'`.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** decifre antes de verificar — a ordem é sanidade → ciphertextHash → InfoHash → (só então) `reassemble`. Decifrar primeiro anula a fronteira de confiança.
> - **NÃO** reimplemente a Merkle Tree nem o slice/encrypt — reutilize `MerkleTree.build` e `Chunker.reassemble` da T-801.
> - **NÃO** compare `Sha256Digest` com `===` (são `Uint8Array`); compare byte a byte — `a === b` em `Uint8Array` testa identidade de referência, não conteúdo.
> - **NÃO** retorne `false` em divergência — sempre lance `BlobIntegrityError` com o `reason` específico.
> - **NÃO** acesse Key Vault, UCAN, rede ou SQLite — a chave chega pronta por parâmetro.

### Pegadinhas conhecidas
- `MerkleTree.build`/`verifyProof` são **async** (`Promise`) na T-801 (decisão 2026-06-23) — use `await`. `getProof`/`root` são síncronos.
- A raiz Merkle **depende da ordem** dos `ciphertextHash`. Ordene os chunks por `index` **antes** de mapear para `ciphertextHash` (igual ao que `reassemble` faz), senão a comparação de InfoHash falha em entrega fora de ordem legítima.
- Comparação de digests: escreva um helper local `equalBytes(a, b)` (compara `length` + cada byte). Não use `Buffer` (mantenha Node-puro/`Uint8Array`, compatível com browser).
- `verifyChunks` **não** recebe a chave — é verificação de hash, não decifração. Só `verifyAndReassemble` recebe `key`.
- Lint `@typescript-eslint/no-non-null-assertion` está ativo — sem `!`; use guardas explícitas (como T-801 faz em `MerkleTree.ts`).
