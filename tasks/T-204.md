## 1. Objetivo
Implementar `NetworkAdapterPort` *(derivado de `tasks/T-004.md` §1 + `tasks/T-109.md` §1)* usando
a interface `WebSocket` nativa do browser e a biblioteca `ws` no Node.js *(derivado de
`docs/plano-de-implementacao.md` §5 T-204)*. O adapter provê transporte binário de mensagens
cruas (`Uint8Array`) sobre WebSocket — o handshake Noise_XX (T-202), o wire format (T-201) e
demais camadas rodam **sobre** este adapter, não dentro dele.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/transport/src/websocket.ts 
---

import { NetworkAdapterPort, PeerId, WireData, MessageHandler } from '@plataforma/protocol';

/** Configuração de conexão — mapeia PeerId → URL WebSocket. */
export interface WsAdapterConfig {
  /** URLs dos peers conhecidos (PeerId → 'ws://host:port'). Usado por connect(). */
  readonly peerUrls?: ReadonlyMap<PeerId, string>;
  /** Porta para listen() (servidor WebSocket). 0 = SO escolhe. */
  readonly listenPort?: number;
  /** Host para listen(). Default: '127.0.0.1'. */
  readonly listenHost?: string;
}
```

> **✅ `WsAdapterConfig` confirmado (Seção 6, decisão 3):** `peerUrls` (mapa explícito PeerId→URL)
> é o mecanismo de resolução. Conexões **entrantes** recebem um **peerId provisório, escopo-de-conexão**
> (decisão 6) — a identidade real (`DevicePeerId`) é estabelecida pela camada Noise_XX (T-202),
> não pelo adapter.

```ts
/**
 * Implementação de NetworkAdapterPort sobre WebSocket.
 *
 * - Node.js: usa a biblioteca `ws`
 * - Browser: usa a global `WebSocket` nativa
 *
 * @implements {NetworkAdapterPort}
 *   (contrato de `packages/protocol/src/ports.ts` — T-004 §1 + T-109 §1)
 */
export class WsAdapter implements NetworkAdapterPort {
  constructor(config: WsAdapterConfig);

  /** Conecta a um peer remoto via WebSocket. Resolve quando a conexão estiver aberta. */
  connect(peerId: PeerId): Promise<void>;

  /** Inicia servidor WebSocket para aceitar conexões entrantes. */
  listen(): Promise<void>;

  /** Envia dados binários para um peer conectado. Rejeita se peer não estiver conectado. */
  send(peerId: PeerId, data: WireData): Promise<void>;

  /**
   * Registra handler para mensagens recebidas.
   * Suporta múltiplos handlers simultâneos (broadcast).
   * @returns função de unsubscribe idempotente.
   *   *(derivado de tasks/T-109.md §1 — multi-subscriber)*
   */
  onMessage(handler: MessageHandler): () => void;

  /** Encerra todas as conexões e o servidor. Limpa todos os handlers. Idempotente. */
  close(): Promise<void>;
}
```

## 6. Feedback de Especificação (Spec Feedback Loop)

> **✅ DECISÕES RESOLVIDAS (2026-06-24 — arquiteto). Zero decisões em aberto.**

1. **Reconexão exponencial → ADIADA para M4 (opção c).** A Seção 5 já proíbe "lógica de reconexão
   pesada"; nenhum doc de arquitetura dita a topologia de tempos (base/max/jitter) e fixar valores
   arbitrários cria defaults errados. **Esta task NÃO implementa reconexão.** *Seam futureproof:*
   `connect(peerId)` é **re-chamável** (idempotente por peer) e, ao detectar `close` do socket, o
   adapter marca o peer como desconectado (`send` passa a rejeitar — caso 7). Um supervisor de
   reconexão (M4) apenas re-invoca `connect()`; nenhuma API nova é necessária agora. "Reconexão
   exponencial" foi **removida do escopo** desta task.

2. **Backpressure básico → delegar ao `bufferedAmount` (opção a).** Tanto `ws` (Node) quanto a
   `WebSocket` do browser expõem `bufferedAmount` (isomórfico). `send()` aplica um guard de
   high-water-mark: se `bufferedAmount` exceder um limite, aguarda o dreno antes de escrever (ou
   rejeita se o socket estiver fechando). **Sem fila custom** (YAGNI — "básico" = nativo da lib).

3. **PeerId → URL → mapa explícito `peerUrls` no config (opção a).** Mantido como na Seção 1.
   Convenção de URL e multiaddr são prematuros (multiaddr/libp2p-addressing chega depois). Mapa
   explícito é o mais simples e testável.

4. **`listen()` → apenas `ws://` local; `wss://` adiado (opção a).** Terminação TLS é preocupação
   de deploy (proxy reverso na frente do peer do sistema), não do adapter em v0. Bind em
   `listenHost`/`listenPort` com `ws://`. `wss://` vira opção de config futura quando houver
   necessidade real.

5. **Node vs Browser → conditional exports via campo `browser` do `package.json` (opção b).** É a
   forma idiomática e amigável a bundlers (Vite/webpack respeitam `browser`); `import()` dinâmico
   com try/catch quebra análise estática de bundler. O construtor do `WebSocket` vem de um módulo
   fino `wsImpl` cuja resolução troca por plataforma; a lógica do `WsAdapter` fica num único arquivo.
   *(ver Seção 3 — `wsImpl.ts` Node + `wsImpl.browser.ts` browser + campo `browser` no package.json).*

6. **Identificação de peer entrante → delegar ao Noise_XX; peerId provisório no adapter (opção c).**
   **Decisão arquitetural canônica:** `caderno-5-transport/01 §2.2.1` afirma que o `DevicePeerId` é
   extraído **pelo handshake Noise_XX** (o adapter "não deveria ler chaves criptográficas"). Para
   conexões **entrantes** (`listen()`), o adapter atribui um **peerId provisório, escopo-de-conexão**
   (ex.: `ws-conn-<n>` ou o address remoto), entregue em `onMessage(provisionalPeerId, data)`. O
   binding de identidade real (`DevicePeerId`) é feito pela camada Noise_XX (T-202), que reconcilia.
   `send(provisionalPeerId, …)` funciona porque pós-`connect`/aceite a conexão é point-to-point (o
   peerId é a alça do socket, não a identidade — consistente com a observação do handover de T-202).

7. **`close()` → rejeitar as Promises de `send` pendentes (opção b).** Contrato honesto: ao
   `close()`, qualquer `send()` não resolvido **rejeita** (fail-fast, sem perda silenciosa).
   Descartar em silêncio (a) esconde perda de dados; drenar com timeout (c) é over-engineering para
   v0.

> rascunho. Resoluções fundadas em `caderno-5-transport/01 §2.2.1` (item 6), Seção 5 desta task
