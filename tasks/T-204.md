# T-204 · Adapter WebSocket

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro — usa `ws` no teste)
- **Capacidade-alvo:** haiku

## 1. Objetivo
Implementar `WebSocketAdapter` no pacote `@plataforma/transport`, implementando `NetworkAdapterPort` de T-004. Isomórfico: usa `ws` em Node, `WebSocket` nativo no browser. Encapsula frames do wire format (T-201) sem inspecionar payload.

### Contratos exatos
```ts
// --- packages/transport/src/websocket.ts ---
import { NetworkAdapterPort, PeerId, WireData, MessageHandler } from '@plataforma/protocol'; // T-004

export class WebSocketAdapter implements NetworkAdapterPort {
  constructor(url: string);
  connect(peerId: PeerId): Promise<void>;
  listen(): Promise<void>;
  send(peerId: PeerId, data: WireData): Promise<void>;
  onMessage(handler: MessageHandler): void;
  close(): Promise<void>;
}
```
- `send`: serializa frame via `encodeFrame` (T-201) e envia como `ArrayBuffer`.
- `onMessage`: recebe `ArrayBuffer`, decodifica via `decodeFrame` (T-201), invoca handler com `peerId`.
- `listen` no Node: cria `ws.Server`; no browser: no-op (cliente apenas).

## 4. Estratégia de Testes
- [x] **Framework:** Vitest (Node puro). Usar `ws` para criar servidor e cliente no mesmo processo.
- [x] **Ambiente:** `pnpm --filter @plataforma/transport test`.

Casos de teste:
1. `connect` + `send` + `onMessage` roundtrip: servidor recebe o que cliente enviou.
2. `listen` aceita conexão e `onMessage` registra callback.
3. `close` no cliente → servidor detecta desconexão (evento `close`).
4. `send` após `close` → erro controlado.
5. `encodeFrame`/`decodeFrame` integrado: dados MessagePack chegam íntegros via WebSocket.
6. `connect` para URL inválida → rejeita Promise (não crasha).
