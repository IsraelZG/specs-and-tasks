# T-009 · Superfície de Controle e Telemetria (Headless)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest`

## 1. Objetivo
Implementar a Superfície de Controle (`ControlPort`) que fornece observabilidade e injeta comandos no Plano de Dados (motor P2P) sem acoplar a lógica à interface visual. Construir o adaptador `WebSocketControlClient` para o browser se conectar ao Hub de QA (`system-peer`).

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Vitest
- [ ] **Métricas/Cobertura:** Testar push de telemetria, parsing de comandos e resolução assíncrona por `correlation_id`.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** misture este canal WS com o transporte de dados do P2P (`T-204`). Este é o plano de controle.
> - **NÃO** instancie interfaces React.

1. **[ControlPort]** Em `@plataforma/core`, defina a interface `ControlPort` com métodos genéricos para particionar `NetworkAdapter` (derrubar conexões mantendo o processo vivo) e enviar eventos estruturados de telemetria (Publish/Subscribe).
2. **[Outbound WS Adapter]** Em `@plataforma/client-sdk`, implemente o `WebSocketControlClient`.
   - Ele deve discar para uma URL de controle (ex: `ws://localhost:3000/control`).
   - Fazer handshake com `client_id` efêmero e um token de sessão escopado.
   - Ouvir mensagens do servidor contendo `{ corr_id, cmd, payload }`.
   - Despachar o comando nativamente na `ControlPort` residente.
   - Retornar o resultado para o servidor com o mesmo `corr_id`.
3. **[Telemetria Ativa]** Habilite que eventos internos da `ControlPort` (ex: "peer connected", "sync complete") sejam empurrados (push) proativamente via WebSocket de volta para o servidor, garantindo observabilidade contínua (inclusive quando a rede de dados estiver particionada artificialmente).
4. Rode e passe nos testes unitários com um WebSocket Server em memória (`ws`).
