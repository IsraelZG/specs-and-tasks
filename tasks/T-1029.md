# T-1029 · TaskService side-effects: trava-decomposto + autoPromoteDependents + auto-close-pai

- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Pacote-alvo:** `apps/nexus-backend` (rode SÓ `pnpm --filter nexus-backend build|test`)
- **Test Runner:** `vitest` (Node puro)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Adicionar quatro *side-effects* automáticos ao `TaskService.transition()`, todos idempotentes, que tornam o pipeline auto-propagante (fim da disciplina manual que travou T-006 e ORQ-02):
1. **Trava de start em pai decomposto:** `start` recusa tasks com `subtasks` não-vazio ou status `draft:decomposed`.
2. **Auto-promote on harden:** ao chegar em `draft:hardened` (via `harden`/`decide`), se a árvore de dependências está toda `done`, promove sozinha para `ready`.
3. **`autoPromoteDependents`:** ao `approve → done`, promove para `ready` todo dependente que está em `draft:hardened` e cujas deps agora estão todas `done`.
4. **`parentAutoClose`:** ao `approve → done` de uma filha, se TODAS as filhas do pai (`parent.subtasks`) estão `done`, encerra o pai (`→ done`).

Depende da máquina de estados da **T-1028** (verbos e status já existentes). NÃO mexe em scripts `.mjs` nem migra dados (T-1030).

### Contratos exatos

```ts
// Novo erro em task.types.ts
export class DecomposedParentStartError extends TaskError {
  constructor(id: string) {
    super(`Task '${id}' é um pai decomposto (tem subtasks ou status draft:decomposed) e não pode ser iniciada. Trabalhe as filhas.`);
    this.name = 'DecomposedParentStartError';
  }
}

// Em task.service.ts — helpers privados:

/** true se TODAS as deps (por id) estão 'done'. Ids inexistentes contam como NÃO-done (conservador). */
private depsAllDone(depIds: string[], byId: Map<string, TaskRecord>): boolean;

/** Transição interna SEM gate de papel — usada pelos auto-side-effects (promote/close mecânicos).
 *  Grava status + ledger + Log, regenera INDEX. Idempotente: no-op se já está em `to`. */
private autoTransition(id: string, to: TaskStatus, logLabel: string, reason: string, agent = 'system'): void;
```

**Regras de cada side-effect (todos NO FIM de `transition()`, depois do write de status/ledger/log, e idempotentes):**

- **(1) Trava start** — NO INÍCIO de `transition()`, ANTES de aplicar a regra: se `action === 'start'` e (`fm.subtasks?.length` OU `current === 'draft:decomposed'`) → lança `DecomposedParentStartError`. (A T-1028 já removeu `draft:*` de `start.from`, então isto cobre o caso raro de um pai com subtasks que de algum modo esteja `ready`/`rework`.)

- **(2) Auto-promote on harden** — se a transição recém-aplicada resultou em `to === 'draft:hardened'` (ações `harden` ou `decide`): carregue todas as tasks, e se `depsAllDone(fm.dependencies, byId)` → `autoTransition(id, 'ready', '[Auto-promovida]', 'deps todas done')`.

- **(3) autoPromoteDependents** — se `action === 'approve'` (task foi a `done`): para cada task `d` do backlog cujo `dependencies` inclui `id`, se `d.status === 'draft:hardened'` e `depsAllDone(d.dependencies, byId)` → `autoTransition(d.id, 'ready', '[Auto-promovida]', 'dep ' + id + ' concluída')`.

- **(4) parentAutoClose** — se `action === 'approve'` e a task tem `fm.parent`: carregue o pai; se todas as `parent.subtasks` estão `done` (checa cada uma no backlog) → `autoTransition(parent.id, 'done', '[Auto-encerrado]', 'todas as filhas done')`. (O pai pode estar em `review` ou `draft:decomposed` — o `autoTransition` não valida `from`, é mecânico.)

**Ordem no `approve`:** aplica approve→done (existente) → merge de branch (existente) → **(3) autoPromoteDependents** → **(4) parentAutoClose**. **Decisão fechada — sem recursão:** `autoTransition` NÃO re-dispara os side-effects (evita reentrância). Dependentes de um pai recém-fechado são pegos na próxima transição ou pelo safety-net batch `arquiteto-promover`. Marque com `// ponytail: auto-close não recursa; batch cobre a cauda`.

### Casos obrigatórios (`task.service.test.ts`):
1. **Trava start:** task com `subtasks: ['X-a','X-b']` em `ready` → `start` lança `DecomposedParentStartError`.
2. **Trava start:** task em `draft:decomposed` → `start` lança `DecomposedParentStartError` (na prática já barrado por `from`, mas o erro específico deve vir).
3. **Auto-promote on harden (deps done):** task `draft:triaged` com dep única `done` → `harden` → status final `ready` (não `draft:hardened`), com Log `[Endurecido]` seguido de `[Auto-promovida]`.
4. **Auto-promote on harden (deps pendentes):** task `draft:triaged` com dep `in_progress` → `harden` → fica em `draft:hardened` (não promove).
5. **autoPromoteDependents:** task A (dep de B) em `review`; B em `draft:hardened` com A como única dep → `approve(A)` → B vira `ready`.
6. **autoPromoteDependents (dep parcial):** B depende de A e C; C ainda `in_progress` → `approve(A)` → B continua `draft:hardened`.
7. **parentAutoClose:** pai P (`subtasks: [F1,F2]`, status `review`); F1 já `done`; `approve(F2)` → P vira `done` com Log `[Auto-encerrado]`.
8. **parentAutoClose (filha pendente):** pai P (`subtasks:[F1,F2]`); F1 ainda `review`; `approve(F2)` → P NÃO fecha.
9. **Idempotência:** rodar `approve` num cenário já resolvido não duplica transições nem quebra (autoTransition no-op se já em `to`).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** use o `transition()` público (com gate de papel) para os auto-movimentos — use o `autoTransition` privado, senão o gate de reviewer barra o auto-close.
> - **NÃO** deixe `autoTransition` recursar nos side-effects (reentrância infinita). Ele só grava status/ledger/log.
> - **NÃO** toque em scripts `.mjs`, migração ou docs (T-1030/T-1031).

### Pegadinhas conhecidas
- **`transition()` já é `async`** (faz merge de branch). Os side-effects são síncronos (só I/O de arquivo) — não precisam de `await`, mas mantenha a assinatura `async`.
- **Carregar o backlog uma vez** por transição (`this.list()` ou leitura direta do dir) e montar `Map<id, record>`; não releia por task (O(n²)).
- **`subtasks`/`parent`** podem não estar tipados em `TaskFrontmatter` — adicione como opcionais. Leia via o parser de frontmatter já usado (`parseFrontmatterNaive` ou `gray-matter`).
- **Anti-drift:** cada `autoTransition` grava ledger + status juntos — não gera drift. Mas garanta que o ledger recebe a entrada ANTES de qualquer outra leitura de status no mesmo `transition()`.
