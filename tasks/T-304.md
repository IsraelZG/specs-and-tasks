## 1. Objetivo
Fornecer guarda de execução para reconciliação concorrente no pacote `@plataforma/core`: budget de tempo + retomada de range + **diff complementar** quando o usuário escreve localmente durante o sync RBSR
*(derivado de `docs/plano-de-implementacao.md` §6 T-304; spec herdada da decomposição T-304 ↔ T-302)*.

> **✅ `hardened` (2026-06-29).** As 5 decisões da §6 foram fechadas pelo arquiteto, aterradas no
> código concreto: a **lineage é append-only imutável** (`nodes.hlc` em todo nó, sem `updated_at`,
> sem DELETE — `schema.ts`) e **não há adapter SQLite/WAL ainda** (só mock de `StoragePort`). Isso
> torna o snapshot uma simples **marca d'água de HLC** e tira WAL/migrations/triggers do escopo.

### Contratos exatos
```ts
// --- packages/core/src/sync/concurrentGuard.ts ---
import type { StoragePort, ULID } from '@plataforma/protocol'; // T-004 / T-102
import type { ClockPort } from '@plataforma/protocol';         // T-003 (budget determinístico em teste)

/** Marca d'água do início do sync. Como a lineage é append-only+HLC, ela basta como "snapshot". */
export interface GuardSnapshot {
  checkpointHlc: string;       // HLC máximo da lineage no início do sync (string HLC, como nodes.hlc)
  createdAt: number;           // ClockPort.now() no beginSync
}

/** Mudanças locais ocorridas DURANTE o sync (tudo é nó novo com hlc > checkpoint). */
export interface DiffComplement {
  added: ULID[];               // ids de nós com hlc > checkpoint cujo entity ainda não existia no snapshot
  modified: ULID[];            // ids de nós com hlc > checkpoint cujo entity_head avançou
  deleted: ULID[];             // ids de nós-tombstone com hlc > checkpoint
}

export interface GuardOptions {
  budgetMs?: number;           // tempo máximo para o sync (default 30_000)
}

export class ConcurrentReconciliationGuard {
  constructor(storage: StoragePort, clock: ClockPort, opts?: GuardOptions);

  /** Lê a marca d'água de HLC (SELECT MAX(hlc) FROM nodes) e o relógio. NÃO abre transação longa. */
  beginSync(): Promise<GuardSnapshot>;

  /** true se (clock.now() - snapshot.createdAt) > budgetMs (caller deve abortar). */
  checkBudget(snapshot: GuardSnapshot): boolean;

  /** Diff complementar via HLC range scan: nós com hlc > snapshot.checkpointHlc. */
  computeDiff(snapshot: GuardSnapshot): Promise<DiffComplement>;

  /** Mescla o resultado RBSR com o diff local sem duplicar (união por id, ordem estável). */
  applyDiff(result: ULID[], diff: DiffComplement): ULID[];

  /** Registra o range RBSR pendente para retomada (estado in-memory na instância). */
  abortSync(lastRangeProcessed: { lo: ULID; hi: ULID }): void;

  /** Limpa o estado de retomada; retorna o range pendente se abortSync foi chamado. */
  commitSync(): { lo: ULID; hi: ULID } | null;
}
```
- `beginSync`: lê `SELECT MAX(hlc) AS hlc FROM nodes` via `StoragePort.exec` + `clock.now()`. **Sem** `BEGIN DEFERRED` (a imutabilidade da lineage dispensa isolamento transacional longo).
- `checkBudget`: comparação de tempo via `ClockPort` (determinístico sob `VirtualClock` no teste).
- `computeDiff`: `SELECT id, node_type, ... FROM nodes WHERE hlc > :checkpoint` + join em `entity_members`/`entity_heads` para classificar added/modified/deleted (tombstone = `node_type` de remoção).
- `applyDiff`: união determinística do resultado RBSR com `added`+`modified` (remove `deleted`).
- **WAL checkpoint e persistência de retomada estão FORA do escopo** (ver §6 D2/D5) — retomada é in-memory; WAL é responsabilidade do futuro adapter SQLite.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest. Storage via mock de `StoragePort` (ou SQLite `:memory:`); relógio via `VirtualClock` (T-003) para budget determinístico.
- [x] **Ambiente:** `pnpm --filter @plataforma/core test`.

Casos (enumerados — derivados do contrato fechado):
1. `beginSync` retorna `checkpointHlc` = MAX(hlc) da lineage atual e `createdAt` = `clock.now()`.
2. `beginSync` numa lineage vazia → `checkpointHlc` = HLC zero/sentinela (sem throw).
3. `checkBudget` com `budgetMs=50`: false logo após `beginSync`; true após avançar o `VirtualClock` 60ms.
4. `computeDiff` detecta **added**: nó de entity nova inserido após o snapshot aparece em `added`.
5. `computeDiff` detecta **modified**: novo head de entity pré-existente aparece em `modified`, não em `added`.
6. `computeDiff` detecta **deleted**: nó-tombstone após o snapshot aparece em `deleted`.
7. `computeDiff` ignora nós com `hlc <= checkpointHlc` (já estavam no snapshot).
8. `applyDiff` mescla resultado RBSR + `added`/`modified` sem duplicar e remove `deleted`; ordem estável.
9. `abortSync(range)` então `commitSync()` retorna o range pendente; `commitSync()` sem abort → `null`.

## 6. Feedback de Especificação

**✅ Status: `hardened`** — as 5 decisões foram fechadas pelo arquiteto em 2026-06-29, **aterradas
no código concreto** (não em suposição). Registradas como fonte canônica daqui pra frente:

### DECIDIDO 1 — Qual "B-Tree" está sendo sincronizada?
**Resolução:** **marca d'água de HLC sobre a lineage append-only.** O snapshot NÃO é a `BTree`
in-memory (T-301) nem uma transação SQL longa; é `MAX(hlc)` da tabela `nodes`. A `BTree` é
re-derivada pelo orquestrador (T-302a). **Fonte:** `packages/core/src/schema.ts` — `nodes.hlc TEXT
NOT NULL` em todo nó + nós imutáveis assinados (sem UPDATE/DELETE) ⇒ a marca d'água identifica
univocamente "o que existia no início do sync". `BEGIN DEFERRED` é desnecessário.

### DECIDIDO 2 — Onde vive o estado de retomada?
**Resolução:** **in-memory na instância do guard** (v0). `checkpointHlc`/`lastRange` ficam em campos
da classe; retomada funciona dentro da sessão. **Sem migration.** Retomada cross-restart é upgrade
futuro (YAGNI — o cenário de crash no meio do sync pode esperar). **Fonte:** `schema.ts` não tem
tabela de sync e não vamos adicionar uma agora.

### DECIDIDO 3 — Mecanismo de diff (added/modified/deleted)
**Resolução:** **HLC range scan puro** — `SELECT ... FROM nodes WHERE hlc > :checkpointHlc`.
**Dispensa `updated_at` (sem migration v4) e triggers.** **Fonte:** a lineage é append-only
(`schema.ts`: sem `updated_at`, sem coluna `deleted`, nós imutáveis): "modificar" = novo nó com head
avançado em `entity_heads`; "deletar" = nó-tombstone. Os três casos são nós novos com `hlc >
checkpoint`, classificados por join em `entity_members`/`entity_heads`.

### DECIDIDO 4 — O que é "range"?
**Resolução:** **RBSR ULID range `{lo: ULID, hi: ULID}`** — consistente com D1 (a B-Tree RBSR é o que
se sincroniza). **Fonte:** `BTree.reconcile` opera por `IdRange{lo,hi}` em ULIDs (T-301).

### DECIDIDO 5 — WAL checkpoint
**Resolução:** **FORA DO ESCOPO de T-304.** Não há adapter SQLite concreto nem WAL no código (busca
em todo o monorepo: zero `PRAGMA`/`journal_mode`; único `StoragePort` é o mock de
`packages/protocol/tests/ports.test.ts`). Adicionar `wal_checkpoint` agora seria acoplar a um backend
inexistente. A gestão de WAL/checkpoint vai para a **task do adapter SQLite** (ver T-DS… nova task de
fundação de storage). T-304 **não** toca `StoragePort` (T-004 intacto). **Fonte:** `StoragePort` só
expõe `exec`/`transaction`/`migrate`.

**Nenhuma decisão em aberto.** Pronta para `/arquiteto-promover` → `ready` → execução por Sonnet.
