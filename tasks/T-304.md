# T-304 · ConcurrentReconciliationGuard

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro)

## 1. Objetivo
Fornecer guarda de execução para reconciliação concorrente no pacote `@plataforma/core`: budget de tempo + checkpoint WAL + retomada de range + diff complementar quando o usuário escreve durante o sync
*(derivado de `docs/plano-de-implementacao.md` §6 T-304; spec herdada da decomposição T-304 ↔ T-302)*.

> **⛔ STATUS ATUAL: `blocked-decision`** — a spec herdada referencia "snapshot da B-Tree" mas o
> repositório concreto (T-301, T-106, T-302a) não estabelece com clareza **qual** "B-Tree" está
> sendo sincronizada. Ver §6 para as 5 decisões em aberto que travam o endurecimento.

### Contratos exatos
```ts
// --- packages/core/src/sync/concurrentGuard.ts ---
import { StoragePort } from '@plataforma/protocol'; // T-004
import { ULID } from '@plataforma/protocol'; // T-102

export interface GuardSnapshot {
  entities: ULID[];            // snapshot dos entity_ids no momento do BEGIN
  checkpointHlc: number;       // HLC máximo no snapshot
  createdAt: number;           // timestamp do snapshot
}

export interface DiffComplement {
  added: ULID[];               // nós inseridos durante o sync
  modified: ULID[];            // nós modificados durante o sync
  deleted: ULID[];             // nós deletados durante o sync
}

export interface GuardOptions {
  budgetMs: number;            // tempo máximo para sync (default 30s)
  walCheckpointMode: 'TRUNCATE' | 'PASSIVE'; // default TRUNCATE
}

export class ConcurrentReconciliationGuard {
  constructor(storage: StoragePort, opts?: GuardOptions);

  /** Inicia transação DEFERRED e captura snapshot da B-Tree. */
  beginSync(): Promise<GuardSnapshot>;

  /** Verifica budget de tempo. Se excedido, aborta e faz wal_checkpoint. */
  checkBudget(): boolean;

  /** Computa diff complementar: o que mudou desde o snapshot. */
  computeDiff(snapshot: GuardSnapshot): Promise<DiffComplement>;

  /** Aplica diff complementar ao resultado da reconciliação. */
  applyDiff(result: ULID[], diff: DiffComplement): ULID[];

  /** Aborta transação e executa wal_checkpoint(TRUNCATE). */
  abortSync(lastRangeProcessed: string): Promise<void>;

  /** Confirma transação e retoma ranges pendentes se houver. */
  commitSync(): Promise<void>;
}
```
- `beginSync`: `BEGIN DEFERRED` via `StoragePort`, captura snapshot da B-Tree (lista de entity_ids + HLC máximo).
- `checkBudget`: mede tempo decorrido desde `beginSync`; se exceder `budgetMs`, retorna true (caller deve abortar).
- `computeDiff`: consulta nós inseridos/modificados/deletados desde o snapshot, comparando com estado atual do banco.
- WAL checkpoint: evita crescimento ilimitado do WAL durante sync lento.
- Cliente da reconciliação usa `applyDiff` para mesclar resultado RBSR com as mudanças locais concorrentes.

## 6. Feedback de Especificação

**⛔ Status: `blocked-decision`** — 5 decisões arquiteturais precisam ser fechadas antes de
endurecer o contrato. Cada uma está mapeada com a fonte canônica e a opção pragmática quando
aplicável.

### Decisão 1 — Qual "B-Tree" está sendo sincronizada?
- **Fonte:** T-301 (BTree em `packages/protocol/src/rbsr/BTree.ts`) vs T-106 (lineage em SQL).
- **Ambigüidade detectada:** a spec original mistura "snapshot da B-Tree" com `BEGIN DEFERRED`
  (que opera sobre o storage SQL, não sobre uma estrutura in-memory). A `BTree` de T-301 é
  uma estrutura em memória construída a partir dos fingerprints do lineage; ela não vive no
  storage e não tem SQL representation.
- **Opções:** (a) Snapshot é da B-Tree em memória → não precisa de transação SQL, só de serializar
  a B-Tree (lista de ids + xorAggregate); (b) Snapshot é do lineage SQL → `BEGIN DEFERRED` faz
  sentido, e a B-Tree é re-derivada do snapshot.
- **Recomendação técnica:** (b) — é o que a spec original implica com `BEGIN DEFERRED` + diff
  via `nodes` table.

### Decisão 2 — Onde vive o estado de snapshot?
- **Fonte:** T-106 schema (`nodes`, `edges`, `entity_heads`, `entity_members` — sem tabela de sync).
- **Ambigüidade detectada:** o spec menciona `lastRangeProcessed` e `checkpointHlc` que
  precisam persistir entre `abortSync` e a próxima retomada; isso requer nova tabela ou coluna.
- **Opções:** (a) Nova migration v3 com `sync_state(snapshot_id, checkpoint_hlc, last_range,
  created_at)`; (b) Tabela in-memory (perdida entre execuções); (c) Reuso de `entity_heads`
  + nova tabela `sync_checkpoints`.
- **Recomendação técnica:** (a) — versionada e portável.

### Decisão 3 — Mecanismo de diff (insert/modify/delete concorrentes)
- **Fonte:** T-106 (schema tem `created_at` em todas as tabelas, mas não `updated_at`).
- **Ambigüidade detectada:** o spec quer detectar modified, mas o schema não tem `updated_at`
  nem triggers de auditoria.
- **Opções:** (a) Adicionar `updated_at` em `nodes` (migration v4); (b) Triggers SQLite que
  preenchem tabela `nodes_audit`; (c) Diff por HLC range scan entre `checkpoint_hlc` e
  `now_hlc`; (d) Diff por snapshot copy da tabela + comparação.
- **Recomendação técnica:** (a) + (c) combinados — `updated_at` para deleted/inserted,
  `hlc` comparison para modified.

### Decisão 4 — O que é "range"?
- **Fonte:** T-301 (BTree.reconcile com `IdRange{lo, hi}` em ULIDs) vs T-108a (lineage parentHash chain).
- **Ambigüidade detectada:** a spec menciona "retomada do range pendente", mas o reconcile RBSR
  opera por ULID range, e o lineage por entity_id chain. Qual é o range persistido?
- **Opções:** (a) RBSR range (`{lo: ULID, hi: ULID}`); (b) Lineage range (lista de entity_ids).
- **Recomendação técnica:** (a) — a B-Tree é o que está sendo sincronizada (decisão 1); range = RBSR.

### Decisão 5 — WAL checkpoint
- **Fonte:** T-004 (`StoragePort` — `exec`/`transaction`/`migrate`, **NÃO** expõe `wal_checkpoint`).
- **Ambigüidade detectada:** `wal_checkpoint(TRUNCATE)` é SQLite-specific. O `StoragePort` é
  genérico (isomórfico, poderia ser Postgres/SQLite/etc.).
- **Opções:** (a) Estender `StoragePort` com `walCheckpoint?(): Promise<...>` (método opcional,
  breaking change em todos adapters); (b) Wrapper SQLite-specific fora do `StoragePort`
  (acoplamento a SQLite, mas não quebra T-004); (c) Tabela de tracking + scheduler em
  application-layer (sem `wal_checkpoint`, mas simula o efeito via `PRAGMA wal_checkpoint(TRUNCATE)`
  direto no adapter SQLite).
- **Recomendação técnica:** (b) — novo arquivo `packages/core/src/sync/sqliteCheckpoint.ts` que
  recebe o `StoragePort` e executa `PRAGMA wal_checkpoint(TRUNCATE)` via `exec()`. Caller é
  responsável por usar somente com backend SQLite. Aplica YAGNI (não generaliza antes de ter
  2º backend).
