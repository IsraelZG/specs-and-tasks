# T-1024 · Worktree pool: reuso de worktrees longevas com node_modules quente (gatilho: concorrência)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Test Runner:** `vitest` (backend) — esta task é 100% backend/infra, sem UI.
- **Capacidade-alvo:** sonnet *(contratos, tipos e plano de testes fixados abaixo; zero decisão em aberto. O `pnpm install` real é injetável nos testes — ver Seção 4 — então a suíte não depende de ambiente pesado.)*

## 1. Objetivo
Evoluir `worktree.service.ts` de **efêmero-por-task** (cria `../.nexus-worktrees/<id>` no `start`,
remove no `approve`) para um **pool de worktrees longevas** que mantêm `node_modules` quente entre
tasks. Hoje cada task isolada paga um `pnpm install` novo (lento no Windows/ARM64 por causa da
floresta de symlinks `.pnpm` — ver `PITFALLS.md` P-005). Com pool, o `pnpm install` roda **uma vez
por slot** (na criação) e só repete se o `pnpm-lock.yaml` mudar; trocar de task é só um
`git checkout` de branch no slot.

**Gatilho de adoção:** isto só compensa quando o Runner virar **concorrente** (vários workers ao
mesmo tempo). O Runner atual (T-1023) é serial e roda no `master`. Esta task **adiciona** o modo
pool atrás de uma flag, **sem remover nem alterar o caminho efêmero atual** (que continua o
default). Decisão de design registrada em sessão de 2026-06-16.

```ts
export class NoWorktreeAvailableError extends Error {
  constructor(poolSize: number) {
    super(`Pool de worktrees esgotado (todos os ${poolSize} slots ocupados).`);
    this.name = 'NoWorktreeAvailableError';
  }
}

export interface PoolSlot {
  slot: string;            // 'slot-1' .. 'slot-N'
  path: string;            // <base>/pool/<slot>
  branch: string | null;   // 'task/<id>' quando ocupado, null quando livre
  lockfileHash: string | null; // sha256 do pnpm-lock.yaml na última provisão; null = nunca provisionado
}

export interface PoolOptions extends WorktreeOptions {
  poolSize?: number;                                   // default: NEXUS_WORKTREE_POOL_SIZE ou 2
  installer?: (slotPath: string) => Promise<void>;     // default: pnpm install --prefer-offline --frozen-lockfile (injetável p/ teste)
}

// Idempotente: cria <base>/pool/<slot-n> (git worktree add --detach) para n=1..poolSize
// e o registro <base>/pool/registry.json se não existirem. Retorna os slots.
export async function ensureWorktreePool(opts?: PoolOptions): Promise<PoolSlot[]>;

// Aloca um slot para a task `id`:
//  - se algum slot já tem branch === `task/<id>` → retorna esse (idempotente);
//  - senão pega o primeiro slot livre (branch === null); se nenhum → lança NoWorktreeAvailableError;
//  - no slot escolhido: `git checkout -B task/<id>` (cria/reseta a branch a partir do HEAD do slot);
//  - provisão: calcula sha256 do pnpm-lock.yaml do root; se slot.lockfileHash !== hash atual →
//    roda opts.installer(slotPath) e grava o novo hash; se igual → PULA (node_modules quente);
//  - persiste o registro. Retorna { branch, path, slot }.
export async function acquireWorktree(id: string, opts?: PoolOptions): Promise<WorktreeInfo & { slot: string }>;

// Libera o slot da task `id`: `git checkout --detach` no slot, marca branch=null no registro.
// NÃO apaga o diretório nem o node_modules (mantém quente). No-op se id não estiver alocado.
export async function releaseWorktree(id: string, opts?: PoolOptions): Promise<void>;
```

Regras de implementação fixas:
- Base do pool: `path.join(resolveBase(opts), 'pool')`. Registro: `<base>/pool/registry.json`.
- `poolSize` resolve de `opts.poolSize ?? Number(process.env.NEXUS_WORKTREE_POOL_SIZE) || 2`.
- Hash do lockfile: `crypto.createHash('sha256').update(fs.readFileSync(path.join(root,'pnpm-lock.yaml'))).digest('hex')`.
- `installer` default:
  `(p) => git-like execFile('pnpm', ['install','--prefer-offline','--frozen-lockfile'], { cwd: p })`
  reutilizando o mesmo padrão `execFile`-em-Promise já usado por `git()` (criar um helper
  `run(cmd, args, cwd)` ou generalizar; NÃO duplicar lógica de spawn). Timeout generoso (≥ 600000ms).
- Reutilizar `getWorktreeList`/`branchExists` existentes; NÃO criar um segundo parser de worktree.

### Pegadinhas conhecidas
- `git checkout -B` em um slot exige que o slot NÃO tenha mudanças não-commitadas; como o pool só
  faz `--detach` no release, os slots ficam limpos — mas garanta `--detach` (não deixar branch
  presa) senão a próxima `acquire` da mesma branch em outro slot falha ("branch já em uso").
- Uma branch `task/<id>` não pode estar em dois slots ao mesmo tempo (regra do git worktree). O
  registro é a fonte da verdade de quem tem o quê — leia-o antes de alocar.
- `crypto`, `os`, `fs`, `path` são imports de node; não esquecer `import crypto from 'crypto'`.
