# C-32 · Corrigir StoragePort->GraphStorePort em syncCoordinator (build de transport quebrado na master)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (pacotes core/protocol) e `playwright` (E2E/Frontend)
- **Capacidade-alvo:** haiku | sonnet | opus-spike *(ver regra "Dimensionamento de Tarefas" no CLAUDE.md: spec sem decisões em aberto, contratos explícitos, sem API externa não-fixada, verificação por comando)*

## 1. Objetivo
Consertar o build **quebrado na master** de `@plataforma/transport`. `src/syncCoordinator.ts:173` passa `this.storage` (tipado `StoragePort`) para `applyNodes()`, que exige `GraphStorePort` → `error TS2345`. A própria classe já tem o campo correto `this.graphStore: GraphStorePort` (linha 69, populado no construtor via `opts.graphStore ?? this.requireGraphStore(storage)`) e a linha 172 logo acima **já o usa**. A linha 173 só usa o campo errado.

É bug de tipo **e** latente de runtime: `this.storage` cru pode não implementar os métodos de grafo; `this.graphStore` é o validado.

**Origem (varredura completa, 2026-07-16):** um `pnpm build` de todo o monorepo (turbo, 31 pacotes) foi rodado como varredura. **15 pacotes verdes**, e a **única** ocorrência real deste padrão (`StoragePort` onde `GraphStorePort` é exigido) é esta linha. O caso irmão em `plugin-workflows/src/durableQueue.ts` **já foi corrigido** na master (commit `e899a0c`). A falha do `transport` **mascarou o build do único dependente, `workers`** (que já usa `StoragePort & GraphStorePort` em `commitCycle.ts`, provavelmente OK) — por isso a verificação exige rebuild completo verde, não só o de `transport`.

## 2. Contexto RAG (Spec-Driven Development)
- `packages/transport/src/syncCoordinator.ts` — o defeito (linha 173) e o padrão correto já presente (linhas 69, 93, 172, 279 `requireGraphStore`).
- [`docs/adr/0015-extensao-graphstore-port.md`](../docs/adr/0015-extensao-graphstore-port.md) — `GraphStorePort` é a porta de grafo; `applyNodes`/`insertNode`/`getLineage` do core exigem-na. Contexto de por que os dois contratos existem.
- Erro literal do tsc: `src/syncCoordinator.ts(173,48): error TS2345: Argument of type 'StoragePort' is not assignable to parameter of type 'GraphStorePort'. Type 'StoragePort' is missing the following properties … putNode, getNode, putEdge, getEdges, and 8 more.`

## 3. Escopo de Arquivos (Inputs e Outputs)
- **[UPDATE]** `packages/transport/src/syncCoordinator.ts` — linha 173: trocar `this.storage` por `this.graphStore` na chamada `applyNodes(nodes, this.storage)`. **Nenhuma outra mudança.**
> Se o rebuild completo (passo de verificação) revelar **novas** ocorrências do mesmo TS2345 em `workers` ou qualquer pacote antes mascarado pela falha do `transport`, corrija-as com o mesmo padrão (usar o binding `GraphStorePort` correto / `& GraphStorePort`) e liste cada arquivo:linha no Handover. > **⚠️ REGRAS DO QUE NÃO FAZER:**

> - **NÃO** tocar em outros arquivos além do necessário para zerar o TS2345 (ver escopo).

### Pegadinhas conhecidas
- O fix é **1 token** (`this.storage` → `this.graphStore`) na linha 173. A linha 172 (`graphStore: this.graphStore`) é o modelo de como referenciar o campo certo.
- `transport` depende de `core`+`protocol` compilados; numa worktree fria rode o build deles antes (P-012). O `pnpm build` da raiz (turbo) já resolve a ordem.
- O único dependente de `transport` é `workers` — só ele pode ter sido mascarado. Rebuild completo confirma.

1. Editar `packages/transport/src/syncCoordinator.ts:173`: `applyNodes(nodes, this.storage)` → `applyNodes(nodes, this.graphStore)`.

3. **Rebuild completo** `pnpm build` (raiz) — precisa ficar **verde em todos os pacotes** (é o que garante que `workers` e qualquer downstream antes mascarado passam). Colar a saída na Seção 8.
