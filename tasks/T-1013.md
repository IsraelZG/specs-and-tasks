# T-1013 · UI Board Jira + Drawer de Task + client de API

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Test Runner:** **Vitest + React Testing Library (JSDOM)** — escolhido entre as duas
  opções que a spec já admitia (Playwright OU RTL/JSDOM); JSDOM evita um browser headless.

## 1. Objetivo
Substituir o board atual (que só mostra o título) por um Kanban real que lê/edita
tasks chamando o backend (T-1011), e consertar os bugs de fetch existentes (URL
hardcoded, ausência de `response.ok`, ausência de UI de erro). Permitir ver o
conteúdo completo da task, sua timeline de Log de Execução, e executar ações
(assign, transition, adicionar nota).

## 2. Contexto RAG (Spec-Driven Development)
- [ ] `apps/nexus-frontend/src/pages/TaskBoard.tsx` (versão atual a substituir)
- [ ] `apps/nexus-frontend/src/App.tsx` (rotas/navegação)
- [ ] Endpoints REST da T-1011 (`/api/tasks`, `/api/tasks/:id`, `.../transition|assign|log`)

## 3. Escopo de Arquivos (Inputs e Outputs)
- **[CREATE]** `apps/nexus-frontend/src/api.ts` — client central: base = `import.meta.env.VITE_API_URL ?? 'http://localhost:3001'`; checa `response.ok`; lança erro tipado; funções `listTasks/getTask/transition/assign/logProgress`.
- **[CREATE]** `apps/nexus-frontend/.env.example` (`VITE_API_URL=http://localhost:3001`)
- **[UPDATE]** `apps/nexus-frontend/src/pages/TaskBoard.tsx` — colunas por status
  (`draft|ready|in_progress|review|rework|done|blocked`); cards com id, título,
  `target_agent`, complexidade, branch e último log; mover card → `transition`.
- **[CREATE]** `apps/nexus-frontend/src/components/TaskDrawer.tsx` — painel lateral:
  markdown do corpo, timeline do Log de Execução (seção 9), e ações
  (assign, transition, adicionar nota) via `api.ts`; estados de loading e erro reais.
- **[CREATE]** `apps/nexus-frontend/src/transitions.ts` — espelho (somente-UI) da máquina
  de estados MGTIA do backend: `TRANSITIONS`, `COLUMNS`, `STATUS_LABEL`, `AGENT_PROFILES`,
  `actionsFor(status)` (ações válidas → botões) e `actionForTarget(from,to)` (deriva a ação
  ao soltar um card numa coluna). A validação canônica continua no backend; este módulo só
  evita oferecer ações impossíveis na UI.
- **[CREATE]** Harness de testes (Vitest + JSDOM + React Testing Library):
  `vitest.config.ts` (environment `jsdom`, plugin react, `setupFiles`),
  `src/test/setup.ts` (`@testing-library/jest-dom` + `cleanup`), e os specs
  `src/transitions.test.ts`, `src/api.test.ts`, `src/pages/TaskBoard.test.tsx`.
  Adicionar `"test": "vitest run"` e as devDeps de teste ao `package.json`; excluir
  `*.test.*` e `src/test` do `tsconfig.app.json` (o build de produção não os compila).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - NÃO hardcodar `http://localhost:3001` nas páginas; usar `api.ts` + env.
> - NÃO chamar `.json()` sem checar `response.ok`.
> - NÃO deixar `catch` vazio: sempre refletir o erro na UI.

2. Refatore `TaskBoard.tsx` para consumir `api.listTasks()` e agrupar por status.
3. Crie `TaskDrawer.tsx` e ligue as ações.
4. Adicione associação de label (`htmlFor`/`id`) e `aria-label` onde aplicável.
