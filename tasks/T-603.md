## 1. Objetivo

### Contratos exatos *(revisão arquiteto 2026-07-12 — D3 dissolvida por fonte canônica; ver §6)*
```ts
// --- packages/workers/src/sync/deterministicCommitter.ts ---
// (D1-transitivo RESOLVIDO: T-602 fixou CommitCycle em @plataforma/workers; este módulo
//  importa CommitCycle, logo mora no MESMO pacote — core importando workers inverteria a seta.)
import type { EntityId, ULID } from '@plataforma/protocol';
import type { Ed25519PrivateKey, Ed25519PublicKey, Ed25519Signature } from '@plataforma/crypto';
import type { CommitCycle } from './commitCycle.js'; // T-602, mesmo pacote

export interface CommitCandidate {
  entityId: EntityId;            // entity_id do AGENTE candidato a committer no ciclo
  changes: Uint8Array[];         // changes a commitar
  authorPubKey: Uint8Array;
  hlc: number;
}

export interface DeterministicTiebreak {
  /** Desempate CANÔNICO (caderno-2-protocol/04 §4, colapso v4; verbete automerge-repo
   *  §Evolução): vence o MENOR entity_id (ordem lexicográfica de bytes) entre os
   *  candidatos ativos do ciclo. Sem hash, sem HLC na eleição — a fórmula
   *  hash(entity_id‖HLC) da versão anterior desta spec não tinha fonte e divergia do cânone. */
  winner: EntityId;
  reason: 'lowest_entity_id' | 'single_author';
}

import type { PreparedCommit } from './commitCycle.js'; // T-602 (F2): lote congelado

export interface CoSignRequest {
  snapshotHash: Uint8Array;      // = PreparedCommit.snapshotHash — ÚNICA coisa que se assina
  /** Vem da SPEC do ativo: co-signers exigidos COM suas chaves públicas. A resolução
   *  EntityId→chave é responsabilidade do chamador (SPEC/perfil) — este módulo só verifica. */
  requiredCoSigners: Map<EntityId, Ed25519PublicKey>;
  signatures: Map<EntityId, Ed25519Signature>; // Ed25519 sobre snapshotHash (caderno-2 §4.1)
}

export interface CoSignRecoveryPolicy {
  /** Prazo assinado pela SPEC; depois dele uma recuperação pode ser proposta. */
  maxAbsenceMs: number;
  mode: 'remaining_unanimous';
}

export interface CoSignRecoveryDecision {
  absentSigner: EntityId;
  action: 'discard' | 'adopt';
  originalChangeHashes: Uint8Array[];
  adoptedChangeHashes?: Uint8Array[];
  adopter?: EntityId;
  deadlineAt: number;
  /** Assinaturas Ed25519 dos restantes sobre o payload de recuperação canônico. */
  approvals: Map<EntityId, Ed25519Signature>;
}

export interface CommitterResult {
  commitNodeId: ULID;
  tiebreak: DeterministicTiebreak;
  coSignatures: CoSignRequest | null; // null se não exige co-assinatura
}

export class DeterministicCommitter {
  /** Ordena candidatos por entity_id lexicográfico ascendente (determinístico: mesmo
   *  conjunto de candidatos → mesma ordem em qualquer peer, sem rede). */
  rankCandidates(candidates: CommitCandidate[]): CommitCandidate[];

  /** FLUXO OBRIGATÓRIO prepare → co-sign → finalize (caderno-2 §4.1 — coleta ANTES de
   *  persistir; snapshot assinado === snapshot persistido, byte a byte):
   *  1. eleição: rankCandidates[0] == eu? senão, retorna sem commitar (não-committer);
   *  2. `prepared = cycle.prepareCommit()` — congela snapshot/hash/lote (T-602 F2);
   *     changes que chegarem daqui em diante ficam para o PRÓXIMO lote, não invalidam este;
   *  3. se co-sign exigida: `requestCoSignature({snapshotHash: prepared.snapshotHash, ...})`
   *     e aguarda `isCoSignComplete` até o timeout (VirtualClock-able via ClockPort do cycle);
   *  4. VERIFICA cada assinatura: `ed25519Verify(pubKey, prepared.snapshotHash, sig)` —
   *     inválida ou sobre outro snapshot ⇒ excluída; K incompleto no timeout ⇒ ABORTA
   *     (prepared descartado; staging intacto — clearStaging só roda no finalize OK);
   *  5. monta as K arestas CO_SIGNED_BY (formato abaixo) e chama
   *     `cycle.finalizeCommit(prepared, coSignEdges)` — nó + AUTHORED + CO_SIGNED_BY na
   *     MESMA transação via insertNodeWithEdges (T-602 F2). Nunca nó primeiro, assinaturas
   *     depois: a janela "nó replicável sem co-assinaturas obrigatórias" violaria o aceite. */
  commit(candidates: CommitCandidate[], options?: {
    requireCoSign?: boolean;
    recoveryPolicy?: CoSignRecoveryPolicy;
  }): Promise<CommitterResult>;

  /** Após o prazo de `recoveryPolicy`, verifica unanimidade dos restantes, materializa as
   *  RECOVERY_DECISION e finaliza somente o snapshot reconstruído/adotado. O payload assinado
   *  inclui ação, ausente, deadline e hashes originais; não pode ser reutilizado para outra
   *  recuperação. */
  recoverAfterAbsence(
    candidates: CommitCandidate[],
    request: CoSignRequest,
    policy: CoSignRecoveryPolicy,
    decision: CoSignRecoveryDecision,
  ): Promise<CommitterResult>;

  /** Envia `snapshotHash` via ephemeral (T-403) e acumula respostas em `signatures`. */
  requestCoSignature(request: CoSignRequest): Promise<void>;

  isCoSignComplete(request: CoSignRequest): boolean;
}
```
- **Persistência de co-assinaturas (D2):** K arestas `edge_type='CO_SIGNED_BY'`
  (`source_id` = co-signer PROFILE, `target_id` = nó do commit, payload = JSON UTF-8
  `{ signer: EntityId, snapshotHash: hex, signature: hex /* Ed25519 do co-signer sobre snapshotHash */ }`).
  A ARESTA é assinada com `signEdge` pela chave do **committer** (Layer 1: `publicKey` da
  aresta = quem grava); a assinatura do **co-signer** vive no payload e é verificável de
  forma independente contra `requiredCoSigners`. Persistidas via `finalizeCommit(prepared,
  coSignEdges)` → `insertNodeWithEdges` — mesma transação do nó. Schema aceita sem migração
  (schema.ts:18).
- **Co-assinatura parcial (timeout):** default = `CommitTrigger.maxPendingMs` (30s). Expirou
  sem K assinaturas → **aborta o ciclo**, changes permanecem no staging (limpeza seletiva de
  T-602 garante isso), re-tenta no próximo gatilho. Após `CoSignRecoveryPolicy.maxAbsenceMs`
  definido pela SPEC, todos os coassinantes restantes podem assinar a mesma decisão de
  recuperação: `discard` reconstrói causalmente sem as mudanças do ausente; `adopt` cria mudança
  nova com chave do adotante. A decisão e hashes são arestas `RECOVERY_DECISION` na mesma transação.
- **Risco documentado (ataque):** a eleição é determinística DADO o mesmo conjunto de
  candidatos; sob reconexão/partição os conjuntos podem divergir e dois peers podem se eleger.
  O cânone trata isso como exceção esperada-rara com rede de segurança estrutural: o fork
  resultante é detectado e curado por T-601 (caderno-2 §4.2). Não tentar prevenir aqui.
- N peers commitando simultaneamente → não geram fork no caso comum (comutativo).

## 4. Estratégia de Testes
- [x] **Framework:** Vitest + VirtualClock (timeout de co-sign).
- [x] **Ambiente:** `pnpm --filter @plataforma/workers test`.

Casos de teste:
1. `rankCandidates` ordena deterministicamente (mesmo input → mesma ordem; permutação do input → mesma ordem).
2. `commit` com candidato único → commit direto sem tiebreak (`reason: 'single_author'`).
3. `commit` com 3 candidatos → vence o menor entity_id lexicográfico (`reason: 'lowest_entity_id'`).
4. Dois peers commitando simultaneamente (mesmo conjunto de candidatos) → ambos elegem o MESMO committer; só ele insere → não geram fork (comutativo).
5. `requestCoSignature` envia `snapshotHash` via ephemeral channel (stub do shell T-403).
6. `isCoSignComplete` com todas as assinaturas → true.
7. `isCoSignComplete` faltando assinatura → false.
8. Co-assinaturas persistidas como arestas CO_SIGNED_BY na MESMA transação do nó (consulta `getEdges({target: nodeId, type: 'CO_SIGNED_BY'})` → K arestas; payload verificável com `ed25519Verify`).
9. Timeout de co-sign (VirtualClock avança maxPendingMs) sem K assinaturas → ciclo aborta, staging intacto, nenhum nó inserido.
10. **Change pós-prepare:** co-editor envia change entre `prepareCommit` e `finalizeCommit` → snapshot persistido = bytes do prepare (hash confere com o assinado); a change vai para o próximo lote.
11. **Assinatura de snapshot divergente (adversarial):** co-signer responde com assinatura sobre OUTRO hash → `ed25519Verify` falha, assinatura excluída; se K fica incompleto → aborta sem persistir nada.
12. **Rollback atômico (adversarial):** falha injetada no `putEdge` de uma CO_SIGNED_BY → zero nó/aresta/projeção persistida (mesma garantia do teste 11 de T-602).
13. Recuperação antes do prazo da SPEC é recusada e preserva staging.
14. Recuperação sem decisão idêntica assinada por todos os coassinantes restantes é recusada.
15. `discard` reconstrói snapshot válido sem as mudanças do ausente e persiste arestas `RECOVERY_DECISION` auditáveis.
16. `discard` causalmente impossível falha fechada, sem persistência parcial.
17. `adopt` exige mudança nova assinada pelo adotante e preserva os hashes da mudança abandonada; bytes e assinatura do ausente não podem ser reutilizados.
18. Falha ao gravar uma aresta de recuperação faz rollback completo de nó, arestas e staging.

**DECIDIDO (2026-07-16) — recuperação após ausência com unanimidade dos restantes.** Cada SPEC
declara `CoSignRecoveryPolicy.maxAbsenceMs`. Antes desse prazo o ciclo fica N-de-N; expiração só
aborta a tentativa e preserva staging. Depois dele, todos os coassinantes restantes devem assinar a
mesma `CoSignRecoveryDecision`: `discard` somente por reconstrução causal desde o último snapshot
finalizado, ou `adopt` por uma nova mudança, revisada e assinada com a chave de quem a adota.

As decisões são gravadas como `RECOVERY_DECISION`, junto das `CO_SIGNED_BY`, na mesma transação do
snapshot. Nunca se reatribui mudança, chave ou assinatura do ausente: o artefato abandonado continua
referenciado por hash para auditoria. Esta é a opção C, formalizada sem falsificar proveniência; o
custo aceito é menor disponibilidade e configuração explícita por SPEC.
