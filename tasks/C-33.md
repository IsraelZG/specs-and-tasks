# C-33 · Corrigir build do bancada (WASM Automerge / vite-plugin-pwa)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (pacotes core/protocol) e `playwright` (E2E/Frontend)
- **Capacidade-alvo:** haiku | sonnet | opus-spike *(ver regra "Dimensionamento de Tarefas" no CLAUDE.md: spec sem decisões em aberto, contratos explícitos, sem API externa não-fixada, verificação por comando)*

## 1. Objetivo
Deixar `pnpm --filter @plataforma/bancada build` **verde**. Hoje o `vite build` do `bancada` falha ao empacotar o WASM do Automerge:
```
[vite-plugin-pwa:build] Could not load .../@automerge/automerge/.../automerge_wasm_bg.wasm:
"ESM integration proposal for Wasm" is not supported currently.
Use vite-plugin-wasm or `.wasm?init` / `.wasm?url`.
```
**Cadeia de import (por que o WASM entra no bundle):** `apps/bancada` importa `@plataforma/transport`, que usa `@automerge/automerge` em `automergeShell.ts`/`portNetworkAdapter.ts`. Ao empacotar o `transport`, o Vite/Rollup encontra o import ESM do `.wasm` e não sabe carregá-lo sem plugin.

Esta é a **3ª dívida basal** desmascarada na série C-31/C-32 (era ocultada pela falha anterior do `transport`). Não é regressão da C-32 — só ficou visível quando o build passou do `transport`.

## 2. Contexto RAG (Spec-Driven Development)
- `apps/bancada/vite.config.ts` — config atual: `react()` + `VitePWA(...)`, sem plugin de WASM.
- `apps/bancada/package.json` — deps atuais: `vite ^6`, `vite-plugin-pwa ^0.21`. Sem `vite-plugin-wasm`.
- `packages/transport/src/automerge/automergeShell.ts` e `portNetworkAdapter.ts` — origem do import `@automerge/automerge` (WASM). **Não alterar** — o fix é no bundler do consumidor, não na lib.
- Erro literal: ver Seção 1.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** editar `packages/transport/**` nem stubar o Automerge — o app precisa do transport real.
> - **NÃO** desligar/rebaixar o `VitePWA` para "resolver" — o WASM não é problema do PWA, é do bundle.
> - **NÃO** adicionar o `.wasm` aos `globPatterns` do workbox achando que é precache — não é o caso.

### Pegadinhas conhecidas
- **Abordagem recomendada:** adicionar `vite-plugin-wasm` **e** `vite-plugin-top-level-await` (o init do Automerge usa top-level await; só o `vite-plugin-wasm` costuma não bastar). Ordem no array: `wasm()` e `topLevelAwait()` **antes** do `VitePWA`. Ex.:
>  ```ts
>  import wasm from "vite-plugin-wasm";
>  import topLevelAwait from "vite-plugin-top-level-await";
>  // plugins: [react(), wasm(), topLevelAwait(), VitePWA({...})]
>  ```
> - Fixe versões compatíveis com `vite ^6` (à época: `vite-plugin-wasm ^3.4`, `vite-plugin-top-level-await ^1.4`). Se o build reclamar de `target`, garanta `build.target` ≥ `es2022` (top-level await exige).
> - Se após os plugins o erro persistir só no **PWA/generateSW** (workbox tentando o `.wasm`), aí sim ajuste o `workbox.globIgnores` para excluir `**/*.wasm` do precache — mas só se o erro mudar para essa fase.
> - `bancada` é ARM64 Windows nativo; `pnpm install` da nova dep ~poucos s (ver memória de ambiente). Não rodar install gigante no meio da tool.

1. Confirmar a fase exata do erro: `pnpm --filter @plataforma/bancada build` e ler se é bundling (Rollup) ou generateSW (PWA).
2. `pnpm --filter @plataforma/bancada add -D vite-plugin-wasm vite-plugin-top-level-await`.
3. Editar `apps/bancada/vite.config.ts`: importar e inserir `wasm()` + `topLevelAwait()` antes do `VitePWA`.
4. `pnpm --filter @plataforma/bancada build` → verde. Rodar `test`. Colar saída na Seção 8.
