# T-901 · PWA produto v0 (apps/web) — prova de consumo do SDK

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+ / browser (PWA)
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `playwright` (E2E/PWA)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Construir um shell PWA instalável e offline-first em `apps/web` que **consome o `client-sdk`** (não a Bancada) para provar que o SDK é utilizável por um produto externo: login nos 3 modelos de identidade, lista/edição de um tipo de `CONTENT` colaborativo, e indicador de conectividade/topologia *(derivado de `docs/plano-de-implementacao.md` §12 — "T-901 · PWA produto v0")*.

### Contratos e Assinaturas Exatas (derivados do client-sdk e crypto)
A interface de consumo exposta pelo `@plataforma/client-sdk` deve seguir o seguinte contrato TypeScript:

```ts
// packages/client-sdk/src/sdk.ts

export interface SDKConfig {
  signalingUrl: string;
  bootstrapPeers: string[];
  dbName: string;
}

export interface Session {
  getPeerId(): string;
  getPersonaId(): string;
}

export interface ConnectivityStatus {
  connected: boolean;
  peerCount: number;
  topology: 'P2P' | 'Relay' | 'Offline';
}

export interface DocumentState {
  id: string;
  title: string;
  content: string;
  lastUpdated: number;
}

export class ClientSDK {
  /** Inicializa o SDK com a configuração de banco e rede. */
  static initialize(config: SDKConfig): Promise<ClientSDK>;

  /** Login Modelo 1: Utiliza mnemônico puro. */
  loginWithMnemonic(mnemonic: string): Promise<Session>;

  /** Login Modelo 2: Utiliza mnemônico cifrado e senha do dispositivo. */
  loginWithEncryptedMnemonic(encrypted: Uint8Array, password: string): Promise<Session>;

  /** Login Modelo 3: Utiliza delegação de dispositivo via pareamento. */
  loginWithDeviceDelegation(peerId: string): Promise<Session>;

  /** Retorna o status de conectividade atual. */
  getConnectivityStatus(): ConnectivityStatus;

  /** Assina mudanças de conectividade/topologia. Retorna função de desinscrição. */
  onConnectivityChange(callback: (status: ConnectivityStatus) => void): () => void;

  /** Assina atualizações de um documento colaborativo. Retorna função de desinscrição. */
  subscribeToDocument(documentId: string, callback: (doc: DocumentState) => void): () => void;

  /** Envia uma mutação/alteração ao documento. */
  updateDocument(documentId: string, changeFn: (doc: any) => void): Promise<void>;

  /** Encerra as conexões e fecha os bancos de dados do SDK. */
  close(): Promise<void>;
}
```

## 4. Estratégia de Testes Estrita
- [x] **Framework:** Playwright (browser real). Unit/JSDOM **não** basta para esta UI.
- [x] **Ambiente:** `pnpm --filter @plataforma/web test:e2e` rodando localmente.
- [x] **Cenários verificáveis (lista numerada):**
  1. **Registro e Carregamento Offline:** Acessar o app `http://localhost:5173/`, verificar se o service worker e manifest estão registrados. Forçar modo offline (`context.setOffline(true)`), recarregar e atestar que a UI carrega do cache.
  2. **Login nos 3 Modelos:**
     - Simular inserção de mnemônico válido BIP39 e atestar login bem-sucedido.
     - Simular salvamento de mnemônico criptografado em localStorage, desbloqueio via senha e atestar login.
     - Simular login por delegação (Device ID e handshake simulado).
  3. **Edição Colaborativa e Sincronização Offline:**
     - Criar/editar um documento `CONTENT:DOCUMENT` online.
     - Desativar rede (`context.setOffline(true)`), realizar edições locais.
     - Reativar rede (`context.setOffline(false)`), verificar se o Automerge reconcilia o estado e as alterações são propagadas.
  4. **Indicador de Conectividade/Topologia:**
     - Verificar se o status muda dinamicamente entre `Offline`, `P2P` e `Relay` sob variações de conectividade simuladas.
- [x] **Métricas de Aceite:** Lighthouse PWA score ≥ 90.
- [x] **Fora de escopo:** Testes reais de gateway de pagamento ou sincronização em servidores remotos reais.

## 5. Instruções de Execução
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** reimplemente lógica de protocolo na app — consuma só a API pública do `@plataforma/client-sdk`.
> - **NÃO** copie a Bancada inteira; a Bancada é ferramenta de teste, `apps/web` é produto.
> - **NÃO** ignore o tratamento de erros no login e na reconectividade.

1. **Configuração e Dependências:**
   - Adicione no `apps/web/package.json` dependência do `@plataforma/client-sdk`, `@plataforma/design-system` (seus componentes/tokens), `react`, `react-dom` e plugins necessários.
   - Configure o `vite.config.ts` com `vite-plugin-pwa` para gerar o service worker e registrar no bootstrap da aplicação.
2. **Desenho da Interface (PWA):**
   - Crie `apps/web/src/App.tsx` que renderiza a interface reativa usando os componentes da Plataforma.
   - Tela de Autenticação com as 3 opções de login.
   - Área Principal contendo:
     - Indicador Visual de Conectividade e Topologia (P2P / Relay / Offline).
     - Editor de Documento Colaborativo (título, conteúdo text-area e histórico de alterações/heads).
3. **Integração com o Client SDK:**
   - Use `ClientSDK.initialize` no bootstrap do app.
   - Conecte o formulário de login aos métodos correspondentes do SDK.
   - Observe a conectividade através do hook/callback de conectividade do SDK.
   - Assine o documento colaborativo usando `subscribeToDocument`.
4. **Escrita dos Testes E2E (Playwright):**
   - Crie `apps/web/tests/pwa-e2e.spec.ts` com os cenários de teste descritos na Seção 4.
   - Use os métodos de interceptação de rede e emulação offline do Playwright (`context.setOffline`).
5. **Validação do Gate:**
   - Execute o comando do Gate de Evidência na Seção 7.
