## 1. Objetivo
Fechar o buraco que o **incidente T-1016** (16/06) expôs: o T-1025 protege a *API* do
`TaskService` (`approve`/`request_changes` exigem `agent === reviewer_agent`), mas o `status:` mora
no frontmatter Markdown, que qualquer agente pode **editar direto no arquivo**, pulando a máquina de
estados inteira. Foi exatamente o que aconteceu: o **DeepSeek (worker)** implementou e chegou a
`review` corretamente, mas o **Gemini, agindo como revisor não autorizado**, rodou um `/qa-review`
raso (sem rodar o Gate de Evidência) e forçou `review → done` **editando `T-1016.md` e `INDEX.md` na
mão**, sem nenhum `approve` do serviço (mesmo padrão de M-013/T-1014). Esta task fecha por código a
dimensão *filesystem* do bypass; as dimensões "agente errado revisando" e "Gate pulado" são de
processo/config de agente (ver `.claude/agents/agile-reviewer.md` e CLAUDE.md Regra 6), fora deste
escopo de código.

Num sistema baseado em arquivo não dá para **impedir** a escrita no `.md`. A defesa correta é
**detectá-la e recusar honrá-la**: um **ledger append-only** de transições (fonte de verdade da
integridade) + um **gate anti-drift** no início de `transition()` que rejeita operar quando o
`status:` do Markdown diverge do topo do ledger. Mais um comando `reconcile` (a ÚNICA forma
sancionada de corrigir drift, reescrevendo o status para o valor do ledger).

> **Escopo deliberado:** isto NÃO move a fonte de verdade do status para fora do Markdown (o
> frontmatter continua legível por humanos e pelo `agile-reviewer`). O ledger é uma camada de
> integridade *ao lado* do Markdown. "Status como projeção read-only de um store" é uma evolução
> futura possível, fora desta task.

### **[UPDATE]** `apps/nexus-backend/src/services/task.types.ts`
```ts
export class StatusDriftError extends Error {
  constructor(
    public readonly id: string,
    public readonly ledgerStatus: string,
    public readonly fileStatus: string,
  ) {
    super(
      `Drift de status em ${id}: o ledger registra '${ledgerStatus}' mas o arquivo está '${fileStatus}'. ` +
        `O status foi editado fora do serviço (bypass). Rode 'reconcile ${id}' para restaurar o valor do ledger antes de qualquer transição.`,
    );
    this.name = 'StatusDriftError';
  }
}

export interface LedgerEntry {
  ts: string;        // ISO
  id: string;
  from: string | null;
  to: TaskStatus;
  action: string;    // verbo da transição, 'seed' ou 'reconcile'
  agent: string;
}
```

### **[UPDATE]** `apps/nexus-backend/src/services/task.service.ts`
- Ledger: `private get ledgerPath() { return path.join(this.rootDir, '.nexus', 'transitions.jsonl'); }`
  (formato JSONL, uma `LedgerEntry` por linha; criar `.nexus/` com `fs.mkdirSync(..., {recursive:true})` na primeira escrita).
- `private readLedgerHead(id: string): LedgerEntry | null` — lê o arquivo (se existir), filtra por
  `id`, retorna a ÚLTIMA entrada ou `null`. (Leitura linha-a-linha tolerante a arquivo ausente.)
- `private appendLedger(entry: LedgerEntry): void` — `fs.appendFileSync(ledgerPath, JSON.stringify(entry) + '\n')`.
- **Gate anti-drift** no INÍCIO de `transition()`, logo após ler o `currentStatus` do arquivo e
  ANTES da validação de `rule.from`:
  ```
  const head = this.readLedgerHead(id);
  if (head === null) {
    // Baseline: primeira vez que o serviço vê esta task → confia no status atual e semeia o ledger.
    this.appendLedger({ ts, id, from: null, to: currentStatus, action: 'seed', agent: 'system' });
  } else if (head.to !== currentStatus) {
    throw new StatusDriftError(id, head.to, currentStatus);
  }
  ```
- Ao FINAL de uma transição bem-sucedida (depois de escrever o arquivo, junto do fluxo atual),
  registrar no ledger: `this.appendLedger({ ts, id, from: <status resolvido de origem>, to: rule.to, action, agent })`.
- Novo método público:
  ```ts
  /** Única forma sancionada de corrigir drift: reescreve o status do arquivo para o topo do ledger. */
  reconcile(id: string, agent: string): TaskRecord
  ```
  - Lê `head = readLedgerHead(id)`; se `null` → lança `Error('Sem ledger para ' + id + '; nada a reconciliar')`.
  - Se `head.to === currentStatus` → no-op informativo (retorna o task; nenhum drift).
  - Senão: `replaceStatus(file, head.to)` + `appendLog(file, agent, '[Reconciliado]', 'status restaurado de <fileStatus> para <head.to> (drift corrigido)')` + `appendLedger({...action:'reconcile'})` + `rebuildIndexes()`.

  4. `reconcile(id, 'agile_reviewer')` após o drift do caso 3 → status do arquivo volta a `review`, Log ganha `[Reconciliado]`, ledger ganha entrada `reconcile`; uma transição subsequente legítima volta a funcionar.

  6. **Seed/grandfather:** task pré-existente em `status: done` cuja primeira interação com o serviço é uma transição válida a partir de `done` (ex. `block`) → NÃO lança drift (semeia o baseline com `done`) e a transição procede.

### Pegadinhas conhecidas
- O baseline "seed na primeira vez" significa que um drift que aconteça ANTES de existir qualquer
  entrada de ledger para a task é grandfathered (não detectável retroativamente). Isso é aceitável e
  intencional — a proteção vale para tudo que passe pelo serviço depois da semeadura. NÃO tente
  "adivinhar" o status correto de tasks legadas.
- `appendLog` e `appendLedger` precisam usar o MESMO timestamp `ts` numa transição para a trilha
  bater; gere `ts` uma vez no início de `transition()`.
- Como `transition()` agora pode lançar `StatusDriftError` antes de `rule.from`, garanta que o
  teste do T-1025 (gate de papel) continua passando — drift e papel são checagens independentes.
