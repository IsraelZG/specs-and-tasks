# T-302b · RBSR exchange: applyNodes (validação Layer 1 + inserção)

## 0. Ambiente de Execução Obrigatório

## 1. Objetivo
Implementar `applyNodes` no pacote `@plataforma/protocol`: recebe nós binários do peer remoto, valida assinatura Layer 1 (T-107) e insere no storage via `insertNode` (T-108a). Rejeita nós com assinatura inválida sem contaminar o grafo local
*(extraído de T-302 §1)*.

### Contratos exatos (derivados de T-107 signature.ts, T-108a lineage.ts, T-004 ports.ts)

```ts
// --- packages/protocol/src/rbsr/exchange.ts (extensão) ---
import { verifyNode, insertNode, canonicalizeNode, hashNode } from '@plataforma/core';
// T-107: signature.ts — verifyNode/SignedNode/canonicalizeNode
// T-108a: lineage.ts — insertNode(StoragePort, SignedNode, parentId?): Promise<void>
import type { SignedNode } from '@plataforma/core';
import type { StoragePort } from '../ports.js'; // T-004

/**
 * Aplica nós binários recebidos no storage local.
 *
 * Para CADA nó:
 *   1. Verifica assinatura Layer 1 (verifyNode — T-107). Se inválida → conta como rejected e segue.
 *   2. Tenta insertNode (Layer 2 — parentHash, HLC, ciclo, entityId). Se lança → conta como rejected e segue.
 *   3. Em sucesso, conta como imported.
 *
 * Nó com id duplicado: insertNode lança "Duplicate node ID" (T-108a lineage.ts) → counted as rejected.
 *
 * @param nodes  Nós binários a importar (SignedNode decodificados — T-302a entrega id+fp via wire;
 *               o caller desta função (futuro) é responsável por fetch do SignedNode completo).
 *               Contrato atual: array de SignedNode já decodificados.
 * @param storage  StoragePort (T-004) do grafo local (SQLite :memory: em teste).
 * @returns Contadores: imported (assinatura válida + inserção ok), rejected (qualquer falha).
 */
export async function applyNodes(
  nodes: readonly SignedNode[],
  storage: StoragePort,
): Promise<{ imported: number; rejected: number }>;
```

## 4. Estratégia de Testes
- [x] **Framework:** Vitest + `@plataforma/testkit` `VirtualClock` + `SeededRandom` + `SimNetwork` (T-005).
- [x] **Storage:** SQLite real via `StoragePort` (T-106 schema já migrado em testkit ou setup por teste).

**Setup por teste** (helper `makeSignedNode`):
- Gerar par Ed25519 (`@plataforma/crypto` — T-101)
- Construir `UnsignedNode` com `id: ULID` (T-102), `hlc: HLCTimestamp` (T-103), `nodeType`, `payload: Uint8Array`, `publicKey`, `parentHash: NodeHash` (32 zero bytes para raiz, senão hashNode(parent))
- `signNode(unsignedNode, privateKey)` → `SignedNode`
- Para nó-filho: parentId = id do nó já inserido; `parentHash = await hashNode(parent)` (T-108a)
- Storage `:memory:` SQLite + `storage.migrate([{version: 1, sql: <T-106 schema>}])`

**Casos de teste** (5 enumerados, todos derivados de T-107/T-108a + casos da §4 original):

1. **3 nós válidos (todos raiz)** — `applyNodes([n1,n2,n3], storage)` → `{imported: 3, rejected: 0}`. Verifica via `storage.exec('SELECT COUNT(*) FROM nodes')` → 3.
2. **1 nó com assinatura adulterada (rejeitada na Layer 1)** — gera nó válido, **muda 1 byte da signature** → verifyNode retorna false → `imported: 0, rejected: 1`. Verifica que storage permanece vazio.
3. **Array vazio** — `applyNodes([], storage)` → `{imported: 0, rejected: 0}`. Edge case sem erro.
4. **Nó duplicado (mesmo id, mesmo parent)** — `applyNodes([n1, n1, n2], storage)` → `{imported: 2, rejected: 1}` (insertNode lança "Duplicate node ID"). Verifica COUNT = 2.
5. **1 nó com parentHash inválido (rejeitado na Layer 2)** — gera nó-filho, **muda 1 byte do parentHash** → assinatura é válida (Layer 1 passa), mas insertNode lança "parentHash mismatch" → `imported: 0, rejected: 1`. Verifica que storage permanece vazio.

## 5. Instruções de Execução (Step-by-Step)
1. **Setup de imports** — `import { verifyNode, insertNode, ... } from '@plataforma/core'` (verificar se `index.ts` de core reexporta — caso contrário adicionar). `import type { StoragePort }` de `@plataforma/protocol`.
2. **Implementar `applyNodes`** em `packages/protocol/src/rbsr/exchange.ts`:
   ```ts
   for (const node of nodes) {
     try {
       const ok = await verifyNode(node);
       if (!ok) { rejected++; continue; }
       await insertNode(storage, node); // parentId omitido → trata como raiz
       imported++;
     } catch {
       rejected++;
     }
   }
   return { imported, rejected };
   ```
3. **Adicionar re-export** em `packages/protocol/src/index.ts`:

## 6. Feedback de Especificação

**Decisão em aberto — wire format de SignedNode completo:**

O `reconcile` de T-302a (já merged) usa o wire format `MSG_NODES_RESPONSE` que codifica
`Array<{ id: string; fingerprint: Fingerprint }>` (apenas id+fp, não o SignedNode completo).
`applyNodes` precisa do `SignedNode` inteiro (com payload, publicKey, signature) para
passar por `verifyNode` (Layer 1) e `insertNode` (Layer 2). Existem 3 opções para resolver:

- **(a)** Evoluir `respondNodes` em `exchange.ts` para enviar SignedNode binário (canonicalizado
  com `canonicalizeNode` — T-107). Wire format muda: `{ id, signedNodeBytes: Uint8Array }[]`.
  Caller decodifica com `canonicalizeNode` reverter (não há reverso hoje) **OU** envia
  SignedNode como JSON MessagePack (codec já em T-212).
- **(b)** Adicionar uma nova mensagem RBSR `MSG_FULL_NODES_REQUEST/RESPONSE` (T-302a extension)
  que carrega o SignedNode canônico (T-107). Mantém `MSG_NODES_RESPONSE` só para BTree sync.
- **(c)** Reusar infraestrutura de get-blob (T-802: Merkle-verified blob reassembly) — chunks
  encriptados + Merkle. Overhead significativo.

**Recomendação técnica:** (b) — preserva o wire format atual (T-302a), adiciona mensagem
dedicada para fetch de SignedNode. **Aguardando decisão do arquiteto** — não há fonte canônica.

**Impacto enquanto aberto:**
- T-302b pode ser implementada e testada unitariamente (5 casos da §4 não dependem do wire format)
- A integração end-to-end com `reconcile` de T-302a fica pendente até decidir (a) / (b) / (c)
- Sugere-se criar T-302c para a escolha arquitetural, ou marcar T-302b como `blocked-decision`
  e resolver antes de promover

**Decisão:** manter T-302b como `hardened` (corpo executável) e criar pendência separada
para a evolução do wire format. `applyNodes` fica testada e pronta — quem chamar passa um
`SignedNode[]` (não bytes), que é a forma natural de uso dentro do monorepo.
