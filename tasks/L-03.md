#### 🔴 BLOQUEANTE

**[B1] `assertFiscalTransition` não é exportada por nenhum barrel → o entregável é inalcançável fora do `core`.**
- `packages/core/src/index.ts` exporta os **cinco** irmãos do módulo (`createMoRContext`, `stepRequiresMoR`, `evaluateBlockingRules`, `checkBlockingRules`, `BlockingError`) — mas **não** `assertFiscalTransition`. `packages/core/src/workflow/index.ts` idem.
- Prova de que é lapso e não design: o **tipo** `FiscalTransition` FOI tornado público (`protocol/src/index.ts`), mas a **função** que o consome não. Exportaram o contrato e esconderam a implementação.
- Sintoma: `tests/mor-hardstop.test.ts` importa por caminho profundo (`../src/workflow/workflow-engine.js`), enquanto o teste irmão `workflow-blocking.test.ts` importa do barrel (`@plataforma/core`). O teste **dribla** o caminho público — por isso o gate fica verde com a API quebrada.
- Por que é bloqueante e não cosmético: a decisão da §6 exige que a transição sem NF-e **"falhe, travando UI e orquestração"**. UI e orquestração vivem **fora** do `core`. Sem export, nada fora do pacote consegue sequer chamar o hard stop — é o `[gate de wiring]` da §7: *primitiva inalcançável = feature NÃO entregue*. Para uma trava fiscal legal (NF-e obrigatória no BR), entregar código morto é o pior modo de falha.
- **Correção:** exportar `assertFiscalTransition` (e o tipo `MoRHardStopResult` / `evaluateMoRHardStop`, se fizer sentido para o consumidor) de `packages/core/src/index.ts` e `packages/core/src/workflow/index.ts`, seguindo o padrão dos irmãos; e **trocar o import do teste para o barrel** (`@plataforma/core`), que é o que prova o caminho público. **A §3 foi emendada** para declarar os barrels — o rework está autorizado a tocá-los, não é ampliação de escopo.
