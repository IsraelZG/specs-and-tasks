## 1. Objetivo
Costurar scheduler (T-1020), gate (T-1021) e adapter (T-1022) com o `TaskService` no loop
**serial supervisionado** (K=1) do ADR: por task, dispara worker → confere que virou `review`
→ roda o gate determinístico → dispara reviewer → em conflito de verdito faz rework (até N) →
escala ao humano nas condições definidas. **Fase 1: NÃO faz merge** (isso é a T-1024/Fase 2);
ao aprovar, a task simplesmente fica `done`.

  ```ts
  import { TaskService } from '../services/task.service.js';
  import type { AgentAdapter } from './agent-adapter.js';

  export type EscalationReason =
    | 'rework_exceeded' | 'paused_or_blocked' | 'agent_crash_or_timeout'
    | 'malformed_run' | 'opus_spike' | 'gate_verdict_mismatch';

  export interface RunnerOptions {
    taskService: TaskService;                          // ledger central
    adapter: AgentAdapter;                             // T-1022
    worktreeFor: (taskId: string) => string;           // cwd por task (injetável; stub nos testes)
    verify: (cwd: string, taskId: string) => Promise<{ ok: boolean; output: string }>;  // T-1021
    maxRework?: number;                                // default 2
    supervised?: boolean;                              // default true
    confirm?: (action: string, taskId: string) => Promise<boolean>;  // usado se supervised
    onEscalate: (taskId: string, reason: EscalationReason, detail: string) => void;
    log?: (line: string) => void;
  }

  export interface TaskOutcome {
    taskId: string;
    final: 'done' | 'escalated';
    rounds: number;                                    // nº de idas ao worker
    reason?: EscalationReason;
  }

  export class WaveRunner {
    constructor(opts: RunnerOptions);
    /** Processa UMA task ponta-a-ponta (serial). Ver máquina abaixo. */
    runTask(taskId: string): Promise<TaskOutcome>;
    /** Calcula o ready-set (via scheduler sobre taskService.listTasks) e roda uma por vez. */
    runWave(): Promise<TaskOutcome[]>;
  }
  ```

### Máquina de `runTask` (determinística)
1. Lê a task. Se `capacity === 'opus-spike'` → `onEscalate(id,'opus_spike')`, retorna escalated.
2. (se `supervised`) `confirm('worker', id)`; se falso → escalated `malformed_run` ("não confirmado").
3. `adapter.run({ role:'worker', taskId:id, cwd:worktreeFor(id), env, timeoutMs })`.
   - `timedOut || exit!==0` → `onEscalate('agent_crash_or_timeout')`, escalated.
4. Relê status no ledger:
   - `review` → segue. `blocked`/`in_progress`(pausa) → `onEscalate('paused_or_blocked')`.
   - qualquer outro → `onEscalate('malformed_run')`.
5. **Gate:** `verify(cwd,id)`. Se `!ok` → `taskService.transition(id,'request_changes','runner',
   'gate: build/test falhou\n'+output[trunc])` → `rounds++`; se `rounds > maxRework` →
   `onEscalate('rework_exceeded')` escalated; senão volta ao passo 2 (worker reassume rework).
6. Gate ok → `adapter.run({ role:'reviewer', ... })`. timeout/exit!=0 → escalated `agent_crash_or_timeout`.
7. Relê status: `done` → **gate final** `verify` de novo; se `!ok` → `onEscalate('gate_verdict_mismatch')`
   escalated; se ok → retorna `{ final:'done', rounds }`.
   `rework` → `rounds++`; `>maxRework` → escalated `rework_exceeded`; senão volta ao passo 2.
   outro → `onEscalate('malformed_run')`.

> O runner transita via `TaskService` SOMENTE como `'runner'` no caso 5 (`request_changes`
> automático do gate). O `start/finish` são do worker; o `approve/request_changes` de mérito
> são do reviewer — o runner não os forja.

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Vitest. Ledger real (`TaskService` num tasks dir temporário) + **adapter FAKE**.
- [ ] **Setup:** crie um `FakeAdapter implements AgentAdapter` cujo `run` recebe um roteiro
  injetado e EFETUA as transições no `TaskService` para simular o agente (ex.: no papel
  `worker` chama `transition(id,'start',...)`+`transition(id,'finish',...)`; no `reviewer`
  chama `approve` ou `request_changes`). `worktreeFor` → um dir tmp qualquer. `verify` → stub
  que devolve `{ok}` programável por teste.
- [ ] **Cobertura:**
  - **happy path:** worker→review, gate ok, reviewer→done → `final:'done'`, `rounds:1`.
  - **gate vermelho:** verify `{ok:false}` na 1ª volta, `{ok:true}` na 2ª; worker conserta →
    `done` com `rounds:2`.
  - **rework excedido:** reviewer sempre `request_changes` → após `maxRework` → `escalated`
    `rework_exceeded`.
  - **opus-spike:** task com `Capacidade-alvo: opus-spike` → `escalated` `opus_spike` sem spawnar.
  - **worker não moveu p/ review** (FakeAdapter no-op) → `escalated` `malformed_run`.
  - **agent timeout:** FakeAdapter devolve `{timedOut:true, exit:null}` → `escalated`
    `agent_crash_or_timeout`.
  - **gate_verdict_mismatch:** reviewer aprova mas o gate FINAL devolve `{ok:false}` → escalated.
  - **supervised:** `confirm` retornando false impede o spawn.
- [ ] **Ambiente:** Node puro, sem rede, sem git real, sem pnpm real (tudo via fakes/stubs).
- [ ] **Fora de Escopo:** merge (T-1024); paralelismo K>1 (Fase 3); invocação real de agentes (T-1017).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - NÃO fazer merge nem tocar git aqui (Fase 1 termina em `done`).
> - NÃO forjar `approve`/`finish` em nome do agente — só o `request_changes` do gate é do `runner`.

> - NÃO usar K>1 — serial (uma task por vez) nesta task.

### Pegadinhas conhecidas
- **`transition` é async** (T-1012) — `await` todas as chamadas ao `TaskService`.
- **Relê o status do ledger após cada spawn** (`getTask(id).frontmatter.status`) — não deduza do
  exit code do agente; o exit só diz se o processo terminou, não o que ele fez.
- **`rounds` conta idas ao worker** (passo 2), não spawns totais. Compare `> maxRework`.
- **Gate roda 2x no caminho feliz:** antes do reviewer (passo 5) e como checagem final (passo 7).
  É proposital (pega regressão entre o gate inicial e o approve).
- **Injete tudo que toca o mundo** (`adapter`, `verify`, `worktreeFor`, `confirm`) para o teste
  rodar 100% offline.
