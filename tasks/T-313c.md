# T-313c · Archive Cargo: GC + migration blind_archives

> Subtask de T-313 (decomposta 2026-06-29, arquiteto opção B). Adiciona a tabela
> `blind_archives` ao schema SQLite de `device_state.db` e implementa o GC por
> `expires_at`. T-313a (pack/unpack/verify) e T-313b (assignCustodian) devem estar
> `done` antes desta (tipos + ring); T-806 (WebTorrent Adapter) depende desta para
> consumir archives via storage.

- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`

- **Test Runner:** `vitest` (Node puro) + SQLite `:memory:` para migration tests
- **Capacidade-alvo:** sonnet

## 1. Objetivo

Adicionar a tabela `blind_archives` ao `device_state.db` (schema de T-106) e
implementar o GC por `expires_at`. A tabela é cega: armazena apenas o `archiveId`,
`ciphertext`, `manifest` e `expiresAt` — **nunca** `pub_key` ou autoria. O GC é
um sweep atômico que remove linhas com `expires_at < now()`, retornando a contagem
de removidos.

*(derivado de `tasks/T-313.md` §1 + §4 caso 9; `tasks/T-106.md` — schema SQLite;
`tasks/T-004.md` — `StoragePort`; `docs/caderno-1-foundation/03-storage-engine.md`)*

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/core/src/archive/blindArchives.ts ---
import type { StoragePort } from '@plataforma/protocol';                // T-004
import type { ArchiveCargo, BlindScopeId } from './archiveCargo.js';   // T-313a

/** Linha da tabela `blind_archives` (D1 do plano de migration). */
export interface BlindArchiveRow {
  /** PK — `archiveId` (blake2s256 do ciphertext). */
  archiveId: Uint8Array;          // BLOB(32) PRIMARY KEY
  /** FK conceitual para `blind_scope_id` (não enforced por SQL — T-313b associa). */
  blindScopeId: BlindScopeId;     // BLOB(32) NOT NULL
  /** Manifesto serializado MessagePack (T-203). */
  manifest: Uint8Array;           // BLOB NOT NULL
  /** Ciphertext completo (zstd + AES-256-GCM + tag). */
  ciphertext: Uint8Array;        // BLOB NOT NULL
  /** Timestamp Unix ms — GC remove onde `expires_at < now()`. */
  expiresAt: number;              // INTEGER NOT NULL
  /** Quando foi inserido (Unix ms) — para auditoria. */
  createdAt: number;              // INTEGER NOT NULL
}

/** ID da migration que cria `blind_archives` — adicionar a `MIGRATIONS` de T-106. */
export const MIGRATION_BLIND_ARCHIVES_ID = 'V003__blind_archives';

/** SQL DDL (apenas a parte da tabela; `device_state.db` é gerenciado por T-106). */
export const MIGRATION_BLIND_ARCHIVES_SQL: string; // CREATE TABLE blind_archives (...)

/** Insere archive. Idempotente por `archiveId` (PRIMARY KEY). Lança se já existe. */
export function putBlindArchive(
  storage: StoragePort,
  cargo: ArchiveCargo,
  expiresAt: number,
): Promise<void>;

/** Busca archive por `archiveId`. Retorna `null` se não existe ou expirado. */
export function getBlindArchive(
  storage: StoragePort,
  archiveId: Uint8Array,
  /** Se `true`, inclui expirados (para debug). */
  includeExpired?: boolean,
): Promise<BlindArchiveRow | null>;

/** Lista archives por `blindScopeId` (não expirados). */
export function listBlindArchivesByScope(
  storage: StoragePort,
  blindScopeId: BlindScopeId,
  limit?: number,
): Promise<BlindArchiveRow[]>;

/**
 * GC atômico: remove archives com `expires_at < now()`.
 * Retorna a quantidade de linhas removidas. Operação em transação única.
 */
export function garbageCollectArchives(
  storage: StoragePort,
  now?: number,   // injetado para testes determinísticos (T-003 VirtualClock)
): Promise<number>;
```

> **Migration:** `MIGRATION_BLIND_ARCHIVES_SQL` é a string DDL que deve ser adicionada
> ao array `MIGRATIONS` exportado de `packages/core/src/schema.ts` (T-106). A migration
> cria a tabela no `device_state.db` ao chamar `migrateLatest(storage)`. T-313c **adiciona**
> a migration; não reescreve o mecanismo de migration (já existe, T-106).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** armazene `pub_key` ou autoria na tabela — a tabela é **cega**. Apenas `archiveId`/`blindScopeId`/`manifest`/`ciphertext`/`expiresAt`/`createdAt`.
> - **NÃO** use timestamps reais (`Date.now()`); use `ClockPort.now()` (T-004) ou `now` injetado (T-003 VirtualClock).
> - **NÃO** reescreva o mecanismo de migration de T-106; apenas adicione uma entrada em `MIGRATIONS`.
> - **NÃO** use `setTimeout`/`setInterval` para "rodar GC periodicamente" — GC é sob demanda (chamada explícita). Scheduler de GC é T-313+.

### Pegadinhas conhecidas
- **PK BLOB em SQLite:** `archiveId` é `Uint8Array(32)`. SQLite tem `BLOB` nativo; o `StoragePort.exec` de T-004 deve aceitar `Uint8Array` como parâmetro. Se T-004 não suportar, documentar a limitação e usar hex/base64 — preferir `Uint8Array` direto.
- **Migration ordering:** T-106 exporta `MIGRATIONS` como array ordenado por ID (`V001`, `V002`, ...). Anexar `V003__blind_archives` **no fim**. **NÃO** reordenar.
- **Transação do GC:** o sweep inteiro deve ser em `storage.transaction(async tx => ...)` (T-004); evita "GC parcial" se falhar no meio.
- **`manifest` MessagePack:** serializar `ArchiveManifest` (T-313a) via codec de T-203 (bigint nativo confirmado por T-212). Reusar `codecEncode`/`codecDecode` exportados de `@plataforma/protocol`.
