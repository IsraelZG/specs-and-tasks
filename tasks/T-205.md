# T-205 · SwarmRegistry em RAM

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo

## 1. Objetivo e Contratos
Criar o registro de peers em RAM (`SwarmRegistry`) indexado por `DevicePeerId`, mantendo estado de conexão, latência (EWMA), tier declarado, score de relay, época de identidade e a tabela auxiliar `device_personas`.
*(derivado de `docs/conceitos/swarm-registry.md`; `docs/plano-de-implementacao.md` §5 T-205)*.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/transport/src/SwarmRegistry.ts ---
import { PeerId } from '@plataforma/protocol';

/** Tier de capacidade declarado pelo peer. */
export type PeerTier = 'light' | 'standard' | 'super';

/** Estado de conexão de um peer no registry. */
export type ConnectionState = 'disconnected' | 'connecting' | 'connected';

/** Entrada de um peer no SwarmRegistry. */
export interface PeerEntry {
  readonly deviceId: PeerId;
  state: ConnectionState;
  /** Latência estimada (EWMA, ms). */
  latencyMs: number;
  /** Tier declarado pelo peer. */
  tier: PeerTier;
  /** Score local de relay (não-transitivo). */
  relayScore: number;
  /** Época de identidade vigente do peer. */
  identityEpochIndex: number;
  /** Timestamp da última atividade (tráfego ou PONG). */
  lastActiveAt: number;
  /** Personas ativas/atestadas neste dispositivo. */
  personas: Set<PeerId>;
  /**
   * true enquanto este peer atua como **system peer provisório** durante a gênese
   * (FPP/T-208 em GENESIS, antes de um segundo peer real entrar). Set/clear via `upsert`.
   * Default ausente/`false`. *(Campo aditivo solicitado por T-208 decisão 4; observável por
   * T-206/207/404.)*
   */
  isProvisionalSystemPeer?: boolean;
}

/** Eventos emitidos pelo SwarmRegistry. */
export type RegistryEvent =
  | { type: 'peer_joined'; deviceId: PeerId }
  | { type: 'peer_left'; deviceId: PeerId }
  | { type: 'peer_state_changed'; deviceId: PeerId; from: ConnectionState; to: ConnectionState }
  | { type: 'persona_added'; deviceId: PeerId; personaId: PeerId }
  | { type: 'persona_removed'; deviceId: PeerId; personaId: PeerId };

export type RegistryEventHandler = (event: RegistryEvent) => void;

/**
 * Registro de peers em RAM — fonte de verdade para decisões de roteamento,
 * eleição de líder e shadowban de relays.
 *
 * Derivado de docs/conceitos/swarm-registry.md §Definição.
 */
export class SwarmRegistry {
  /** Adiciona ou atualiza peer. Se já existe, mescla campos (não sobrescreve
   *  `relayScore` nem `lastActiveAt` sem delta explícito). */
  upsert(deviceId: PeerId, partial: Partial<Omit<PeerEntry, 'deviceId'>>): void;

  /** Remove peer do registry (evicção ou close limpo). */
  remove(deviceId: PeerId): void;

  /** Retorna entrada ou undefined. */
  get(deviceId: PeerId): PeerEntry | undefined;

  /** Todos os peers atualmente registrados. */
  all(): PeerEntry[];

  /** Peers com `state === 'connected'`. */
  connected(): PeerEntry[];

  /** Atualiza `lastActiveAt` para agora (heartbeat implícito). */
  touch(deviceId: PeerId, now?: number): void;

  /** Registra callback para eventos de entrada/saída/mudança de estado. */
  onEvent(handler: RegistryEventHandler): () => void;

  /** Tamanho atual do registry. */
  get size(): number;
}
```

> **device_personas:** a tabela auxiliar de personas por dispositivo é representada
> pelo campo `personas: Set<PeerId>` em `PeerEntry`. Derivado de `docs/plano-de-implementacao.md`
> §5 T-205 ("tabela auxiliar device_personas: personas ativas/atestadas por dispositivo").

> **Em RAM apenas:** NÃO persistir no SQLite. O registry é volátil, reconstruído a
> cada inicialização do peer. Derivado do spec original T-205 §5.

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [x] **Framework:** Vitest (Node puro).
- [x] **Ambiente do Teste:** Node puro, `pnpm --filter @plataforma/transport test`.

Casos de teste (numerados):
1. **Registro vazio:** `new SwarmRegistry()` → `size === 0`, `all()` retorna `[]`, `connected()` retorna `[]`.
2. **Upsert insere peer:** `upsert('peer-A', { state: 'connected', tier: 'standard' })` → `get('peer-A')` retorna entrada com campos fornecidos e defaults (`latencyMs: 0`, `relayScore: 0`, `identityEpochIndex: 0`).
3. **Upsert atualiza peer:** `upsert('peer-A', { state: 'connecting' })` → `get('peer-A').state === 'connecting'`, campos não passados mantêm valores anteriores.
4. **Remove peer:** `upsert('peer-A', {})`; `remove('peer-A')` → `get('peer-A') === undefined`, `size === 0`.
5. **Get inexistente:** `get('peer-X')` → `undefined`.
6. **all() retorna todos:** upsert 3 peers → `all().length === 3`.
7. **connected() filtra:** upsert 2 connected + 1 disconnected → `connected().length === 2`.
8. **touch atualiza lastActiveAt:** `upsert('peer-A', {})`; `touch('peer-A', 1000)` → `get('peer-A').lastActiveAt === 1000`.
9. **Evento peer_joined:** handler registrado antes do upsert → chamado com `{ type: 'peer_joined', deviceId: 'peer-A' }`.
10. **Evento peer_left:** handler registrado; remove peer → `{ type: 'peer_left', deviceId: 'peer-A' }`.
11. **Evento peer_state_changed:** upsert com state='connected'; upsert com state='disconnected' → evento `{ type: 'peer_state_changed', from: 'connected', to: 'disconnected' }`.
12. **Persona add/remove:** `upsert('peer-A', { personas: new Set(['persona-1']) })` → evento `persona_added`. Atualizar personas → eventos correspondentes.
13. **Unsubscribe de eventos:** `const unsub = registry.onEvent(handler)`; `unsub()`; upsert peer → handler NÃO é chamado.
14. **Multi-handler:** dois handlers registrados → ambos recebem o mesmo evento.
