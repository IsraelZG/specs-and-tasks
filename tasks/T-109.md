# T-109 · NetworkAdapterPort multi-subscriber (onMessage com unsubscribe)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (pacote `protocol` é Node puro, sem browser)
- **Capacidade-alvo:** sonnet *(mudança de assinatura de interface + semântica de concorrência de handlers — pequena mas exige atenção a edge cases, não é puro boilerplate)*

## 1. Objetivo
A T-004 (`status: done`) definiu `NetworkAdapterPort` com `onMessage(handler: MessageHandler): void` — um único handler, e o próprio JSDoc do método documenta que "Sobrescreve callback anterior". Isso é insuficiente: qualquer código que precise observar mensagens de rede para mais de uma finalidade (ex.: um listener de protocolo + um listener de métricas/debug) hoje colide — o segundo `onMessage()` silenciosamente substitui o primeiro.

Esta task altera o contrato para multi-subscriber: `onMessage` aceita múltiplos handlers simultâneos e retorna uma função de `unsubscribe` para remover apenas aquele handler.

**Por que agora:** nenhum adapter concreto (`T-204` WebSocket, `T-402` WebRTC) foi implementado ainda — a interface só tem implementações dummy/teste. Migrar agora é uma mudança de assinatura sem custo de migração real (zero call sites em produção a atualizar). Adiar para depois de T-204/T-402 existirem tornaria a mudança uma quebra de API com retrabalho nos dois adapters.

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Vitest, Node puro (sem browser, sem JSDOM)
- [ ] **Métricas/Cobertura:** todos os casos numerados abaixo devem ter um `it()` correspondente

- [ ] **Fora de Escopo:** qualquer teste de transporte real (WebSocket, WebRTC, NAT) — isso pertence a T-204/T-402

### Casos de teste (numerados — implementar 1:1)
1. Múltiplos handlers registrados via `onMessage` recebem a mesma mensagem quando o adapter emite uma (broadcast — nenhum handler "rouba" a mensagem do outro).
2. A função retornada por `onMessage` remove **apenas** aquele handler — outros handlers registrados continuam recebendo mensagens subsequentes normalmente.
3. Chamar a função de unsubscribe duas vezes é um no-op idempotente (não lança erro, não afeta outros handlers).
4. Um handler registrado **depois** de uma mensagem já emitida não recebe replay dessa mensagem passada — só recebe mensagens futuras.
5. `close()` no adapter limpa todos os handlers registrados; nenhum handler é chamado após `close()` resolver.
6. A implementação dummy/stub de teste compila com a nova assinatura (`onMessage(handler): () => void`) sem `any`/cast — prova de que o contrato é type-safe.
7. Código que ignora o valor de retorno de `onMessage()` (não captura a função de unsubscribe) continua compilando — TypeScript permite descartar um valor de retorno não-`void`, então chamadores que não precisam de unsubscribe não são forçados a mudar.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - Não implemente debounce, buffer ou fila de mensagens — isso é puramente sobre fan-out de múltiplos handlers, não sobre semântica de entrega.
> - Não adicione prioridade/ordem garantida entre handlers — a ordem de invocação pode ser a de registro, mas não documente isso como contrato (mantém liberdade de implementação para os adapters reais).
> - Não toque em `packages/protocol/src/ports.ts` fora de `MessageHandler`/`NetworkAdapterPort`/JSDoc relacionado — não refatore outras portas do mesmo arquivo (ex.: `StoragePort`, se existir no mesmo módulo).

### Pegadinhas conhecidas
- O JSDoc atual de `onMessage` diz "Sobrescreve callback anterior" — isso precisa ser **removido/corrigido** junto com a assinatura, ou o próximo agente que ler só o comentário (sem ler o código) vai implementar o adapter errado.
- `MessageHandler` (tipo do handler) **não muda** — é o retorno de `onMessage` que muda de `void` para `() => void`. Não confunda os dois.
- Ao trocar `Set<MessageHandler>` (ou estrutura equivalente) internamente no dummy de teste, cuidado com handlers duplicados (mesma referência de função registrada duas vezes) — decida e documente explicitamente no teste se isso conta como 1 ou 2 assinaturas (sugestão: tratar como 2 assinaturas independentes, cada `onMessage()` retorna um unsubscribe próprio, mesmo que a função seja a mesma referência — assim o comportamento é previsível por chamada, não por identidade de função).
