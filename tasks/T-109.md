# T-109 · NetworkAdapterPort multi-subscriber (onMessage com unsubscribe)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
A T-004 (`status: done`) definiu `NetworkAdapterPort.onMessage(handler): void` — single handler, "Sobrescreve callback anterior". Esta task altera o contrato para multi-subscriber: `onMessage` aceita múltiplos handlers simultâneos e retorna uma função de `unsubscribe` para remover apenas aquele handler.

**Por que agora:** os adapters de *produção* (T-204 WebSocket, T-402 WebRTC) ainda não existem — então o único implementador real do port hoje é a **`SimNetwork`** (testkit, T-005 `done`). Migrar o contrato agora custa só atualizar a SimNetwork, e ela vira a **implementação de referência** do padrão multi-subscriber que T-204/T-402 vão copiar. Adiar encareceria a migração a cada novo adapter.

> **Esta task migra a SimNetwork junto (Seção 3).** Ela implementa `NetworkAdapterPort`; sem migrá-la, o testkit deixa de compilar contra o contrato novo. Quem muda o contrato conserta os implementadores — não se deixa um implementador real para trás.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/protocol/src/ports.ts (UPDATE) ---

// MessageHandler (sem mudança)
export type MessageHandler = (peerId: PeerId, data: WireData) => void;

// NetworkAdapterPort (onMessage muda o retorno)
export interface NetworkAdapterPort {
  connect(peerId: PeerId): Promise<void>;
  listen(): Promise<void>;
  send(peerId: PeerId, data: WireData): Promise<void>;
  /**
   * Registra callback para mensagens recebidas.
   * Suporta múltiplos handlers simultâneos (broadcast — nenhum handler rouba a mensagem do outro).
   * @returns função de unsubscribe para remover **apenas** este handler.
   * @remarks Chamar a função de unsubscribe duas vezes é idempotente.
   */
  onMessage(handler: MessageHandler): () => void;
  close(): Promise<void>;
}
```

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [x] **Framework:** Vitest, Node puro
- [x] **Fora de Escopo:** WebSocket, WebRTC, NAT, transporte real (T-204/T-402)

### Casos de teste (numerados — implementar 1:1 em `ports.test.ts`)
1. Múltiplos handlers registrados via `onMessage` recebem a mesma mensagem (broadcast).
2. Unsubscribe remove **apenas** aquele handler — outros continuam.
3. Unsubscribe chamado 2× é idempotente.
4. Handler registrado **depois** de uma mensagem não recebe replay.
5. `close()` limpa todos os handlers — nenhum handler é chamado após `close()` resolver.
6. Stub/dummy compila com a nova assinatura sem `any`/cast (prova de type-safety).
7. Código que ignora o valor de retorno de `onMessage()` continua compilando (TS permite descartar `() => void`).

### Casos adicionais — SimNetwork (implementador real; em `SimNetwork.subscriber.test.ts`)
8. Dois handlers registrados via `onMessage` no mesmo peer da SimNetwork recebem a mesma mensagem (broadcast real, não só no stub).
9. `unsubscribe()` de um handler remove **apenas** ele; o outro continua recebendo.
10. `close()` no adapter limpa **todos** os handlers — nenhum recebe após `close()`.
11. **Timing multi-subscriber (M1, herdado de T-005):** `const un = b.onMessage(h1); a.send('B', x) /* sem await */; un(); b.onMessage(h2);` → após a entrega, `h2` 1× e `h1` 0× (o conjunto de handlers é lido **na entrega**, não no `send`).
12. Os 12 casos de `SimNetwork.test.ts` (T-005) seguem verdes — com o caso 9 já na forma `unsubscribe`.

### Pegadinhas conhecidas
- O JSDoc atual "Sobrescreve callback anterior" precisa ser **removido/corrigido** junto com a assinatura, ou o próximo agente implementa errado.
- `MessageHandler` **não muda** — é o retorno de `onMessage` que muda.
- Handlers duplicados (mesma função registrada 2×): tratar como 2 assinaturas independentes (cada `onMessage()` retorna unsubscribe próprio).
- **SimNetwork é implementador real — migrá-la não é opcional.** `onMessage` passa de `internal.onMessage = handler` (substitui) para `handlers.add(handler)` + retorno `() => { handlers.delete(handler); }` (idempotente). O `deliverMsg` itera um **snapshot do conjunto no momento da entrega** (`[...handlers]`) — preserva M1/M2: handler que deu `unsubscribe` antes da entrega não é chamado; `close()` esvazia o conjunto.
- **Caso 9 de T-005 muda de FORMA, não de intenção:** single-handler "swap por substituição" → multi-subscriber "swap por `unsubscribe()` + novo `onMessage`". Continua testando leitura-na-entrega. **Não** altere os outros 11 casos.

1. **[TDD]** Em `ports.test.ts`, ajuste o dummy/stub e adicione os casos 1–7.
2. Implemente a nova assinatura em `ports.ts` e atualize o JSDoc.
3. **[Migração SimNetwork]** Em `packages/testkit/src/SimNetwork.ts`: troque `onMessage: MessageHandler | null` por `Set<MessageHandler>`; `onMessage(h)` faz `handlers.add(h)` e retorna `() => { handlers.delete(h); }`; `deliverMsg` itera `[...handlers]` (snapshot) chamando cada um; `close()` faz `handlers.clear()`.
4. Atualize o caso 9 de `SimNetwork.test.ts` para a forma `unsubscribe`; crie `SimNetwork.subscriber.test.ts` (casos 8–12).
