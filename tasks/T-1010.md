# T-1010 · Serviço de Tasks (máquina de estados MGTIA) no Nexus

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo
- **Test Runner:** `vitest`

## 1. Objetivo
Centralizar **toda** a escrita de tasks num serviço reusável no `nexus-backend`,
que se torna a única fonte de verdade do ciclo de vida MGTIA. O serviço deve fazer,
numa única operação atômica: reescrever o `status` no frontmatter, fazer append
datado no Log de Execução (seção 9) e regenerar o `tasks/INDEX.md`. Esse serviço
substitui a lógica hoje espalhada em `tools/scripts/manage-task.mjs` (que só faz
start/pause/finish) e passa a suportar o ciclo de vida completo.

2. Defina a máquina de estados em `task.types.ts`:
   - status: `draft | ready | in_progress | review | rework | done | blocked`
   - ações → transições: `start` (draft|ready→in_progress), `pause` (in_progress→in_progress, grava handoff), `finish` (in_progress→review), `approve` (review→done), `request_changes` (review→rework), `block` (*→blocked), `unblock` (blocked→ready). `logProgress` não muda status.
3. Implemente `task.service.ts`:
   - `getTasksDir()` resolve `tasks/` a partir de uma raiz configurável (ROOT_DIR), default = raiz do repo; permitir override por arg/env p/ testes.

   - Append de log no formato existente: `- **[YYYY-MM-DDTHH:mm]** - *${agent}* - \`[${rotulo}]\`: ${message}`.
   - Após cada mutação, regenerar o INDEX (portar `rebuildIndexes`).
4. Refatore e garanta tipagem estrita (sem `any`).
