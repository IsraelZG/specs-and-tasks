# T-021 · Arcabouço de E2E Playwright compartilhado (`@plataforma/testkit`) + showcase como 1º consumidor

> **Esta task entrega uma FERRAMENTA, não testes de um único app.** O E2E de UI nasce como um preset compartilhado em `@plataforma/testkit`, consumido por **qualquer** app do superapp (`apps/web`, `apps/bancada`, `apps/system-peer` quando saírem de stub; ver T-009 casos 8–9, hoje adiados). O **showcase do design-system é apenas o primeiro consumidor** — escolhido por ser o único alvo renderável hoje. As specs E2E concretas (os 5 componentes) provam a ferramenta; elas **não** são o ponto, o preset reutilizável é.

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `playwright` (browser real — Chromium headless no CI)
- **Capacidade-alvo:** sonnet *(o preset é uma função fina sobre `defineConfig`; o julgamento está em qual interação real cada componente precisa — drag, foco, transição — não repetível por find-and-replace)*

## 1. Objetivo
Criar o **arcabouço de testes de UI E2E do superapp** como um preset compartilhado e fazer o showcase consumi-lo:

1. **A ferramenta:** `createPlaywrightConfig(...)` em `@plataforma/testkit` (pacote workspace já existente) — uma função fina sobre o `defineConfig` nativo do Playwright que fixa os defaults do projeto (Chromium, `webServer` com reuse local / start no CI, `retries` no CI, reporter, `forbidOnly` no CI). Qualquer app passa só o seu comando de boot e a baseURL. **Sem classe, sem plugin system** — uma função que retorna `PlaywrightTestConfig`.
2. **O 1º consumidor:** `apps/design-system-showcase/playwright.config.ts`, um arquivo de ~5 linhas que chama o preset apontando para o harness de boot já existente `pnpm --filter design-system-showcase dev` (T-012 caso 3 / T-015 smoke caso 5).
3. **A prova:** specs E2E dos 5 componentes do design-system onde o JSDOM da T-018 é insuficiente (drag real, focus-trap real, clique-fora com stacking context real).
4. **O CI:** um pipeline task `test:e2e` no Turbo, de modo que **todo app com script `test:e2e`** entra no CI automaticamente (showcase hoje, `bancada`/`web` depois — sem editar o CI de novo).

**Por que `@plataforma/testkit` e não um config local no design-system:** um `playwright.config.ts` enfiado em `packages/design-system` não é compartilhável — seria a "reinvenção em cada app" que esta task existe para evitar. O testkit já é o pacote de utilitários de teste do monorepo (`@plataforma/testkit`, usado por client-sdk/core/etc.); o preset de E2E mora lá pelo mesmo motivo.

**Por que não é uma task de E2E de fluxo de produto:** hoje `apps/web`, `apps/bancada` e `apps/system-peer` são stubs sem UI renderável — não há fluxo real para testar ainda. O único alvo renderável é o showcase. Esta task não inventa fluxos inexistentes; cobre a lacuna real (interações que `@testing-library/react` + JSDOM da T-018 não simula com fidelidade) **e** deixa a ferramenta pronta para os apps reais herdarem.

### Contrato exato (assinatura TS fixada do preset)

```ts
// --- packages/testkit/src/playwright.ts ---
import { defineConfig, type PlaywrightTestConfig } from "@playwright/test";

export interface UiE2EOptions {
  /** Comando que sobe o app sob teste. Ex.: "pnpm --filter design-system-showcase dev". */
  webServerCommand: string;
  /** URL onde o app responde quando pronto. Ex.: "http://localhost:5173". */
  baseURL: string;
  /** Diretório dos specs, relativo ao config do consumidor. Default: "./e2e". */
  testDir?: string;
}

/**
 * Preset de E2E compartilhado do superapp. Todo app consome isto em vez de
 * escrever um playwright.config do zero.
 * Defaults fixados (consumidor NÃO repete): projeto único Chromium; webServer
 * com { command: webServerCommand, url: baseURL, reuseExistingServer: !process.env.CI };
 * retries: process.env.CI ? 2 : 0; forbidOnly: !!process.env.CI; reporter "line".
 */
export function createPlaywrightConfig(opts: UiE2EOptions): PlaywrightTestConfig {
  return defineConfig({
    testDir: opts.testDir ?? "./e2e",
    forbidOnly: !!process.env.CI,
    retries: process.env.CI ? 2 : 0,
    reporter: "line",
    use: { baseURL: opts.baseURL },
    projects: [{ name: "chromium", use: { /* devices['Desktop Chrome'] */ } }],
    webServer: {
      command: opts.webServerCommand,
      url: opts.baseURL,
      reuseExistingServer: !process.env.CI,
    },
  });
}
```
> O corpo acima é a referência; o Worker pode ajustar detalhes do `projects[0].use` (ex.: importar `devices` de `@playwright/test`), mas **a assinatura e os defaults são fixos**.

### Lista de componentes desta task (apenas onde browser real importa)
1. **Slider** — arraste real de pointer (mousedown na thumb → mousemove com coordenadas → mouseup) deve mover o valor; JSDOM não tem layout real para calcular posição.
2. **Carousel** — swipe/drag real entre slides; depende de cálculo de largura real do container, que JSDOM não renderiza.
3. **Resizable** — arraste do handle de resize altera dimensão real do painel; depende de layout real.
4. **Modal** — foco real preso dentro do diálogo (focus-trap) ao pressionar Tab repetidamente atravessando o DOM real, incluindo elementos fora do componente que deveriam ficar inacessíveis — diferente do teste JSDOM da T-018, que verifica o listener de teclado, não o resultado real do foco do browser.
5. **Sheet** e **Popover** — comportamento de `z-index`/clique-fora real depende de stacking context calculado pelo motor de renderização, não simulável com fidelidade em JSDOM.

- [x] **Fora de Escopo:** qualquer asserção visual/pixel (isso é T-020/gate visual); componentes não listados; fluxos de produto em `apps/*` (não existem ainda); E2E da Bancada (T-009 casos 8–9 — consumidor futuro do mesmo preset, fora desta task).

Casos de teste (numerados — um por componente + ferramenta + infra):

**Ferramenta (preset compartilhado):**
0. `createPlaywrightConfig({ webServerCommand: "x", baseURL: "http://localhost:1" })` retorna um `PlaywrightTestConfig` com `webServer.command === "x"`, `webServer.url === "http://localhost:1"`, `testDir === "./e2e"` (default) e `reuseExistingServer === !process.env.CI`. *(É o teste que prova que a ferramenta é genérica, não hardcoded para o showcase. Vitest no `@plataforma/testkit`.)*

**Slider (arraste real de pointer):**
1. `mousedown` na thumb, `mousemove` com coordenadas calculadas a partir de `boundingBox()`, `mouseup` → valor do slider alterado para o esperado (ex.: de 50% para 75%). *(Derivado de T-018 §4 Tier A2.)*

**Carousel (swipe/drag entre slides):**
2. `mousedown` no container, `mousemove` com deslocamento horizontal > 50px, `mouseup` → slide visível mudou (`data-active` ou índice diferente).

**Resizable (drag do handle):**
3. `mousedown` no handle, `mousemove` com coordenadas calculadas, `mouseup` → `boundingBox().width` diferente do inicial.

**Modal (focus-trap com Tab real):**
4. Modal aberto → ciclo completo de Tab até `document.activeElement` voltar ao primeiro focável do modal; nenhum elemento fora do modal recebe foco.

**Sheet e Popover (stacking context / clique-fora):**
5. Sheet aberto → clique fora → Sheet fecha (ausente do DOM ou `hidden`).
6. Popover aberto → clique fora → Popover fecha.

**Infra (CI):**
7. `pnpm turbo run test:e2e` no CI (runners `ubuntu-latest` x64 e `ubuntu-24.04-arm` arm64 da matriz T-019) → todos os specs passam. Browser via `npx playwright install --with-deps chromium`. O Turbo seleciona o showcase hoje e qualquer app futuro com `test:e2e` sem novo edit.

### Pegadinhas conhecidas
- A baseURL do preset precisa casar com a porta real do `dev` do showcase (Vite). Confirme a porta no `vite.config.ts` do showcase (ou fixe-a) antes de escrever `baseURL`; o `webServer.url` é o que o Playwright usa para saber que o app subiu.
- Drag real no Playwright é `page.mouse.move` + `down`/`up` em sequência com coordenadas absolutas — não há helper único de "drag"; calcule a partir do `boundingBox()` real de cada componente, não de valores fixos.
- Focus-trap do Modal: `page.keyboard.press('Tab')` repetidamente + `page.evaluate(() => document.activeElement)` a cada passo — o ciclo completo (último focável → volta ao primeiro) é o que prova o trap.
- CI precisa instalar os browsers do Playwright (`npx playwright install --with-deps chromium`) nos dois runners da matriz T-019 — step novo, não vem de cache de pnpm/node_modules.
- O preset é importado por um config `.ts` na raiz de um app; garanta que `@plataforma/testkit` resolva por `workspace:*` no `package.json` do showcase (mesmo padrão do consumo da lib pela T-012).

Decisão de não bloquear a task inteira por uma diferença de gesto num componente; reviewer decide se quer um follow-up para drag-to-scroll no próprio componente.
