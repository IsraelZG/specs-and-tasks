## 4. Estratégia de Testes

- **Fila:** três processos concorrentes nunca sobrepõem a região crítica; ordem FIFO observável;
  processo morto não deixa lock permanente.
- **Merge:** fixture Git temporária prova `--no-commit`; gate vermelho restaura exatamente o SHA
  base; gate verde cria um único merge commit.
- **Gate:** `backend` do Estaleiro executa `test:integration` sem `test:e2e`; `full` mantém E2E;
  mudança de HEAD/índice durante o gate invalida o artefato.
- **ARM64:** auditoria PE dos binários do caminho de teste documentada no handover; não instalar
  dependência nova só para substituir emulação sem prova de ganho/compatibilidade.
- **Fora de escopo:** não executar a suíte Playwright completa durante a implementação desta
  infraestrutura; usar testes de contrato leves e inspeção dos comandos gerados.

## 5. Instruções de Execução

1. Escrever primeiro os testes da fila e da transação.
2. Implementar a fila somente com Node stdlib.
3. Fazer `gate.mjs` entrar automaticamente na fila; token reentrante evita deadlock no integrador.
4. Tornar `worktree merge` transacional e rejeitar invocação sem gate explícito.
5. Adicionar os perfis de impacto sem mudar silenciosamente o significado de specs antigas:
   ausência de `test_profile` continua conservadora (`full`).
6. Atualizar o processo: worker roda gate escopado; reviewer valida o artefato/tree e roda sondas
   direcionadas; integrador serial executa o gate definitivo na composição com a master.

### Não fazer

- Não adicionar Redis, daemon, banco ou dependência de locking.
- Não usar `turbo --concurrency=1`/`workers:1` como substituto da fila entre processos.
- Não deixar merge commitado quando o gate falhar.
- Não inferir ausência de E2E apenas de `target_agent`; usar `test_profile` explícito.
- Não remover E2E de tasks antigas sem reendurecimento; fallback é `full`.

## 6. Decisões de Arquitetura

- Uma única fila de **validação** por clone/máquina atende worktrees e master.
- Integração segura é uma região crítica única: sync → merge sem commit → gate → commit → push.
- `test_profile`: `backend` (sem browser), `ui` (UI + E2E), `full` (produto/standalone completo).
- Backend que muda contrato observável deve declarar `ui` ou `full`; backend interno usa `backend`.

## 7. Definition of Done

- [x] Dois gates concorrentes são serializados entre master e qualquer worktree.
- [x] Dois reviewers não conseguem intercalar merges/gates na master.
- [x] Gate vermelho não altera o HEAD da master.
- [x] Gate backend do Estaleiro não chama Playwright.
- [x] Gate full continua incluindo Playwright.
- [x] Reviewer não repete suíte completa quando há artefato válido, salvo risco/sonda explícita.
- [x] Testes leves da infraestrutura passam sem executar UI pesada.
