# T-1020 · Runner: scheduler (ready-set + pick) do grafo de tasks

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Test Runner:** `vitest`
- **Capacidade-alvo:** haiku *(módulo puro, sem I/O; assinaturas e regras explícitas abaixo)*

## 1. Objetivo
Módulo **puro** que, dado o conjunto de tasks (lido via `TaskService.listTasks`), calcula
quais estão prontas para um worker (deps satisfeitas, não bloqueadas), quais estão em review,
e escolhe um lote de tamanho `k`. Sem I/O, sem rede — só lógica de grafo, 100% testável.

## 3. Escopo de Arquivos (Inputs e Outputs) — assinaturas FIXAS
- **[CREATE]** `apps/nexus-backend/src/runner/scheduler.ts`
  ```ts
  import type { TaskStatus } from '../services/task.types.js';

  export interface SchedulableTask {
    id: string;
    status: TaskStatus;
    dependencies: string[];   // IDs que precisam estar 'done'
    blocks: string[];         // IDs que ESTA task bloqueia
    targetAgent?: string;
    capacity?: 'haiku' | 'sonnet' | 'opus-spike';
  }

  /**
   * PRONTA p/ worker se TODAS as condições valerem:
   *   - status ∈ {'draft','ready','rework'}  (os estados que `start` aceita)
   *   - toda id em `dependencies` existe no conjunto E está 'done'
   *   - nenhum "blocker de entrada" incompleto: para todo Y no conjunto cujo
   *     `Y.blocks` inclui esta task, Y.status === 'done'
   * Tasks com id de dependência ausente do conjunto → NÃO prontas (conservador).
   */
  export function computeReadySet(tasks: SchedulableTask[]): SchedulableTask[]

  /** Tasks em status 'review' (aguardando QA). */
  export function computeReviewSet(tasks: SchedulableTask[]): SchedulableTask[]

  /**
   * Escolhe até `k` tasks do ready-set, em ordem ESTÁVEL por `id` (localeCompare),
   * excluindo as cujos ids estão em `inFlightIds`. k<=0 → []. 
   */
  export function pickBatch(ready: SchedulableTask[], k: number, inFlightIds: string[]): SchedulableTask[]

  /** Mapeia um TaskRecord (frontmatter) para SchedulableTask. `capacity` é best-effort:
   *  procurar no corpo a linha "Capacidade-alvo: <x>" via regex; ausente → undefined. */
  export function toSchedulable(record: { frontmatter: { id: string; status: TaskStatus;
    dependencies?: string[]; blocks?: string[]; target_agent?: string }; body?: string }): SchedulableTask
  ```

- [ ] **Cobertura:**
  - `computeReadySet`: task `draft` com dep `done` → pronta; com dep `in_progress` → NÃO;
    task com um blocker de entrada (outra task cujo `blocks` a inclui) ainda não-`done` → NÃO;
    dep ausente do conjunto → NÃO; status `review`/`in_progress`/`done` → NÃO no ready-set.
  - `computeReviewSet`: só retorna status `review`.
  - `pickBatch`: respeita `k`, ordena por id, exclui `inFlightIds`, `k<=0` → `[]`.
  - `toSchedulable`: extrai `capacity` da linha "Capacidade-alvo:"; sem a linha → `undefined`.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - NÃO ler arquivos nem chamar `TaskService` aqui — receba `SchedulableTask[]` pronto.

### Pegadinhas conhecidas
- **Duas condições de bloqueio, não uma.** Uma task pode estar liberada por `dependencies`
  mas ainda ser bloqueada por um "blocker de entrada" (outra task com `blocks: [estaId]`).
  Cheque **as duas**.
- **`rework` é runnable.** Não esqueça `rework` no conjunto de status iniciáveis (o worker
  reassume tasks em rework).
- **Conservador com deps ausentes:** se um id de dependência não está no conjunto recebido,
  trate como NÃO-pronta (não assuma `done`).
