# T-203 · Codec canônico compartilhado (MessagePack/`msgpackr`) para payloads de protocolo

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo — pacote alvo: `packages/protocol/`
- **Test Runner:** `vitest` (Node puro, isomórfico)
- **Capacidade-alvo:** sonnet *(a API é pequena, mas exige acertar as opções de interop do `msgpackr` — `useRecords:false`, `bigint`, binário — que são justamente o motivo de centralizar; errar uma opção quebra portabilidade P2P silenciosamente)*

## 1. Objetivo
Extrair, para uma **única primitiva compartilhada**, o encode/decode MessagePack que hoje cada consumidor (wire format T-201, snapshot T-308) configuraria por conta própria. O `msgpackr` já é a lib escolhida (T-201), mas suas opções de interop **não são default-safe**: o modo `useRecords` (extensão proprietária do `msgpackr`) quebra interop com implementações não-JS, e o tratamento de `bigint` (usado pelos timestamps HLC — `int64`) precisa ser explícito. Centralizar evita que cada call-site reinvente — e divirja — dessas opções.

Esta task **não** cobre o framing de rede (length-prefix/FrameType/versão — isso é T-201) nem a serialização canônica de assinatura (concatenação binária em ordem léxica — isso é T-107, deliberadamente **não** MessagePack; ver T-107 §1). É só o codec MessagePack de payloads.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/protocol/src/codec.ts 
---

/**
 * Codifica um valor em MessagePack com as opções canônicas do projeto.
 * Opções FIXADAS (ver Seção 6): useRecords=false (interop não-JS), bigint nativo
 * (int64, p/ HLC), binário (Uint8Array) preservado como bin, sem variableMapSize quirks.
 * Determinístico: a MESMA entrada produz SEMPRE os mesmos bytes (necessário para o
 * checksum estável do snapshot T-308 e para dedup).
 */
export function encode(value: unknown): Uint8Array;

/**
 * Decodifica bytes MessagePack produzidos por `encode` (ou por outra impl. compatível).
 * NÃO lança em entrada malformada estrutural do MessagePack? -> ver contrato: LANÇA
 * `CodecError` em bytes inválidos; o caller (ex.: decodeFrame da T-201) é quem decide
 * se converte isso num resultado de quarentena. (Mantém esta primitiva simples.)
 */
export function decode(bytes: Uint8Array): unknown;

/** Erro de decode (bytes não são MessagePack válido). */
export class CodecError extends Error {}
```

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** Vitest (Node puro).
- [ ] **Ambiente do Teste:** `pnpm --filter @plataforma/protocol test`.
- [ ] **Fora de Escopo:** framing de rede (T-201), compressão (T-308), canonicalização de assinatura (T-107).

Casos de teste (numerados):
1. Roundtrip de tipos primitivos: `null`, boolean, number, string, array, objeto aninhado → `decode(encode(x))` é igual a `x`.
2. **bigint:** `encode(123n)` / objeto com campo `bigint` (ex.: `{ createdAt: 9007199254740993n }`) → roundtrip preserva o `bigint` exato (não vira `number` com perda de precisão).
3. **binário:** `Uint8Array` (ex.: uma assinatura de 64 bytes) → roundtrip preserva os bytes idênticos como tipo binário (não array de números).
4. **Determinismo:** `encode(x)` chamado 100× com a mesma entrada produz Uint8Array byte-a-byte idêntico.
5. **Determinismo entre objetos equivalentes construídos por caminhos diferentes:** dois objetos com os mesmos campos/valores produzem o mesmo buffer (documentar explicitamente se isso depende de ordem de inserção — se o `msgpackr` não normaliza ordem de chave, o teste deve construir os objetos na mesma ordem e o contrato deve declarar "determinístico para entrada idêntica", não "canônico por valor").
6. **Interop não-JS (useRecords desligado):** o buffer de um objeto simples NÃO contém a extensão de records do `msgpackr` (verificar que o primeiro byte é um map header MessagePack padrão `0x8_`/`0xde`/`0xdf`, não uma ext tag proprietária).
7. `decode` de bytes inválidos (ex.: `Uint8Array([0xc1])` — byte nunca-usado do MessagePack) → lança `CodecError`, não erro genérico.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** use `JSON.stringify` nem invente concatenação binária — é MessagePack via `msgpackr`.
> - **NÃO** deixe `useRecords` no default `true` do `msgpackr` — isso emite ext tags proprietárias que peers não-JS não decodificam (quebra interop P2P silenciosamente). Fixe `useRecords: false`.
> - **NÃO** deixe `bigint` cair no comportamento default — configure explicitamente para int64 (HLC depende disso).
> - **NÃO** acople este codec a `SignedNode`/`SignedEdge` nem a qualquer tipo de domínio — ele é genérico (`unknown`), senão `protocol` passaria a depender de `core` (ciclo).

### Pegadinhas conhecidas
- `msgpackr` por padrão decodifica int64 que cabe em `number` como `number`; para garantir o roundtrip de `bigint` do HLC, use a opção de int64-as-bigint do `msgpackr` (`int64AsType: 'bigint'` ou equivalente da versão instalada — confirmar na doc da versão pinada, não chutar o nome da flag).
- Use uma instância configurada (`new Packr({...})`) e exporte funções que a reusam — não crie uma instância nova por chamada (perde determinismo de opções e custa GC).
- `protocol` é camada-base; **não** importe de `core`/`crypto`/`media`. O codec é genérico de propósito.
