# T-019 · Matriz de CI multiplataforma (linux x64 + linux arm64)

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner:** `vitest` (pacotes core/protocol) e `playwright` (E2E/Frontend)
- **Capacidade-alvo:** sonnet *(edição de YAML de matriz é simples, mas exige entender corretamente o que cada runner hospedado oferece — risco real de configurar algo que não existe)*

## 1. Objetivo
O CI atual (`.github/workflows/ci.yml`) roda um único job (`ci`) em `runs-on: ubuntu-latest` (linux x64). O produto, porém, deve rodar em **linux/x64 e linux/arm64** em produção — e hoje nada no pipeline garante que o código (em especial dependências nativas, como `better-sqlite3` usado pelo `StoragePort`/schema SQLite da T-106) realmente compila e passa os testes em arm64. Um regressão específica de arquitetura (ex.: binário nativo não disponível/não compilável para arm64) só seria descoberta em produção, não em CI.

Esta task adiciona **um segundo job ao CI rodando em `ubuntu-24.04-arm`** (runner GitHub-hosted, linux arm64), mantendo o job existente em `ubuntu-latest` (linux x64) intacto. **Não adiciona Windows ao CI** — o ambiente de desenvolvimento local é Windows ARM64 (ver `Docs/PITFALLS.md` e `superapp/CLAUDE.md`), mas isso é uma característica da máquina do desenvolvedor, não um alvo de deploy do produto; não existe runner Windows-ARM64 hospedado no GitHub Actions, e adicionar Windows x64 não testaria nada que reflita onde o produto efetivamente roda.

## 3. Escopo de Arquivos (Inputs e Outputs)
- **[READ]** `.github/workflows/ci.yml`
- **[READ]** `package.json` (raiz) — confirmar os scripts `lint`/`build`/`test` que o CI invoca
- **[UPDATE]** `.github/workflows/ci.yml` — converter o job único em uma `strategy.matrix` com `os: [ubuntu-latest, ubuntu-24.04-arm]`, ou (alternativa equivalente) duplicar o job em dois `runs-on` distintos. Preferir `strategy.matrix` — menos duplicação.

Não toque em `.github/workflows/*.yml` além de `ci.yml`. Não adicione um terceiro job/branch de matriz (Windows, macOS) — fora de escopo desta task.

## 4. Estratégia de Testes Estrita (Test-Driven Development)
- [ ] **Framework:** N/A no sentido de teste de código — a "verificação" desta task é o próprio CI executando com sucesso nas duas arquiteturas
- [ ] **Métricas/Cobertura:** o job `ubuntu-24.04-arm` precisa completar `pnpm install && pnpm lint && pnpm build && pnpm test` com exit code 0, exatamente como o job x64 já faz
- [ ] **Ambiente do Teste:** os dois runners do GitHub Actions (`ubuntu-latest` e `ubuntu-24.04-arm`) — não há ambiente local equivalente a arm64 linux para reproduzir antes do push (a máquina de dev é Windows ARM64, arquitetura diferente do runner linux/arm64 de CI)
- [ ] **Fora de Escopo:** qualquer teste de performance comparando x64 vs arm64; qualquer teste de Windows

### Casos de verificação (numerados — confirmar via execução real do workflow, não apenas leitura do YAML)
1. O job `ubuntu-latest` (x64) continua passando sem nenhuma mudança de comportamento (regressão zero no pipeline existente).
2. O novo job arm64 (`ubuntu-24.04-arm`) completa `pnpm install --frozen-lockfile` sem erro — esse é o ponto onde uma dependência nativa sem binário pré-compilado para arm64 apareceria primeiro (rebuild via `node-gyp`, que precisa de toolchain de compilação disponível no runner — confirmar que `ubuntu-24.04-arm` tem isso por padrão ou adicionar um step de `apt-get install build-essential` se necessário).
3. O job arm64 completa `pnpm lint`, `pnpm build` e `pnpm test` com exit code 0.
4. O cache do Turborepo (`actions/cache@v4`) usa uma chave que diferencia os dois runners (a chave atual usa `${{ runner.os }}` — confirmar que isso já diferencia, ou adicionar `${{ runner.arch }}` explicitamente, já que `runner.os` é `Linux` para ambos x64 e arm64; sem isso os dois jobs colidem/sobrescrevem o cache um do outro).

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - Não remova o job/runner `ubuntu-latest` existente — a crítica original a este pipeline nunca foi sobre remover x64, foi sobre faltar cobertura de outra arquitetura.
> - Não adicione `windows-latest` ou qualquer runner Windows à matriz — não é alvo de produção, é só o ambiente de desenvolvimento local.
> - Não troque o trigger (`on: push`/`pull_request`) nem os branches monitorados — fora de escopo.

### Pegadinhas conhecidas
- `runner.os` no GitHub Actions retorna `Linux` tanto para `ubuntu-latest` quanto para `ubuntu-24.04-arm` — uma chave de cache `${{ runner.os }}-turbo-...` sem o `arch` faz os dois jobs da matriz competirem pelo mesmo cache, potencialmente restaurando um binário compilado para a arquitetura errada. Use `${{ runner.os }}-${{ runner.arch }}-turbo-...`.
- `ubuntu-24.04-arm` é um runner **GitHub-hosted** (não self-hosted) — não confundir com a necessidade de configurar um runner próprio; basta declarar o nome na matriz.
- Dependências nativas (`better-sqlite3` e qualquer outra com binário pré-compilado) podem precisar de rebuild via `node-gyp` em arm64 se não houver binário pré-built publicado para essa combinação — se o `pnpm install` falhar especificamente nesse passo, isso é justamente o tipo de regressão que esta task existe para capturar; documente o erro na Seção 6 em vez de mascarar (ex.: não adicione `--ignore-scripts` para "fazer passar").

1. Edite `.github/workflows/ci.yml`: troque `runs-on: ubuntu-latest` fixo por uma `strategy.matrix.os: [ubuntu-latest, ubuntu-24.04-arm]` e `runs-on: ${{ matrix.os }}`.
2. Ajuste a chave de cache do Turborepo para incluir `runner.arch` (ver Pegadinhas).
3. Abra um PR e confirme nos *checks* do GitHub que **ambos** os jobs da matriz completam com sucesso (isso é o Gate de Evidência desta task — ver Seção 7, que aqui é "saída do CI real", não um comando local, já que não há runner arm64 linux disponível na máquina de dev Windows ARM64).
4. Cole o link do run do workflow (ambos os jobs verdes) na Seção 8.
