## 1. Objetivo
Compactar nós/edges iniciais de um contexto/especificação em um snapshot binário para acelerar o onboarding de novos peers, conforme o modelo de snapshots do RBSR
*(derivado de `docs/caderno-2-protocol/03-set-reconciliation-protocol.md` §5 "Snapshots de Bootstrap"; `docs/caderno-5-transport/01-p2p-transport-and-reconciliation.md` §Gênese P2P)*.

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/core/src/snapshot.ts ---
import type { SignedNode, SignedEdge } from './signature.js'; // T-107

/** Metadados do snapshot — cabeçalho que precede os dados binários. */
export interface SnapshotHeader {
  magic: 0x53504254;        // "SPBT" em BE — identificador de formato
  version: 1;               // versão do formato
  contextId: string;        // ULID do contexto/SPECIFICATION de origem
  createdAt: bigint;        // HLC timestamp do momento da geração
  nodeCount: number;        // número de nós no snapshot
  edgeCount: number;        // número de arestas no snapshot
  compression: 'gzip' | 'none'; // algoritmo de compressão (gzip nativo via node:zlib — ver Seção 6)
  checksum: Uint8Array;     // SHA-256 do corpo (MessagePack, ANTES da compressão)
}

/** Snapshot completo — cabeçalho + corpo binário compactado. */
export interface Snapshot {
  header: SnapshotHeader;
  body: Uint8Array;  // corpo MessagePack (codec T-203), comprimido com gzip ou 'none'
}

/**
 * Gera um snapshot de bootstrap a partir de um conjunto de nós e arestas assinados.
 * Derivado de: caderno-2-protocol/03 §5 — snapshot consolida tabelas nodes/edges
 *              em estado podado (metadados + assinaturas, sem payloads pesados).
 */
export function createBootstrapSnapshot(
  nodes: SignedNode[],
  edges: SignedEdge[],
  contextId: string,
  compression?: 'gzip' | 'none',
): Promise<Snapshot>;

/**
 * Reidrata um snapshot em nós e arestas.
 * Descomprime, verifica checksum e decodifica o wire format.
 */
export function hydrateSnapshot(snapshot: Snapshot): Promise<{
  nodes: SignedNode[];
  edges: SignedEdge[];
}>;
```

Casos de teste (numerados):
1. `createBootstrapSnapshot` com 0 nós e 0 arestas → snapshot com `nodeCount=0, edgeCount=0`, checksum válido.
2. Roundtrip: criar snapshot com N nós e M arestas → `hydrateSnapshot` retorna os mesmos N/M.
3. Assinaturas dos nós/arestas são preservadas no roundtrip (verifyNode/verifyEdge passam após hidratação).
4. `hydrateSnapshot` rejeita snapshot com checksum incorreto (corpo adulterado).
5. `hydrateSnapshot` rejeita snapshot com magic bytes errados.
6. Snapshot compactado com `gzip` é menor que com `none` para conjuntos > 10 nós.
7. Snapshot gerado com `compression: 'gzip'` → hidratação produz os mesmos dados que sem compressão.
8. `SnapshotHeader.createdAt` é um HLC timestamp > 0.
9. Roundtrip de `createdAt` (`bigint`) preserva o valor exato após `hydrateSnapshot` — depende do codec T-203 tratar `int64` como `bigint`.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** use JSON para o corpo do snapshot. O corpo é MessagePack via o **codec compartilhado** (`encode`/`decode` de `@plataforma/protocol`, T-203) — não chame `msgpackr` direto nem invente concatenação binária.
> - **NÃO** use `zstd` nem adicione dependência de compressão — use **gzip nativo** do `node:zlib` (`gzipSync`/`gunzipSync`). Ver decisão na Seção 6.
> - **NÃO** inclua payloads pesados no snapshot — apenas metadados + assinaturas (estado podado).
> - **NÃO** implemente integração com RBSR ou rede nesta task.

1. **[TDD]** Crie `packages/core/tests/snapshot.test.ts` com os 9 casos da Seção 4.
2. Implemente `SnapshotHeader` e `Snapshot` types em `packages/core/src/snapshot.ts`.
3. Serialize o corpo com `encode({ nodes, edges })` do codec compartilhado (T-203) — preserva `bigint` (HLC) e binário (assinaturas).
4. Implemente `createBootstrapSnapshot`: `codec.encode` → SHA-256 checksum (sobre o MessagePack, antes de comprimir) → `gzipSync` se `compression==='gzip'`.
5. Implemente `hydrateSnapshot`: valida magic → `gunzipSync` se gzip → verifica checksum sobre o MessagePack descomprimido → `codec.decode`.
6. Re-exporte em `packages/core/src/index.ts`.
7. Rode build + test (Seção 7) e cole saída.

## 6. Feedback de Especificação (Spec Feedback Loop)
> **✅ DECISÃO #1 RESOLVIDA (2026-06-22 — arquiteto) — encoding do corpo:** o corpo é serializado
> pelo **codec MessagePack compartilhado da T-203** (`@plataforma/protocol`), não por `msgpackr`
> direto, não por concatenação binária custom, e **não** pelos wire frames da T-201 (snapshot é um
> blob persistido, não um frame de rede — a API frame-tipada da T-201 não se aplica). Por isso a
> dependência de **T-201 foi removida** e substituída por **T-203**; T-201 e T-308 agora consomem o
> mesmo codec de forma independente. Sem migração de encoding futura.

> **✅ DECISÃO #2 RESOLVIDA (2026-06-22 — arquiteto) — compressão = gzip nativo:** o caderno §5 **não
> especifica** algoritmo de compressão (não é contrato de interop), e o runtime exigido é Node v20+,
> que **não tem zstd nativo** (zstd em `node:zlib` só a partir do Node 22.15). Decisão: usar **gzip via
> `node:zlib`** (`gzipSync`/`gunzipSync`) — nativo, cross-platform, **sem addon nativo** (evita o risco
> de rebuild arm64 que a T-019 monitora na máquina dev Windows ARM64 e na matriz de CI), e dá
> compressão real (casos 6/7). O header (`compression`) carrega o algoritmo, então adicionar zstd no
> futuro é retrocompatível (novo valor de enum), sem quebrar snapshots gzip existentes.
>
> **Por que não:** `@mongodb-js/zstd` = addon nativo (risco arm64, peso); `none`-only = perde o ganho
> de tamanho que é o propósito do snapshot. Sem decisões em aberto.

> **✅ DECISÃO #3 RESOLVIDA (2026-06-25 — arquiteto, ADR 0003) — modelo de persistência byte-level:**
> **Opção A — envelope binário puro auto-descritivo** (`serializeSnapshot`/`deserializeSnapshot`
> com `magic "SPBT"` + `version`). Ver [`docs/adr/0003-snapshot-persistence-model.md`](../docs/adr/0003-snapshot-persistence-model.md)
> (Status: aceito). Layout: `[magic:4B BE][version:1B u8][compression:1B u8][reserved:2B][contextIdLen:4B BE][contextId:UTF-8][nodeCount:4B BE][edgeCount:4B BE][createdAt:8B BE][checksum:32B][bodyLen:4B BE][body:variável]`.
> Implementado em T-308-rework-2 (M2), com cross-link normativo ao ADR 0002 (bulk-import deve
> passar por `insertNode`, invariante U3 — para tasks futuras de hidratação persistente).
