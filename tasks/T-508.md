## 1. Objetivo
Implementar conector SMTP no peer do sistema: interface `NotificationConnector` + implementação SMTP self-hostable; templates spec-driven; creds fora do grafo; conteúdo bearer com TTL curto/single-use
*(derivado de `docs/plano-de-implementacao.md` §8 T-508)*.

### Contratos exatos
```ts
// --- packages/core/src/notifications/connector.ts ---
export type NotificationChannel = 'email' | 'sms' | 'push';

export interface NotificationPayload {
  to: string;                    // destinatário (email, telefone, etc.)
  channel: NotificationChannel;
  template: string;              // nome do template
  params: Record<string, string>; // parâmetros do template
  bearerToken: string;           // conteúdo bearer com TTL curto/single-use
  tokenExpiryMs: number;         // TTL do token (default 300000 = 5min)
}

export interface NotificationConnector {
  /** Envia notificação. */
  send(payload: NotificationPayload): Promise<{ success: boolean; messageId?: string }>;

  /** Verifica saúde do conector. */
  healthCheck(): Promise<boolean>;
}
```

```ts
// --- apps/system-peer/src/connectors/smtpConnector.ts ---
import { NotificationConnector, NotificationPayload } from '@plataforma/core';

export interface SmtpConfig {
  host: string;
  port: number;                  // 587 (STARTTLS) ou 465 (SSL)
  secure: boolean;
  auth: {
    user: string;
    pass: string;
  };
  from: string;                  // endereço de origem
  templates: Record<string, SmtpTemplate>; // spec-driven
}

export interface SmtpTemplate {
  subject: string;
  bodyHtml: string;              // template com {{placeholders}}
  bodyText: string;              // fallback plain text
}

export class SmtpConnector implements NotificationConnector {
  constructor(config: SmtpConfig);

  /** Envia email via SMTP. */
  send(payload: NotificationPayload): Promise<{ success: boolean; messageId?: string }>;

  /** Verifica conexão SMTP. */
  healthCheck(): Promise<boolean>;
}
```
- Templates spec-driven: definidos em config JSON, não hardcoded.
- Creds SMTP fora do grafo: armazenados em env vars ou arquivo de config seguro.
- Conteúdo bearer: token single-use com TTL curto (ex: link de convite, link de recovery).
- Integração com mailhog no docker-compose para testes.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest + mailhog (docker-compose).
- [x] **Ambiente:** `pnpm --filter @plataforma/system-peer test`.

Casos de teste:
1. `send` com template de convite → email entregue via mailhog.
2. `send` com template de recovery → contém token single-use.
3. Template com placeholders → renderiza corretamente.
4. `healthCheck` com SMTP acessível → true.
5. `healthCheck` com SMTP inacessível → false.
6. Creds SMTP não trafegam no grafo (asserção).
7. Token expirado → rejeitado como single-use.
8. Template inexistente → erro descritivo.
