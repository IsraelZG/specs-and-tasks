# T-212 · Codec MessagePack — suporte nativo a bigint

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo — pacote alvo: `packages/protocol/`
- **Test Runner:** `vitest` (Node puro, isomórfico)
- **Capacidade-alvo:** haiku *(contrato API é o mesmo da T-203; é só consertar a serialização — assinatura e tipagem não mudam)*

## 1. Objetivo
Fazer `encode(value)` do codec MessagePack (T-203) **serializar `bigint` nativamente como int64**, sem que o caller precise pré-converter para string. Hoje `int64AsType: 'bigint'` no `msgpackr` afeta apenas a decodificação; o encode lança "Do not know how to serialize a BigInt" quando recebe um objeto com campo `bigint` (verificado em T-308 caso 3 e em debug direto em `tests/codec.debug.test.ts`).
*(Contexto: T-308 precisou contornar o defeito pré-convertendo `bigint → string` antes de `encode`. T-201 também pode ser afetada quando o payload do frame incluir um `bigint`.)*

### Contratos exatos (assinaturas TS fixadas)

```ts
// --- packages/protocol/src/codec.ts ---
// Mesmas assinaturas da T-203. Sem mudança de API.
export function encode(value: unknown): Uint8Array;
export function decode(bytes: Uint8Array): unknown;
export class CodecError extends Error {}
```

A **mudança** é de **comportamento**, não de assinatura:
- **Antes:** `encode({ hlc: 123n })` lança `TypeError: Do not know how to serialize a BigInt`.
- **Depois:** `encode({ hlc: 123n })` produz um buffer MessagePack onde `hlc` está como int64, e `decode(buf)` retorna o objeto com `hlc` como `bigint`.

- [x] **Ambiente do Teste:** `pnpm --filter @plataforma/protocol test`.

Casos de teste (numerados):
1. `encode(123n)` (bigint raiz) → buffer MessagePack → `decode(buf)` retorna `123n`.
2. `encode({ hlc: 1700000000000n, n: 1 })` (bigint em objeto) → roundtrip preserva `hlc` como `bigint` exato (sem perda de precisão).
3. `encode({ a: { b: { c: 9999999999999n } } })` (bigint aninhado em 3 níveis) → roundtrip preserva.
4. **Regressão — `int64` sem bigint:** `encode({ n: 1700000000000 })` (number > Number.MAX_SAFE_INTEGER) → roundtrip retorna `1700000000000n` (porque `int64AsType: 'bigint'` na decode).
5. **Regressão — binário:** `Uint8Array` roundtrip preserva (T-203 caso 3).
6. **Determinismo:** `encode` 100× com mesma entrada → mesmo buffer (T-203 caso 4).
7. **Regressão — payload misto:** `encode({ s: 'x', n: 42, b: true, bi: 1n, arr: [1n, 2n] })` roundtrip completo.

## 5. Instruções de Execução (Step-by-Step)
> **⚠️ REGRAS DO QUE NÃO FAZER:**
> - **NÃO** use JSON.stringify nem concatenação binária — o payload é MessagePack via `msgpackr`.
> - **NÃO** mude a assinatura de `encode`/`decode` — o caller não pode quebrar.
> - **NÃO** adicione dependência nova — `msgpackr` já tem o que precisa (a flag correta já existe na versão pinada).
> - **NÃO** mexa em wireFormat (T-201) nem em outros codecs — só `packages/protocol/src/codec.ts`.

### Pegadinhas conhecidas
- A flag do `msgpackr` que ativa serialização de bigint em encode tem nome sutilmente diferente de `int64AsType` (que só afeta decode). Verificar na doc da **versão pinada** qual é — nomes comuns: `useBigIntAsBigInt`, `bigintAsString`, ou simplesmente `forceIntegerToBigInt`. O log da falha "Do not know how to serialize a BigInt" é o sinal de que a flag não foi setada.
- **NÃO** chutar o nome da flag. Se a doc do `msgpackr` instalado não listar a opção, abrir issue no upstream e considerar fallback para T-308 (manter workaround) — mas a opção existe (confirmado em fórums do projeto, embora com nome pouco intuitivo).
- Cuidado com `useRecords:false` (T-203) — essa flag continua obrigatória; o que muda é só a flag de bigint.
- `package.json`: não adicionar dep — `msgpackr` já é dep de `protocol` (via T-203).
