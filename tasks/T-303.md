## 1. Objetivo
Implementar `RangeFooter` e rodada de desafio no pacote `@plataforma/protocol`: footer `{count, checksum}` em toda transferência de range; divergência dispara re-sync em modo desafio com nonce por sessão
*(derivado de `docs/plano-de-implementacao.md` §6 T-303)*.

### Contratos exatos
```ts
// --- packages/protocol/src/rbsr/rangeFooter.ts ---
import { Fingerprint } from './BTree.js'; // T-301a

export interface RangeFooter {
  rangeStartId: string;         // ULID do início do range
  rangeEndId: string;           // ULID do fim do range (exclusive)
  count: number;                // nº de nós no range
  checksum: Fingerprint;        // SHA-256 concatenado de todos os (id‖signature) do range
  sessionNonce: Uint8Array;     // 16 bytes — nonce da sessão para desafio
}

/** Calcula RangeFooter para um range da B-Tree. */
export function computeRangeFooter(
  tree: RBSRBTree,
  rangeStartId: string,
  rangeEndId: string,
  sessionNonce: Uint8Array,
): RangeFooter;

/** Verifica se dois footers para o mesmo range são divergentes. */
export function footersDiverge(a: RangeFooter, b: RangeFooter): boolean;

/** Modo desafio: re-sync com nonce por sessão (fora do caminho rápido). */
export function challengeResync(
  localFooter: RangeFooter,
  remoteFooter: RangeFooter,
  sessionNonce: Uint8Array,
): Promise<ChallengeResult>;
```
- `computeRangeFooter`: percorre a B-Tree no range, concatena `id‖signature` de cada nó, faz SHA-256.
- `footersDiverge`: compara `count` e `checksum`. Se coincidentes mas divergence persistir, entra modo desafio com nonce.
- Modo desafio: re-sync com nonce por sessão, sem invalidar cache do root no caminho rápido.
- Vetor adversarial 5: footer divergente com fingerprint coincidente → rodada de desafio.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest (Node puro).
- [x] **Ambiente:** `pnpm --filter @plataforma/protocol test`.

Casos de teste:
1. `computeRangeFooter` para range com 3 nós → checksum determinístico.
2. `computeRangeFooter` para range vazio → count=0, checksum zero.
3. `footersDiverge` com count diferente → true.
4. `footersDiverge` com checksum diferente → true.
5. `footersDiverge` com footers idênticos → false.
6. Modo desafio: footers coincidentes mas dados divergentes (vetor adversarial 5) → detectado via nonce.
7. Modo desafio: resolução correta (dados do remote aceitos após verificação nonce).
8. Cache do root preservado no caminho rápido (não entra desafio sem divergência).
