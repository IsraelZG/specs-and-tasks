## 1. Objetivo
Implementar `RangeFooter` e rodada de desafio no pacote `@plataforma/protocol`: footer `{count, checksum}` em toda transferência de range; divergência dispara re-sync em modo desafio com nonce por sessão
*(derivado de `docs/plano-de-implementacao.md` §6 T-303)*.

### Contratos exatos (derivados de T-301 BTree.ts)

**Observação derivada:** T-301 (BTree, já done) já provê `BTree.rangeFooter(range): {count, checksum}` —
`BTree.ts:181`. O T-303 **estende** esse padrão adicionando `sessionNonce` e adiciona a função
`computeRangeFooter` que computa **incluindo o nonce** no digest (para a rodada de desafio).
A versão do T-301 é a base do "caminho rápido" (sem desafio); T-303 adiciona a versão com nonce.

> **Discrepância com a spec original:** a spec antiga diz "concatena `id‖signature`", mas o
> `BTreeNode` em T-301 só carrega `{id, fingerprint}` — a assinatura Ed25519 está em
> `SignedNode` no core/lineage (T-107) e não no RBSR. **Decisão pragmática** (registrada em §6):
> checksum = SHA-256(`rangeStartId ‖ rangeEndId ‖ nonce ‖ fp_1 ‖ fp_2 ‖ … ‖ fp_n`) onde `fp_i`
> é o `Fingerprint` (32 bytes) do nó. Isso detecta divergência no conjunto de fingerprints
> sem precisar do `SignedNode` no caminho RBSR.

```ts
// --- packages/protocol/src/rbsr/rangeFooter.ts (NOVO) ---
import { BTree, IdRange, Fingerprint } from './BTree.js'; // T-301 (já done)

/**
 * RangeFooter estendido: metadata de range + digest com nonce (para desafio).
 *
 * - `count` e `checksum` (sem nonce) = T-301 `BTree.rangeFooter` (caminho rápido, já existente).
 * - `checksumWithNonce` = digest incluindo o nonce da sessão (modo desafio, T-303).
 */
export interface RangeFooter {
  readonly range: IdRange;          // [lo, hi] efetivo do range
  readonly count: number;            // nº de nós no range
  readonly checksum: Fingerprint;    // SHA-256 de `id_1 ‖ id_2 ‖ … ‖ id_n` (caminho rápido, T-301)
  readonly checksumWithNonce: Fingerprint; // SHA-256 incluindo o nonce (modo desafio, T-303)
  readonly sessionNonce: Uint8Array; // 16 bytes — nonce da sessão
}

/**
 * Calcula RangeFooter para um range da B-Tree, incluindo digest com nonce.
 *
 * - Caminho rápido: usa `tree.rangeHeader(range)` para count + percorre nós para ids.
 * - Digest com nonce: SHA-256(`startId ‖ endId ‖ nonce ‖ fp_1 ‖ fp_2 ‖ … ‖ fp_n`).
 *
 * Lança `RangeEmptyError` (T-301) se o range não tem elementos — caller decide entre
 * tratar como footer vazio (count=0, checksum = SHA-256 vazio) ou propagar.
 */
export function computeRangeFooter(
  tree: BTree,
  range: IdRange,
  sessionNonce: Uint8Array,
): RangeFooter;

/**
 * Compara dois footers para o mesmo range.
 *
 * Retorna:
 *   - `'equal'` se count E checksum (caminho rápido) coincidem.
 *   - `'divergent-fast'` se count difere OU checksum (caminho rápido) difere
 *     → divergência óbvia, sem entrar em modo desafio.
 *   - `'divergent-challenge'` se count+checksum coincidem MAS checksumWithNonce difere
 *     → vetor adversarial 5 (footer coincidente mas dados divergentes); caller DEVE
 *     entrar em modo desafio.
 */
export type FooterComparison = 'equal' | 'divergent-fast' | 'divergent-challenge';

export function compareRangeFooters(
  a: RangeFooter,
  b: RangeFooter,
): FooterComparison;
```

3. **`compareRangeFooters` com count diferente** — mesmo checksum mas count diverge → `'divergent-fast'`.
4. **`compareRangeFooters` com checksum (rápido) diferente** — count igual mas checksum diverge → `'divergent-fast'`.
5. **`compareRangeFooters` com checksumWithNonce diferente mas checksum igual** — vetor adversarial 5: dois ranges com mesmos IDs e mesmo `checksum` (caminho rápido) mas fingerprints de algum nó diferem (simulando atacante que "passa" o caminho rápido). → `'divergent-challenge'`.
6. **`compareRangeFooters` idêntico** — todos campos coincidem → `'equal'`.

## 6. Feedback de Especificação

**Decisão A (pragmática, já aplicada):** checksum usa `Fingerprint` (32 bytes) do nó, não a
`signature` Ed25519 raw. Justificativa: `BTreeNode` (T-301) só carrega `(id, fingerprint)`.
A signature está em `SignedNode` (T-107, core/lineage) e não está disponível na camada RBSR
sem um round-trip adicional. O `Fingerprint` JÁ É derivado da signature (via `hashNode` em T-108a),
então a garantia criptográfica é equivalente em termos de detecção de divergência.

**Decisão B (ABERTO — escalado para arquiteto): protocolo de `challengeResync`.**

A spec original menciona "modo desafio" com nonce por sessão, mas não define:
- (1) Que mensagem RBSR carrega o challenge? (estender T-302a com `MSG_FOOTER_CHALLENGE`?)
- (2) Como o respondente prova conhecimento do nonce? (resposta inclui HMAC-SHA256 do nonce + checksum?)
- (3) O que acontece se o respondente se recusar ou errar o nonce? (kick + re-handshake? blacklist temporário?)
- (4) Como evitar replay attack do challenge? (nonce já é suficiente? incluir timestamp?)

Sem essas 4 respostas, `challengeResync` é só um esqueleto. **Recomendação:** criar T-303b
(filho de T-303) como `blocked-decision` + `opus-spike` com entregável = ADR descrevendo o
protocolo. Worker NÃO implementa `challengeResync` enquanto B não estiver decidido.

**Decisão C (pragmática):** Vetor adversarial 5 está coberto pelo caso de teste 5 da §4
(detecção via `checksumWithNonce` diferente). A "resolução" do desafio (substituir o nó
adulterado pelo correto) é responsabilidade de **T-302b** (applyNodes) + protocolo de
resync (T-303b, aberto), não desta T-303.

**Ações:**
- T-303: implementar `computeRangeFooter` + `compareRangeFooters` (6 casos de §4) — escopo desta task.
- T-303b (a criar): ADR + spike para protocolo `challengeResync` — escopo separado, `blocked-decision`.
