## 1. Objetivo
Implementar `ZenEngine` embarcado no pacote `@plataforma/core`: motor de regras (GoRules Zen WASM) para validação de invariantes transacionais — ex: saldo(t+1) = saldo(t) + delta, nunca negativo sem colateral
*(derivado de `docs/plano-de-implementacao.md` §4 T-604; T-601 — resolução de fork usa Zen para validar merge)*.

### Contratos
```ts
// --- packages/core/src/rules/ZenEngine.ts 
---
import { evaluateUnaryExpressionSync } from '@gorules/zen-engine'; // ^1.0.0-beta.3 — decisão §6, corrigida 2026-07-02

export interface ZenRule {
  id: string;
  expression: string; // expressão da Zen Expression Language nativa da lib (evaluateUnaryExpressionSync) — NÃO JSON logic. Ex.: "result.balance >= 0"
  onFailure: 'reject' | 'warn';
}

export interface ZenContext {
  facts: Record<string, unknown>;   // estado anterior
  changes: Record<string, unknown>;  // delta proposto
  result: Record<string, unknown>;   // estado resultante
}

export class ZenEngine {
  constructor();

  /** Carrega regras. */
  loadRules(rules: ZenRule[]): void;

  /** Avalia contexto contra regras. Retorna true se todas passam. */
  evaluate(ctx: ZenContext): Promise<{ passed: boolean; failures: ZenRule[] }>;

  /** Invariante T1: saldo nunca fica negativo (anterior + delta >= 0). */
  static readonly INVARIANT_T1: ZenRule;
}
```
- `evaluate`: para cada regra carregada, chama `evaluateUnaryExpressionSync(rule.expression, ctx)` (função exportada por `@gorules/zen-engine` — retorna `boolean`); se `false` e `onFailure='reject'`, entra em `failures`.
- Invariante T1: `result.balance >= 0` — rejeita transação que tornaria saldo negativo.
- Isomórfico: engine roda em Worker (não na thread principal), mas o contrato da classe é síncrono.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest (Node puro).

Casos de teste:
1. `evaluate` com saldo suficiente → `passed: true`.
2. `evaluate` com saldo negativo → `passed: false`, `failures` contém T1.
3. Regra customizada: `changes.amount <= facts.limit` → rejeita se ultrapassa.
4. Múltiplas regras: uma falha, outras passam → `passed: false`, só a falha listada.

## 5. Instruções de Execução
1. Escreva testes com `ZenContext` mockados.
2. Implemente `ZenEngine` (`packages/core`, não confundir com a classe `ZenEngine` da lib) delegando cada regra a `evaluateUnaryExpressionSync` de `@gorules/zen-engine`.
3. Defina `INVARIANT_T1`.
4. Re-exporte. Rode build + test.

## 6. Feedback de Especificação

### Decisões de Arquiteto (2026-07-01, resolvidas via `/arquiteto-decisoes`, com pesquisa)

**Decisões #1+#2 — pacote + carregamento: DECIDIDO `@gorules/zen-engine-wasm` em todo lugar.**
~~Pesquisa confirmou que o pacote real é `@gorules/zen-engine` (bindings nativos NAPI, por
plataforma) e que existe um pacote WASM separado, `@gorules/zen-engine-wasm`, feito para
browser/Worker.~~ *(ver correção abaixo)*

**CORREÇÃO (re-endurecimento, 2026-07-02) — a pesquisa original estava errada.** Verificado
diretamente no repo `gorules/zen` (`bindings/nodejs/package.json` via GitHub API): **não existe
pacote `@gorules/zen-engine-wasm`**. É um pacote único, **`@gorules/zen-engine`**
(`napi.targets` inclui `wasm32-wasi-preview1-threads`; `package.json` tem `"browser": "browser.js"`
apontando pro build WASI/WASM) — o bundler resolve Node vs. browser automaticamente pelo campo
`browser`, um só pacote em todo runtime (Node/browser/Worker), sem instalar nada "-wasm" à parte.
Fonte: [gorules/zen bindings/nodejs](https://github.com/gorules/zen/tree/master/bindings/nodejs),
`index.d.ts` (classes `ZenEngine`/`ZenDecision`/`ZenDecisionContent`, funções
`evaluateExpressionSync`/`evaluateUnaryExpressionSync`).

**Decisão #3 — Schema `ZenRule`: RESOLVIDA (re-endurecimento, 2026-07-02).** Em vez de montar um
grafo JDM completo (decision table/tree) para uma checagem de invariante booleana simples, esta
spec usa a função standalone exportada pela lib, **`evaluateUnaryExpressionSync(expression: string,
context: any): boolean`** — citação exata de `index.d.ts` — para avaliar `ZenRule.expression` na
**Zen Expression Language** nativa da lib (não JSON logic, não JDM). Casos como
`"result.balance >= 0"` e `"changes.amount <= facts.limit"` (§4) já são sintaxe válida dessa
linguagem. JDM/`ZenDecisionContent`/`ZenEngine` (a classe) ficam fora de escopo desta task — são
para grafos de decisão multi-nó, não o caso de uso aqui.
