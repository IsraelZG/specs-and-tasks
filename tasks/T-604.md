# T-604 · Zen Engine Embarcado + Invariante T1

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm`
- **Monorepo:** Turborepo
- **Test Runner:** `vitest` (Node puro)
- **Capacidade-alvo:** sonnet

## 1. Objetivo
Implementar `ZenEngine` embarcado no pacote `@plataforma/core`: motor de regras (GoRules Zen WASM) para validação de invariantes transacionais — ex: saldo(t+1) = saldo(t) + delta, nunca negativo sem colateral
*(derivado de `docs/plano-de-implementacao.md` §4 T-604; T-601 — resolução de fork usa Zen para validar merge)*.

### Contratos
```ts
// --- packages/core/src/rules/ZenEngine.ts ---

export interface ZenRule {
  id: string;
  expression: string; // expressão Zen (JSON logic)
  onFailure: 'reject' | 'warn';
}

export interface ZenContext {
  facts: Record<string, unknown>;   // estado anterior
  changes: Record<string, unknown>;  // delta proposto
  result: Record<string, unknown>;   // estado resultante
}

export class ZenEngine {
  constructor();

  /** Carrega regras. */
  loadRules(rules: ZenRule[]): void;

  /** Avalia contexto contra regras. Retorna true se todas passam. */
  evaluate(ctx: ZenContext): Promise<{ passed: boolean; failures: ZenRule[] }>;

  /** Invariante T1: saldo nunca fica negativo (anterior + delta >= 0). */
  static readonly INVARIANT_T1: ZenRule;
}
```
- `evaluate`: roda todas as regras carregadas contra `ctx`; se `onFailure='reject'`, retorna `passed=false`.
- Invariante T1: `result.balance >= 0` — rejeita transação que tornaria saldo negativo.
- Isomórfico: engine roda em Worker (não na thread principal), mas o contrato da classe é síncrono.

## 4. Estratégia de Testes
- [x] **Framework:** Vitest (Node puro).

Casos de teste:
1. `evaluate` com saldo suficiente → `passed: true`.
2. `evaluate` com saldo negativo → `passed: false`, `failures` contém T1.
3. Regra customizada: `changes.amount <= facts.limit` → rejeita se ultrapassa.
4. Múltiplas regras: uma falha, outras passam → `passed: false`, só a falha listada.

## 6. Feedback de Especificação

### Decisões de Arquiteto (2026-07-01, resolvidas via `/arquiteto-decisoes`, com pesquisa)

**Decisões #1+#2 — pacote + carregamento: DECIDIDO `@gorules/zen-engine-wasm` em todo lugar.**
Pesquisa confirmou que o pacote real é `@gorules/zen-engine` (bindings nativos NAPI, por
plataforma — `@gorules/zen-engine-win32-x64-msvc` etc.) e que existe um pacote **WASM separado**,
`@gorules/zen-engine-wasm`, feito para browser/Worker (carregado via top-level `await`, é o padrão
oficial da lib). Como o projeto é browser-first isomórfico (T-004b: OPFS+WASM no browser), usa-se
`@gorules/zen-engine-wasm` consistentemente em Node/browser/Worker — um só caminho de execução,
comportamento idêntico em qualquer runtime, sem binário nativo por-plataforma para empacotar na
PWA. Fontes: [npm @gorules/zen-engine](https://www.npmjs.com/package/@gorules/zen-engine),
[GitHub gorules/zen](https://github.com/gorules/zen),
[docs.gorules.io/reference/nodejs](https://docs.gorules.io/reference/nodejs).

**Decisão #3 — Schema `ZenRule`: NÃO é decisão de arquiteto.** O formato JDM (JSON Decision Model)
é documentado oficialmente em
[docs.gorules.io/reference/json-decision-model-jdm](https://docs.gorules.io/reference/json-decision-model-jdm)
e os tipos (`ZenDecisionContent` etc.) vêm exportados do próprio pacote. O re-endurecimento desta
task deve citar a assinatura exata dos tipos exportados por `@gorules/zen-engine-wasm` em vez de
inventar um "JSON logic" próprio — `expression` na `ZenRule` desta spec deve referenciar o formato
real da lib, não um schema customizado.
