# T-020 · Spike: decidir o gate de regressão visual (Lookout vs Playwright `toHaveScreenshot`) → ADR

> **Esta é uma SPIKE, não uma task de implementação.** O entregável é uma **ADR** (Architecture Decision Record) que escolhe a ferramenta do gate de regressão visual do `packages/design-system` **com base em evidência**, mais a task de implementação subsequente já recortada. Esta task **não** instala binário, **não** escreve `lookout.config.yaml`, **não** mexe no CI de produção. Ela existe porque a versão anterior pré-comprometia com o Lookout a partir de um RFC em rascunho e tinha **duas confirmações não resolvidas** que são, por natureza, perguntas de spike (Seção 6). Conforme `Docs/CLAUDE.md` ("Dimensionamento"): o que exige Opus → spike (entregável = ADR/PoC com critério claro). É isto.

## 0. Ambiente de Execução Obrigatório
- **Runtime:** Node.js v20+
- **Package Manager:** `pnpm` (NÃO USE npm ou yarn)
- **Monorepo:** Turborepo (`pnpm build`, `pnpm test`, `pnpm lint` na raiz afetam todos os pacotes)
- **Test Runner (do PoC, se houver):** `playwright` (sobe o showcase via `pnpm --filter design-system-showcase dev`, o harness da T-012/T-015) + investigação do binário externo **Lookout** (Go, `alexmchughdev/lookout`)
- **Capacidade-alvo:** **opus-spike** — envolve julgamento de arquitetura (escolha de ferramenta, leitura de trade-off de infra/custo/plataforma), não é mecânico. **Não** despachar para Haiku/Sonnet.

## 1. Objetivo
Decidir, com evidência e registrar numa ADR, **qual ferramenta** implementa o gate de regressão visual do `packages/design-system`:

- **Opção A — Lookout** (`alexmchughdev/lookout`): binário Go que "levanta um navegador (Chromium) de forma determinística, tira uma screenshot e passa para um modelo de visão computacional, retornando um veredito binário estrito: **Pass ou Fail**" — proposta da Seção 3.1 do RFC `docs/rfcs/rfc- plugin_architecture_blueprint - draft.md` (ainda em `draft`, **não absorvido** via `/absorver-rfc`). Argumento do RFC: veredito por modelo de visão reduz falso-positivo de "loop infinito de CSS" de agentes de IA.
- **Opção B — Playwright `toHaveScreenshot`**: regressão visual nativa do Playwright, que a **T-021 já traz para o CI** (Playwright + Chromium nos dois runners da matriz T-019). Diff de pixel, cross-platform nativo, **zero toolchain marginal**, sem credencial.

**Por que esta decisão precisa ser uma spike e não uma escolha já feita:** a versão anterior desta task assumia o Lookout como certo citando um RFC em rascunho, mas listava (corretamente) duas confirmações bloqueantes que ninguém resolveu — exatamente as perguntas que um spike responde antes de comprometer infraestrutura. Enquanto isso, a alternativa (B) já entra "de graça" porque a T-021 põe Playwright+Chromium no CI de qualquer forma. Decidir A sem responder às duas confirmações é alucinar operação a partir de intenção.

## 4. Critério de Decisão Estrito (o que a ADR precisa resolver)
A ADR **escolhe A ou B** preenchendo esta tabela com fatos verificados (não com o que o RFC "pretende"):

| Eixo | Lookout (Opção A) | Playwright `toHaveScreenshot` (Opção B) |
|---|---|---|
| Veredito | Modelo de visão (semântico) — argumento do RFC | Diff de pixel |
| Infra marginal | **Segundo toolchain**: binário Go + modelo de visão | **~Zero** — já entra com a T-021 |
| Plataforma | **Confirmação #1** (binário x64 **e** arm64 nos runners da T-019?) | Nativo, cross-platform |
| Credencial/custo | **Confirmação #2** (modelo de visão exige API paga/credencial?) | Nenhuma |
| Status da fonte | RFC **rascunho, não-absorvido** | Capacidade nativa, já no CI via T-021 |

**Regra de decisão (gate do spike):**
- Se **ambas** as confirmações da Seção 5 derem **verde** (binário arm64+x64 disponível **e** modelo de visão sem credencial/custo bloqueante) → a ADR **pode** escolher Lookout, pesando o ganho do veredito semântico contra o segundo toolchain.
- Se **qualquer uma** der **vermelho** → a ADR escolhe **Playwright `toHaveScreenshot`** (fallback pragmático: zero infra marginal, sem credencial, nativo nas duas arquiteturas), rodando dentro do **preset compartilhado da T-021** (`@plataforma/testkit`, showcase como 1º consumidor) — não um config novo —, e registra explicitamente que o Lookout foi descartado pela confirmação que falhou.
- **Empate técnico** (ambas verdes, sem diferencial decisivo de qualidade de veredito comprovado) → preferir B por menor custo de operação/manutenção. O ônus da prova é da opção que adiciona um toolchain.

**Fora de escopo da spike:** implementar o gate escolhido; capturar baselines; escrever YAML/specs; mexer no CI de produção. Tudo isso vai para a task de implementação que a ADR recorta (Seção 7).

## 5. Investigação Obrigatória (as duas confirmações = corpo do spike)
> Estas são as **duas perguntas que a versão anterior deixou como caixas não marcadas na Seção 6**. Aqui elas são o trabalho, não um pré-requisito. **Responda com evidência (link de release, README, saída de comando), nunca com suposição.**

1. **Confirmação #1 — Plataforma.** O binário Lookout (`alexmchughdev/lookout`) tem release pré-compilado (ou caminho de build viável em CI) para `linux/amd64` (runner `ubuntu-latest`) **E** `linux/arm64` (runner `ubuntu-24.04-arm`)? Evidência: link da página de releases mostrando os artefatos por arquitetura, ou prova de `go install`/build nos dois runners. **Não assuma** que `go install` resolve arm64 sem verificar.
2. **Confirmação #2 — Modelo de visão / credencial.** Qual modelo de visão computacional o Lookout usa internamente e ele exige credencial/custo **não provisionado neste repositório** (chave de API paga, conta externa)? O RFC **não especifica**. Evidência: trecho do README/config do binário, ou issue/doc do projeto. Se exigir credencial não provisionada, isso é **vermelho** para a Opção A — não contorne com um "modelo grátis substituto" não documentado.

### Pegadinhas conhecidas (para a ADR registrar, valham para A ou B)
- **Aceitação de nova baseline:** qualquer ferramenta de regressão visual vira falso-positivo permanente se não houver processo de "aceitar nova baseline" quando o design muda legitimamente (ex.: T-013 trocando tokens). A ADR deve dizer como cada opção lida com isso (`lookout` em modo de re-baseline vs `playwright --update-snapshots`).
- **Portais (Tier B do T-018 — `Popover`, `Tooltip`, `DropdownMenu`):** renderizam fora do container raiz; a opção escolhida tem que fotografar `document.body` inteiro. Anotar como cada uma resolve.
- **Determinismo cross-arch:** diff de pixel (B) pode divergir entre x64/arm64 por antialiasing de fontes; veredito semântico (A) tende a ser mais estável a isso — **mas só se a Confirmação #1 passar**. A ADR pondera os dois lados.

### Passos
1. Ler a Seção 3.1 do RFC (argumento pró-Lookout) e a T-021 (Playwright já no CI).
2. Resolver **Confirmação #1** (plataforma) com evidência de release/build.
3. Resolver **Confirmação #2** (modelo/credencial) com evidência de doc do binário.
4. (Opcional) PoC descartável para sustentar a evidência — ex.: provar `toHaveScreenshot` disponível, ou baixar/rodar `lookout --help` num runner. Não mergear.
5. Aplicar a **regra de decisão** da Seção 4 e escrever `docs/adr/ADR-0001-gate-regressao-visual.md`: decisão, as duas confirmações com evidência, a tabela de trade-off preenchida, e o **recorte da task de implementação** (Seção 7).
