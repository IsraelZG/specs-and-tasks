## 1. Objetivo

`apps/bancada/src/components/tabs/SyncTab.tsx`) por uma UI funcional que exibe o estado de

última rodada) e dispara reconciliação manual via botão
*(derivado de T-008 — Bancada; T-302a — RBSR exchange state machine)*.

import { useSyncStatus } from '../hooks/useSyncStatus.js'; // NOVO hook, criado nesta task

/** Status de sync exposto para a UI via hook. */
export interface SyncStatus {
  readonly localRootXor: string | null;   // hex (32 bytes) do root fingerprint local
  readonly remoteRootXor: string | null;  // hex do root fingerprint remoto (null = sem sessão)
  readonly sessionState: string | null;    // RBSRState atual (T-302a) ou null
  readonly pendingRanges: readonly { lo: string; hi: string }[]; // ranges com divergência
  readonly lastImported: number;          // nós importados na última reconciliação
  readonly lastSyncedAt: number | null;   // timestamp da última reconciliação concluída
}

/** Hook que conecta a UI ao estado RBSR real (criado em T-311). */
export function useSyncStatus(): SyncStatus;

/** Componente da aba. SEM props (mesmo padrão dos outros tabs). */
export function SyncTab(): JSX.Element;

## 6. Feedback de Especificação

**Decisão 1 (YAGNI / stub):** A UI real precisa de um `useSyncStatus` que conecte ao `RBSRSession`
real (T-302a) e/ou ao `WaveCoordinator` (T-306). Mas T-306 ainda não está integrado à bancada
(e T-306 também está em fase de endurecimento). **Recomendação:** T-311 entrega a UI funcional
+ o esqueleto do hook `useSyncStatus` com **stub controlado** (retorna valores fixos via mock
em teste, valores default em runtime). A integração real é uma task separada (`T-311a`?) que
conecta o hook ao RBSRSession. Isso mantém T-311 implementável **HOJE** sem bloquear em T-306.

**Decisão 2 (Forçar Partição, fora de escopo):** a spec original menciona "Botão 'Forçar
Partição' → dispara `ControlPort.setPartition`". `ControlPort` é de **T-009a (status: draft,
não done)**. Sem T-009a, não há API para forçar partição. **Aplicado:** removido do escopo
desta task. A feature deve entrar em T-009a (que já está prevista) ou numa task filha de T-311
(depois de T-009a). Sugestão: criar T-311a para a feature de partição.

**Decisão 3 (T-302 → T-302a):** A spec original listava `T-302` como dependência, mas T-302 é
parent shell (`decomposed`). O motor concreto é T-302a (já done). **Aplicado:** ajustado
`dependencies: [T-008, T-302a]` no frontmatter.

**Decisão 4 (padrão de props):** A spec original passava `session` como prop, mas todos os
outros tabs da Bancada (`IdentidadeTab`, `RedeTab`, `AuthTab`, `DadosTab`, `CenariosTab`)
seguem o padrão de **zero props** com hook para acesso a estado. **Aplicado:** `SyncTab()`
sem props + `useSyncStatus()` hook.

**Ações:**
- T-311 implementável com as 4 decisões aplicadas; nenhuma decisão em aberto.
- Dependência adicional real (sutil): `@plataforma/protocol` para o tipo `RBSRState` (já é
  dep da bancada via T-008).
